## 一、三条路径对比

### 路径 A：重写 Odoo 运行时 (❌ 不可行)

在 TypeScript 中重新实现 Odoo 的完整 ORM + 视图引擎 + 安全系统。

| 维度 | 评估 |
|------|------|
| 代码量 | 30,000+ 行核心 + 测试 |
| 时间 | 12-18 个月（全职单人） |
| 风险 | 极高 — Odoo 每半年发新版，API 持续变化 |
| 兼容性 | 永远追不上 — 标准模块有 320 万行业务逻辑 |
| 维护成本 | 天文数字 |

**结论：理智上不可行。** 20 年 / 数千人 / 320 万行代码的工程奇迹，不是一个可重写的目标。

### 路径 B：Odoo Bridge (✅ 最务实)

**思路**：不重新实现 Odoo，而是把 Odoo 作为后台服务，飞达做前端壳。

```
┌──────────────────────────────────────────────┐
│                  飞达 (Node.js)               │
│                                              │
│  ┌─────────┐   ┌──────────┐   ┌───────────┐ │
│  │ React UI │ ← │ API 桥接  │ → │ JSON-RPC  │ │
│  │ (表单/列表)│   │ (Express) │   │ Client    │ │
│  └─────────┘   └──────────┘   └─────┬─────┘ │
│                                     │        │
└─────────────────────────────────────┼────────┘
                                      │ HTTP
                         ┌────────────▼────────┐
                         │   Odoo 服务          │
                         │   (Python, 端口8069) │
                         │                      │
                         │  ┌────────────────┐  │
                         │  │ ORM + 视图引擎  │  │
                         │  │ + 工作流 + 安全 │  │
                         │  │ + 报表 + 向导   │  │
                         │  └────────────────┘  │
                         │  完整 Odoo 兼容 ✅    │
                         └──────────────────────┘
```

| 维度 | 评估 |
|------|------|
| 代码量 | 1,000-2,000 行（桥接层） |
| 时间 | 1-2 周 |
| 兼容性 | **100%** — Odoo 本尊运行 |
| 缺点 | 需要 Odoo 服务运行（Odoo 是免费的） |
| 维护 | 极低 — 只需维护 JSON-RPC 调用层 |

**实现方式**：
```typescript
// 飞达端: 代理 Odoo 调用
router.get('/odoo/proxy/:model/search', async (req, res) => {
  const result = await odooRpc.call('execute_kw', [
    db, uid, password,
    req.params.model, 'search_read',
    [domain], { fields, limit, offset }
  ]);
  res.json(result);
});
```

Odoo 的 XML-RPC/JSON-RPC API 自 v7.0 以来非常稳定（10年+未变）。

### 路径 C：选择性运行时 (⚠️ 折中)

只实现最有价值的运行时特性，其余保持静态导入。

| 特性 | 价值 | 复杂度 | 建议 |
|------|------|--------|------|
| 菜单树 ↔ 分层导航 | ⭐⭐⭐⭐⭐ | 低 | ✅ v2.0 |
| 表单布局 ↔ 分组/页签 | ⭐⭐⭐⭐⭐ | 低 | ✅ v2.0 |
| 列表视图 ↔ 显示列 | ⭐⭐⭐⭐ | 低 | ✅ v2.0 |
| **计算字段** (@depends) | ⭐⭐⭐⭐ | 中 | ✅ v2.1 |
| **约束** (@constrains) | ⭐⭐⭐⭐ | 中 | ✅ v2.1 |
| **Onchange** (@onchange) | ⭐⭐⭐ | 中 | ✅ v2.2 |
| Domain 过滤器 | ⭐⭐⭐ | 高 | ⚠️ v2.3 |
| 工作流/状态机 | ⭐⭐ | 极高 | ❌ |
| 报表引擎 | ⭐⭐ | 极高 | ❌ |
| 安全/权限 | ⭐⭐ | 高 | ❌ |
| Python 业务方法 | ⭐⭐⭐⭐⭐ | 不可能 | ❌ 需路径B |

## 二、我的推荐：路径 B（Odoo Bridge）+ 路径 C（选择性运行时）混合

### 推荐方案：分三层

```
第1层 — 静态导入 (v2.0, 1周)
  XML解析 → 菜单树 + 表单布局 + 模型分类
  效果: 48平铺表 → 分层导航 + 分组表单
  ⚡ 不需要 Odoo 运行

第2层 — 选择性运行时 (v2.1-v2.2, 2-4周)
  实现: 计算字段 / 约束 / 基本 onchange
  方式: TypeScript 实现有限的计算引擎
  效果: 会计科目的余额自动计算、必填校验
  ⚡ 不需要 Odoo 运行

第3层 — Odoo Bridge (v3.0, 1-2周)
  方式: 代理 Odoo JSON-RPC 调用
  效果: 100% 兼容 — 所有业务逻辑可用
  ⚡ 需要 Odoo 服务运行
```

### 用户体验路径

```
用户导入 account 模块

  第1层 (静态) → 导航: Customers → Invoices → 表单(分组+页签)
               → 48个表精简为18个 + 30个隐藏关联表
               
  第2层 (运行时) → 发票金额自动计算(计算字段)
                → 借贷平衡校验(约束)
                → 选择客户时自动填充地址(onchange)

  第3层 (桥接) → 点击「确认过账」→ 调用 Odoo action_post()
               → Odoo 执行完整的过账逻辑(创建日记账分录/更新余额/生成税金)
               → 返回结果 → 飞达显示
```

### 三层协同工作

| 操作 | 处理层 | 是否需要 Odoo |
|------|--------|-------------|
| 浏览菜单/打开表单 | 第1层 | ❌ |
| 查看/编辑字段 | 第1+2层 | ❌ |
| 字段自动计算 | 第2层 | ❌ |
| 数据校验 | 第2层 | ❌ |
| 复杂业务操作(过账/对账) | 第3层 | ✅ |
| 生成报表 | 第3层 | ✅ |
| 工作流推进 | 第3层 | ✅ |

## 三、Odoo 是免费的吗？

**是的。** Odoo Community Edition 是 **LGPL v3** 开源许可，完全免费使用：

- 下载：https://github.com/odoo/odoo
- 安装：`pip install odoo` + PostgreSQL
- 启动：`python odoo-bin --addons-path=addons -d mydb`
- JSON-RPC 端口：8069（默认）

Odoo 可以作为"后台服务"静默运行，飞达只通过 JSON-RPC 与之通信。用户完全不需要知道 Odoo 的存在。

## 五、如果坚持"不依赖 Odoo"

那就走路径 C（选择性运行时），但明确边界：

| ✅ 可以实现 | ❌ 永远无法实现 |
|-----------|-------------|
| 菜单/表单/列表结构 | Python 业务方法（320万行） |
| 计算字段（公式型） | 工作流状态机 |
| 简单约束（必填/唯一） | Domain 表达式引擎 |
| 基本 Onchange | QWeb 报表 |
| CRUD + 关联 | 权限/记录规则 |
| | 复杂税务计算 |
| | 银行对账匹配算法 |

**如果接受这些边界**，v2.x 系列可以做到"80% 的场景可用，复杂操作走手动"。

但如果你真的追求"完美"，桥接 Odoo 是唯一合理的答案。
