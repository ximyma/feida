# 飞达智能 HR 系统 — 开发手册

> 面向二次开发者的架构与扩展指南（v1.0.0）。

## 1. 整体架构

飞达是 **单体全栈应用**：后端以 `server/standalone.ts` 为入口，使用 Express 4 风格的 `Router` 挂载于 `/api`；前端是 React 19 SPA（Vite 构建），生产模式下由后端同端口静态托管。

```
请求 → standalone.ts (Express)
        ├─ /api/*      → apiRouter()（业务路由，按模块分区）
        ├─ /uploads/*  → express.static(uploadDir)
        └─ /*          → 静态托管 dist/client + SPA fallback (index.html)
```

- **数据库**：`better-sqlite3`，同步 API，文件 `data/ehr.db`。封装在 `server/modules/database/database.service.ts`。
- **AI 服务**：`server/ai-service.js`（LLM / 知识库混合检索 / 智能体），通过 SSE 流式输出。

## 2. 目录与构建

| 路径 | 作用 |
|------|------|
| `server/standalone.ts` | 主入口，含全部 `/api` 路由与种子逻辑 |
| `server/modules/database/database.service.ts` | 建表 / 迁移 / 种子 |
| `server/seed-cms-shop.ts` | CMS + 商城种子 |
| `client/src/app.tsx` | 前端路由表 |
| `client/src/components/Layout.tsx` | 布局 + 菜单 + 权限过滤 |
| `client/src/hooks/usePermission.tsx` | 前端权限核心 |
| `client/src/i18n/` | 多语言词典与 Provider |

**脚本**
```bash
npm run dev          # 开发：vite(8080) + tsx watch(3000)
npm run build        # build:server(tsc) + build:client(vite)
npm start            # node dist/server/standalone.js
npm run type:check   # tsc --noEmit
```
> 安装依赖必须用 `npm install --legacy-peer-deps`（见 README 说明）。

## 3. 数据库约定

### 3.1 建表与迁移
- 建表语句集中在 `database.service.ts` 的初始化方法中，首次启动执行。
- 兼容性迁移使用 `ALTER TABLE ... ADD COLUMN`（带 `PRAGMA table_info` 存在性检查），保证旧库升级不报错。
- 切勿手动删除字段；新增列用 `migrateColumns()` 模式。

### 3.2 通用 CRUD
- 白名单 `ALLOWED` 内的表支持 `GET /:table`、`POST /:table`、`GET/PUT/DELETE /:table/:id`，字段透传存储。
- 业务表命名前缀：`emp_*`(人事)、`shop_*`(商城)、`cms_*`(CMS)、`ai_*`(AI)。

### 3.3 关键 schema 约定
- `shop_orders` 实际字段：`pay_method / paid_at / shipped_at / order_status / pay_status`（非 `payment_method` 等）。
- `shop_goods` 增强列：`title_color / sub_title / gift_points / usage_guide / mobile_content`。
- `roles` 含 `siteScope`（JSON 数组，`["*"]`=全部站点）。

## 4. 权限系统（RBAC）

### 4.1 后端
- 目录：`RBAC_CATALOG`（cms/shop/hr/sys 细粒度点），`RBAC_ROLES`（内置角色），`RBAC_SITES`（main/shop/portal）。
- 种子：`ensureRbacSeeded()` 幂等，启动调用；内置角色 `siteScope` 始终以代码定义为准（自定义角色由 API 维护）。
- 关键端点：
  - `GET /api/rbac/sites` — 站点目录
  - `GET /api/rbac/roles` / `GET /api/rbac/roles/:id` — 角色（含 `siteScope`）
  - `POST /api/rbac/roles` / `PUT /api/rbac/roles/:id` — 支持 `siteScope`（仅允许 `*` 或已登记站点码）
  - `POST /api/rbac/resolve` — 传 `roleIds`，返回 `permissions`(并集) + `siteScope`(并集, `*` 优先) + `sites`

### 4.2 前端
`client/src/hooks/usePermission.tsx`：
```ts
const { can, hasPermission, siteScope, canAccessSite } = usePermission();
// can('cms:article:delete') → 按钮显隐
// canAccessSite('shop')     → 站点作用域判断
```
- 初始化：本地 `computePermissions` 先算（防闪烁）→ 异步 `fetch('/api/rbac/resolve')` 合并权威结果。
- `PermissionButton` / `PermissionGuard` / `filterMenuByPermission` 组件可直接使用。

## 5. 多语言（i18n）

- 后端词典 `I18N_EN` + 端点 `GET /api/i18n/available`、`GET /api/i18n/messages?lang=en`。
- 前端：`client/src/i18n/index.tsx`，`useI18n()` 返回 `t(key)`；中文以原文为 key，未译回退原文。
- 新增文案：后端 `I18N_EN` 与前端 `EN_DICT` **同步**添加，并在 UI 用 `t('中文文案')` 包裹。

---

## 6. API 约定

- 统一前缀 `/api`。
- 鉴权：`POST /api/auth/login` → 返回 `user`（含 `roleIds`）；前端存 `localStorage`。
- 响应风格：成功 `{ success: true, data }` 或 `{ ... }`；失败 `{ success: false, message }` 或 HTTP 错误。
- **路由顺序陷阱**：新端点必须注册在通用 catch-all `/:table` 之前，否则被吞。

---

## 7. 如何新增一个业务模块

1. **后端**：在 `standalone.ts` 的 `apiRouter()` 中（catch-all 之前）用 `router.get/post(...)` 注册；如需持久化，直接在白名单表或新增表（在 `database.service.ts` 建表）。
2. **前端页面**：在 `client/src/pages/<Module>/` 新建页面，于 `app.tsx` 注册路由，在 `Layout.tsx` 菜单添加项（用 `filterMenuByPermission` 过滤）。
3. **权限**：在 `RBAC_CATALOG` 增加模块权限点；前端用 `can('module:action')` 控制按钮。
4. **i18n**：UI 文案用 `t()` 包裹，词典补英文。
5. **测试**：新增 `test-<module>.js` 脚本（Node `fetch`，`BASE` 默认 `http://localhost:3400`）。

## 9. 常见问题

- **前端编译报 `failed to resolve import "antd"`**：依赖被 prune。用 `npm install antd@^5 @ant-design/icons@^5 es-toolkit@^1 sharp pg mysql2 --legacy-peer-deps` 恢复，切勿整体重装。
- **生产模式首页显示“请先运行 npm run build”**：未构建或静态路径错误。确认 `npm run build` 已执行，且 `standalone.ts` 静态路径为 `path.join(__dirname, '../client')`。
- **端口被占用**：`netstat -ano | findstr :3400` 找 PID，`taskkill /PID <id> /F` 结束。
