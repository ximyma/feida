# 36-订单与支付：库存联动、线下支付确认

## 学习目标

学完本篇，你将能够：

- 讲清「下单 → 扣库存 → 支付 → 发货」这条状态机的每个节点由谁触发、改了哪些字段。
- 看懂 `applyStockChange` 为什么是库存变动的**唯一入口**，以及它如何同时更新商品总库存、仓库库存与流水日志。
- 区分 `pay_status`（支付状态）与 `order_status`（订单状态）两套独立状态，避免把「已付款」误当成「已发货」。
- 配置并跑通**线下支付**场景：用户提交对公转账后，订单进入待确认态，由管理员 `confirm-pay` 确认收款才真正生效。

本篇基于 Sowork AI 企业智能ERP系统 v1.2.0。本地启动请使用 `npm install --legacy-peer-deps`，服务端口 3000，管理后台默认账号 `admin/admin123`。

## 核心概念

商城模块大量使用命名路由（统一以 `/api/shop-*` 前缀），本篇涉及的关键路由如下：

| 路由 | 方法 | 作用 |
| --- | --- | --- |
| `/api/shop-orders` | POST | 创建订单（写入 `shop_orders` + `shop_order_items`） |
| `/api/shop-orders/:id` | PUT | 管理员改订单状态/支付状态/物流 |
| `/api/shop-orders/:id/pay` | POST | 用户发起支付（在线或线下） |
| `/api/shop-orders/:id/confirm-pay` | POST | 管理员确认线下收款 |
| `/api/shop-warehouse/adjust` | POST | 手动库存调整，同样走 `applyStockChange` |
| `/api/shop-stock-logs` | GET | 查询库存流水 |

两个状态维度务必分开理解：

- `pay_status`：钱到没到。取值 `unpaid` / `offline_pending` / `paid`。
- `order_status`：货走到哪了。取值 `pending` / `offline_pending` / `paid` / `shipped` / `completed` / `cancelled`。

🔑 记住：一笔订单可以「钱已付、货待发」（`pay_status=paid` 且 `order_status=paid`），也可以「货发了、退款中」。两套状态各管一摊，互不等价。

## 界面布局与操作流程

管理后台的订单入口在 `ShopAdminPage.tsx` 的 `orders` 标签页（默认标签页）。左侧统计卡展示总订单、待付款、已发货、已完成数量；右侧 `Table` 列出全部订单，列包含订单号、订单状态、操作按钮。

订单详情弹窗（`ShopAdminPage.tsx:849`）做了三件事：

1. 展示订单号、客户、收货地址、金额、商品清单。
2. 当 `pay_status === 'offline_pending'` 时，黄色提示条出现「该订单采用**线下支付**，等待管理员确认收款后订单方可生效」，并给出「确认已收款」按钮。
3. 下方「订单追溯」用 `Timeline` 渲染 `shop_order_logs`，让每一次状态流转都可回看。

用户侧「我的订单」在 `ShopOrdersPage.tsx`，用 `Steps` 时间线呈现四段进度：提交订单 → 付款成功 → 商家发货 → 确认收货。待付款订单显示「去支付」，已发货显示「确认收货」，已取消不可再操作。

典型操作流：

```
用户下单 ──POST /api/shop-orders──> order_status=pending, pay_status=unpaid
   │ (库存此时未扣)
   ▼
用户支付 ──POST /api/shop-orders/:id/pay──> 在线: pay_status=paid + finalizePaidOrder 扣库存
                                          └> 线下: pay_status=offline_pending + order_status=offline_pending
   │                                                          │
   │                                                          ▼
   │                              管理员确认 ──POST confirm-pay──> pay_status=paid + order_status=paid + 扣库存
   ▼
管理员发货 ──PUT /api/shop-orders/:id {order_status:'shipped'}──> 写 shipped_at、物流单号
   │
   ▼
用户确认收货 ──PUT /api/shop-orders/:id {order_status:'completed'}──> 写 receive_time
```

## 底层逻辑与数据模型

`shop_orders` 关键字段（见 `standalone.ts:10797`）：

| 字段 | 含义 | 写入时机 |
| --- | --- | --- |
| `order_no` | 订单号 `SO`+时间戳 | 创建时 |
| `total_amount` / `pay_amount` | 总额 / 实付 | 创建时计算 |
| `pay_status` | 支付状态 | pay / confirm-pay 改写 |
| `order_status` | 订单状态 | 各状态机节点改写 |
| `pay_method` | 支付方式 `online`/`offline` | pay 改写（注意不是 `payment_method`） |
| `paid_at` | 支付时间 | `pay_status` 变 `paid` 时 |
| `shipped_at` | 发货时间 | `order_status` 变 `shipped` 时 |

> **字段名纠偏**：请求体里参数叫 `payment_method`（`standalone.ts:10820`），但落库列名是 `pay_method`。读源码时不要被这两个名字迷惑——入参经 `if (payment_method) updates.pay_method = payment_method;` 转存。

库存三张表的关系：

```
shop_goods.stock          <- 商品总库存（面向买家展示）
shop_warehouse_goods.stock <- 某仓库该商品库存（面向仓管）
shop_stock_logs           <- 变动流水（防超卖、可追溯）
        ▲
        └── 全部由 applyStockChange 写入
```

## 源码剖析

### 1. 库存变动唯一入口 `applyStockChange`

位于 `standalone.ts:8872-8889`。它一次性更新三处：商品总库存、指定仓库库存、流水日志，并对负数做了 `Math.max(0, ...)` 兜底防止库存变负。

```ts
// standalone.ts:8872
const applyStockChange = (warehouseId, goodsId, skuCode, num, type, remark, operator) => {
  const g = db.findById('shop_goods', goodsId);
  if (g) db.update('shop_goods', goodsId, { stock: Math.max(0, (g.stock || 0) + num) }); // 商品总库存
  let after = 0;
  if (warehouseId) {
    // 找到该仓库该商品行，没有就新建
    const allWg = db.findWhere('shop_warehouse_goods', { warehouse_id: warehouseId }) as any[];
    let wg = allWg.find((x: any) => x.goods_id === goodsId);
    if (!wg) { wg = { id: 'swg_' + Date.now() + '_' + ..., warehouse_id: warehouseId, goods_id: goodsId, sku_code: skuCode || '', stock: 0, freeze_stock: 0 }; db.insert('shop_warehouse_goods', wg); }
    after = Math.max(0, (wg.stock || 0) + num);
    db.update('shop_warehouse_goods', wg.id, { stock: after, sku_code: skuCode || wg.sku_code || '' }); // 仓库库存
  }
  db.insert('shop_stock_logs', { // 流水：type 可为 order/return/adjust/in/out
    id: 'sl_' + Date.now() + '_' + ..., warehouse_id: warehouseId, goods_id: goodsId,
    sku_code: skuCode || '', type, num, after_stock: after, remark, operator, created_at: new Date().toISOString()
  });
};
```

`num` 为正表示入库、为负表示出库。订单支付扣减传 `-(quantity)`，退货回滚传 `+quantity`。

### 2. 支付成功后的统一钩子 `finalizePaidOrder`

位于 `standalone.ts:10894-10902`。它定位默认仓库，遍历订单明细，**逐一调用 `applyStockChange` 扣减**。这就是为什么下单那一刻不扣库存、而在支付成功这一刻才扣——避免用户占着库存却不付款。

```ts
// standalone.ts:10894  finalizePaidOrder 内
const wh = (db.findOne('shop_warehouse', { is_default: 1 }) as any) || (db.findAll('shop_warehouse') as any[])[0];
const items = db.findWhere('shop_order_items', { order_id: order.id }) as any[];
items.forEach((it: any) => {
  applyStockChange(wh ? wh.id : null, it.product_id, it.sku || '', -(it.quantity || 0), 'order', '订单支付扣减 #' + order.order_no, 'system'); // standalone.ts:10899
});
```

### 3. 线下支付与确认收款

`pay` 路由（`standalone.ts:10904`）里，若 `payment_method === 'offline'`，订单只进入待确认态，**不触发扣库存**：

```ts
// standalone.ts:10917
if (pm === 'offline') {
  db.update('shop_orders', req.params.id, {
    pay_status: 'offline_pending', pay_method: 'offline',
    order_status: 'offline_pending', paid_at: null
  });
  logOrder(req.params.id, 'pay_offline_pending', '提交线下支付，待管理员确认', 'user');
  return; // 注意：没有 finalizePaidOrder
}
```

真正的扣库存发生在 `confirm-pay`（`standalone.ts:10939`）被管理员调用之后：

```ts
// standalone.ts:10939
router.post('/shop-orders/:id/confirm-pay', (req, res) => {
  const order = db.findById('shop_orders', req.params.id);
  if (order.pay_status !== 'offline_pending') { res.status(400).json({ error: '订单不是线下待确认状态' }); return; }
  db.update('shop_orders', req.params.id, {
    pay_status: 'paid', paid_at: new Date().toISOString(), order_status: 'paid' // standalone.ts:10949
  });
  logOrder(req.params.id, 'paid', '管理员确认线下收款，订单生效', 'admin');
  finalizePaidOrder(order); // 到这里才扣库存
  res.json({ success: true, message: '已确认收款，订单生效' });
});
```

前端 `handleConfirmPay`（`ShopAdminPage.tsx:254`）正是 `fetch('/api/shop-orders/${id}/confirm-pay', { method: 'POST' })`，成功后本地把 `pay_status/order_status` 同步为 `paid`。

## 原理剖析

**为什么库存扣减必须走唯一入口？** 如果允许在「手动调整」「订单支付」「退货回滚」三处各自直接 `db.update('shop_goods', ...)`，会出现两个后果：第一，仓库库存与商品总库存容易对不齐；第二，没有任何流水，`shop_stock_logs` 一片空白，事后无法解释「库存怎么少了」。统一收口到 `applyStockChange` 后，每一次增减都强制落一条 `type` 明确的日志，`after_stock` 记录变动后余量，超卖排查、对账审计都靠它。

**为什么在下单时不扣库存，而在支付成功时扣？** 下单只是「占位意图」。若下单即扣，大量未付款订单会把库存锁死，真实想买的客户看到「缺货」而流失。改为「支付成功才扣」，配合 `pay_status` 与 `order_status` 分离，系统既能区分「已拍下未付」与「已付款待发」，又能在支付后才真正减少可售库存。线下支付因为收款动作滞后，所以把扣库存再推迟到 `confirm-pay` 之后——逻辑一致，只是触发点后移。

**支付状态与订单状态为什么要分开？** 支付是财务事件，订单流转是履约事件。对公转账场景下，「钱已到账」(`pay_status=paid`) 与「货已发出」(`order_status=shipped`) 由不同角色、在不同时间确认。两套字段解耦，让财务对账看 `pay_status`，仓储发货看 `order_status`，互不阻塞。

## 管理者视角

- 🚀 **库存联动防超卖**：因为你只能经由 `applyStockChange` 改库存，且每次扣减都有流水，管理者在「仓库」标签页打开 `shop_stock_logs` 就能看到每一笔「订单支付扣减 / 退货回滚」的来龙去脉，盘亏时一键溯源。
- ✅ **线下支付适配对公转账**：大客户常走银行转账而非在线支付。`offline_pending` 态让订单先挂起，财务核实到账后由管理员点「确认已收款」，订单才进入可发货状态，既兼容传统结算又不被坏账风险拖累。
- 🔑 **订单状态透明可追溯**：每个关键动作都写进 `shop_order_logs`（创建、支付、发货、收货、取消、确认收款），管理者在订单详情「订单追溯」里看到完整时间线，客户纠纷时有据可查。

## 注意事项

- ⚠️ **支付状态与订单状态是两回事**：看到 `order_status=paid` 只代表「已付款待发货」，不代表「已发货」；`pay_status=paid` 也不等于货已到客户手上。对账与履约请各取所需字段。
- 🔑 **库存扣减必须唯一入口**：任何新增业务（如拼团、秒杀）需要动库存时，请调用 `applyStockChange`，不要图省事直接 `db.update('shop_goods')`，否则流水缺失、仓库与总库存迟早对不上。
- ✅ **线下支付需管理员确认防刷**：`offline_pending` 态的订单**不会**自动扣库存、不发货。务必在核实银行到账后再点 `confirm-pay`，否则可能出现「未收款却出库」的资金风险。
- ⚠️ **字段名陷阱**：入参 `payment_method`，落库 `pay_method`；下单接口只支持在线/线下两种，扩展其他线下渠道时记得在 `pay` 与 `confirm-pay` 两处同步处理状态机。

## 小结与练习

本篇你掌握了 Sowork AI 企业智能ERP系统 商城子系统的订单—支付—库存联动全链路：下单仅占位，`pay` 在线支付或 `confirm-pay` 线下确认后才经 `applyStockChange` 唯一入口扣库存；`pay_status` 与 `order_status` 两套状态解耦，分别服务财务与履约。
