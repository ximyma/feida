## 核心概念

5 大营销都是"活动定义 + 用户行为记录"两段式设计：

| 营销 | 一句话 | 核心表 |
|------|--------|--------|
| 优惠券 | 满减/折扣，先领后用 | `shop_coupons` / `shop_user_coupons` |
| 秒杀 | 限时低价，库存独立 | `shop_seckill` |
| 拼团 | 多人成团享低价 | `shop_group_buy` / `shop_group_buy_records` |
| 砍价 | 邀好友帮忙砍到低价 | `shop_bargain` / `shop_bargain_records` / `shop_bargain_helps` |
| 分销 | 多级佣金 + 邀请码裂变 | `shop_distribution_*` |

它们都挂在商城模块下，i18n 映射在 `server/standalone.ts:3796-3797`。前端在 `client/src/pages/Admin/ShopAdminPage.tsx:69` 的 `validTabs` 里以 `coupons / seckill / groupbuy / bargain / distribution` 五个 tab 呈现。

## 界面布局与操作流程

后台入口：`商城管理 → 营销` 系列 tab。导航关系如下：

```
商城管理(ShopAdminPage)
├─ 优惠券管理  tab=coupons      POST /api/shop-coupons
├─ 秒杀管理    tab=seckill      POST /api/shop-seckill
├─ 拼团管理    tab=groupbuy     POST /api/shop-group-buy
├─ 砍价管理    tab=bargain      POST /api/shop-bargain
└─ 分销管理    tab=distribution 配置/等级/分销商/佣金订单/提现
```

常用操作流：

- 优惠券：后台"新增优惠券"填名称、类型、面值、门槛、发放量 → 商城端用户"领取"(扣 `stock`) → 下单时"使用"(状态 `unused→used`)。
- 秒杀：选商品、填秒杀价与 `seckill_stock` → 前台限时展示并独立扣减。
- 拼团：填拼团价、`group_size` → 用户"开团"(`/open`) → 他人"参团"(`/join`)。
- 砍价：填起始价与 `floor_price` → 用户"发起"(`/start`) → 好友"帮砍"(`/help`)。
- 分销：配置开关/层级/基数 → 用户"申请"(`/apply`，可带邀请码) → 产生订单后自动结算佣金。

## 底层逻辑与数据模型

活动类（券/秒杀/拼团/砍价）统一用 `goods_id` 绑定商品，时间窗用 `start_time / end_time`。用户参与侧各有独立记录表：

```
shop_coupons ──< shop_user_coupons (user_id, coupon_id, status)
shop_group_buy ──< shop_group_buy_records (members(JSON), current/target_count)
shop_bargain ──< shop_bargain_records ──< shop_bargain_helps (amount)
```

分销是最复杂的一棵链：

```
shop_distribution_config  (is_open, level_mode, commission_base)
shop_distribution_levels  (rate1, rate2, rate3 三级比例)
shop_distribution_members (invite_code, parent_id, 团队/佣金字段)
   └─ 上级链: member.parent_id → 另一 member
shop_distribution_orders  (distributor_id, buyer_id, distribute_level, commission)
shop_distribution_withdraw(金额, 账户, status: pending/done/rejected)
```

🔑 关键字段：`parent_id` 构造成员上下级树；`invite_code` 是裂变入口；`withdrawable` 是"可提现余额"，与 `total_commission / withdrawn` 三字段共同防纠纷。

## 源码剖析

分销佣金结算是整套营销里最值得精读的真实逻辑。它发生在"支付成功"统一钩子里，沿买家上级链逐级计算一/二/三级佣金(`server/standalone.ts:10861-10902`)：

```ts
const cfg = db.findById('shop_distribution_config', 'dc_default');
if (cfg && cfg.is_open && order.user_id) {
  const base = cfg.commission_base === 'goods'
    ? (order.total_amount || 0) : (order.pay_amount || order.total_amount || 0);
  const maxLevel = cfg.level_mode || 1;
  let node = db.findOne('shop_distribution_members', { user_id: order.user_id });
  let lv = 1;
  while (node && node.parent_id && lv <= maxLevel) {
    const parent = db.findOne('shop_distribution_members', { user_id: node.parent_id });
    if (!parent) break;
    const lvl = parent.level_id ? db.findById('shop_distribution_levels', parent.level_id) : null;
    const rate = lvl ? (lv === 1 ? lvl.rate1 : lv === 2 ? lvl.rate2 : lvl.rate3) : 0;
    if (rate > 0 && base > 0) {
      const commission = Math.round(base * rate * 100) / 100;
      db.insert('shop_distribution_orders', {/* 记一笔佣金订单 */});
      db.update('shop_distribution_members', parent.id, {
        total_commission: (parent.total_commission||0) + commission,
        withdrawable: (parent.withdrawable||0) + commission
      });
    }
    node = parent; lv++;
  }
}
```

调用点有两处：`/shop-orders/:id/pay`（`standalone.ts:10934`，在线支付）与 `confirm-pay`（`standalone.ts:10955`，线下确认收款）。即"钱到账才发佣金"。

提现则扣减 `withdrawable` 并写申请单(`standalone.ts:8833-8842`)，管理员打款 `/done` 或拒绝 `/reject`(拒绝会把钱退回 `withdrawable`，`standalone.ts:8844-8852`)。

## 原理剖析

优惠券的"抵扣规则"由 `type / value / min_amount` 三字段定义（`standalone.ts:8568`）：`type='discount'` 为折扣率、`type='fullreduce'` 为满减额，`min_amount` 是门槛。商城端领取会扣减 `stock` 并防重复领(`standalone.ts:11438-11462`)。🚀 注意：订单主流程的 `discount_amount` 默认 0，券的抵扣在"领用"模型上闭环，结算页按 `type/value` 计算，二者字段同源、互相对账。

秒杀/拼团/砍价本质是"价格覆盖 + 库存隔离"：秒杀价 `seckill_price`、拼团价 `group_price`、砍价落地价 `current_price` 都覆盖原价，且各自带独立库存字段，避免冲击商品主库存。拼团用 `members` JSON 数组 + `current_count/target_count` 判成团；砍价用 `remain * (0.05~0.25)` 随机单刀金额逼近 `floor_price`(`standalone.ts:8698`)。

分销原理是"树遍历 + 比例分层"：`commission_base` 决定基数(实付或商品额)，`level_mode` 限制向上遍历层级，`rate1/2/3` 给出每级比例。团队统计用递归 `collect(parent_id)`(`standalone.ts:8859`) 汇总直推与间推。

## 管理者视角

- ✅ **组合提 GMV**：秒杀引流、拼团冲量、砍价拉新、优惠券锁客，四者叠加可显著拉升客单与复购。
- ✅ **分销裂变获客**：邀请码 + 三级佣金让老客带新客，获客成本低于广告投放。
- 🔑 **佣金透明防纠纷**：每笔佣金都是一条 `shop_distribution_orders` 记录，`total_commission/withdrawable/withdrawn` 三方对账，提现拒绝自动回滚，杜绝"算错账"。

建议：新活动先用小库存灰度，观察成团率/砍价完成率后再放量。

## 注意事项

⚠️ **秒杀必须防超卖**：秒杀库存与商品主库存是两套，下单扣减要锁定"支付成功"节点(`finalizePaidOrder` 统一扣库存)，不要在"创建订单"时扣，否则未支付会虚占。

🔑 **分销层级务必合规**：`level_mode` 建议 ≤ 3 级，且佣金来自真实商品销售(`commission_base`)，避免"拉人头返利"触碰传销红线。

✅ **优惠券可限品类/门槛**：用 `min_amount` 设门槛、"限制适用商品"可控制让利范围，别把满减设成无门槛无上限，否则会被薅羊毛。

⚠️ 砍价 `floor_price` 要高于成本价，且帮砍随机幅度需在合理区间，防止被脚本一次砍穿。

## 小结与练习

本篇讲了 5 大营销：优惠券(满减/折扣)、秒杀(限时低价)、拼团(多人成团)、砍价(邀请砍价)、分销(多级佣金+邀请码+团队+提现)。核心记住三句话：营销=活动表+记录表；分销=上级链遍历发佣金、钱到才结算；秒杀=独立库存、支付才扣。
