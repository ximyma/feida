# 03-整体架构与请求生命周期

本篇带你拆解 **Sowork AI 企业智能ERP系统** v1.2.0 的服务端骨架：一个 HTTP 请求从进来到响应返回，究竟经过了哪些环节。读完你既能给老板讲清「为什么这么便宜」，也能上手改代码。

## 学习目标

你将在本篇掌握四点：

1. 画出「请求从进来到响应返回」的完整链路图。
2. 看懂中间件链与 `apiRouter()` 挂载顺序。
3. 理解 catch-all `/api/:table` 通配路由如何做白名单与字段过滤。
4. 记住一条铁律：命名路由必须在 catch-all 之前注册，否则被吞。

## 核心概念

Sowork AI 企业智能ERP系统是一个 **单体架构（Monolith）**：前端静态文件与后端 API 跑在同一个 Node 进程里，监听 `3000` 端口。它的核心设计哲学是「**用一张表的 CRUD 通配路由，覆盖几乎全部业务**」，而不是为每个模块写一堆子路由。

三个你必须先认识的概念：

| 概念 | 含义 | 出现位置 |
| --- | --- | --- |
| `apiRouter()` | 集中式路由工厂，返回装载了所有 API 的 `express.Router` | standalone.ts:208 |
| catch-all `/api/:table` | 用 `:table` 占位符把表名当路径，动态做 CRUD | standalone.ts:4065 |
| `ALLOWED` 白名单 | 只允许白名单内的表名被通配路由访问 | standalone.ts:1060 |

🔑 一句话：业务路由不走 `/api/hr` 这种按域前缀划分的子路由，而是统一走 `/api/:table`。

## 界面布局与操作流程

作为使用者，你不需要关心路由细节。打开浏览器访问 `http://localhost:3000`，登录默认账号 `admin/admin123`（或 HR 主管 `hr_admin/hr123456`），即可进入 SPA 单页应用。

后台操作流程其实很简单：

```
浏览器 ──> 加载 /index.html (SPA) ──> 前端调 /api/:table ──> 返回 JSON ──> 前端渲染表格
```

你点「员工列表」，前端就发一个 `GET /api/employees`；点「保存」，前端发 `POST /api/employees`。**你看到的每一个表格，背后几乎都对应一个数据库表和一个 `/api/:table` 请求。**

## 底层逻辑与数据模型

数据模型是「**表即资源**」。每个业务实体（`employees`、`salaries`、`contracts`…）都是一张 SQLite 表，URL 里的 `:table` 直接等于表名。路由层不关心业务字段，字段校验交给 `PRAGMA table_info` 动态探测。

为什么能这样？因为 Sowork AI 企业智能ERP系统用的是「**驱动抽象 + 自动建表**」思路（下一篇《04-数据库层》详述）。路由只负责「拿到表名 → 校验白名单 → 读写表」，与具体表结构解耦。

## 源码剖析

下面贴真实片段，逐段解释。

### 1. 中间件链

```ts
// standalone.ts:159-161
app.use(express.json());
app.use(loginSecurityMiddleware);
const upload = multer({ dest: uploadDir });
```

- 第 159 行：`express.json()` 把请求体解析成 `req.body`，后面的 CRUD 才能读到 JSON。
- 第 160 行：`loginSecurityMiddleware`（来自 `modules/login-security`）做登录安全防护，例如防爆破、限流。它在 `apiAuthMiddleware` 之前，属于全局前置守卫。

🚀 注意顺序：JSON 解析必须在所有业务中间件之前，否则 `req.body` 为空。

### 2. apiRouter() 挂载

```ts
// standalone.ts:11856-11892
app.use('/uploads', express.static(uploadDir));

app.use('/api', apiAuthMiddleware, apiRouter());

app.use('/addons', express.static(path.join(__dirname, '..', '..', 'addons'), { maxAge: 0, etag: false }));

// 静态文件服务 ...
app.get('*', (_req, res) => {
  // SPA fallback：所有非 /api 的 GET 都回 index.html
  res.sendFile(indexPath);
});

const PORT = Number(process.env.SERVER_PORT) || 3000;
const HOST = process.env.SERVER_HOST || '0.0.0.0';
server.listen(PORT, HOST, () => { ... });
```

关键三件事：

1. 第 11858 行：`app.use('/api', apiAuthMiddleware, apiRouter())`。**`/api` 前缀下先过 `apiAuthMiddleware` 鉴权，再过 `apiRouter()` 业务路由。** 这是「鉴权在内、业务在外」的洋葱模型。
2. 第 11876 行：`app.get('*')` 是 SPA fallback——凡是不匹配 `/api` 和静态资源的请求，都返回前端 `index.html`，交给前端路由处理。
3. 第 11885-11892 行：监听 `0.0.0.0:3000`，`0.0.0.0` 表示接受所有网卡连接，便于容器/局域网部署。

### 3. catch-all CRUD：以 GET 为例

```ts
// standalone.ts:4065-4088
router.get('/:table', (req, res, next) => {
  const { table } = req.params;
  const RESERVED_ROUTES = new Set(['workflows', 'approval', 'workflow', 'ai', 'mrp', 'sitemap.xml']);
  if (table.includes('-') || RESERVED_ROUTES.has(table)) { return next(); }
  if (!ALLOWED.includes(table)) { res.json({ error: 'Invalid table' }); return; }
  const skipKeys = new Set(['_', 'limit', 'offset', 'page', 'pageSize', 'sort', 'order', 'fields']);
  const filters = Object.fromEntries(Object.entries(req.query).filter(([k]) => !skipKeys.has(k)));
  // ... 调用 db.findAll / db.findWhere ...
  res.json(data);
});
```

逐段解释：

- 第 4067 行：`RESERVED_ROUTES` 是「保留路由」集合。像 `workflows`、`ai` 这种已有专属实现（非简单表 CRUD）的路径，`return next()` 放行给后面的命名路由处理，避免被通配吞掉。
- 第 4069 行：含连字符的 `table`（如 `email-campaigns`）同样 `next()` 放行——它们有独立路由。
- 第 4070 行：**白名单校验**。不在 `ALLOWED` 里的表名直接返回 `{ error: 'Invalid table' }`，这是防越权访问任意表的安全闸门（`ALLOWED` 定义在 standalone.ts:1060，列了上百张合法表）。
- 第 4071-4072 行：把 `limit/offset/sort` 等分页排序关键字剔除，剩余 query 当作等值过滤条件（`findWhere`）。

### 4. POST 的字段过滤（PRAGMA）

```ts
// standalone.ts:4097-4114 (节选)
router.post('/:table', (req, res, next) => {
  // ... 白名单/保留路由判断 ...
  let data: Record<string, any> = { ...rawBody };
  const tableInfo = db.query(`PRAGMA table_info("${table.replace(/"/g, '""')}")`);
  const cols = tableInfo.map((c: any) => c.name);
  data = Object.fromEntries(Object.entries(data).filter(([k]) => cols.includes(k)));
  if (cols.includes('createdAt')) data.createdAt = new Date().toISOString();
  res.json(db.insert(table, data));
});
```

这段是「**动态字段过滤**」的精髓：用 `PRAGMA table_info` 查出目标表真实存在的列，把请求体里多传、错传的字段全部剔除，再补上 `createdAt` 时间戳。它让前端少传一个字段也不会报错，多传也不会脏写数据库。⚠️ 注意对表名做了 `"` 转义（`replace(/"/g, '""')`），防止 SQL 注入。

## 原理剖析

### 完整请求链路图

```
┌─────────────────────────────────────────────────────────────┐
│ 浏览器 GET /api/employees?limit=20                            │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
[1] express.json()          解析 body（standalone.ts:159）
[2] loginSecurityMiddleware 登录安全防护（standalone.ts:160）
[3] apiAuthMiddleware       鉴权：校验 token/session（standalone.ts:11858）
[4] apiRouter() 路由匹配
        │
        ├─ 命中命名路由?  → 专用 handler（如 /api/workflows）
        └─ 未命中且 :table 合法 → catch-all /api/:table
[5] ALLOWED 白名单校验    不在表? 返回 {error:'Invalid table'}
[6] RESERVED_ROUTES 判断 保留名? next() 放行
[7] 数据库层 db.findAll/insert   经 PRAGMA 字段过滤
[8] res.json(data)        序列化为 JSON 返回
                            ▼
                  浏览器收到数据 → 前端渲染
```

### 集中式 apiRouter() 的取舍

Sowork AI 企业智能ERP系统选择「**一个 `apiRouter()` 集中返回所有路由**」，而非「每个模块一个 `router = express.Router()` 再 `app.use('/api/hr', hrRouter)`」。

✅ 优点：
- 新增一张表无需新建子路由文件，加进 `ALLOWED` 即可对外提供 CRUD。
- 白名单、字段过滤、时间戳等横切逻辑只写一遍，统一治理。
- 上手成本极低，开发者一眼看全所有接口。

⚠️ 代价：
- 路由文件巨大（单文件上万行），大型团队协作时易冲突。
- 无法按域做独立的中间件/鉴权粒度。
- 命名路由与通配路由挤在同一文件，顺序管理要靠纪律（见铁律）。

对中小型企业 ERP 而言，这套取舍是划算的：**用工程复杂度换开发速度与一致性**。

### 🔑 铁律：命名路由必须在 catch-all 之前注册

Express 的路由是「自上而下、先匹配先执行」。catch-all `router.get('/:table')` 会把 `/api/workflows` 也匹配成 `table='workflows'`。所以代码里用 `RESERVED_ROUTES` 和 `return next()` 在通配内部「让路」。

但更稳妥的做法是：**所有命名路由（如 `/api/workflows/:id/send`）必须写在 `router.get('/:table')` 之前**。一旦写反，专有逻辑会被通配路由吞掉，返回的是普通表 JSON 而非你的业务结果。这正是 `:table` 放在文件靠后位置（standalone.ts:4065）的原因——几百行命名路由排在它前面。

## 管理者视角

从成本与运维角度，单体架构意味着：

| 维度 | 影响 | 呼应 00 篇 |
| --- | --- | --- |
| 部署成本 | 一个进程、一个端口（3000），无需微服务编排，运维接近零 | 低 TCO |
| 资源占用 | 前后端同进程，内存/CPU 共享，小机器也能跑 | 降本增效 |
| 扩展性 | 整体横向复制即可，但无法单独扩某个模块 | 中小型够用 |
| 故障隔离 | 一个模块崩溃可能拖垮整体 | 需做好监控 |

简言之，Sowork AI 企业智能ERP系统用「单体 + 通配路由」把 **首年 IT 支出压到很低**，适合成长型企业；当业务量上规模后，再按模块拆子路由也不迟。

## 注意事项

⚠️ `ALLOWED` 白名单是安全边界。任何新表想通过 API 访问，必须先加进白名单，否则直接 403 式拒绝。

⚠️ 表名会拼进 `PRAGMA table_info("...")` 语句。虽然做了双引号转义，但**永远不要**把用户输入直接当表名，必须过 `ALLOWED` 校验。

✅ 部署前务必 `npm install --legacy-peer-deps`（v1.2.0 依赖树有 peer 冲突，加该参数才能装成功），再 `npm run build && npm start`。

✅ 默认账号仅用于演示，上线前务必修改 `admin/admin123` 与 `hr_admin/hr123456`。

## 小结与练习

本篇你学到了：请求经过「JSON 解析 → 登录安全 → 鉴权 → 通配路由 → 白名单 → 数据库字段过滤 → JSON 响应」八步；`apiRouter()` 集中式挂载在 `/api` 下，用 `/api/:table` 覆盖全表 CRUD；命名路由必须先于 catch-all 注册。

🚀 动手练习：
1. 在 `ALLOWED` 数组中新增一张自定义表名，重启服务后访问 `/api/<你的表名>`，验证白名单生效。
2. 故意把某个命名路由挪到 catch-all 之后，观察返回差异，体会铁律。
3. 用 `curl 'http://localhost:3000/api/employees?limit=2'` 抓一条真实响应，对照链路图标注每一步。
