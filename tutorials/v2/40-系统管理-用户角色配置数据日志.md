# 40. 系统管理——用户/角色/配置/数据/审计/登录日志

## 学习目标

你将从代码层面掌握 Sowork AI 企业智能ERP系统（下文简称 Sowork AI）中「系统管理」模块的结构与原理。读完后你应该能够：

- 理解 `/system` 下的六大核心子功能及其对平台治理的作用。
- 在 `client/src/pages/SystemPage/` 与 `server/standalone.ts` 中定位真实路由与接口。
- 看懂用户创建、角色绑定、审计日志写入的真实代码路径。
- 以管理者视角评估账号生命周期、合规与等保要求。

> 环境约定：版本 **v1.2.0**，默认账号 **admin/admin123**，前端端口 **3000**，依赖安装使用 `npm install --legacy-peer-deps`。

## 核心概念

系统管理是 Sowork AI 的「治理中枢」。它只对管理员开放，承载六个方面：

| 子功能 | 路由 | 作用 |
| --- | --- | --- |
| 系统概览 | `/system` | 全局状态仪表盘 |
| 用户管理 | `/system/users` | 账号生命周期 |
| 角色权限 | `/system/roles` | RBAC（呼应第 08 篇） |
| 系统配置 | `/system/config` | 站点名/主题等（呼应 i18n） |
| 数据管理 | `/system/data` | 备份、导入导出、VACUUM |
| 审计日志 | `/system/logs` | 操作留痕 |
| 登录日志 | `/system/login-logs` | 安全审计 |

权限模型上，系统菜单仅对两类用户可见：`super_admin`（超级管理员）与 `tech_admin`（技术管理员）。判定逻辑在 `client/src/components/Layout.tsx:399`：

```ts
const isAdmin = currentUser.userType === 'super_admin' || currentUser.userType === 'tech_admin';
```

并在 `Layout.tsx:577` 仅当 `isAdmin` 时才渲染系统入口：`{isAdmin && renderNewModuleNav('/system')}`。

## 界面布局与操作流程

侧边栏的 `/system` 菜单共 13 个子项，定义在 `Layout.tsx:232-247`，核心六项路由注册于 `client/src/app.tsx:350-356`：

```
/system                → SystemOverviewPage   系统概览
/system/users          → UserManagePage       用户管理
/system/roles          → RoleManagePage       角色权限
/system/config         → SystemConfigPage     系统配置
/system/data           → DataManagePage       数据管理
/system/logs           → AuditLogPage         审计日志
/system/login-logs     → LoginLogPage         登录日志
```

你日常的操作流如下：

1. **用户管理**：点击「+ 添加用户」，填写用户名、姓名、手机号、类型（普通员工/技术管理员/超级管理员）、状态。保存即调用 `POST /api/users`。
2. **角色权限**：点击「编辑权限」，勾选细粒度权限点与站点范围（全部站点或指定站点），保存即 `PUT /api/roles/:id`。
3. **系统配置**：新增键值对（如 `company_name`），按分类（基础/考勤/薪资/系统）归类，前端直接读写 `system_config` 表。
4. **数据管理**：执行导出、导入、一键初始化或压缩，底层命中 `/api/db/*` 系列接口。
5. **审计/登录日志**：按模块、操作、状态筛选，支持分页与详情查看。

## 底层逻辑与数据模型

所有页面都是「通用 CRUD 壳」：它们通过 `fetch('/api/<表名>')` 读写对应数据表，无需为每个页面单独写后端。后端把这些表列入白名单 `ALLOWED`（`standalone.ts:1060` 起），其中系统模块包含：

```
'system_config','audit_logs','login_logs','data_backups','scheduled_tasks'
```

关键数据表结构：

- `users`：账号主表，`userType` ∈ {super_admin, tech_admin, employee}，`status` ∈ {active, inactive, locked, pending}。
- `roles`：角色表，`permissionIds` 存权限点 JSON，`siteScope` 存站点范围（空或含 `*` 视为全部站点）。
- `audit_logs`：审计日志，字段含 `userId, username, action, module, detail, ip, timestamp`。
- `login_logs`：登录日志，字段含 `username, ip, success, failReason`。

角色权限采用 RBAC 目录播种机制。启动时 `ensureRbacSeeded()`（`standalone.ts:3866`）把内置角色与权限点写入数据库，默认含 `super_admin / hr_admin / dept_manager / employee`（`standalone.ts:3850` 起）。

## 源码剖析

**用户创建**走通用接口。前端在 `UserManagePage.tsx:41-53` 组装表单并请求：

```ts
// UserManagePage.tsx:48-52 新增用户
body.id = `user_${Date.now()}`;
body.createdAt = new Date().toISOString();
await fetch(`/api/${TABLE}`, { method: 'POST', ... });
```

**角色绑定**则在 `RoleManagePage.tsx:71-80` 把勾选的权限点序列化为 `permissionIds` 与 `siteScope`：

```ts
// RoleManagePage.tsx:73
const body = { ...form, permissionIds: JSON.stringify(selectedPerms),
  siteScope: JSON.stringify(selectedSites.length ? selectedSites : ['*']) };
```

后端 `POST /rbac/roles`（`standalone.ts:3924`）会校验 `code` 唯一，并过滤非法站点码。系统角色（type='system'）禁止删除（`standalone.ts:3953`）。

🔑 **审计日志写入**是理解「操作留痕」的关键。登录时服务端直接在 `standalone.ts:1034` 落一条审计：

```ts
db.addAuditLog({ userId: user.id, username: user.username, realName: user.realName,
  action: 'login', module: 'system', detail: `用户 ${user.realName} 登录`, ip: req.ip });
```

底层 `addAuditLog` 定义在 `database.service.ts:4203-4210`，向 `audit_logs` 插入 12 个字段含 `userAgent` 与时间戳。

🚀 **登录日志**由安全模块负责。`login-security.ts:46-74` 区分成功与失败：

```ts
// 登录失败计数并写入 login_logs
db.insert('login_logs', { id: `log_${Date.now()}`, username, ip, success: 0, fail_reason: reason });
// 成功后清除锁定计数
db.insert('login_logs', { id: `log_${Date.now()}`, username, ip, success: 1 });
```

`/api/db/initialize`（`standalone.ts:6343`）在清空业务数据后，也会向 `audit_logs` 追加一条 `db_initialize` 记录，确保「谁初始化了数据库」可追溯。

## 原理剖析

系统管理的本质是「以最小权限 + 全程留痕保障平台可控」。它把治理拆成两条线：

- **授权线**：用户 → 角色 → 权限点（RBAC 三层）。用户本身不直接挂权限，只挂 `roleIds`；角色解析在 `POST /rbac/resolve`（`standalone.ts:3957`）中对多个角色的权限点做并集、站点做合并（含 `*` 即全部站点）。
- **留痕线**：凡是敏感操作（登录、改密、数据初始化）都被强制写入 `audit_logs` 或 `login_logs`，与业务数据物理隔离，无法被普通 CRUD 壳随意改动。

数据管理页的「核心表保护」也体现了这一思想：`DataManagePage.tsx:5-10` 定义 `SAFE_TABLES`（users/roles/permissions/system_config 等），清空操作遇到核心表直接拒绝（`DataManagePage.tsx:163-168`）。

## 管理者视角

作为管理者，系统管理是你做**合规底座**的地方：

- **账号生命周期**：入职建账号、调岗改角色、离职停用（status=inactive/locked）。默认 admin 是系统最高权限，必须第一时间改密。
- **审计日志支撑内控与等保**：审计日志完整记录「谁、在什么时间、对哪个模块、做了什么」，是等保 2.0「安全审计」控制项的直接证据。登录日志则用于发现暴力破解与异常 IP。
- **配置即治理**：站点名、主题等集中在 `system_config`，与 i18n 多语言联动，避免散落在代码里难以管控。

## 注意事项

⚠️ **默认 admin 密码必改**。账号 `admin/admin123` 仅用于初始化，投产前请通过「修改密码」或 `POST /auth/change-password` 强制更换。

🔑 **审计日志不可随意清空**。审计日志是合规证据链，清空需二次确认（`AuditLogPage.tsx:71-80`），且应在有备份与审批的前提下进行。

✅ **定期备份数据**。数据管理页提供一键导出 JSON（`DataManagePage.tsx:107-128`）与 `/api/db/compress` 压缩，呼应第 45 篇；建议在每次重大操作前先导出。

⚠️ **核心表受保护**。初始化或清空前确认非核心表，用户/角色/权限/系统配置表不可被清空，避免治理瘫痪。
