## 学习目标

- 你能进入统计分析页，按「员工 / 考勤 / 薪资 / 报表 / 数据源」维度出图。

## 核心概念

仪表盘解决「一屏掌握经营健康度」，数据统计解决「按维度深挖原因」。两者共享同一套后端聚合服务。

| 概念 | 在哪 | 一句话 |
| --- | --- | --- |
| 首页仪表盘 | 路由 `/`（index） | 轻量 HR 看板，6 张 KPI + 饼图/折线 |
| 统计分析页 | 路由 `/statistics` | 多维报表，含报表管理与数据源配置子页 |
| 聚合接口 | `GET /api/dashboard/stats` | 首页用的精简指标包 |
| 统计接口 | `GET /api/statistics` | 全量维度统计 `getStatistics()` |
| 报表定义 | `report_definitions` 表 | 一张图绑定哪张表、什么图表类型 |
| 数据源 | `data_sources` 表 | 内部库 / Excel / CSV / API 四种来源 |

🔑 记住：`/dashboard/stats` 是「首页专用速览」，`/statistics` 是「分析师专用全量」。前者字段少、算得快；后者维度多、覆盖 8 大业务域。

## 界面布局与操作流程

**首页（登录后默认页）**：顶部欢迎区显示当前用户与今日考勤率，下面是 6 张 KPI 卡片。

```
首页 /（DashboardPage）
├─ 欢迎区：你好，{姓名} 👋 + 今日考勤率
├─ KPI 卡片 ×6：员工总数 / 在职 / 本月新入职 / 离职 / 待审批 / 月工资总额
├─ 快捷入口：员工档案、合同、考勤、薪酬、招聘、培训、审批、统计分析
├─ 图表区：部门人员分布（饼） + 近6个月入职趋势（折线）
└─ 右侧栏：待办提醒 / 公告通知 / 系统概览 / AI 智能洞察
```

操作路径：登录 → 自动落在首页 → 点右上「统计分析」或快捷入口的 📊 进入 `/statistics`。

**统计分析页 `/statistics`** 用 Tabs 切分七个分析面：

```
/statistics（StatisticsPage）
├─ 数据总览 overview：6 张 Statistic + 7 张分布/趋势图
├─ 报表管理 reports：report_definitions 表格，可增删改/预览
├─ 数据源配置 datasource：data_sources 表格，可测试连接
├─ 常用配置 config：report_configs 全局配置
└─ 员工 / 考勤 / 薪资 三个专项分析 Tab
```

🚀 你新建一张报表时，只需选「数据表」+「图表类型（饼/柱/线/表）」，无需写 SQL——这正是与第 10 篇 BI 呼应的低代码思路。

## 底层逻辑与数据模型

首页只聚合 HR 核心表。它并不真正去碰 ERP 库存表或商城订单表，而是对 `employees`、`salaries` 做轻量计数。流程如下：

```
浏览器 fetch('/api/dashboard/stats')
        │
        ▼
server/standalone.ts:2028  router.get('/dashboard/stats')
        │  db.findAll('employees') / db.findAll('salaries')
        │  + 一条 SQL: SELECT department, COUNT(*) ... GROUP BY department
        ▼
返回 JSON → DashboardPage 的 stats 状态 → 渲染 KPI + 饼图
```

统计分析页则走「重」路径：`/api/statistics` 直接调用 `getStatistics()`，一次性算完员工、考勤、薪资、招聘、绩效、请假、合同、培训八大域。它返回的对象结构（节选自 `StatisticsPage.tsx:72`）：

```ts
interface StatisticsData {
  overview: any;                       // 顶部 6 指标聚合
  deptDistribution: {...}[];           // 部门分布
  eduDistribution: {...}[];            // 学历结构
  salaryDistribution: {...}[];         // 薪资档位
  hireTrendData: { month, value }[];   // 入职趋势近12月
  attendanceTrendData: { month, normal, late, absent }[]; // 考勤趋势近6月
  // ... 共 13 个维度字段
}
```

✅ 报表定义驱动图表的核心字段：`tableName`（绑定哪个内部统计结果）+ `chartType`（pie/bar/line/table）。预览时 `handlePreviewReport` 用 `dataMap[report.tableName]` 取出对应数组直接喂给 ECharts（`StatisticsPage.tsx:302`）。

## 源码剖析

先看首页聚合接口的真实片段（`server/standalone.ts:2028`）：

```ts
router.get('/dashboard/stats', (_req, res) => {
  const employees = db.findAll('employees');
  const salaries = db.findAll('salaries');
  const currentMonth = new Date().toISOString().slice(0, 7);
  const monthPayroll = salaries.filter((s: any) => s.month === currentMonth)
    .reduce((sum: number, s: any) => sum + (s.netSalary || 0), 0);
  res.json({
    totalEmployees: employees.length,
    activeEmployees: employees.filter((e: any) => e.status === 'active').length,
    pendingEmployees: employees.filter((e: any) => e.status === 'pending').length,
    monthPayroll,
    departmentStats: db.query('SELECT department, COUNT(*) as count FROM employees GROUP BY department'),
  });
});
```

这段逻辑很直白：`findAll` 把全表读进内存，用 JS 的 `filter/reduce` 算人数与月薪，再用一条 `GROUP BY` 出部门分布。对于中小数据量是够用的；数据量很大时建议改成纯 SQL 聚合。

再看「数据源连接测试」的真实逻辑（`server/modules/database/database.service.ts:4406`）——它解释了为什么内部数据源点「测试」会返回字段数：

```ts
testDataSourceConnection(id: string): any {
  const ds = this.db.prepare('SELECT * FROM data_sources WHERE id = ?').get(id) as any;
  if (!ds) return { success: false, message: '数据源不存在' };
  if (ds.type === 'internal') {
    try {
      const tableInfo = this.db.prepare(`PRAGMA table_info(${ds.tableName})`).all();
      if (tableInfo && tableInfo.length > 0) {
        this.db.prepare('UPDATE data_sources SET status=?, lastTestedAt=? WHERE id=?')
          .run('connected', new Date().toISOString(), id);
        return { success: true, message: '连接成功，字段数：' + tableInfo.length, fields: tableInfo };
      }
    } catch (e: any) { /* 标记 error 并返回失败信息 */ }
  }
  return { success: true, message: '数据源配置已保存' };
}
```

🔑 注意：内部数据源用 `PRAGMA table_info(表名)` 验证表真实存在；`excel/csv/api` 类型目前只做「配置已保存」的占位校验。如果你要接真实 Excel，需要在此分支补文件读取逻辑。

前端怎么把数据变成图？以部门饼图为例（`DashboardPage.tsx:189`）：

```tsx
<ReactECharts option={{
  tooltip: { trigger: 'item', formatter: '{b}: {c}人 ({d}%)' },
  series: [{
    type: 'pie', radius: ['35%', '65%'],
    data: stats?.departmentStats?.map(d => ({ name: d.department, value: d.count })) || [],
  }]
}} style={{ height: 280 }} />
```

## 原理剖析

为什么分两个接口？这是「首页快」与「分析全」的权衡。

- 首页聚合把计算压在内存里（`findAll` + `filter`），省去多表 JOIN，首屏毫秒级。
- 统计分析的 `getStatistics()`（`database.service.ts:4213`）则是「一次请求、八域同算」。例如司龄分布不是 SQL 直接出的，而是把 `hireDate` 拉到前端侧用 `getTime()` 差值分段（`database.service.ts:4239`）；考勤趋势用近 6 月的循环里跑 18 条 `prepare` 查询（`database.service.ts:4276`）。

这种「聚合服务 + 前端图表库」的架构，和第 10 篇 BI 是一脉相承的：BI 把指标定义存进知识库再预警，本页把指标定义存进 `report_definitions` 再出图。区别在于 BI 偏「智能洞察」，本页偏「人工看板」。

⚠️ 要拓展到 ERP 库存/商城销售额，只需在 `getStatistics()` 里追加对 `inventory`、`orders`、`shopxo_*` 等表的查询，并给前端 `StatisticsData` 增加对应字段——接口契约不变，前端加一个 Tab 即可。

## 管理者视角

这是老板最该每天打开的一页。一屏之内你能回答三个问题：

1. **人够不够**：员工总数、在职、本月新入职 vs 离职，待审批堆了多少。
2. **钱发没发**：月工资总额（首页以「万」为单位，统计分析页有应发/实发/均值）。
3. **健康不健康**：今日的考勤率、近 6 月入职趋势、合同本月到期数（`AI 智能洞察` 卡会直接提示「有 N 份劳动合同即将到期」）。

✅ 数据驱动决策的关键：别只看 KPI 数字，要看趋势图。入职趋势连续 3 月下滑，比单月「本月新入职 +2」更有预警价值。

## 注意事项

⚠️ 仪表盘数据来自各业务表，受 RBAC 权限控制。普通员工看到的「员工总数」可能是本人部门口径，管理员才看全公司——统计口径以登录角色为准。

🔑 统计口径要对齐。例如「在职人数」在首页用 `status='active'`，在 `/statistics` 同样用 `status='active'`；但「月考勤迟到」首页硬编码显示 `96.8%` 演示值，而统计分析页用真实 `todayLateCount`。两页口径不一致时，以统计分析页的真实聚合为准，别拿首页的展示数字做决策。

✅ 卡片可定制：首页 KPI 写死在 `DashboardPage.tsx:155` 的 `grid` 里，想加「ERP 库存金额」「商城今日销售额」卡片，复制一个 `KPICard` 并接新接口即可。

⚠️ 数据源测试只验证内部库的表存在性；外部 Excel/CSV/API 尚未实现真实连通性校验，生产接入前需补齐 `testDataSourceConnection` 的对应分支。
