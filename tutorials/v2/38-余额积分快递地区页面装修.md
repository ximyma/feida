# 38-余额积分快递地区页面装修

> 适用产品：**Sowork AI 企业智能ERP系统**（版本 v1.2.0）。本篇只讲商城后台的五大支撑模块：余额、积分、快递、地区、页面装修（DIY）。

## 学习目标

读完本篇，你将能够：

- 在后台为会员充值余额、发放积分，并理解它们如何参与下单抵扣。
- 配置快递公司，理解物流轨迹的对接方式。
- 使用省市区三级联动选择器，明白 1399 条地区数据如何支撑运费与收货地址。
- 用拖拽式页面装修，自定义商城首页的 banner、商品、文本区块。
- 从源码层面看懂命名路由与数据表之间的映射关系。

## 核心概念

这五个模块不是孤立功能，而是商城运转的「底盘」：

| 模块 | 作用 | 核心数据表 | 命名路由 |
|------|------|-----------|----------|
| 余额 | 会员预存资金，下单时抵扣 | `shop_balance_logs` / `shop_users.balance` | `/api/shop-balance` |
| 积分 | 消费返积分，积分抵现 | `shop_user_points` / `shop_point_logs` | `/api/shop-points/add` |
| 快递 | 物流公司维护与轨迹查询 | `shop_express_companies` | `/api/shop-express` |
| 地区 | 省市区三级联动，运费计算 | `shop_region`（1399 条） | `/api/shop-region` |
| 页面装修 | 首页 DIY 拖拽布局 | `shop_page_design` | `/api/shop-page-design` |

后台入口在「商城」模块的 `ShopAdminPage`，关键 tab 为 `warehouse`（仓库）、`express`（快递）、`decoration`（装修）。地区选择器是通用组件，散落在收货地址等表单中。

## 界面布局与操作流程

后台路由结构（ASCII 树）：

```
商城后台 (ShopAdminPage)
├── 仓库管理  warehouse    → /api/shop-warehouses
├── 快递管理  express      → /api/shop-express
└── 页面装修  decoration   → /api/shop-page-design
地区选择  RegionCascader   → /api/shop-region（任意表单嵌入）
余额/积分  会员维度        → /api/shop-balance、/api/shop-points/add
```

余额操作流程：进入会员详情 → 调 `/api/shop-balance` 充值（`type=recharge`），系统写入一条 `shop_balance_logs` 并把金额累加进 `shop_users.balance`。

积分操作流程：用户下单支付后，后端按规则调用 `/api/shop-points/add`，对 `shop_users.points` 增减并留痕到 `shop_point_logs`。

快递管理：在 `express` tab 新增物流公司（名称、编码、官网、电话），数据进 `shop_express_companies`，可启用/停用/删除。

页面装修：在 `decoration` tab 点「新建页面」，给页面一个 `page_key`（如 `home`），然后追加三类区块——`banner`（横幅）、`goods`（商品）、`text`（富文本），用 ↑/↓ 调整顺序，保存即生效。

## 底层逻辑与数据模型

余额采用「流水 + 冗余字段」双写。每次变动只追加一条 `shop_balance_logs`，同时把 `shop_users.balance` 同步刷新，查询时既可按流水看明细，也可直接读冗余值，避免实时累加。

积分同理：`shop_user_points` 存当前总分，`shop_point_logs` 存每条变动（获得/消费）。注意系统里有两套积分接口——会员维度 `shop_points/add` 直接改 `shop_users.points`，另一套 `shop_user_points` 是更细的账户视图，二者通过 `user_id` 关联。

快递公司表结构：

```
shop_express_companies
├── id        exp_ + 时间戳
├── name      顺丰速运 / 圆通快递 ...
├── code      SF / YTO ...
├── is_enabled 是否启用
└── website / phone
```

地区表 `shop_region` 是自关联树：`id` / `parent_id` / `level`(1省 2市 3区) / `code` / `sort_order`。种子数据来自 `server/region-data.json`，共 **1399 条**，覆盖全国省市区。

页面装修表 `shop_page_design`：`page_key` 唯一标识页面，`blocks` 用 JSON 字符串存区块数组，每个区块形如 `{id, type, title, image/url/goods_ids/content}`。

## 源码剖析

**① 地区三级联动（懒加载）**

后端按 `parent_id` 过滤，并对 `sort_order` 排序：

```ts
// server/standalone.ts:8950
router.get('/shop-region', (req, res) => {
  const parentId = (req.query.parent_id as string) || '0';
  let list = db.findWhere('shop_region', { parent_id: parentId }) as any[];
  list.sort((a: any, b: any) => (a.sort_order || 0) - (b.sort_order || 0));
  res.json(list);
});
```

前端 `RegionCascader` 组件首次加载省级（`parent_id=0`），用户展开某省时才请求其子级，且 `level>=3` 标记为叶子节点停止加载：

```ts
// client/src/components/RegionCascader.tsx:14
const loadChildren = async (parentId: string): Promise<any[]> => {
  const res = await fetch(`/api/shop-region?parent_id=${encodeURIComponent(parentId)}`);
  const data = await res.json();
  return data.map((r: any) => ({
    value: r.name, label: r.name, id: r.id, isLeaf: r.level >= 3,
  }));
};
```

**② 余额累加逻辑**

```ts
// server/standalone.ts:11780
router.post('/shop-balance', (req, res) => {
  const { user_id, amount, type, remark } = req.body;
  db.insert('shop_balance_logs', { id: 'bal_' + Date.now(), user_id, amount, type: type || 'recharge', remark: remark || '', created_at: new Date().toISOString() });
  const user = db.findById('shop_users', user_id);
  if (user) db.update('shop_users', user_id, { balance: (user.balance || 0) + amount, updated_at: new Date().toISOString() });
  res.json({ success: true });
});
```

**③ 页面装修渲染（前台 DIY）**

后台保存的 `blocks` JSON，前台首页按 `page_key=home` 拉取并解析。`goods` 区块会并发补全商品详情：

```ts
// client/src/pages/ShopHomePage/ShopHomePageV2.tsx:79
fetch('/api/shop-page-design?page_key=home').then(r => r.json()).then(async (list) => {
  const page = list.find((p: any) => p.status !== 0) || list[0];
  let blocks = typeof page.blocks === 'string' ? JSON.parse(page.blocks) : (page.blocks || []);
  const resolved = await Promise.all(blocks.map(async (b: any) => {
    if (b.type === 'goods' && Array.isArray(b.goods_ids)) {
      b.goods = (await Promise.all(b.goods_ids.map(gid => fetch(`/api/shop-goods/${gid}`).then(r => r.ok ? r.json() : null)))).filter(Boolean);
    }
    return b;
  }));
  setDecorationBlocks(resolved);
});
```

渲染时按 `type` 分支输出不同 UI（`ShopHomePageV2.tsx:340` 起）：`banner` 渲染图片（带跳转）、`text` 用 `dangerouslySetInnerHTML` 注入富文本、`goods` 渲染商品卡片网格。装修改动只改 `shop_page_design`，前台无需发版即可换版。

## 原理剖析

为什么地区用懒加载而不是一次性下发 1399 条？因为省→市→区层级深、数据量大，一次性拉取既慢又占带宽。懒加载「用到才取」，每次只返回当前层级子节点，体验更顺滑，也减轻了首屏压力。

为什么余额/积分用「流水 + 冗余字段」？流水是事实来源，可审计、可对账；冗余字段是为读性能——下单抵扣时直接读 `shop_users.balance`，不必每次 `SUM` 流水。两者写入在同一事务语义下保持一致。

页面装修的本质是「配置驱动视图」：把首页结构抽象成 `{type, payload}` 的区块数组，存数据库。前台只写一个通用渲染器，新增版式不必改代码，显著降低改版成本。

快递模块当前轨迹是模拟数据（`standalone.ts:11403` 的 `tracks` 为固定样例），真实环境应替换为第三方物流 API；公司列表既可走内置静态列表（`standalone.ts:11386`），也可用 `shop_express_companies` 持久化版本（`standalone.ts:11751`）。

## 管理者视角

- **余额与积分提升复购与粘性**：会员预存余额降低下次购买决策成本；积分抵现制造「还能再省」的心理锚点，把一次性交易变成长期关系。
- **快递对接保障履约**：准确的物流公司与轨迹，是售后、超时赔付、客户信任的前提。务必维护公司列表的编码与启用状态。
- **DIY 装修降低改版成本**：运营/市场可自行拖拽首页，无需研发排期。大促换皮、活动专题页都能分钟级上线，把「技术资源」释放回核心业务。
- **地区数据是隐形的基础设施**：它不直接产生 GMV，却支撑收货地址校验、运费模板、区域限购等所有下游逻辑，缺失会导致下单链路断裂。

## 注意事项

⚠️ **余额提现需风控**：余额可充值可抵扣，若开放提现必须接入审核与限额，防止洗钱与刷单套利。当前系统未实现提现接口，仅做充值与消费抵扣。

🔑 **积分规则要清晰**：获得比例、有效期、抵现上限、是否可叠加优惠券，都应在「会员等级」配置中写清，避免客诉。两套积分接口（`shop_users.points` 与 `shop_user_points`）务必统一来源，勿重复记账。

✅ **地区数据用于运费计算**：`shop_region` 的 `code` 与 `level` 是运费模板、区域限售的键。请勿手工改库，升级请用 `region-data.json` 重新种子化，保持 1399 条完整性。

🚀 **装修富文本有 XSS 风险**：`text` 区块用 `dangerouslySetInnerHTML` 注入 HTML（`ShopHomePageV2.tsx:354`），运营输入须过滤脚本，建议后台增加白名单 sanitizer。
