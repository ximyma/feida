# 27-采购管理：供应商分组 / 供应商 / 订单 / 入库

## 学习目标

你将掌握采购管理的四个子功能，并能回答三个问题：

- 供应商分组与供应商档案如何分层管理？
- 一张采购订单从创建到审批经历了哪些状态？
- 为什么只有「采购入库」才会让库存真正增加？

读完后，你可以独立走完「供应商 → 采购订单 → 入库 → 库存联动」的完整闭环，并理解第 25 篇《仓储管理》中 `stock_in` 的来龙去脉。

## 核心概念

采购管理是连接「外部供应商」与「内部库存」的桥梁。它把花钱买料的过程拆成两个动作：**先下单（承诺）**，**后收货（落库）**。

四个子功能分工如下：

| 子功能 | 路由 | 本质 | 关键表 |
| --- | --- | --- | --- |
| 供应商分组 | `/purchase/supplier-groups` | 分类容器 | `supplier_groups` |
| 供应商档案 | `/purchase/suppliers` | 交易主体 | `suppliers` |
| 采购订单 | `/purchase/orders` | 采购承诺 | `purchase_orders` + `purchase_order_items` |
| 采购入库 | `/purchase/receipts` | 实物落库 | `purchase_receipts` + `purchase_receipt_items` |

🔑 一句话记忆：**分组管类别，供应商管对象，订单管计划，入库管实货。**

## 界面布局与操作流程

左侧菜单「采购管理」下挂四个子页，文件位于 `client/src/pages/PurchasePage/`：

```
采购管理 (/purchase)
├── 供应商分组  SupplierGroupsPage   → /purchase/supplier-groups
├── 供应商档案  SuppliersPage        → /purchase/suppliers
├── 采购订单    PurchaseOrdersPage   → /purchase/orders
└── 采购入库    PurchaseReceiptsPage → /purchase/receipts
```

路由注册在 `client/src/app.tsx:318-321`。四个子页逐一说明：

**① 供应商分组（SupplierGroupsPage）**
最轻量的字典表，仅含 `code`、`name`、`remark`。你先在这里建「鞋材类」「包装类」等分类，再在供应商档案里挂到对应分组。

**② 供应商档案（SuppliersPage）**
供应商主数据。字段很全：`code`、`name`、`short_name`、`group_id`（归属分组）、`contact_person`、`phone`、`email`、`address`、`tax_no`、`bank_name`、`bank_account`、`payment_terms`、`status`。列表支持按分组、状态、关键字（编码/名称/联系人/电话）过滤。

**③ 采购订单（PurchaseOrdersPage）**
核心计划单。选供应商 → 填期望交期、收货仓库、付款方式 → 加多行明细（物料/SKU、数量、单价）。保存后状态为 `pending`，可走审批（呼应第 21 篇《流程审批》）或点「审批」直接置为 `approved`。

**④ 采购入库（PurchaseReceiptsPage）**
基于某张采购订单收货。选订单后系统带出待收明细，你录实际到货数量、批次号，保存为 `pending` 草稿；点「审批」才真正写入库存。

## 底层逻辑与数据模型

采购模块用 6 张表串联（注意：它们不在 catch-all 的 `ALLOWED` 表里，而是走专用路由，挂载在 `/api` 前缀下）：

```
supplier_groups ──< suppliers           (1:N 分组↔供应商)
purchase_orders ──< purchase_order_items (1:N 订单↔明细)
purchase_receipts ──< purchase_receipt_items (1:N 入库↔明细)
purchase_orders 1──1 purchase_receipts  (1:N 订单↔入库，按 order_id 关联)
purchase_receipt_items . order_item_id → purchase_order_items (回写已收数量)
purchase_receipt_items ──> inventory    (审批时按仓库+物料/SKU 落库存)
```

关键外键：
- `suppliers.group_id` → `supplier_groups.id`
- `purchase_order_items.order_id` → `purchase_orders.id`
- `purchase_receipts.order_id` → `purchase_orders.id`
- `purchase_receipt_items.order_item_id` → `purchase_order_items.id`

采购订单状态机：

```
pending ──approve──> approved ──(有入库)──> received ──(全收齐)──> completed
   │                                                                       
 cancel                                                            
   └────────────────────────> cancelled
```

`purchase_orders.payment_status` 初始为 `unpaid`，与第 18 篇财务应付联动（本文不展开）。

## 源码剖析

后端全部集中在 `server/standalone.ts` 的「采购管理」段，挂载路径为 `/api/supplier-groups`、`/api/suppliers`、`/api/purchase-orders`、`/api/purchase-receipts`。

**供应商分组创建**（`standalone.ts:7665`）：校验 `code`+`name`，生成 `id = sg_${Date.now()}`，写入 `supplier_groups`。

**采购订单创建**（`standalone.ts:7775`）：先校验供应商与明细非空，生成 `code = PO + 时间戳后8位`，遍历 `items` 累加 `total_amount`，再逐行插入 `purchase_order_items`，并初始化 `delivery_qty: 0`（已收数量，后续由入库回写）。

🚀 入库触发库存增加的核心逻辑，在审批接口 `router.post('/purchase-receipts/:id/approve')`（`standalone.ts:7902-7961`）：

```ts
// standalone.ts:7907-7930  逐行更新库存
items.forEach((item: any) => {
  let inventory = db.findWhere('inventory', {
    warehouse_id: receipt?.warehouse_id,
    ...(item.sku_id ? { sku_id: item.sku_id } : { material_id: item.material_id })
  });
  if (inventory.length > 0) {
    const inv = inventory[0];
    const newQty = (inv.qty || 0) + (item.qty || 0);          // 累加库存
    db.update('inventory', inv.id, { qty: newQty, ... });
  } else {
    db.insert('inventory', { ... qty: item.qty || 0 ... });    // 首次建库存行
  }
  // 回写订单明细已收数量
  if (item.order_item_id) {
    const orderItem = db.findById('purchase_order_items', item.order_item_id);
    db.update('purchase_order_items', item.order_item_id, {
      delivery_qty: (orderItem.delivery_qty || 0) + (item.qty || 0)
    });
  }
});
```

随后（`standalone.ts:7943-7958`）把入库单置为 `approved`，并据订单所有明细是否「已收 ≥ 应收」决定订单状态为 `completed` 还是 `received`。这正是第 25 篇 `stock_in` 的同源逻辑——**库存只认入库审批，不认采购订单**。

## 原理剖析

为什么设计成「订单不直接加库存」？因为订单只是承诺，供应商可能分批、少量、甚至违约送货。若下单即入库，库存将虚假膨胀，导致第 28 篇生产工单领料时出现「有单无料」。

因此系统用双状态分离：

- **计划态**（purchase_orders）：反映「该买多少」，受预算与审批约束。
- **实态**（purchase_receipts → inventory）：反映「真实到货多少」，是库存的唯一切入点。

`delivery_qty` 字段是连接两者的铰链：每张入库回写对应订单明细的已收量，订单据此自动推进到 `received`/`completed`。这套「承诺—实收」双轨模型，也是后续应付、生产齐套分析的数据基础。

## 管理者视角

- 🔑 **供应商管理降采购成本**：分组便于比价与集中议价；供应商档案里的 `payment_terms`（账期）、`tax_no` 直接影响资金占用与进项税，是采购谈判的抓手。
- ✅ **采购订单控预算**：订单在 `pending`/`approved` 阶段即形成采购承诺额，结合审批流（第 21 篇）可卡住超额下单，避免盲目备料。
- 🚀 **入库联动库存保障生产物料**：只有入库审批通过，`inventory` 才增加，生产计划（第 28 篇）才能可靠地排产领料，形成「采购供料—生产耗料」的闭环。

## 注意事项

- ⚠️ **入库才加库存**：保存入库草稿（`pending`）不会动库存，必须点「审批」触发 `stock_in`。很多新手误以为建了入库单库存就涨了，实际只在审批后更新。
- 🔑 **供应商评估要用分组**：把同类供应商归入同一 `supplier_groups`，便于后续做交付准时率、质量合格率对比，支撑选优汰劣。
- ✅ **采购订单可关联审批（第 21 篇）**：大额采购建议走 `purchase-orders/:id/approve` 之外的正式审批流，留下合规痕迹。
- ⚠️ **批次号要录准**：`purchase_receipt_items.batch_no` 会带入 `inventory`，影响后续先进先出与质量追溯，漏填会给仓储带来麻烦。

## 小结与练习

你已看清采购管理的四张面孔与一条主线：**供应商分组 → 供应商档案 → 采购订单（承诺）→ 采购入库（落库）→ 库存增加**。核心要诀是「订单管计划、入库管实货」，库存只听入库审批的话。

动手练习：
1. 新建一个供应商分组「五金类」，在其下登记一家供应商，并设置 30 天账期。
2. 为该供应商开一张采购订单（2 行物料），提交审批。
3. 基于该订单做一笔采购入库，只收其中 1 行，点审批——观察 `inventory` 是否增加、订单状态是否变为 `received` 而非 `completed`。
4. 再补一张入库收齐剩余数量，确认订单自动变为 `completed`。
