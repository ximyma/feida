# 06 后端引擎：中间件、路由注册与登录安全

本文是 **Sowork AI 企业智能ERP系统** v1.2.0 后端引擎系列的第 6 篇，带你从中间件链、路由注册一路钻到登录安全的代码实现。无论你是想把系统用好的企业管理者，还是负责二次开发的技术人员，读完都能说清「一次登录请求到底经过了几道防线」。

> 运行环境速记：默认账号 `admin/admin123`，服务端口 `3000`，安装依赖请用 `npm install --legacy-peer-deps`。本系统不按 `/api/hr` 这类域前缀划分业务，所有业务表统一走 catch-all 路由 `/api/:table`。

## 学习目标

- 说清请求从进来到落到业务处理函数，经过了哪几层中间件。
- 读懂 `login-security.ts` 的三件事：失败计数、账号锁定、IP 限速。
- 读懂 `api-auth.ts` 的 token 校验逻辑与开发态绕过陷阱。
- 作为管理者，能向团队解释「账号锁定 + 限速」对防数据泄露与合规的意义。

## 核心概念

| 概念 | 一句话解释 | 所在文件 |
| --- | --- | --- |
| 中间件链 | 请求在到达路由前依次穿过的「安检层」 | `standalone.ts` |
| IP 限速 | 同一 IP 每分钟最多尝试 N 次，超出即拦截 | `login-security.ts:78` |
| 账号锁定 | 连续失败超阈值，账号锁定 M 分钟 | `login-security.ts:32` |
| Token 鉴权 | 登录后每个写操作须带 Bearer Token | `api-auth.ts:77` |
| catch-all 路由 | `/api/:table` 承载所有业务表 CRUD | `standalone.ts:4065` |

三道防线（由外到内）：

```
请求 ──▶ ① IP 限速 ──▶ ② 账号锁定 ──▶ ③ Token 鉴权 ──▶ 业务路由 /api/:table
```

## 界面布局与操作流程

登录安全虽是后端逻辑，但与管理界面强相关。你日常接触的入口有三处：

- **登录页**：填写 `admin / admin123`，POST 到 `/api/auth/login`。
- **个人中心 → 修改密码**：POST 到 `/api/auth/change-password`，需先验证原密码。
- **注册页（员工自助）**：POST 到 `/api/auth/register`，按手机号创建员工账号。

后端的认证路由集中在 `standalone.ts:1029` 起：

```
server/standalone.ts
└─ apiRouter()                         (line 208)
   ├─ POST /auth/login                 (line 1029)  ← 登录
   ├─ POST /auth/register              (line 1039)  ← 员工注册
   └─ POST /auth/change-password       (line 1050)  ← 改密
```

注意：这些路由都挂在 `/api` 之下（`standalone.ts:11858`），所以完整路径是 `/api/auth/login`。

## 底层逻辑与数据模型

登录安全依赖两张表，由 `initLoginSecurity` 在建库时创建（`login-security.ts:21`）：

```
login_logs(登录流水)
├─ id, username, ip, success(0/1)
├─ fail_reason, user_agent, created_at

login_lockouts(锁定状态)
├─ id, username(UNIQUE), locked_until, fail_count, created_at
```

用户主表 `users` 的密码字段存的是 PBKDF2 派生串，格式 `pbkdf2:<salt>:<hash>`（`standalone.ts:11835`），哈希参数：`PBKDF2_ITERATIONS=100000`、`KEYLEN=64`、`DIGEST=sha512`（`standalone.ts:11828`）。明文密码永不落库。

Token 则落在 `api_tokens` 表，由 `initAuth` 创建（`api-auth.ts:23`），字段含 `user_id / token / scopes / expires_at / last_used_at`。

## 源码剖析

### 防线一：账号失败计数与锁定

入口是 `recordLoginFailure`（`login-security.ts:46`）。每次登录失败，先把事件写进 `login_logs`，再给该用户的 `fail_count + 1`：

```ts
// login-security.ts:52-67
const existing = (db.query(
  'SELECT id, fail_count FROM login_lockouts WHERE username = ?', [username]) as any[])[0];
const newCount = (existing?.fail_count || 0) + 1;

if (existing) {
  if (newCount >= MAX_FAILURES) {            // 达到 5 次 → 锁定
    const lockUntil = new Date(Date.now() + LOCK_MINUTES * 60000).toISOString();
    db.update('login_lockouts', existing.id, { fail_count: newCount, locked_until: lockUntil });
  } else {
    db.update('login_lockouts', existing.id, { fail_count: newCount });
  }
}
```

阈值常量在文件顶部：`MAX_FAILURES = 5`、`LOCK_MINUTES = 30`（`login-security.ts:11`）。即连续 5 次错密码，账号冻结 30 分钟。

解锁检查在 `isAccountLocked`（`login-security.ts:32`）。它读 `locked_until`，若已过期就删除记录放行，否则返回剩余分钟数：

```ts
// login-security.ts:35-42
const until = new Date(row.locked_until);
if (until < new Date()) {
  db.delete('login_lockouts', username);     // 过期自动清锁
  return { locked: false };
}
const remaining = Math.ceil((until.getTime() - Date.now()) / 60000);
return { locked: true, remainingMinutes: remaining };
```

登录成功时 `recordLoginSuccess` 会删除锁定记录、清零计数（`login-security.ts:71`）。

### 防线二：同 IP 限速（内存计数器）

`checkIpRateLimit`（`login-security.ts:78`）用一张内存 Map 做滑动窗口：

```ts
// login-security.ts:77-92
const ipRateMap = new Map<string, { count: number; resetAt: number }>();
export function checkIpRateLimit(ip: string) {
  const now = Date.now();
  const entry = ipRateMap.get(ip);
  if (!entry || now > entry.resetAt) {        // 窗口已过 → 重置
    ipRateMap.set(ip, { count: 1, resetAt: now + RATE_WINDOW_SEC * 1000 });
    return { allowed: true };
  }
  entry.count++;
  if (entry.count > RATE_LIMIT_PER_IP) {      // 单窗口超 10 次 → 拦截
    return { allowed: false, waitSeconds: Math.ceil((entry.resetAt - now) / 1000) };
  }
  return { allowed: true };
}
```

常量 `RATE_LIMIT_PER_IP = 10`、`RATE_WINDOW_SEC = 60`（`login-security.ts:13`）：单 IP 每分钟最多 10 次登录尝试，超出返回 `429`。

### 防线三：Token 鉴权中间件

`apiAuthMiddleware`（`api-auth.ts:77`）夹在 `/api` 与业务路由之间：

```ts
// api-auth.ts:87-103
const authHeader = req.headers['authorization'] || '';
const token = authHeader.startsWith('Bearer ') ? authHeader.slice(7) : '';
if (!token) {
  if (process.env.NODE_ENV !== 'production') return next();  // ⚠️ 开发态绕过
  return res.status(401).json({ success: false, error: '缺少 API Token ...' });
}
const verified = verifyToken(token);
if (!verified) return res.status(403).json({ success: false, error: 'API Token 无效或已过期' });
req.apiUser = verified;
next();
```

`verifyToken`（`api-auth.ts:55`）查 `api_tokens` 表，校验过期时间并刷新 `last_used_at`。注意 GET 请求、`/api/ai/`、`/api/health` 被直接放行（`api-auth.ts:79`）。

### 中间件链与路由挂载

中间件在 `standalone.ts:159` 串起：

```ts
// standalone.ts:159-161
app.use(express.json());
app.use(loginSecurityMiddleware);          // 登录专用安检
```

真正的 API 路由在文件末尾挂载（`standalone.ts:11858`）：

```ts
// standalone.ts:11858
app.use('/api', apiAuthMiddleware, apiRouter());
```

`apiRouter()` 内部先注册 i18n、RBAC 等专属路由，**最后才注册 catch-all**（`standalone.ts:3772`、`3814` 注释明言「注册于 catch-all 之前，避免被 /:table 拦截」），业务 CRUD 落到 `standalone.ts:4065` 起的 `/:table` 系列。

## 原理剖析

把三道防线串成一次登录请求的时序：

```
客户端           loginSecurity       apiAuth        登录路由           DB
  │  POST /api/auth/login  │                │                │
  ├───────────────────────▶│                │                │
  │                        │ IP限速?        │                │
  │                        │ 账号锁定?      │                │
  │                        │ (429/423 可拦) │                │
  │                        ├───────────────▶│                │
  │                        │                │ 有Token?       │
  │                        │                │ 开发态可绕过   │
  │                        │                ├───────────────▶│
  │                        │                │ 校验密码(PBKDF2)│
  │                        │                │ 失败→recordLoginFailure
  │                        │                │ 成功→recordLoginSuccess
  │  ◀── 401 / 200 ────────┤◀───────────────┤◀───────────────┤
```

两道前置防线（限速、锁定）在到达 token 校验与密码比对之前就拦截恶意流量，既保护数据库少挨打，也让暴力破解在时间上「打不动」。

## 管理者视角

🔑 **防内部数据泄露**：账号锁定意味着即使有人拿到了员工工号，也无法靠「试密码」拖库——试 5 次就锁 30 分钟，且每次失败都进 `login_logs` 审计流水，谁、从哪个 IP、什么时间、为什么失败，一查便知。

🚀 **抗暴力破解**：IP 限速把「一秒千次」压到「一分钟十次」，破解成本从分钟级拉到无法承受。结合锁定，攻击者只能换来一堆 `423 Locked`。

✅ **合规价值**：登录流水 + 锁定记录天然满足等保/ISO27001 里「登录失败处理」「审计日志留存」的条款。向审计方展示 `login_logs` 与 `login_lockouts` 两张表即可证明「系统具备账号保护能力」。

## 注意事项

⚠️ **限速是内存级的**：`ipRateMap` 只是进程内 Map（`login-security.ts:77`）。若你用多实例 / 负载均衡部署，各实例计数器互不共享，限速会失效。需要横向一致时，请把 `checkIpRateLimit` 换成 Redis 等共享存储，算法骨架可直接复用。

🔑 **默认 admin 密码必须改**：出厂 `admin/admin123` 仅用于演示。上线第一件事就是改密码（走 `/api/auth/change-password`），否则任何人都可登录后台。

✅ **锁定时长可配**：`MAX_FAILURES`、`LOCK_MINUTES`、`RATE_LIMIT_PER_IP`、`RATE_WINDOW_SEC` 都在 `login-security.ts:11` 顶部常量，按企业安全策略调整即可，无需改逻辑。

⚠️ **开发态 Token 绕过**：`api-auth.ts:92` 在 `NODE_ENV !== 'production'` 时跳过 Token 校验。生产环境务必设 `NODE_ENV=production`，否则写接口形同裸奔。

## 小结与练习

你已掌握：中间件链 `express.json → loginSecurityMiddleware → apiAuthMiddleware → apiRouter`；三道防线分别是 IP 限速、账号锁定、Token 鉴权；业务统一收口到 `/api/:table`。

动手练习：

1. 把 `LOCK_MINUTES` 改成 60，重启服务，连续 5 次输错密码，观察返回 `423` 与剩余分钟数。
2. 用 `curl` 在 60 秒内对 `/api/auth/login` 请求 11 次，验证第 11 次返回 `429`。
3. 在 `api-auth.ts` 给 `verifyToken` 加一行 `console.log`，登录后调用任意 POST `/api/:table`，确认中间件确实被触发。
