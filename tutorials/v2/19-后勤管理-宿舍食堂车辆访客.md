# 19. 后勤管理：宿舍 / 食堂 / 车辆 / 访客

> **Sowork AI 企业智能ERP系统**（以下简称本系统，版本 v1.2.0）的后勤管理模块，把员工吃、住、行、访四类零散事务统一进一张数据网。你不再需要在 Excel 里手抄宿舍分配表、在微信群里接龙订餐、在门卫本上登记访客。本文带你从界面一路钻到源码，把四个子页逐一讲清。

## 学习目标

- 理解后勤模块在系统内的数据归属与路由事实。
- 能在界面上完成宿舍分配、水电费分摊、食堂订餐记账、车辆使用登记、访客登记四类操作。
- 读懂 `meal_orders` 与 `dormitory_bills` 背后的代码逻辑，知道费用如何落到员工头上。
- 从管理者视角评估「食宿后勤一体化」对管理成本与安全的价值。

## 核心概念

后勤模块由四张主数据表 + 若干业务表组成，全部走 catch-all 路由 `/api/:table`，没有 `/api/hr` 这类按域前缀的拆分。

| 子页 | 核心表（ALLOWED 注册） | 业务含义 |
| --- | --- | --- |
| 宿舍管理 | `dormitories`、`dormitory_assignments`、`dormitory_bills` | 房间、床位分配、水电费账单 |
| 食堂管理 | `canteens`、`meal_menus`、`meal_orders`、`meal_records` | 食堂、每日菜谱、员工订餐与消费 |
| 车辆管理 | `vehicles`、`vehicle_usage` | 车辆档案与用车登记 |
| 访客管理 | `visitors` | 访客预约、签到、签出 |

✅ 记住一点：数据一律通过 `fetch('/api/<表名>', ...)` 直读直写，后端不区分「人力资源域」或「后勤域」，只认表名白名单。

## 界面布局与操作流程

侧边栏 `/logistics` 下挂四个子页（见 `client/src/components/Layout.tsx:98-104`）：

```
/logistics          后勤首页
 ├─ /logistics/dormitory   宿舍管理
 ├─ /logistics/canteen     食堂管理
 ├─ /logistics/vehicle     车辆管理
 └─ /logistics/visitor     访客管理
```

路由在 `app.tsx:248-252` 映射到 `LogisticsPage` 及四个子组件。

**宿舍管理（DormitoryPage）** 内部分四个 Tab：宿舍管理、床位分配、水电费管理、入住记录。新增宿舍时会按「床位数量」自动生成对应床位（`DormitoryPage.tsx:198-214`）。在床位 Tab 里点「入住」并填入员工姓名与工号，即完成分配；点「分摊」把当月水电费按入住人数均摊。

**食堂管理（CanteenPage）** 分食堂管理、菜谱管理、订餐管理、统计概览四个 Tab。员工在「订餐管理」选食堂、日期、餐别，勾选菜品后提交，系统自动算出 `totalPrice` 并写入 `meal_orders`。

**车辆管理（VehiclePage）** 是车辆档案页，登记车牌、车型、年检/保险到期日。用车登记落在 `vehicle_usage` 表（需审批，见注意事项）。

**访客管理（VisitorPage）** 登记访客姓名、被访人、部门、目的、身份证号，到访时「签到」、离场时「签出」，全程可追溯。

## 底层逻辑与数据模型

四个子页共享同一套 CRUD 模式：列表查询带 `page/pageSize/search/status` 参数，单条写 `POST/PUT /api/<table>`，删 `DELETE /api/<table>/<id>`。

宿舍分配如何关联员工？核心是 `employeeId` 字段：床位记录 `dormitory_beds` 持有 `employeeId/employeeName`，入住时同步写一条 `dormitory_occupancy` 入住记录（`DormitoryPage.tsx:237-270`）。水电费账单 `dormitory_bills` 按 `dormitoryId + month` 存总额，分摊时按该宿舍 `checked_in` 的入住人数均分，写回每条入住记录的 `waterShare/electricityShare`。

食堂消费如何记账？`meal_orders` 一条记录 = 一名员工某餐的订餐，含 `employeeId/employeeName/canteenId/mealType/dishes/totalPrice/status`。消费金额随订单落地，统计页（StatsTab）聚合 `meal_orders` 与 `meal_records` 即得出个人/食堂账。

```
宿舍分摊链路：
bill(宿舍+月) ──按入住人数均分──> occupancy.waterShare / electricityShare
食堂记账链路：
订餐表单 ──选菜算价──> meal_orders(totalPrice) ──聚合──> 统计概览
```

## 源码剖析

🔑 看 `meal_orders` 的提交逻辑（`CanteenPage.tsx:436-462`）：前端先把所选菜品 `selectedDishes` 与菜谱 `menu.dishes`（JSON 字符串）比对，用正则 `/¥([\d.]+)/` 把每道菜的单价累加出 `selectedTotal()`，再整单写入。

```ts
// CanteenPage.tsx:436-462 节选
const handleSubmitOrder = async () => {
  const vals = await form.validateFields();
  if (selectedDishes.length === 0) { messageApi.warning('请至少选择一道菜'); return; }
  const menu = selectedMenu();
  const res = await fetch('/api/meal_orders', {
    method: 'POST', headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      id: 'mo_' + Date.now(),
      menuId: menu?.id || '',
      canteenId: vals.canteenId,
      employeeId: 'emp-1', employeeName: '当前用户',
      date: vals.date || today, mealType: vals.mealType,
      dishes: JSON.stringify(selectedDishes),
      totalPrice: selectedTotal(),   // 前端聚合出的金额
      status: 'ordered',
    }),
  });
};
```

再看宿舍水电费分摊（`DormitoryPage.tsx:301-333`）：取该宿舍 `checked_in` 的入住记录，水费/电费各自除以人数，逐条 `PUT` 回写入住记录，并把账单状态从 `pending` 置为 `distributed`。

```ts
// DormitoryPage.tsx:309-322 节选
const perWater = (selectedBill.waterFee || 0) / occRecords.length;
const perElectric = (selectedBill.electricityFee || 0) / occRecords.length;
for (const rec of occRecords) {
  await fetch(`/api/${OCCUPY_TABLE}/${rec.id}`, {
    method: 'PUT',
    body: JSON.stringify({ waterShare: perWater, electricityShare: perElectric }),
  });
}
```

注意：源码中床位的实际读写用的是 `dormitory_beds` / `dormitory_occupancy`，而 `dormitory_assignments` 同样在后端 ALLOWED 白名单中注册（`standalone.ts:1088`），两者是同源扩展，分配思想一致——都靠 `employeeId` 关联员工。

## 原理剖析

后端对后勤表的处理与其他表无差别：`standalone.ts:1087-1089` 把九个后勤表并列写进 ALLOWED 数组，路由命中 catch-all `/api/:table` 后由自研 ORM 引擎直接落库。也就是说，界面上每点一次「入住」「订餐」「签到」，本质都是一次对白名单表的 CRUD，没有独立后勤服务进程。

这种「表即接口」的设计让四块业务共享同一套权限、分页、校验逻辑，但也意味着费用分摊、审批流这类业务规则必须写在前端或触发器里——当前分摊均摊规则就在前端 `handleDistributeSubmit` 中完成。理解这一点，你就能预测：任何新增后勤业务，只要往 ALLOWED 加一个表名 + 写一个页组件即可。

## 管理者视角

🚀 员工食宿后勤一体化显著降低管理成本：宿舍分配、水电费、食堂消费全部用 `employeeId` 串成一条线，HR 与行政不再各建一套 Excel。月底导出入住记录即可核算宿舍成本，聚合 `meal_orders` 即可核算食堂补贴。

✅ 访客登记保安全：`visitors` 表记录被访人、部门、身份证号、签到/签出时间，形成完整可追溯的到访链。谁的访客、何时进、何时走，一查便知，满足厂区/园区的安保合规要求。

## 注意事项

⚠️ **宿舍账单周期**：`dormitory_bills.month` 是 `2026-01` 这样的字符串，分摊前务必确认月份唯一。重复添加同宿舍同月账单会导致二次分摊、费用翻倍。

🔑 **车辆使用需审批**：`vehicles` 仅存车辆档案，实际用车登记在 `vehicle_usage`，且按规范应走审批流。不要把「车辆可用」误当作「可直接开走」，用车前需登记用途、驾驶人并获批。

✅ **访客记录可追溯**：`visitors` 的 `checkInTime`/`checkOutTime` 由系统时间戳写入，不可手改。请勿随意 `DELETE` 已签出记录，否则破坏安保追溯链。

## 小结与练习

你已掌握：后勤四子页的界面入口与路由事实、宿舍分配如何借 `employeeId` 关联员工、食堂消费如何以 `meal_orders.totalPrice` 记账、以及 `standalone.ts:1087-1089` 的表注册机制。

练习：
1. 新增一间 4 床位宿舍，分配两名员工入住，添加一张水电费账单并执行分摊，核对两人 `waterShare` 是否均为均摊值。
2. 在食堂订餐管理中下单一餐，观察 `meal_orders` 新增记录的 `totalPrice` 是否等于所选菜品单价之和。
3. 登记一名访客并完成签到、签出，验证 `visitors` 状态从 `pending`→`checked_in`→`checked_out` 流转。
