# 44-测试与质量保障

## 学习目标

你将从代码级别理解 Sowork AI 企业智能ERP系统（本文首现全称，下文简称系统）v1.2.0 的端到端测试体系。读完后，你能够：

- 说出 25 个 `test-*.js` 脚本分别覆盖哪些业务模块；
- 看懂极简断言 `ok(name, cond)` 的工作原理；
- 用 `BASE` 环境变量指向不同环境跑测试；
- 理解「副作用验证」——不仅查接口返回，还要查数据库是否真的发生变化；
- 为新增功能编写配套回归测试，守住院本不回退。

## 核心概念

系统的测试不是单元测试，而是一套**端到端（E2E）脚本**：它们直接启动服务后，用 `fetch` 打真实 HTTP 接口，断言业务结果。

三个关键概念先记牢：

| 概念 | 含义 | 价值 |
| --- | --- | --- |
| `BASE` | 被测服务地址环境变量，默认 `http://localhost:3000` | 一套脚本测本地、预发、生产 |
| `ok(name, cond)` | 极简断言：条件为真计通过，否则计失败 | 零依赖，Node 直接跑 |
| 副作用验证 | 断言后回查数据库/库存/状态日志 | 防止「接口说成功，数据没动」 |

🔑 这套体系最大的特点：不引入测试框架（无 jest/mocha），每个脚本都是独立可执行的 Node 文件，降低开源项目贡献者的上手门槛。

## 界面布局与操作流程

测试没有图形界面，运行流程是一条直线：

```
启动服务 → 设置 BASE → 运行 test-*.js → 看通过/失败计数 → 退出码决定 CI 是否阻断
```

你在本机实测的顺序如下：

```bash
# 1. 安装依赖（注意版本兼容）
npm install --legacy-peer-deps

# 2. 启动系统（端口 3000，账号 admin/admin123）
npm run dev        # 或 npm start

# 3. 另开终端，跑单个测试脚本
BASE=http://localhost:3000 node test-rbac.js

# 4. 跑全部测试（脚本独立，用循环串起来）
for f in test-*.js; do node "$f" || echo "FAIL: $f"; done
```

脚本退出码为 `0` 表示全通过，`1` 表示有失败——这是 CI 流水线里质量门禁的底层依据。

## 底层逻辑与数据模型

每个测试脚本都遵循同一套骨架，差异只在断言内容：

```text
require('./fetch-polyfill')   // 兼容旧 Node 的 fetch
const BASE = process.env.BASE || 'http://localhost:3000'
let pass = 0, fail = 0
function ok(name, cond) { ... }   // 计数器 + 日志
async function main() {
  // 1. 准备数据（如创建订单、商品）
  // 2. 调接口并断言返回
  // 3. 回查数据库验证副作用
  // 4. 清理测试数据
  process.exit(fail === 0 ? 0 : 1)
}
main().catch(e => process.exit(1))
```

数据模型层面，测试直接消费系统已有的业务表：商品库存 `shop-goods.stock`、售后单 `shop-order-aftersale.status`、库存流水 `shop-stock-logs`、角色权限 `rbac/roles` 与 `rbac/resolve`。脚本不需要额外建表，它们把系统当成黑盒，通过 HTTP 读写同一套数据。

## 源码剖析

先看断言核心 `ok()` 的真实定义。文件 `test-rbac.js:5`：

```js
function ok(name, cond, extra) {
  if (cond) { pass++; console.log(`  ✅ ${name}`); }
  else { fail++; console.log(`  ❌ ${name}` + (extra ? ` — ${extra}` : '')); }
}
```

它接收一个可读名称、一个布尔条件、可选的失败详情，成功加 `pass` 失败加 `fail`，仅此而已。系统里 `ok` 有函数式、箭头式、带 `extra` 三种变体，但逻辑完全一致。

再看一条典型权限断言，`test-rbac.js:52-54`——验证超级管理员角色 resolve 后确实包含细粒度删除权限点：

```js
const r5b = await fetch(`${BASE}/api/rbac/resolve`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ roleIds: ['role_super_admin'] })
});
const d5b = await r5b.json();
ok('resolve super_admin 含 cms:article:delete', d5b.permissions.includes('cms:article:delete'));
```

这是「返回即真」的典型：接口返回了权限数组，断言数组里是否含有目标点。

更有价值的是**副作用验证**。文件 `test-aftersale.js:57-58` 在退货退款走完后，不是只看接口返回 `refunded`，而是回去查商品库存是否真的回滚：

```js
const gReturned = await api('GET', '/api/shop-goods/' + goodsId);
if (stockBefore !== null)
  ok(gReturned.data.stock === stockBefore,
     '商品库存已回滚(=' + stockBefore + ', 实际 ' + gReturned.data.stock + ')');
```

注意脚本在开头先记录 `stockBefore`（`test-aftersale.js:23`），走完退货链路后再比对。这正是「不仅查返回，还查数据库是否真的变化」的范例。同理，仅退款场景会断言**不新增**退货回滚日志（`test-aftersale.js:80-81`），反向验证库存不动。

## 原理剖析

为什么不放单元测试？因为系统的业务逻辑大量写在服务端路由与数据库事务里，纯函数少。E2E 脚本用真实请求覆盖「请求→路由→事务→落库→回查」全链路，成本最低、最贴近用户视角。

`BASE` 的设计让同一脚本在本地 `:3000`、同事机器、CI 容器里无缝切换。历史上系统曾用 `:3400` 端口，因此源码注释里偶见 3400 字样，当前 v1.2.0 统一以 `:3000` 为默认——你设 `BASE` 即可覆盖，无需改代码。

🚀 副作用验证的原理本质是**双重确认**：接口响应码只证明「调用被处理」，而回查业务表证明「状态机真的迁移、库存真的扣减」。二者缺一都可能漏掉「假成功」类 bug，比如事务回滚了但返回 200。

## 管理者视角

从管理角度看，这 25 个脚本累计承载了 **240+ 条断言**，覆盖 RBAC、售后、仓库、分销、CMS、权限点、i18n 等核心模块。它们的价值在于：

- **发版不回归**：每次上线前跑一遍，任何旧功能被破坏会立刻报错，退出码 `1` 直接阻断发布；
- **质量门禁**：开源项目里，贡献者提交新功能时配套测试，审阅者一眼能看出覆盖是否到位；
- **低成本可信**：无需搭建复杂 CI 框架，一条 `for` 循环就能当门禁用，对小团队尤其友好。

对管理者而言，要求「新增功能必须配套测试」比要求「写文档」更可执行，因为它能被机器验证。

## 注意事项

⚠️ 跑测试前必须先启动系统服务，并设置 `BASE` 指向正确地址；服务没起或端口错，脚本会因连接失败直接抛异常退出。

🔑 断言一定要验证**副作用**而非仅看返回码/返回字段。返回 200 不等于数据真变了，库存扣减、状态迁移、流水记录必须回查确认。

✅ 新增任何功能请同步添加对应测试脚本或用例，这是防止回归最廉价的手段，也是社区贡献的默认门槛。

⚠️ 依赖安装请用 `npm install --legacy-peer-deps`，否则部分 peer 依赖冲突会导致装包失败、测试脚本无法运行。

## 小结与练习

你已学会 Sowork AI 企业智能ERP系统 v1.2.0 的测试体系：以 `BASE` 指向服务、用 `ok()` 做极简断言、靠回查数据库落实副作用验证。25 个脚本构成发版前的质量门禁。

练习：
1. 启动服务后运行 `node test-warehouse.js`，观察库存断言如何回查；
2. 给一个你想新增的小功能（如「优惠券核销」）设计三条 `ok()` 断言，其中至少一条验证数据库副作用；
3. 把 `BASE` 设成本地 `:3000` 之外的值，体会环境切换。
