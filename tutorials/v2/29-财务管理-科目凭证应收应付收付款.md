# 29-财务管理：科目 / 凭证 / 应收 / 应付 / 收付款（含分析会计）

> 本文基于 **Sowork AI 企业智能ERP系统** v1.2.0 编写。运行环境：账号 `admin/admin123`，服务端口 `3000`，依赖安装使用 `npm install --legacy-peer-deps`。路由事实：系统**没有** `/api/erp` 这类按域前缀的分组；业务数据走统一的 catch-all 路由 `/api/:table`（受 `ALLOWED` 白名单约束）；财务分析会计模块位于 `addons/analytic_odoo`，由 `.py` 与 `.js` 双形态模型共同构成。

读完本文，你将能够：

## 核心概念

财务管理的本质是**用一套可平衡的账本记录企业的资金来龙去脉**。五个子功能按数据流顺序如下：

```
科目(accounts)   账簿结构：资产/负债/权益/收入/费用
      │
      ▼
凭证(journal_entries + journal_items)   每笔业务的借贷记录
      │
      ├─────────────► 应收(ar_invoices)   客户欠你的钱
      │                     │
      │                     ▼
      │                 收款(payments, type=receive)
      │
      └─────────────► 应付(ap_invoices)   你欠供应商的钱
                            │
                            ▼
                        付款(payments, type=pay)

分析会计(account_analytic_*)   给每笔金额打上 项目/部门 维度标签（横向切片）
```

- **科目**：账本的骨架，决定钱分几类（资产/负债/权益/收入/费用）。
- **凭证**：一笔业务的最小记账单元，由一条 `journal_entries` 头 + 多条 `journal_items` 行组成。
- **应收**：已开票未收款的客户欠款；**应付**：已收票未付款的供应商欠款。
- **收付款**：真实资金流动，会回写发票的 `paid_amount` 与状态。
- **分析会计**：在「财务会计」之上叠加「管理维度」，例如「这笔费用属于哪个项目、哪个部门」。

## 界面布局与操作流程

财务模块入口在左侧导航「财务管理」（`client/src/components/Layout.tsx` 约 178–185 行），五个子页由 `client/src/pages/FinancePage/` 实现：

| 子页 | 路由 | 对应组件 | 主要操作 |
|------|------|----------|----------|
| 科目管理 | `/finance/accounts` | `AccountsPage.tsx` | 新增/编辑/删除科目，设类型与层级 |
| 凭证管理 | `/finance/journal-entries` | `JournalEntriesPage.tsx` | 录分录、过账、查看试算平衡 |
| 应收发票 | `/finance/ar-invoices` | `ARInvoicesPage.tsx` | 开应收票、跟踪回款 |
| 应付发票 | `/finance/ap-invoices` | `APInvoicesPage.tsx` | 录进项票、跟踪付款 |
| 收付款管理 | `/finance/payments` | `PaymentsPage.tsx` | 登记收款/付款，关联发票 |

**一笔业务的标准操作流（以销售为例）：**

1. 在「科目管理」确认存在「应收账款」「主营业务收入」等科目。
2. 在「应收发票」开一张 `AR` 发票（状态 `unpaid`）。
3. 在「凭证管理」录一张凭证：借 应收账款、贷 主营业务收入，保存为 `draft`。
4. 点「过账」把凭证状态置为 `posted`，科目余额随之变化。
5. 客户打款后，在「收付款」登记一笔 `type=receive` 的收款并关联该发票，发票状态自动变为 `paid` 或 `partial`。

🚀 提示：`JournalEntriesPage.tsx:68-71` 在前端就做了借贷相等校验，不相等直接拦截提交。

## 底层逻辑与数据模型

财务相关表在 `server/standalone.ts` 的 `ALLOWED` 白名单（约 1128–1129 行）中注册，包括 `account_chart`、`journal_entries`、`journal_items`、`payment_methods`、`tax_rates`；运行时实际通过专用路由读写的核心表为：

| 表名 | 字段要点 | 说明 |
|------|----------|------|
| `accounts` | `code,name,type,parent_id,level,balance` | 科目主档，初始 `balance=0` |
| `journal_entries` | `code,entry_date,description,total_debit,total_credit,status` | 凭证头，`status`: draft/posted |
| `journal_items` | `entry_id,account_id,account_code,debit,credit` | 凭证分录行，一对多 |
| `ar_invoices` | `customer_id,total_amount,paid_amount,status,due_date` | 应收，`status`: unpaid/partial/paid |
| `ap_invoices` | `supplier_id,total_amount,paid_amount,status` | 应付，同三态 |
| `payments` | `type(receive/pay),invoice_id,amount,payment_method` | 收付款流水 |

分析会计的模型在 `addons/analytic_odoo/models/`：

- `account_analytic_account.js` / `analytic_account.py`：`account_analytic_account`（分析账户，挂到某个 `plan`）。
- `account_analytic_plan.js` / `analytic_plan.py`：`account_analytic_plan`（分析维度计划，如「项目」「部门」）。
- `account_analytic_line.js` / `analytic_line.py`：`account_analytic_line`（分析分录，把金额关联到具体维度）。

这些 `.py` 是 Odoo 原生模型定义、`.js` 是转换到本系统的模型描述，二者共同支撑「财务会计 + 分析会计」的双账本结构。

## 源码剖析

**1. 凭证创建：自动汇总借贷并写入分录行**

`server/standalone.ts:8274-8307` 的 `POST /api/journal-entries`：

```ts
const id = `je_${Date.now()}`;
const code = `JE${Date.now().toString().slice(-8)}`;
let total_debit = 0, total_credit = 0;
items?.forEach((item) => {            // 先累加借贷
  total_debit += item.debit || 0;
  total_credit += item.credit || 0;
});
db.insert('journal_entries', { id, code, ..., total_debit, total_credit, status: 'draft' });
items?.forEach((item) => {            // 再逐行写 journal_items
  db.insert('journal_items', {
    id: itemId, entry_id: id, account_id: item.account_id,
    account_code: item.account_code, account_name: item.account_name,
    debit: item.debit || 0, credit: item.credit || 0, remark: item.remark
  });
});
```

注意：创建阶段**只算合计、不校验平衡**（真正的借方必须等于贷方由前端拦截 + 业务规范保证），状态保持 `draft`，余额尚未变动。

**2. 凭证过账：借贷差额更新科目余额**

`server/standalone.ts:8309-8328` 的 `POST /api/journal-entries/:id/post`：

```ts
items.forEach((item) => {
  const account = db.findById('accounts', item.account_id);
  if (account) {
    const newBalance = (account.balance || 0) + (item.debit || 0) - (item.credit || 0);
    db.update('accounts', item.account_id, { balance: newBalance });
  }
});
db.update('journal_entries', req.params.id, { status: 'posted', posted_at: ... });
```

这一段是复式记账落地的关键：借增加余额、贷减少余额，凭证一旦 `posted`，科目余额即被改写，且不可轻易回退。

**3. 收付款回写发票状态**

`server/standalone.ts:8485-8525` 的 `POST /api/payments`：根据 `type` 找到对应 `ar_invoices` 或 `ap_invoices`，累加 `paid_amount` 并换算状态：

```ts
const newPaid = (invoice.paid_amount || 0) + amount;
const status = newPaid >= invoice.total_amount ? 'paid'
              : newPaid > 0 ? 'partial' : 'unpaid';
db.update('ar_invoices', invoice_id, { paid_amount: newPaid, status });
```

这正是「财务业务一体」的代码体现：收款动作直接驱动应收发票从「未收」走向「已收」。

## 原理剖析

**复式记账的不变式**：任意一张已过账凭证，其 `total_debit === total_credit`。系统用 `GET /api/trial-balance`（`standalone.ts:8331-8349`）按 `account_code` 汇总所有 `journal_items`，返回 `summary.balanced` 标志——这也是「借贷平衡」在代码层的直接投影。

**业务到财务的映射**：销售单产生应收（`ar_invoices`），采购单产生应付（`ap_invoices`），收付款（`payments`）则把「债权债务」兑现为「现金流」。三者通过 `invoice_id` 关联，形成可追溯的闭环。

**分析会计的横向切片**：财务会计回答「钱从哪来、到哪去」，分析会计回答「为了什么花」。一笔 `journal_items` 可同时携带 `analytic_account_id`，在 `account_analytic_line` 中沉淀为「项目/部门成本」，让同一笔钱既进总账、又进管理报表。

## 管理者视角

作为管理者，你不必逐行写代码，但要抓住三条价值线：

- **财务业务一体**：销售→应收→收款、采购→应付→付款自动贯通，不需要财务手工二次抄录，杜绝「业务账」与「财务账」两张皮。
- **实时现金流可视**：应收/应付的 `status`（unpaid/partial/paid）与收付款流水，让你随时看清「谁还欠我、我欠谁、已回款多少」。
- **合规记账**：凭证 `draft→posted` 的状态机 + 借贷平衡校验，保证每笔记录可审计、可追溯，满足内控与税务合规要求。

🔑 管理建议：让业务人员在业务发生时就生成应收/应付，财务只负责过账与核对，把「记账」变成「确认」而非「重做」。

## 注意事项

⚠️ **借贷必须平衡**：凭证的 `total_debit` 与 `total_credit` 必须相等，否则总账与试算平衡表会出现偏差，过账前应再次核对。

🔑 **凭证不可随意删除**：凭证是法律责任载体，系统设计上以「过账/留痕」替代物理删除，确需更正应做红冲或调整凭证，保留原始记录。

✅ **科目体系可定制**：`accounts` 表的 `type`（资产/负债/权益/收入/费用）与 `parent_id` 层级支持按企业实际需要搭建科目树，不必拘泥于模板。

⚠️ **收付款务必关联发票**：不关联 `invoice_id` 的收付款不会回写应收/应付状态，会造成「钱收了但发票还显示未收」的对账混乱。

✅ **分析维度提前规划**：`account_analytic_plan`（项目/部门）应在上线初期定义好，事后补维度成本高且历史数据难回填。
