# 26-销售管理：客户订单发货退货

## 学习目标

你将从代码层面理解 Sowork AI 企业智能ERP系统 v1.2.0 的「销售管理」模块，逐一看清 5 个子功能（客户分组、客户档案、销售订单、发货管理、退货管理）背后的数据表与接口，并能描出一条完整的销售闭环：客户 → 订单 → 发货（扣库存）→ 退货（回库）。

读完后你应该能回答：销售订单和订单行是什么关系？发货单如何挂到订单上？库存到底在哪一步被扣减？

## 核心概念

销售管理的本质是「以客户为中心，把意向变成出库」。它在系统里被拆成 5 张主表与 3 张明细表：

| 子功能 | 主表 | 明细表 | 关键外键 |
| --- | --- | --- | --- |
| 客户分组 | `customer_groups` | — | — |
| 客户档案 | `customers` | — | `group_id` |
| 销售订单 | `sales_orders` | `sales_order_items` | `customer_id` |
| 发货管理 | `deliveries` | `delivery_items` | `order_id`、`order_item_id` |
| 退货管理 | `returns` | `return_items` | `order_id`、`delivery_id`、`customer_id` |

🔑 一个核心认知：本模块只管「单据状态」，真正的库存增减发生在仓储模块（见 25 篇）。两者通过 `inventory_transactions.source_doc_id` 关联，这是理解闭环的钥匙。

## 界面布局与操作流程

在前端 `Layout.tsx` 的侧边栏中，销售模块挂载在 `/sales` 下，共 5 个入口（`client/src/components/Layout.tsx:156-163`）：

```
/sales          销售管理（概览）
/sales/customer-groups  客户分组
/sales/customers        客户档案
/sales/orders           销售订单
/sales/deliveries       发货管理
/sales/returns          退货管理
```

你的日常操作流是：

1. 先建**客户分组**（如 VIP、普通、经销商），设定折扣与信用额度。
2. 在分组下建**客户档案**，填写联系人、税号、开户行。
3. 为客户开**销售订单**，录入多行明细（款式/颜色/尺码/数量/单价）。
4. 订单审核后，基于它生成**发货单**；发货动作把订单推进到 `shipping`/`shipped`。
5. 客户退回部分货物时，开**退货单**挂到原发货单，质检通过后回库。

## 底层逻辑与数据模型

所有销售接口都注册在同一个 Express Router 上，并在 `standalone.ts:11858` 以 `app.use('/api', apiAuthMiddleware, apiRouter())` 挂载，所以实际请求路径是 `/api/customers`、`/api/sales-orders` 等。

⚠️ 注意：销售这些表**不在** `ALLOWED` 通用 CRUD 白名单里（`standalone.ts:1060-1140`）。白名单只服务于 catch-all 的 `/api/:table` 通用接口；销售模块走的是显式域路由，控制更精细。这就是「无 `/api/erp` 按域前缀、数据走 catch-all `/api/:table`」的真实体现——两套机制并行。

5 张主表的关系树如下：

```
customer_groups(id, discount, credit_limit)
   └─ customers(id, group_id, available_credit)
         └─ sales_orders(id, customer_id, status, total_amount)
               ├─ sales_order_items(order_id, sku_id, qty, unit_price)
               └─ deliveries(order_id, warehouse_id, status)
                     ├─ delivery_items(delivery_id, order_item_id, sku_id, qty)
                     └─ [仓储出库] inventory_transactions(type=out, source_doc_id=delivery)
               └─ returns(order_id, delivery_id, customer_id, status)
                     ├─ return_items(return_id, sku_id, qty)
                     └─ [仓储入库] inventory_transactions(type=in, source_doc_id=return)
```

`customers` 在建档时把 `credit_limit` 同时写入 `available_credit`（`standalone.ts:7383`），这就是后续应收账款与信用控制的起点。

## 源码剖析

**1) 销售订单的创建。** 接口 `POST /api/sales-orders` 在 `standalone.ts:7437` 起。它会先校验客户与明细，再算总额、写主表、写多行明细（`standalone.ts:7451-7482`）：

```ts
const id = `so_${Date.now()}`;
const code = `SO${Date.now().toString().slice(-8)}`;
let total_amount = 0;
items.forEach((item) => { total_amount += (item.qty||0)*(item.unit_price||0); });
total_amount = total_amount * (1 - (discount||0));
db.insert('sales_orders', { id, code, customer_id, customer_name, ...,
  total_amount, discount: discount||0, paid_amount: 0,
  status: 'pending', payment_status: payment_status||'unpaid', ... });
items.forEach((item) => {
  db.insert('sales_order_items', {
    id: `soi_${Date.now()}_...`, order_id: id, sku_id: item.sku_id,
    qty: item.qty||0, unit_price: item.unit_price||0,
    amount: (item.qty||0)*(item.unit_price||0), ... });
});
```

🚀 看重点：主表与明细是「先主后从」两步走，明细靠 `order_id` 指回主表。订单初始为 `pending`，需经 `POST /api/sales-orders/:id/approve` 变为 `approved` 才能发货。

**2) 发货单如何联动订单状态。** `POST /api/deliveries` 在 `standalone.ts:7529`，创建时立即把订单推进到 `shipping`（`standalone.ts:7559`）：

```ts
db.insert('deliveries', { id, code, order_id, warehouse_id, ..., status:'pending', ... });
items.forEach((item) => db.insert('delivery_items', {
  delivery_id: id, order_item_id: item.order_item_id, sku_id: item.sku_id, qty: item.qty||0, ... }));
db.update('deliveries', id, { total_qty });
db.update('sales_orders', order_id, { status: 'shipping' });   // ← 联动订单
```

发货完成 `POST /api/deliveries/:id/complete` 会把订单置为 `completed`（`standalone.ts:7576`）。注意 `delivery_items.order_item_id` 把发货行精确对应到某条订单行，支持分批发货。

**3) 库存到底在哪扣？** 源码里 `deliveries` 的 `ship`/`complete` 只改自身状态（`standalone.ts:7564-7579`），**并不会**自动调用 `applyStockChange`。`applyStockChange`（`standalone.ts:8872`）是为电商 `shop_*` 商品表服务的，与 ERP 仓储无关。

真正的库存引擎是仓储模块的 `POST /api/inventory/transaction`（`standalone.ts:7053`）：

```ts
db.insert('inventory_transactions', { ..., type, qty, source_doc_id, source_doc_type, ... });
const inv = db.findWhere('inventory', { warehouse_id, ...(sku_id?{sku_id}:{material_id}) });
const newQty = type==='in' ? (inv.qty||0)+qty : (inv.qty||0)-qty;   // 入加出减
db.update('inventory', inv.id, { qty: newQty, ... });
```

✅ 所以闭环是**两步协调**：你在销售模块点「发货」推进单据；再到仓储模块对这张 `deliveries` 做一笔 `type='out'`、`source_doc_id=发货单id` 的出库，库存才真正下降。退货同理，做一笔 `type='in'`、`source_doc_id=退货单id` 的入库回滚。`source_doc_*` 字段就是跨模块追溯的桥梁。

## 原理剖析

为什么不直接在发货时扣库存？因为 ERP 里「单据确认」和「实物出库」是两个业务时点：仓库可能分多批发、可能拣货后发现缺件。把扣减留在仓储事务里，能复用车间的批次、货位、盘点逻辑，也避免销售随便改库存。

同理，退货必须走 `returns` 这张独立单据，而不是直接给库存加数——因为退回的货物要质检、可能报废，且要与原发货单对账。这种「单据驱动、仓储执行」的分层，正是降低超卖的核心：`sales_orders` 只描述「想卖多少」，`inventory_transactions` 才描述「真实走了多少」，两者用 `source_doc` 对账，天然能算出未发货占用。

## 管理者视角

- **销售漏斗**：用 `sales_orders.status`（pending→approved→shipping→completed/cancelled）就能画出转化漏斗，cancel 率高的客户/分组值得复盘。
- **应收账款前置**：订单建好就有 `total_amount` 与 `paid_amount(0)`，配合 `customers.available_credit`，可在发货前就做信用拦截，把「先货后款」的风险关进笼子。
- **订单驱动库存联动**：销售不直接动库存，但通过 `source_doc_id` 把每张发货单绑到一笔出库事务，管理者能随时追问「这张单的货到底出了没有」，从机制上降低超卖与飞单。

## 注意事项

⚠️ **发货才扣库存**：销售模块的「发货」只推进单据状态，库存下降发生在仓储出库事务。别把「已发货」误当成「已扣库存」。

🔑 **退货要质检**：`returns` 有 pending→approved→completed 三态，只有完成态才应触发回库入库事务；质检不合格的部分不要进库存。

✅ **客户分组支撑差异化定价**：`customer_groups.discount` 与 `credit_limit` 是分组级默认值，建档时落到客户；做促销或授信时优先改分组，再让客户继承，省去逐户维护。

⚠️ **明细外键别断**：发货行 `delivery_items.order_item_id` 必须指向真实订单行，否则分批发货、对账会乱。删除订单时会级联删 `sales_order_items`（`standalone.ts:7509`），已发货的订单请勿直接删。

## 小结与练习

你已看清销售管理 5 子功能的表结构与接口，并理解「单据状态在销售、库存增减在仓储、用 `source_doc_id` 串联」的设计哲学。

练习：
1. 用 admin/admin123 登录（端口 3000，依赖 `npm install --legacy-peer-deps`），新建一个「VIP」分组（折扣 0.9），在其下建客户并开一张订单。
2. 调 `POST /api/deliveries` 生成发货单，观察 `sales_orders.status` 是否变 `shipping`。
3. 在仓储模块对这张发货单做一笔 `type='out'` 出库事务，验证 `inventory.qty` 下降，并确认 `inventory_transactions.source_doc_id` 指向该发货单。
4. 开一张退货单，完成后做对应 `type='in'` 入库，形成完整闭环。
