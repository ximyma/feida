# 15-考勤管理：班次 / 排班 / 规则 / 打卡 / 假期 / 加班 / 统计 / 日报

本篇基于 **Sowork AI 企业智能ERP系统** v1.2.0 讲解考勤模块。你本地启动：`npm install --legacy-peer-deps` 后 `npm run dev`，后台服务默认端口 **3000**，用 `admin/admin123` 登录即可进入考勤菜单。

考勤模块是命名路由组 `/attendance/*`，所有底层表的读写走 catch-all 路由 `/api/:table`，表名必须在 `ALLOWED` 白名单中注册（`server/standalone.ts:1060-1140`，其中考勤相关表集中在 `1065-1068`：shift_types、schedules、attendance_records、leave_records、leave_balances、overtime_records、shift_change_requests、attendance_rules、check_locations、daily_attendance_reports、monthly_attendance_summary、attendance_devices、leave_rule_configs 等）。

## 学习目标

- 说清考勤 8 个子页的界面入口、操作流程与底层表。
- 读懂打卡（`clock-in`/`clock-out`）如何写 `attendance_records` 并判定迟到、早退。
- 理解排班如何批量生成"打卡计划"，请假如何自动扣减年假余额。
- 掌握日报聚合引擎如何把打卡、请假、加班合并成部门报表。
- 从管理者视角看考勤数据如何驱动薪资核算与用工合规。

## 核心概念

考勤数据是一条链：先配**班次（shift_types）**，再按班次做**排班（schedules）**，员工据排班**打卡**写入 `attendance_records`；请假、加班走审批后落入 `leave_records` / `overtime_records`；引擎按日聚合出 `daily_attendance_reports`。这张链路图要记牢：

```
班次 shift_types ──┐
                   ├─▶ 排班 schedules ──▶ 打卡 attendance_records
请假 leave_records ─┤                              │
加班 overtime_records┤                              ▼
                   └─▶ 规则 attendance_rules ──▶ 日报 daily_attendance_reports
```

### 考勤数据模型总览表

下表是考勤模块的核心表与关键字段（均已注册进 `ALLOWED`）：

| 核心表 | 路由/接口 | 关键字段 | 说明 |
|--------|-----------|----------|------|
| `shift_types` | `/attendance/shift`、`/shift-types` | name、type、startTime、endTime、workHours、isActive | 班次定义，迟到/早退阈值在引擎里另有默认 |
| `schedules` | `/attendance/schedule`、`/schedule/batch-generate` | employeeId、date、shiftTypeId、scheduledStart、scheduledEnd、isRestDay、isHoliday | 打卡计划，逐员工逐日 |
| `attendance_records` | `/attendance/clock-in`、`/attendance/clock-out` | employeeId、date、clockIn、clockOut、workHours、lateMinutes、earlyLeaveMinutes、status | 实际打卡流水 |
| `leave_records` | `/attendance/leave` | employeeId、leaveType、startDate、endDate、days、status | 请假申请与审批 |
| `leave_balances` | `/attendance/leave-balances`、`/leave/balances` | employeeId、year、leaveType、totalDays、usedDays、availableDays | 年假等余额 |
| `overtime_records` | `/attendance/overtime` | employeeId、date、overtimeType、hours、status | 加班申请与审批 |
| `attendance_rules` | 引擎读取（id=`default`） | lateThreshold（默认15）、defaultWorkHours | 迟到阈值等规则 |
| `daily_attendance_reports` | `/attendance/daily-report(s)` | date、department、normalCount、lateCount、absentCount、overtimeCount、data(JSON) | 考勤日报 |
| `attendance_devices` | `/attendance/devices` | deviceType、config、status、lastSyncAt | 打卡设备（钉钉/企微/飞书/中控/APP） |
| `leave_rule_configs` | `/attendance/leave-rules` | annualDays、carryoverDays、maxDays、accrueMonth | 年假规则全局配置 |

🔑 一个易踩的坑：状态字段存在**两套词汇表**。`attendance_records.status` 由打卡接口写入中文（`正常`/`迟到`/`早退`/`迟到早退`），而 `daily_attendance_reports.data` 里的 `details.status` 由日报引擎写入英文（`normal`/`late`/`early`/`absent`/`leave`/`overtime`/`rest`/`holiday`）。首页与统计页为了兼容二者，代码里都同时判断了中英两种写法（如 `AttendancePage.tsx:98`）。你做二次开发或对接时，务必先确认自己读的是哪张表，否则会出现"明明迟到了却统计不出来"的怪现象。

## 考勤首页（概览）

考勤首页对应 `client/src/pages/AttendancePage/AttendancePage.tsx`，路由 `/attendance`（`app.tsx:214`）。它是整个模块的驾驶舱。

### 界面布局
顶部是标题与日期选择器；其下是 5 张统计卡片（今日出勤、迟到、请假、缺勤、加班）；再下面是"⚡ 快捷操作"九宫格（日报、排班、班次、请假、加班、统计）；主体是一个带 4 个 Tab 的卡片：📊 考勤总览、📝 考勤记录、⏱️ 打卡签到、📅 排班管理。

### 操作流程
你在"考勤总览"看到应出勤人数、实际出勤、迟到/早退、缺勤与总工时/加班工时的环形出勤率图。点"⏱️ 打卡签到"可亲自上班打卡、下班签退；点"📝 考勤记录"可补录、编辑、删除某条打卡。需要深入配置时，点九宫格任意图标即可跳转对应子页。

### 数据模型与字段
首页本身不新建表，它聚合读取 `attendance_records`、`schedules`、`employees`。打卡状态字段 `status` 同时兼容中文（`正常`/`迟到`/`请假`/`缺勤`/`加班`）与英文（`normal`/`late`/`early`/`absent`/`leave`/`overtime`），见 `AttendancePage.tsx:37-44` 的 `statusMap`。

### 源码剖析
打卡按钮直接调用后端：

```ts
// AttendancePage.tsx:117 上班打卡
const res = await fetch('/api/attendance/clock-in', {
  method: 'POST',
  body: JSON.stringify({ employeeId, employeeName, location: '公司' }),
});
// AttendancePage.tsx:149 下班签退
await fetch('/api/attendance/clock-out', { method: 'POST', body: JSON.stringify({ employeeId }) });
```

"今日概况"卡片通过 `loadData`（`AttendancePage.tsx:70`）并行拉取 `/api/attendance_records`、`/api/schedules`、`/api/employees`、`/api/attendance/today/:employeeId`，在内存里用 `selectedDate` 过滤后做统计（`AttendancePage.tsx:94-105`）。

### 原理剖析
首页是一个"只读聚合视图"：它不持有业务规则，只是把已存在的打卡与排班数据按日期切片展示。把高频操作（打卡、看概况）放在首页，是为了让一线员工无需理解排班/规则等后台概念也能完成日常动作。

## 班次管理

班次管理对应 `ShiftTypeConfigPage.tsx`，路由 `/attendance/shift`（`app.tsx:215`），底层主表 `shift_types`。

### 界面布局
上方 4 张统计卡：班次总数、已启用、已停用、平均工时。下方一张大表格，列含班次名称、类型（早班/中班/晚班/行政班/弹性班 Tag）、上班时间、下班时间、工作时长、启用状态开关、说明、操作。右上角"新增班次"按钮，表格上方有搜索框与刷新。

### 操作流程
你点"新增班次"→ 弹窗填**班次名称**（如"早班A"）、**班次类型**（下拉）、**上班时间**（如 09:00）、**下班时间**（如 18:00）、**工作时长**（可选，提示自动算）、**启用状态**（开关）→ 保存即写 `shift_types`。列表里可随时编辑、删除，或用开关一键启停。

### 数据模型与字段
`shift_types` 的关键字段：`name`、`type`、`startTime`、`endTime`、`workHours`、`isActive`、`description`。注意前端表单**没有** `lateThreshold` 字段（见 `ShiftTypeConfigPage.tsx:183-219`），迟到阈值实际由 `attendance_rules` 与日报引擎统一控制，班次只管时间窗口。后端还有 `getShiftTypes`/`createDefaultShiftTypes`（`standalone.ts:1949-1971`），首次访问会自动建默认班次。

### 源码剖析
保存逻辑走 catch-all 路由，新增是 `POST /api/shift_types`，编辑是 `PUT /api/shift_types/:id`：

```ts
// ShiftTypeConfigPage.tsx:76
const url = editing ? `/api/${TABLE}/${editing.id}` : `/api/${TABLE}`;
const method = editing ? 'PUT' : 'POST';
// ShiftTypeConfigPage.tsx:88 删除
await fetch(`/api/${TABLE}/${id}`, { method: 'DELETE' });
```

### 原理剖析
班次是整条链的最上游"模板"。它把"几点上、几点下"固化为可复用对象，排班时只需引用 `shiftTypeId`，不必重复填时间。这样当上班时间调整，改一个班次即可影响所有引用它的排班。

## 排班管理

排班管理对应 `ScheduleManagementPage.tsx`，路由 `/attendance/schedule`（`app.tsx:216`），底层主表 `schedules`。

### 界面布局
顶部 4 张卡：排班总数、早班、中班、晚班。右侧操作按钮有三个：年假管理、批量排班、新增排班。表格列含员工、部门、日期、班次 Tag、上班、下班、操作。列表上方支持按员工搜索、按班次筛选。

### 操作流程
- **单条排班**：点"新增排班"→ 选员工、日期、班次、状态 → 保存写 `schedules`。
- **批量排班**：点"批量排班"→ 弹窗选员工（多选，留空取前 10 名）、起止日期、班次、重复模式（每天/每周/每两周）、排除休息日（星期多选）、是否覆盖已有 → 点"批量生成"。
- **年假管理**：点"年假管理"→ 看年假余额表，或切到"批量更新"选年度执行按入职年限重算。

### 数据模型与字段
`schedules` 关键字段：`employeeId`、`employeeName`、`department`、`date`、`shiftTypeId`、`scheduledStart`、`scheduledEnd`、`isRestDay`、`isHoliday`。前端批量请求体（`ScheduleManagementPage.tsx:198-210`）带的 `pattern`、`restDays`、`overwrite` 会原样转给引擎。年假余额表 `leave_balances` 字段：`totalDays`、`usedDays`、`availableDays`。

### 源码剖析
批量排班落在 `standalone.ts:1900`：

```ts
// standalone.ts:1900
router.post('/schedule/batch-generate', async (req, res) => {
  const { employeeIds, startDate, endDate, shiftTypeId, ... } = req.body;
  const result = await batchGenerateSchedules(db, { employeeIds, startDate, endDate, shiftTypeId, shiftTypeName, pattern, restDays, overwrite });
  res.json(result);
});
```

引擎 `batchGenerateSchedules`（`schedule-engine.ts:91-185`）按 `generateDateRange`（`190` 起）把日期区间按 `pattern` 展开、跳过 `restDays` 指定的星期，再逐员工 `db.insert('schedules', ...)`，遇已存在且非覆盖则 `skipped++`。年假批量更新走 `standalone.ts:1984` → `batchUpdateLeaveBalances`。

### 原理剖析
排班的本质是"把班次按时间轴铺到每个人头上"，形成一张**打卡计划表**。打卡时只需 `employeeId + date` 查 `schedules` 取出 `scheduledStart` 当作基准时间——这就是"以排班为基准"的由来。批量生成而非手工逐条，避免重复劳动，并支持排除周日等休息日。`pattern` 让周期排班（如固定每周早班）一次搞定。

## 考勤规则

考勤规则对应 `AttendanceRulesPage.tsx`（**不是**同名但未被路由的 `AttendanceRulePage.tsx`），路由 `/attendance/rules`（`app.tsx:217`）。它是一个统一配置中心。

### 界面布局
页面内部分 5 个 Tab：**班次 / 排班 / 假期 / 免考勤 / 设备**。班次、排班 Tab 提供增删改表格；假期 Tab 展示年假规则（入职满1年享受 N 天、每年递增、上限、结转上限）与余额；免考勤 Tab 管理免打卡员工；设备 Tab 管理打卡设备（中控/钉钉/企微/飞书/APP）的注册、测试连接、同步。

### 操作流程
- **看/改年假规则**：切到"假期"Tab，系统从 `/api/attendance/leave-rules` 拉取 `leave_rule_configs`，你可改年假天数、结转天数、上限、计入月份后保存。
- **管设备**：切到"设备"Tab，点某设备"测试连接"会调 `/attendance/devices/:id/test`，填好凭证后点"同步"调 `/attendance/devices/:id/sync`。
- 班次、排班 Tab 与前面的班次管理、排班管理功能互补，可在此处统一维护。

### 数据模型与字段
本页涉及三张表：`leave_rule_configs`（annualDays、carryoverDays、maxDays、accrueMonth，见 `AttendanceRulesPage.tsx:108-113`）、`attendance_devices`（deviceType、config、status、lastSyncAt、syncCount）、以及它复用的 `shift_types`/`schedules`。注意真实生效的迟到阈值是 `attendance_rules` 表（id=`default`），由日报引擎读取。

### 源码剖析
设备相关 API 集中在 `standalone.ts:1270-1470`：

```ts
// standalone.ts:1273 拉取设备，无数据则初始化 5 个默认设备（APP 为 active，其余 unconfigured）
// standalone.ts:1311 测试连接
router.post('/attendance/devices/:id/test', ...)
// standalone.ts:1349 从第三方同步打卡
router.post('/attendance/devices/:id/sync', ...)
```

年假规则读写在 `standalone.ts:1537`（GET，初始插入 `leave_rule_global`：`annualDays:15, carryoverDays:5, maxDays:30, accrueMonth:1`）与 `:1553`（PUT）。

### 原理剖析
把"班次、排班、年假规则、免考勤、设备"收进一个"考勤规则"总台，是因为它们都属于**约束与配置层**而非日常流水。设备 Tab 解决"打卡数据从哪来"——无论钉钉/企微/中控，同步后都统一进入 `attendance_records`，保证数据同源。⚠️ 需说明：仓库里另有一个 `AttendanceRulePage.tsx`（操作 `attendance_rules` 迟到/缺勤阈值+处罚），但 `app.tsx:217` 并未把它挂到路由上，属于冗余文件；`attendance_rules` 仍被日报引擎消费（id=`default`）。

## 打卡（含设备）

打卡分两端：员工在首页"⏱️ 打卡签到"点按钮，或设备同步；后端核心是 `clock-in`/`clock-out`（`standalone.ts:1143-1251`）。

### 界面布局
前端打卡 UI 在首页"⏱️ 打卡签到" Tab：一个大时钟、上班打卡/下班签退按钮，下方显示今日上班、下班时间及班次状态。设备端的注册与同步则在考勤规则页"设备" Tab。

### 操作流程
- **手动打卡**：在首页点"上班打卡"→ 调 `POST /attendance/clock-in`；下班点"下班签退"→ 调 `POST /attendance/clock-out`。
- **设备打卡**：在考勤规则页"设备" Tab 给钉钉/企微/中控填好凭证并"测试连接"，通过后"同步"把第三方记录拉进本系统。

### 数据模型与字段
打卡写入 `attendance_records`，关键字段：`clockIn`、`clockOut`、`workHours`、`lateMinutes`、`earlyLeaveMinutes`、`status`、`lateCount`、`isRestDay`、`isHoliday`、`remark`。设备表 `attendance_devices` 的 `config` 以 JSON 存凭证（host/port、appKey/appSecret、corpId 等）。

### 源码剖析
上班打卡（`standalone.ts:1143-1200`）先查当日是否已打卡，再取排班作基准算迟到：

```ts
// standalone.ts:1158 取当日排班作基准
const schedule = db.findWhere('schedules', { employeeId, date: today }) as any[];
const shift = schedule[0] || { shiftTypeId: 'st_1', shiftTypeName: '标准班', scheduledStart: '09:00', scheduledEnd: '18:00' };
// standalone.ts:1165 计算迟到分钟（注意：>0 即判迟到，无宽限）
const lateMinutes = (nowHour*60+nowMin) - (schedHour*60+schedMin);
const status = lateMinutes > 0 ? '迟到' : '正常';
db.insert('attendance_records', record);   // 写打卡记录
updateDailyReport(db, today, record);       // 增量更新部门日报
```

下班签退（`standalone.ts:1202-1251`）用 `clockIn/clockOut` 算 `workHours`，对比 `scheduledEnd` 判定早退，并把状态拼接为"迟到早退"。设备同步（`standalone.ts:1349`）对钉钉/企微分别取 token 后拉取记录，统一调 `importAttendanceRecord`（`standalone.ts:6596`）落库。

### 原理剖析
打卡本质是"以排班为基准的时间比较"：`scheduledStart` 与当前时间换算为分钟差即迟到分钟；签退同理得早退分钟。🔑 所有来源（APP、钉钉、企微、中控）最终都汇入同一张 `attendance_records`，这是后续统计与薪资口径一致的前提。⚠️ 这里有一处需你留意的不一致：`clock-in` 判定迟到是 `lateMinutes > 0` 即"迟到"、**无宽限也无阈值**；而日报引擎（`attendance-report-engine.ts:472-475`）用了 **5 分钟宽限**且需达到 `lateThreshold`（默认15）才算迟到。同一个人可能打卡时显示"迟到"、日报里却判"正常(稍迟)"。

## 假期管理

假期管理对应 `LeaveManagementPage.tsx`，路由 `/attendance/leave`（`app.tsx:218`），底层主表 `leave_records`，余额在 `leave_balances`。

### 界面布局
顶部 4 张卡：待审批、已通过、已拒绝、请假总天数。下方表格列：员工、请假类型（年假/病假/事假/婚假/产假/… Tag）、开始/结束日期、天数、原因、状态（待审批/已通过/已拒绝/已撤销 Badge）、操作（详情、通过、拒绝、编辑、删除）。点"新增请假"弹窗，点"详情"开抽屉看完整信息并可直接审批。

### 操作流程
你点"新增请假"→ 选员工、请假类型、起止日期（系统自动算天数）、天数、原因、状态（默认待审批）→ 保存写 `leave_records`。在列表或抽屉里对"待审批"记录点"通过"或"拒绝"（拒绝需填原因），状态改为 `approved`/`rejected`。

### 数据模型与字段
`leave_records` 关键字段：`employeeId`、`leaveType`、`startDate`、`endDate`、`days`、`reason`、`status`、`approvedAt`、`rejectReason`。余额表 `leave_balances`：`year`、`leaveType`、`totalDays`、`usedDays`、`availableDays`、`pendingDays`。

### 源码剖析
审批通过时前端只改状态（`LeaveManagementPage.tsx:132-140`）；真正的余额扣减发生在后端审批链路调用 `deductLeaveBalance`（`schedule-engine.ts:518-547`）：

```ts
// schedule-engine.ts:526 按 lb_{employeeId}_{year}_{leaveType} 定位余额
const balanceId = `lb_${employeeId}_${year}_${leaveType}`;
const balance = db.findById('leave_balances', balanceId);
if (balance.availableDays < days) return { success: false, message: '年假余额不足' };
db.update('leave_balances', balanceId, {
  availableDays: balance.availableDays - days,
  usedDays: balance.usedDays + days,
});
```

驳回则调 `restoreLeaveBalance`（`schedule-engine.ts:552-577`）把天数加回 `availableDays`。

### 原理剖析
请假走"先落记录、审批后动余额"的两段式：提交时只写 `leave_records`（pending），审过才改 `leave_balances`，驳回则回补——这样避免误扣。余额 ID 用 `lb_员工_年_类型` 复合键，天然按年隔离、按类型隔离。年假天数由 `calculateAnnualLeaveDays`（`schedule-engine.ts:415-418`，公式 `baseDays + (年限-1)*increment`，封顶 `maxDays`）按入职年限自动算。

## 加班管理

加班管理对应 `OvertimeManagementPage.tsx`，路由 `/attendance/overtime`（`app.tsx:219`），底层主表 `overtime_records`。

### 界面布局
顶部 4 张卡：待审批、已通过、加班总时长、工作日加班。表格列：员工、加班类型（工作日/休息日/节假日加班 Tag）、加班日期、时长(h)、原因、状态、操作（详情、通过、拒绝、编辑、删除），布局与假期页高度一致。

### 操作流程
你点"新增加班"→ 选员工、加班类型、加班日期、时长(小时，最小0.5)、原因 → 保存写 `overtime_records`（状态待审批）。在列表或详情抽屉里"通过/拒绝"。审批通过后的加班时长会进入日报的 `overtimeCount`。

### 数据模型与字段
`overtime_records` 关键字段：`employeeId`、`overtimeType`、`date`、`startTime`、`endTime`、`hours`、`reason`、`status`、`approvedBy`、`approvedAt`、`rejectReason`。注意它存的是**小时数**，与请假按"天"不同。

### 源码剖析
审批接口同为行内 `PUT /api/overtime_records/:id` 改状态（`OvertimeManagementPage.tsx:104-123`）。加班数据被日报引擎在 `attendance-report-engine.ts:146` 通过 `getOvertimeRecords(db, date)` 取走，并在 `calculateEmployeeAttendance` 中：若员工在休息日且存在加班，状态判为 `overtime`（`attendance-report-engine.ts:405-418`）；普通日有审批通过加班则标"正常(加班)"（`:514-518`）。

### 原理剖析
加班与请假都是"申请—审批—计入报表"的模式，但维度不同：请假扣的是时间额度（余额），加班增加的是工时与报酬依据。把加班类型分为工作日/休息日/节假日，是为了后续薪资按不同倍率计薪（休息日、节假日通常更高）。

## 考勤统计

考勤统计对应 `AttendanceStatisticsPage.tsx`，路由 `/attendance/statistics`（`app.tsx:220`），数据来自 `attendance_records` 与 `employees`，纯前端聚合。

### 界面布局
顶部右侧是部门下拉与"月份"选择器。下方 6 张统计卡：应出勤天数、实际出勤天数、出勤率、迟到人次、早退人次、请假天数。再下是"🏢 部门考勤对比"表和"⚠️ 考勤异常明细"表（可按迟到/早退/旷工/请假筛选，支持导出）。

### 操作流程
你选好月份与部门 → 页面用 `selectedMonth` 过滤 `attendance_records`（`AttendanceStatisticsPage.tsx:49-51`），实时算出出勤率与各异常人次。在异常明细里选"迟到/早退/旷工/请假"快速定位问题记录。

### 数据模型与字段
统计读 `attendance_records`（employeeId、date、clockIn、clockOut、lateMinutes、earlyLeaveMinutes、status）与 `employees`（department、status）。出勤率公式（`AttendanceStatisticsPage.tsx:83`）：`normalCount / (workDays × 在职人数) × 100`。

### 源码剖析
全部在浏览器内存计算，无独立后端接口：

```ts
// AttendanceStatisticsPage.tsx:72 统计
const lateCount = monthRecords.filter(r => (r.lateMinutes||0) > 0).length;
const earlyCount = monthRecords.filter(r => (r.earlyLeaveMinutes||0) > 0).length;
// AttendanceStatisticsPage.tsx:104 部门对比
deptMap[dept] = { shouldAttend: deptEmps.length * stats.workDays,
                  actualAttend: deptRecs.filter(r => r.clockIn).length, ... };
```

### 原理剖析
统计页是"分析层"，它不写数据，只从原始打卡记录里派生指标。把计算放在前端，好处是筛选/切换部门即时响应、无需后端往返；代价是大数据量时依赖一次性全量拉取。部门对比表用"应到 vs 实到"直观暴露出勤短板。

## 日报查询

日报查询对应 `AttendanceDailyReportPage.tsx`，路由 `/attendance/daily-report`（`app.tsx:221`），底层主表 `daily_attendance_reports`。

### 界面布局
顶部两个 Tab：列表、批量生成。列表是日报表格，列含日期、部门、应到人数、正常、迟到、早退、缺勤、请假、加班，每行可"查看详情/重新生成"。整体统计条汇总所选区间的总人数、正常、迟到、缺勤、请假。

### 操作流程
你点"查看详情"→ 弹窗展示该日每位员工的明细（来自 `data` JSON 字段解析）。需要补数时点"重新生成"调 `POST /api/attendance/daily-report`；切到"批量生成" Tab 选日期区间 → 调 `POST /api/attendance/daily-report/batch` 一次性回填历史日报。

### 数据模型与字段
`daily_attendance_reports` 关键字段：`date`、`department`（"全公司"或具体部门）、`totalEmployees`、`normalCount`、`lateCount`、`earlyLeaveCount`、`absentCount`、`leaveCount`、`overtimeCount`、`restDayCount`、`holidayCount`、`data`（JSON 明细数组）、`updatedAt`。

### 源码剖析
生成入口 `standalone.ts:1573` 调 `generateDailyAttendanceReport`（`attendance-report-engine.ts:110`）：

```ts
// attendance-report-engine.ts:164 逐员工判定状态
for (const emp of employees) {
  const detail = calculateEmployeeAttendance(emp, date, attendanceRecords,
    schedules, leaveRecords, overtimeRecords, rules);
  details.push(detail);
}
// attendance-report-engine.ts:188 日报ID与 upsert
const reportId = department ? `dar_${department}_${date...}` : `dar_all_${date...}`;
db.update(...) || db.insert('daily_attendance_reports', reportData);
```

列表 `GET /api/attendance/daily-reports`（`standalone.ts:1588`）把 `data` 字段 JSON 解析为 `details` 返回。

### 原理剖析
日报引擎是"以员工为中心的多源合并"：同一员工一天只有一条明细，状态按优先级判定——**请假 > 休息日/节假日 > 无打卡缺勤 > 打卡明细**（`calculateEmployeeAttendance`：`attendance-report-engine.ts:385` 先查请假，`400` 查休息/节假日，`447` 无记录即缺勤）。明细 JSON 化存入 `data`，既保留聚合计数便于列表展示，又保留逐人明细供详情钻取。

## 与其他模块的衔接

考勤不是孤岛，它向上游与下游都连着其它模块，理解这些衔接能帮你排障时少走弯路。

- **人事模块**：`employees` 是考勤一切计算的锚点。打卡、排班、日报都靠 `employeeId` 关联员工；年假余额按 `entryDate`（入职日期）算年限；部门统计依赖 `employees.department`。员工信息错了，考勤会全盘错。
- **薪酬模块**：日报与统计里的迟到、早退、加班工时，最终汇入工资表的加减项。你在本篇看到的 `overtimeCount`、`lateMinutes` 就是薪资引擎的取数来源。
- **审批/流程模块**：请假、加班的"通过/拒绝"本质是审批动作。当前版本在前端直接改 `status` 落库，若日后接入正式工作流，只需把审批节点接到同一张 `leave_records`/`overtime_records` 即可，数据模型已预留 `approvedAt`、`rejectReason`、`approvedBy`。
- **系统/合规模块**：`audit_logs`、`login_logs` 记录操作与登录，配合 `attendance_records` 构成从"谁在何时打卡"到"日报怎么算"再到"工资怎么发"的完整可追溯链条，满足审计与劳动法留存要求。

🔑 记住一点：考勤模块对外只通过 `attendance_records`、`leave_records`、`overtime_records`、`daily_attendance_reports` 这几张表"输出结论"，其余模块来读即可，耦合被牢牢控制在表边界内。

## 管理者视角

🚀 考勤数据直接驱动薪资：日报的 `overtimeCount`、迟到早退统计会进入薪酬核算的工资项与加班费计算；月度考勤统计是发薪与绩效的依据。一个真实的闭环是——今天某人休息日加班 4 小时，加班管理审批通过 → 日报 `overtimeCount`+1 且 `data` 明细标"正常(加班)" → 月末薪酬按休息日倍率计发加班费。

⚠️ 规避虚报：设备打卡需先注册并 `status='active'`（`standalone.ts:1273` 默认设备多为 `unconfigured`，仅 APP 内置为 `active`）；未注册设备无法同步，避免手工随意补卡。APP 打卡可配 GPS/WiFi 半径（`config` 中 `radiusMeters`）。

🔑 合规工时：日报自动汇总"正常/迟到/早退/缺勤/请假/加班/休息/节假日"八类计数，班长每日核对，HR 月末导出，满足劳动法对工时与加班记录的留存要求。

📌 数据可追溯：所有打卡无论来源都进 `attendance_records`，配合 `audit_logs`/`login_logs`，形成从原始打卡到日报再到薪资的完整证据链。

🔑 路由设计取舍：前端用命名路由组 `/attendance/*` 组织 8 个子页，让菜单与 URL 清晰对应；后端却把所有表的增删改查收敛到 catch-all `/api/:table`，仅对少数复杂动作（打卡、批量排班、设备同步、日报生成）单独开路由。这种"前端细分、后端收敛"的折中，既保证了页面路由的可读性，又避免了为每张表都写一套 CRUD 接口。代价是你要分清：简单读写走 `/api/:table`，业务动作才有专属 `/attendance/...` 或 `/schedule/...` 路径。

## 注意事项

⚠️ 打卡设备需先注册：`attendance_devices` 初始只有 5 个默认设备且多为 `unconfigured`，钉钉/企微需填 AppKey、CorpID 等凭证并通过连接测试，飞书需企业自建应用权限，否则 `sync` 返回失败或同步 0 条（`standalone.ts:1368-1456`）。

🔑 请假余额 `leave_balances` 自动核算：`batchUpdateLeaveBalances`（`schedule-engine.ts:423`）按入职日期算年假天数并结转上年额度（上限 `maxCarryOver`）；请年假经 `deductLeaveBalance` 实时扣减，驳回经 `restoreLeaveBalance` 回补，无需手工改表。

✅ 日报聚合便于班长核对：每日一键生成 `daily_attendance_reports`，缺勤/迟到一目了然，是月度薪资与合规审计的可靠底表。

⚠️ 迟到阈值默认 15 分钟，且日报引擎有 5 分钟宽限（`attendance-report-engine.ts:354`、`:472`）。改阈值要在"考勤规则"页改 `attendance_rules`（id=`default`）才对日报生效；但要注意 `clock-in` 写记录时并不读该阈值，二者口径不完全一致，排障时别混淆。

⚠️ 年假规则存在两套入口：排班管理的"年假管理"走 `/leave/annual-rules`（`getAnnualLeaveRule`），考勤规则页的假期 Tab 走 `/attendance/leave-rules`（`leave_rule_configs`）。维护时建议以考勤规则页为准，避免两处不一致。

你已掌握：8 个子页的路由、表与操作流程；打卡如何写 `attendance_records` 并判定迟到早退；排班如何批量生成打卡计划；请假如何自动扣/回补余额；日报聚合引擎的状态优先级与合并逻辑；以及管理者视角下的数据驱动与合规要点。
