## 核心概念

🚀 Sowork AI 商城是 ShopXO 6.9 免费版的「超集」。ShopXO 免费版里优惠券、秒杀、拼团、砍价、分销都是付费插件、不含实现；Sowork AI 全部自建，故功能覆盖且超越它。

你需要先建立三个心智模型：

| 概念 | 含义 | 落脚数据 |
| --- | --- | --- |
| 商品主数据 | 一个可售标的，含标题、价格、库存、相册等 | `shop_goods` |
| 商品变体 | 同一商品下按规格拆分的 SKU | `shop_goods_skus` |
| 分类模式 | 单分类 or 多分类命中 | `category_mode` / `category_ids` |

商城是系统内的 B2C 销售渠道，与 ERP 库存台账（`shop_warehouse_goods`）通过 `product_id + sku` 实时联动。

## 界面布局与操作流程

后台入口来自 `client/src/components/Layout.tsx` 的 `/shop-admin` 菜单（约 201–215 行），所有子页共用一个页面 `/admin/shop`，靠 `?tab=` 切换：

```
/shop-admin
├─ 商城管理        /admin/shop
├─ 商品管理        /admin/shop?tab=goods      ← 本篇主角
├─ 订单管理        /admin/shop?tab=orders
├─ 售后 / 评价 / 品牌 / 优惠券 / 秒杀 / 分销
├─ 仓库 / 快递 / 页面装修
└─ 🛒 访问商城     /shop
```

进入商品 Tab 后，点「新建商品」弹出表单。它的内部是一个多 Tab 面板：基础、规格、参数、相册、SEO、扩展……你拖拽相册缩略图即可排序，点「设主图」设定首图。

操作流：新建商品 → 填基础信息（价格/库存/分类/品牌）→ 规格页生成 SKU → 相册页上传并排序 → 保存。代码里每次切换 Tab 触发 `loadData()`（`ShopAdminPage.tsx:152-154`）。

## 底层逻辑与数据模型

商品主表 `shop_goods` 关键字段（来自 `POST /api/shop-goods`，`standalone.ts:10383`）：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `category_mode` | string | `single` 单分类 / `multi` 多分类 |
| `category_ids` | JSON 文本 | 多分类时存分类 ID 数组 |
| `images` | JSON 文本 | 相册图片 URL 数组（顺序即展示序） |
| `spec_images` | JSON 文本 | `{规格值: 图片URL}` 规格图片绑定 |
| `spec_data` / `param_data` | JSON 文本 | 规格组 / 商品参数 |
| `price` / `stock` / `status` | 标量 | 售价、库存、上架状态 |

SKU 落到 `shop_goods_skus`，每行含 `goods_id`、`spec_values`（如 `"红/大"`）、`price`、`stock`、`sku_code`。注意 `spec_values` 以 JSON 文本存储，前端「笛卡尔积生成」按钮帮你在 `ShopAdminPage.tsx:967` 自动组合规格。

## 源码剖析

**1. 商品创建：多分类命中逻辑。** 后端对 `category` 筛选同时比对主分类与 `category_ids` 数组（`standalone.ts:10316-10319`）：

```ts
if (category) goods = goods.filter((g: any) => {
  if (g.category_id === category) return true;
  try { return (JSON.parse(g.category_ids || '[]')).includes(category); } catch { return false; }
});
```

保存时，多分类模式下会自动把第一个分类回填为主分类（`ShopAdminPage.tsx:304-307`）：

```ts
if (values.category_mode === 'multi') {
  const ids: string[] = Array.isArray(values.category_ids) ? values.category_ids : [];
  values.category_id = ids[0] || '';
}
```

**2. 相册拖拽排序。** `reorderImages` 用数组 splice 实现（`ShopAdminPage.tsx:133-139`）：

```ts
const reorderImages = (from: number, to: number) => {
  if (from < 0 || from === to || from >= goodsImages.length) return;
  const arr = [...goodsImages];
  const [m] = arr.splice(from, 1);
  arr.splice(to, 0, m);
  syncImages(arr);   // 同步进表单字段 images
};
```

**3. 规格图片绑定。** `spec_images` 是「规格值→图片」映射，通过 `syncSpecImages` 写入表单（`ShopAdminPage.tsx:143`）：

```ts
const syncSpecImages = (map: Record<string, string>) => {
  setSpecImages(map);
  goodsForm.setFieldValue('spec_images', JSON.stringify(map));
};
```

后端落库时直接 `JSON.stringify(spec_images || {})`（`standalone.ts:10401`）。这就是商品详情页「选红色就显示红图」的数据来源。

## 原理剖析

✅ 为什么多分类要靠 `category_ids` 数组 + 主分类双写？因为列表筛选既要 `category_id === category` 的精确命中，也要 `category_ids.includes()` 的扩展命中，双写保证两种查询路径都快。

商品与 ERP 产品档案（`product_skus`）的关系：商城商品 `id` 即 ERP 侧的 `product_id`。订单支付成功后，`finalizePaidOrder` 调用 `applyStockChange`（`standalone.ts:10899`），按 `product_id + sku` 扣减 `shop_warehouse_goods`：

```ts
applyStockChange(wh ? wh.id : null, it.product_id, it.sku || '', -(it.quantity || 0), 'order', ...);
```

该方法同步更新商品总库存 `shop_goods.stock` 与仓库明细，并写 `shop_stock_logs`（`standalone.ts:8872-8889`）——这就是「电商下单、ERP 库存实时联动、不超卖」的实现原理。售后退货则反向回滚（`:10963`）。

## 管理者视角

🔑 从经营角度看三件事：

- **渠道拓展**：商城是低成本 B2C 增量渠道，与 ERP 共享库存，无需另建一套货品体系。
- **商品丰富度**：相册排序、规格图片、参数详情直接决定转化率，建议每个 SKU 都配规格图。
- **库存安全**：因为支付即扣减 ERP 库存，改价、下架、调库存都要谨慎，避免前端展示与实际可用量背离。

## 注意事项

⚠️ **改价影响在售订单**：已支付订单的金额不会随商品改价回滚，但改价会影响后续下单；促销价 `promotion_price` 与时段 `promotion_start/end` 请先小流量验证。

🔑 **多分类需要索引与回填**：`category_ids` 是 JSON 文本，筛选靠 `JSON.parse(...).includes()`，量大时建议约束分类数量并依赖主分类 `category_id` 做快速命中。

✅ **图片拖拽排序提升体验**：相册数组顺序即前台展示顺序，保存前务必把主图拖到第一位或点「设主图」，否则首图可能不是你想推的卖点图。

⚠️ **规格图片键名要一致**：`spec_images` 的键必须与 SKU 的 `spec_values` 取值严格对应，否则前台选规格不换图。

## 小结与练习

本篇你掌握了：商城是 ShopXO 免费版的超集；商品主数据在 `shop_goods`，变体在 `shop_goods_skus`；多分类靠 `category_mode`+`category_ids` 双写；相册拖拽与规格图片绑定是前端数组/映射同步；订单支付通过 `applyStockChange` 联动 ERP 库存。

3. 模拟下单支付，到仓库库存列表确认 `shop_warehouse_goods` 相应 `product_id/sku` 库存已扣减。
