# Sowork AI 企业智能ERP系统 v1.2.0 部署与运维教程

本篇带你从零把 **Sowork AI 企业智能ERP系统**（以下简称本系统）部署到生产环境，并掌握备份、升级与容灾。我们将对照真实工程文件讲清三种部署方式，让你既会"点按钮"，也懂"背后的线"。

## 学习目标

- 掌握三种部署方式：Docker 容器化、绿色版一键启动、原生 Node 启动。
- 理解数据持久化与备份机制，能独立完成升级与容灾演练。
- 从管理者视角评估私有化部署的成本、合规价值与运维投入。

## 核心概念

- 构建产物：服务端编译到 `dist/server/standalone.js`，前端静态资源在 `dist/client`。
- 数据目录：运行时位于 `/app/data`（容器内），宿主映射为 `./data`，库文件为 `*.db`（默认 SQLite）。
- 运行环境：Node 22，监听端口 3000，默认管理员账号 `admin/admin123`。
- 依赖红线：安装必须 `npm install --legacy-peer-deps`，否则 peer 冲突会直接导致装包失败。

## 界面布局与操作流程

部署拓扑（以 Docker 为例）：

```
宿主机 ./data  ──挂载──►  容器 /app/data   (SQLite *.db 持久化)
浏览器 :3000  ──►  容器 3000  ──►  node dist/server/standalone.js
                              └─ dumb-init 作 PID 1，回收僵尸进程
```

日常运维的四条主路径：

1. 启动：`docker compose up -d`，或双击 `启动系统.bat`。
2. 访问：`http://<服务器IP>:3000`，用 `admin/admin123` 登录。
3. 监控：`docker logs -f <容器名>` 观察启动日志与异常堆栈。
4. 停止：`docker compose down`，或前台运行时按 `Ctrl+C`。

## 底层逻辑与数据模型

系统同时支持 SQLite 与 PostgreSQL（依赖含 `better-sqlite3`、`pg`）。默认走 SQLite，库文件直接落盘到 data 目录：

| 维度 | SQLite（默认） | PostgreSQL |
|------|----------------|------------|
| 库文件 | `./data/*.db` | 外部实例 |
| 备份方式 | 拷贝文件 / 定时快照 | `pg_dump` |
| 适用场景 | 单机、轻量、私有化 | 高并发、集群化 |

数据卷挂载使容器销毁后业务数据不丢——compose 里 `./data:/app/data` 即宿主目录与容器目录双向同步。无论你用哪种部署方式，真正要保护的就是这个 data 目录。

## 源码剖析

Dockerfile 是多阶段构建：构建阶段装全量依赖并编译，运行阶段只保留产物与运行时依赖。

`d:/feida/Dockerfile:2-12`（构建阶段）：
```
FROM node:22-alpine AS builder
RUN apk add --no-cache python3 make g++
WORKDIR /app
COPY package.json ./
RUN npm install --legacy-peer-deps
COPY . .
RUN npm run build:server && npm run build:client
RUN npm prune --omit=dev
```

`d:/feida/Dockerfile:14-30`（运行阶段）：
```
FROM node:22-alpine
RUN apk add --no-cache dumb-init
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
RUN mkdir -p /app/data
ENV SERVER_PORT=3000
EXPOSE 3000
VOLUME ["/app/data"]
ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "dist/server/standalone.js"]
```

三处关键设计：① 用 `dumb-init` 作 PID 1，避免信号丢失与僵尸进程堆积；② `VOLUME ["/app/data"]` 声明持久层；③ `--legacy-peer-deps` 从头贯穿装包，保证依赖可解析。

数据卷是备份核心。`d:/feida/docker-compose.yml:9-15` 中 `volumes` 字段 `./data:/app/data` 把宿主目录挂进容器，`restart: unless-stopped` 让容器意外退出后自动拉起。

`package.json` 的 scripts 定义了完整生命周期（`d:/feida/package.json:7-16`）：
```
dev         前后端并发热更新（concurrently）
build       build:server + build:client
build:server  tsc 编译，并拷贝 ai-service 等运行时 JS
build:client  vite 打包前端静态资源
start        node dist/server/standalone.js
```

绿色版 `启动系统.bat` 的逻辑等价于：检测无 `node_modules` 则装依赖 → 检测无 `dist` 则构建 → 用 `netstat` 清理 3000 端口占用 → `node dist/server/standalone.js` 启动，并在 `EADDRINUSE` 时自动重试一次。

## 原理剖析

为什么必须 `--legacy-peer-deps`？React 19 与部分 `radix-ui`、`tiptap` 等子依赖的 peer 声明尚未对齐，npm 默认的严格 peer 校验会直接报错中断。该参数放宽 peer 冲突，是项目既定约束，不可省略。

多阶段构建的价值：构建阶段需要 `python3/make/g++` 编译原生模块（如 `better-sqlite3`、`sharp`），运行阶段仅保留 alpine 基础镜像加产物，镜像更小、攻击面更低。

数据持久化的本质：容器文件系统是临时的，只有挂载卷（bind mount / volume）能跨容器生命周期存活。你的业务数据、上传附件都落在 `/app/data`，务必挂到卷上。

## 管理者视角

🔑 私有化部署的核心卖点是**数据不出域**。本系统可完全运行在您的内网服务器，客户资料、薪酬、合同等敏感数据不经过任何第三方云，满足合规与审计要求。

运维人力：Docker 方式一次编排长期受益，适合有专职运维的团队；绿色版适合无运维资源的中小企业，双击即用；原生 Node 适合已有 Node 环境的开发者自托管。

备份与容灾：SQLite 单文件天然易备份，但也是单点故障源。请每日定时拷走 `./data` 并异地留存；若用 PG，定期 `pg_dump` 并实际验证可恢复性。没有备份的运维等于裸奔。

## 注意事项

⚠️ 升级必用 `npm install --legacy-peer-deps`，漏掉会装包失败，切勿用纯 `npm install`。

🔑 升级前先备份数据卷 / `./data` 目录（或 PG dump）。构建与重启期间服务短暂不可用，建议在业务低峰期操作。

✅ 生产环境务必设置 `NODE_ENV=production`。该模式下服务端启用第 06 篇所述的严格 Token 校验与产物优化；开发态的宽松校验在生产下不生效，遗漏会带来安全隐患。

升级标准流程：
```
1. 备份 ./data        ── 数据安全底线，先备后动
2. 拉取 / 拷贝新代码
3. npm install --legacy-peer-deps   ← 红线，不可省略
4. npm run build
5. 重启：docker compose up -d --build   或   重跑 启动系统.bat
6. 验证：访问 :3000，用 admin/admin123 登录确认
```
