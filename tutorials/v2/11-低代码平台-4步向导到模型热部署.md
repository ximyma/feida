# 第 11 篇：低代码平台——4 步向导到模型热部署

Sowork AI 企业智能ERP系统（下文简称「本系统」）v1.2.0 的低代码平台，让你在浏览器里点几下就能搭出一个完整业务应用：命名模块、加字段、配关系、一键生成。生成的不是页面 Demo，而是真正可存数据的标准 CRUD 模型，并且**无需重启服务**即可热部署到运行中的 ORM。

本文面向两类读者：企业管理者看「业务人员如何零代码搭应用」，技术开发看「向导如何变成模型文件再热注册到 ORM」。你会读到真实的源码片段、一张全链路 ASCII 图，以及字段热更新为什么不用重启。

## 核心概念

低代码平台本质是一个「模型生成器 + 热部署器」。它把你在界面上的点击，翻译成符合本系统 ORM 规范的模型定义，再写到磁盘并即时注册。

三个关键词：

| 概念 | 含义 | 对应产物 |
|------|------|----------|
| 模块（Module） | 一个独立业务应用，如「客户关系管理」 | `addons/<模块名>/` 目录 |
| 模型（Model） | 一张业务数据表，如「联系人信息」 | `models/<表名>.js` + 数据库表 |
| 热部署（Hot Deploy） | 写盘后即时注册进运行中的 ORM，不重启 | `__feida_models.register()` |

🔑 核心公式：**向导配置 → 模型文件 → ORM 注册 → 自动建表 → 前端菜单**。这五步，前两步发生在磁盘，后三步发生在内存与数据库。

> ⚠️ 注：表中的 `__feida_models` 是 `server/standalone.ts` 中真实的全局 ORM 注册表变量（更名前命名，非产品名）；产品对外名称统一为 **Sowork AI 企业智能ERP系统**。详见大纲"历史内部命名说明"。

## 界面布局与操作流程

你在浏览器打开 `http://localhost:3000`，用账号 `admin` / `admin123` 登录后，进入「应用管理 → 低代码搭建」。整个流程是 4 步向导：

```
第1步  命名模块      填模块名 / 图标 / 颜色        → 决定 addons/ 目录名
第2步  加字段        可视化增删字段 + 选类型       → 决定 _fields 内容
第3步  配关系        多对一 / 一对多关联           → 写入 app.json 的 relations
第4步  生成          点「生成并部署」              → 写文件 + 注册 ORM + 建表
```

每一步的提交，都打到了后端 `standalone.ts` 的不同路由：

- 第 1、2、4 步 → `POST /lowcode/create-module`
- 运行中点「加字段」→ `POST /lowcode/add-field`
- 第 3 步关系 → `POST /apps/:app/relations`
- 最终发布 → `POST /lowcode/deploy`

生成完成后，前端「应用」列表（`GET /apps/list`）会多出一个带图标的应用，点进去就是该模块的菜单（`app.json` 里的 `menu` 数组）。

## 底层逻辑与数据模型

低代码生成物就是标准 ORM 模型，和第 05 篇讲的「模型即表」完全一致。以真实生成的 `addons/app_mru06doi/models/contactinfo.js` 为例，它的结构是这样的：

```js
// 低代码生成: contactinfo
exports.model = {
  "_name": "contactinfo",
  "_description": "联系人信息",
  "_fields": {
    "contact_name": { "type": "char", "label": "联系人姓名", "required": false },
    "customer_id":  { "type": "many2one", "label": "所属客户", "relation": "customer" },
    "mobile":       { "type": "char", "label": "手机" },
    "created_at":   { "type": "char", "label": "创建时间", "default": "CURRENT_TIMESTAMP" }
    // ... 其余字段省略
  }
};
```

注意三点：

1. 模型定义是纯 `JSON` 序列化产物，字段类型用的是 `char` / `integer` / `many2one` 等 ORM 类型名。
2. `many2one` 字段带 `relation`，指向关联表——这就是第 3 步「配关系」落地的形式。
3. 同目录还有 `manifest.json`（模块元信息）和 `app.json`（前端菜单/图标配置）。

`app.json` 里的 `menu` 数组，直接驱动前端菜单渲染：

```json
{ "label": "联系人信息", "table": "contactinfo", "icon": "📋", "desc": "管理联系人信息" }
```

也就是说：**菜单 = 一张表 + 一个标签**，前端拿到 `table` 名就去打 catch-all 路由 `/api/:table` 做增删改查。

## 源码剖析

下面贴真实片段，标注 `file:line`，建议对照阅读。

### 1. create-module：写模型文件 + 注册 ORM + 生成 manifest

`server/standalone.ts:708` 起，核心动作是把向导下发的 `models` 数组逐个写成 `.js` 文件，并立即注册：

```ts
// server/standalone.ts:737-747
const content = '// 低代码生成: ' + finalDef._name + '\n' +
  'exports.model = ' + JSON.stringify(finalDef, null, 2) + ';\n';
fs.writeFileSync(filePath, content, 'utf-8');
const reg = (global as any).__feida_models;
if (reg) {
  reg.register(modelDef);              // 注册进运行中的 ORM
  const def = reg.getDef(modelDef._name);
  if (def) { def._addon = true; def._addonName = moduleName; }
}
```

紧接着生成 `manifest.json`（`standalone.ts:750-755`），`application: true` 让它作为独立应用出现。

✅ 这里已经做了「注册即生效」，但真正的「重新扫描目录 + 清缓存」是在 `deploy` 阶段。

### 2. add-field：ALTER TABLE + 同步内存 _fields

`server/standalone.ts:760` 的 `add-field`，是「热更新」的关键。它做两件事：改数据库表结构，改内存字段定义。

```ts
// server/standalone.ts:767-775
const typeMap = { char:'TEXT', integer:'INTEGER', float:'REAL',
  boolean:'INTEGER DEFAULT 0', many2one:'TEXT DEFAULT \'\'', /* ... */ };
const sqlType = typeMap[fieldDef.type] || 'TEXT';
rawDb.exec('ALTER TABLE "' + table + '" ADD COLUMN "' + fieldName + '" ' + sqlType);
const reg = (global as any).__feida_models;
if (reg) {
  const def = reg.getDef(table);
  if (def) def._fields[fieldName] = fieldDef;   // 直接改内存，不重启
}
```

字段类型映射表 `typeMap` 决定了界面上的「类型」如何落为 SQLite 列类型。这正呼应注意事项里的 🔑 类型映射要准。

### 3. deploy：清缓存 + 重新加载

`server/standalone.ts:848` 的 `deploy` 是发布动作。它先清掉所有模型文件和 `app.json` 的 require 缓存，再调用注册表的 `loadFromModule`：

```ts
// server/standalone.ts:857-872
for (const f of fs.readdirSync(modelsDir)) {        // 1. 清 require 缓存
  if (f.endsWith('.js')) delete require.cache[require.resolve(...)];
}
delete require.cache[require.resolve(appJson)];       // 2. 清 app.json 缓存
const loaded = reg.loadFromModule(dir);              // 3. 重新加载模型
```

### 4. loadFromModule：支持三种导出写法

`model-registry.ts:50-86` 是注册表从目录加载模型的实现。它兼容 `exports.models` / `exports.model` / `export default` 三种写法：

```ts
// server/modules/orm/model-registry.ts:61-77
delete require.cache[require.resolve(fp)];   // 关键：清缓存才能加载最新文件
const mod = require(fp);
if (mod.models && Array.isArray(mod.models)) modelDefs.push(...mod.models);
else if (mod.model && mod.model._name)       modelDefs.push(mod.model);
else if (mod.default && mod.default._name)   modelDefs.push(mod.default);
for (const modelDef of modelDefs) {
  if (modelDef._name) { this.register(modelDef); /* ... */ }
}
```

## 原理剖析

把上面四段串起来，就是完整的全链路。请对照这张 ASCII 图：

```
┌──────────────────────────────────────────────────────────────┐
│  浏览器 4 步向导                                                │
│  ①命名模块 ②加字段 ③配关系 ④生成                              │
└───────────────┬──────────────────────────────────────────────┘
                │ POST /lowcode/create-module
                ▼
       写磁盘：addons/<模块>/models/<表>.js
                │  （JSON.stringify 的 exports.model）
                ▼
       写磁盘：manifest.json + app.json(menu)
                │
                ▼
       reg.register(modelDef)  ──►  ORM 内存注册表 __feida_models
                │                     （模型即表，立刻可被 /api/:table 查询）
                ▼
       POST /lowcode/deploy
                │  1. 清 require 缓存  2. 清 app.json 缓存
                │  3. loadFromModule(dir)
                ▼
       自动建表（SQLite 表随模型存在）/ ALTER TABLE 加列
                │
                ▼
       GET /apps/list 返回新应用
                │
                ▼
   前端菜单出现：点击 → /api/:table 标准 CRUD 生效
```

**为什么加字段不用重启？** 秘密在 `add-field` 的第二个动作：`def._fields[fieldName] = fieldDef`。ORM 的字段定义是存在内存对象上的普通属性。直接给这个对象塞一个新键，下一次查询/写入就认识这个新列了。数据库侧用 `ALTER TABLE ... ADD COLUMN` 同步物理表，两侧在同一请求内完成，所以业务无感知。

**为什么 deploy 还要清 require 缓存？** 因为 Node 的 `require` 会缓存已加载的模块。如果不 `delete require.cache`，第二次 `require` 同一文件拿到的还是旧定义。`loadFromModule` 在 `model-registry.ts:62` 也做了同样的事，确保「重新加载」是真的重新读盘。

## 管理者视角

站在管理者角度，低代码平台的价值不是「少写几行代码」，而是**把需求到上线的链条从「排期 → 开发 → 测试 → 发布」压缩成「业务人员自己点完即上线」**。

以「客户跟进」为例：销售总监想要一个记录每次拜访的应用，过去要提需求、等 IT 排期、开发两周。现在他自己在向导里：命名「客户关系管理」、加「联系人 / 商机 / 跟进记录」三张表、配好「跟进记录 → 客户」的多对一关系，点生成——当天就能用。

🚀 释放的 IT 人力，可以从「重复搬砖式 CRUD 开发」转向「复杂业务逻辑、报表、集成」等高价值工作。这也和第 05 篇「模型即表」形成呼应：正因为底层把「一张表」抽象成了统一模型，前端 CRUD 才能完全由数据驱动，业务人员点出来的东西和开发手写的，在运行时是同一种对象。

## 注意事项

⚠️ **低代码生成的是标准 CRUD**。向导能搞定列表、表单、基础关联；但审批流、复杂计算、外部接口对接这类逻辑，仍需开发在模型或控制器里补代码，别指望向导包办一切。

🔑 **字段类型映射要准**。`add-field` 的 `typeMap`（`standalone.ts:767`）把 `integer` 映射成 `INTEGER`、`many2one` 映射成 `TEXT`。如果你在界面选错类型，落库的列类型就错，后续统计或排序可能出问题。生成前务必核对类型。

✅ **生成即建表，无需手写迁移**。模型文件一旦写盘并注册，表结构随之确定；后续加字段走 `add-field` 自动 `ALTER TABLE`。你不需要写任何 SQL 迁移脚本。

另外两点实操提醒：删除应用走 `DELETE /apps/unlink/:moduleName`，系统禁止删除内置的 `demo_erp` 模块（`standalone.ts:1006`）；低代码模块统一带 `Category: LowCode`，在 `apps/list` 里可与手写 addon 区分。
