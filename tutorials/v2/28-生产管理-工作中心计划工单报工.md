# 28. 生产管理：工作中心 / 计划 / 工单 / 报工

本篇带你以代码视角吃透 **Sowork AI 企业智能ERP系统** v1.2.0 的生产管理模块。你将看到「计划→工单→报工」这条生产闭环在前后端上如何落地，以及 MRP 的 BOM 展开路由 `/mrp/explode` 如何把产品 BOM 炸开成物料需求。

> 环境约定：服务端口 `3000`，初始账号 `admin/admin123`，前端依赖安装用 `npm install --legacy-peer-deps`。路由遵循统一事实——**没有** `/api/erp` 这种按域前缀的入口；普通数据走 catch-all 路由 `/api/:table`，而 MRP 有独立命名路由 `/mrp/explode`。

## 学习目标

- 说出生产模块 4 个子页（工作中心、计划、工单、报工）各自对应的表与路由。
- 看懂 `计划 → 工单 → 报工` 的生产闭环，以及工单如何呼应第 24 篇 PLM 的 BOM。
- 读懂 `/mrp/explode` 如何从 `product_boms` / `product_bom_items` 展开物料需求。
- 站在管理者视角，理解产能平衡、进度追踪与人工成本核算。

## 核心概念

生产管理回答三个问题：**在哪生产、生产什么、生产得怎样**。

| 概念 | 对应子页 | 核心表 | 命名路由 |
|------|----------|--------|----------|
| 产能单元 | 工作中心 | `work_centers` | `/api/work-centers` |
| 生产安排 | 生产计划 | `production_plans` | `/api/production-plans` |
| 执行单据 | 生产工单 | `production_work_orders` | `/api/work-orders` |
| 进度与成本 | 报工管理 | `work_order_operations` | `/api/work-orders/:id/start`·`/finish` |

🔑 注意表名与路由名的差异：表用下划线（`production_work_orders`），路由用连字符（`/work-orders`）。这些生产表**没有**进入 catch-all 的 `ALLOWED` 白名单，而是由专属命名路由直接服务——这正是 Sowork AI 企业智能ERP系统的设计：生产这类强业务路由走专属 handler，而非通用 `/api/:table`。

## 界面布局与操作流程

左侧菜单在 `client/src/components/Layout.tsx` 的 `/production` 分组里挂载了 4 个子页（第 171–177 行）：

```
生产管理 (/production)
├── 工作中心   → /production/work-centers
├── 生产计划   → /production/plans
├── 生产工单   → /production/work-orders
└── 报工管理   → /production/reporting
```

入口页 `ProductionPage.tsx` 用卡片导航把四个子功能串起来。你在界面上的每一次「新增 / 审批 / 派工 / 报工」，背后都对应一条专属路由。整体操作流是一条单向闭环：

```
生产计划 (planned_qty)  ──审批──▶  生产工单 (派工/开工)  ──报工──▶  实际产量 + 工时
        │                                                            │
        └──────────── 依据 BOM 展开物料需求 (/mrp/explode) ◀─────────┘
```

你在「生产计划」里登记款号与计划数量；在「生产工单」里把计划拆成对应工序/工作中心的执行单；在「报工管理」里记录每道工序的开工、完工、良品数与工时。

## 底层逻辑与数据模型

四个核心表的关系如下：

```
work_centers (工作中心)
      ▲ work_center_id
      │
production_plans (计划) ──plan_id──▶ production_work_orders (工单)
                                         │ work_order_id
                                         ▼
                                  work_order_operations (工序/报工)
```

- `work_centers`：字段 `code, name, type(默认 sewing), status(默认 active), remark`。
- `production_plans`：字段 `name, style_id, style_code, style_name, planned_qty, start_date, end_date, status`。初始 `draft`，调 `/production-plans/:id/approve` 后变 `approved`（server `standalone.ts:8050`）。
- `production_work_orders`：字段含 `plan_id, style_id, process_id, work_center_id, scheduled_qty, actual_qty, status`。状态机：`pending → dispatched → in_progress → completed`。
- `work_order_operations`：工单派工（`/work-orders/:id/dispatch`）时自动生成，记录 `actual_qty, scrap_qty, start_time, end_time`，是报工数据的落点。

🚀 工单与 BOM 的呼应：工单本身存的是「款号 + 工序 + 工作中心」，而它需要的物料来自 PLM 模块（第 24 篇）维护的 `product_boms` / `product_bom_items`。系统通过 `/mrp/explode` 把款号对应的 BOM 炸开，得到该工单的物料需求清单。

## 源码剖析

### 1. 四个子页的专属路由注册

工作中心是典型的 CRUD 样板（`standalone.ts:7966` 起）：

```ts
router.get('/work-centers', (req, res) => {          // standalone.ts:7966
  const { type, status } = req.query;
  let centers = db.findAll('work_centers');
  if (type)   centers = centers.filter((c:any)=>c.type===type);
  if (status) centers = centers.filter((c:any)=>c.status===status);
  res.json(centers);
});
```

生产计划多了「审批」动作（`standalone.ts:8050`）：

```ts
router.post('/production-plans/:id/approve', (req, res) => {  // standalone.ts:8050
  db.update('production_plans', req.params.id, {
    status: 'approved',
    approved_at: new Date().toISOString()
  });
  res.json({ success: true });
});
```

工单派工时，会顺带生成一条工序操作记录（`standalone.ts:8113`）：

```ts
router.post('/work-orders/:id/dispatch', (req, res) => {  // standalone.ts:8113
  const { worker_id, worker_name } = req.body;
  const order = db.findById('production_work_orders', req.params.id);
  if (order) {
    db.insert('work_order_operations', {
      id: `woop_${Date.now()}`, work_order_id: req.params.id,
      seq_no: 1, operation_id: order.process_id,
      operation_name: order.process_name, status: 'pending',
      worker_id, worker_name, actual_qty: 0, scrap_qty: 0,
      created_at: new Date().toISOString()
    });
  }
  db.update('production_work_orders', req.params.id, { status: 'dispatched', dispatched_at: new Date().toISOString() });
  res.json({ success: true });
});
```

报工的两步（`start` / `finish`）落在 `standalone.ts:8150` 与 `8165`。`finish` 会把工序的 `actual_qty`、`scrap_qty`、`end_time` 写回，并累加工单的 `actual_qty`；当良品数达到 `scheduled_qty` 时工单自动置为 `completed`（`:8182`–`:8194`）。

### 2. /mrp/explode 的 BOM 展开逻辑（重点）

这是本篇核心。`/mrp/explode` 在 `standalone.ts:4028`，它**不进 catch-all**，而是在 catch-all 之前注册的命名路由（见第 4027 行注释「注册于 catch-all 之前」）：

```ts
router.get('/mrp/explode', (req, res) => {            // standalone.ts:4028
  const { productId, qty } = req.query;
  const qtyNum = Number(qty) || 1;
  const boms = db.findWhere('product_boms', { product_id: productId }) as any[];
  if (!boms.length) { res.json({ productId, qty: qtyNum, materials: [], message: '无BOM数据' }); return; }
  const result: Array<{ material: string; code: string; unitQty: number; totalQty: number; unit: string }> = [];
  for (const bom of boms) {
    const items = db.findWhere('product_bom_items', { bom_id: bom.id }) as any[];  // standalone.ts:4035
    for (const item of items) {
      result.push({ material: item.material_name, code: item.material_code,
                    unitQty: item.quantity, totalQty: item.quantity * qtyNum, unit: item.unit });
    }
  }
  res.json({ productId, qty: qtyNum, materials: result, bomCount: boms.length });
});
```

逻辑拆解（三层）：

1. 用 `product_id` 在 `product_boms` 中查出该产品的 BOM 头（可能有多个版本）。
2. 对每个 BOM 头，用 `bom_id` 在 `product_bom_items` 查出其下级物料行。
3. 把每行的 `quantity` 乘以计划数量 `qtyNum`，得到 `totalQty`（总物料需求），回写 `materials` 数组。

调用示例：`GET /mrp/explode?productId=P001&qty=100`，返回 P001 所需的全部物料及总用量。✅ 这正是工单排产时补料、请购的输入依据。

⚠️ 注意：该展开是**单层（single-level）**的——它只炸开产品直接挂在 BOM 上的物料，不会递归展开「物料本身又是半成品」的多级 BOM。需要多级时，你要在外层循环里对半成品再调一次 `/mrp/explode`。

## 原理剖析

把四块拼起来看，Sowork AI 企业智能ERP系统的生产引擎遵循「**计划驱动、工单执行、报工闭环**」：

- **计划层**只回答「做多少」。`production_plans` 不绑定具体工位，它在审批后才下达到工单。
- **工单层**回答「谁在哪做哪道工序」。一个计划可拆成多张工单，分别派往不同工作中心；派工瞬间落 `work_order_operations`，把「计划量」翻译成「可执行的工序任务」。
- **报工层**回答「做得怎么样」。开工/完工两个动作驱动状态机，并把良品数、废品数、工时记进工序行，使工单 `actual_qty` 单调累加直至达标。
- **MRP 桥接**：BOM 来自 PLM（第 24 篇），`/mrp/explode` 是连接「产品设计」与「生产补料」的枢纽，把树状 BOM 压平成物料×用量的扁平清单。

数据通道上，四个子页均走各自的专属连字符路由；而 BOM 表 `product_boms` / `product_bom_items` 则注册进了 `ALLOWED`（`:1133`、`:1136`），因此也能经 catch-all 的 `/api/:table` 直接读写——这就是「通用数据走 `/api/:table`、强业务走命名路由」的混合路由策略。

## 管理者视角

- **计划平衡产能**：制定 `production_plans` 时，要对照 `work_centers` 的可用产能（类型、状态）。把超过产能的计划硬塞进工单，只会造成在制积压。
- **工单追踪进度**：`production_work_orders.status` 从 `pending` 一路走到 `completed`，配合 `work_order_operations` 的开工/完工时间，你能实时看到每张单卡在哪道工序。
- **报工核算人工成本**：`start_time` / `end_time` 给出真实工时，`actual_qty` / `scrap_qty` 给出产出与损耗。报工数据是人工成本与效率分析的第一手来源，向下直接喂给第 29 篇财务管理的成本核算。

## 注意事项

⚠️ **工单需关联 BOM**：派发工单前，请确认对应款号在 PLM 已维护 `product_boms` / `product_bom_items`，否则 `/mrp/explode` 返回「无BOM数据」，无法算出物料需求。

🔑 **工作中心产能受限**：`work_centers` 的 `type`/`status` 决定它能否承接工单。停用的中心（`status≠active`）不应再派工，否则排产失真。

✅ **报工驱动成本核算**：只有走完 `start → finish` 的报工，工单才会累加 `actual_qty` 并可能自动 `completed`；漏报工会让产量与成本统计出现缺口。

🚀 排产口诀：先核 BOM（explode 验证物料），再核产能（工作中心），然后下计划、派工单、收报工。

## 小结与练习

本篇你掌握了：

1. 生产四子页 = 工作中心 + 计划 + 工单 + 报工，各自表与专属路由。
2. 闭环：`production_plans → production_work_orders → work_order_operations`。
3. `/mrp/explode`（`standalone.ts:4028`）通过 `product_boms`→`product_bom_items` 单层展开物料需求。
4. 路由策略：强业务走命名路由，BOM 表走 `/api/:table` catch-all。
