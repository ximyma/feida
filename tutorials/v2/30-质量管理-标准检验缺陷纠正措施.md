# 30-质量管理：标准/检验/缺陷/纠正措施

> **Sowork AI 企业智能ERP系统** v1.2.0 代码级双读者教程。运行环境：账号 `admin/admin123`，服务端口 `3000`，安装依赖 `npm install --legacy-peer-deps`。本文仅聚焦于质量管理模块，不讲其他模块。

## 学习目标

你将学会使用质量管理四大子功能，并理解它们如何串成一条"质量闭环"。

- 能区分 `质量标准 / 质量检验 / 缺陷管理 / 纠正措施` 四个子页的用途。
- 能说清一条物料从"来料检验"到"CAPA 关闭"的数据流转。
- 能在源码层面定位：检验单如何判定合格/不合格、缺陷如何关联到检验、纠正措施如何闭环。

## 核心概念

质量管理不是孤立的"打分"，而是一条可追溯的链路：

```
质量标准(判定依据) ──▶ 质量检验(按标准判合格/不合格)
        │                      │
        │                      ▼
        │                 缺陷管理(登记不合格项)
        │                      │
        ▼                      ▼
   依据标准 ──────────▶ 纠正措施(CAPA 闭环, 防再发)
```

四个概念一句话定义：

| 概念 | 数据表 | 作用 |
|------|--------|------|
| 质量标准 | `quality_standards` | 定义"什么叫合格"的判定尺度 |
| 质量检验 | `quality_inspections` + `quality_inspection_items` | 按标准逐项检，给出合格/不合格结论 |
| 缺陷管理 | `quality_defects` | 把不合格项登记成可追踪的缺陷单 |
| 纠正措施 | `quality_corrective_actions` | 针对缺陷制定 CAPA，验证后关闭 |

## 界面布局与操作流程

左侧导航 `质量管理` 下挂四个子页（`client/src/components/Layout.tsx:222-228`）：

```
质量管理
 ├─ 质量标准      /quality/standards
 ├─ 质量检验      /quality/inspections
 ├─ 缺陷管理      /quality/defects
 └─ 纠正措施      /quality/corrective-actions
```

入口页 `QualityPage.tsx` 用四张卡片展示各模块记录数（标准数、检验数、待处理缺陷数）。点击卡片进入对应子页。

四个子页的操作流：

1. **质量标准**：点"新增"填编码、名称、类型（product 等）、描述。系统写入 `status: 'active'`。
2. **质量检验**：选类型（来料/过程/成品），逐行录"标准值 / 实际值 / 单项判定"，保存后系统自动汇总结论。
3. **缺陷管理**：从某张检验单新增缺陷，填类型、级别、描述、数量、责任人。
4. **纠正措施**：选关联缺陷、措施类型、截止日、责任人，保存后状态为"待执行"。

⚠️ 注意：四个子页的"新增"弹窗都做了非空校验，缺陷和措施必须选关联对象才能提交。

## 底层逻辑与数据模型

后端在 `server/standalone.ts` 用独立路由暴露这四个功能（注意：它们**不在**通用 catch-all 路由 `ALLOWED` 白名单里，而是专用端点，统一挂在 `/api` 前缀下，见 `server/standalone.ts:11858` 的 `app.use('/api', ...)`）。

核心字段映射：

```
quality_standards      : id, code, name, type, description, items, status, created_at
quality_inspections    : id, code(QI...), type, reference_code, inspector,
                         total_items, passed_items, failed_items, result
quality_inspection_items: id, inspection_id, standard_id, item_name,
                         standard_value, actual_value, result, remark
quality_defects        : id, code(DF...), inspection_id, defect_type,
                         defect_level, description, quantity, status
quality_corrective_actions: id, defect_id, action_type, description,
                         responsible_person, due_date, status, effect_evaluation
```

🔑 三处外键是闭环的关键：
- 检验明细 `standard_id` → 引用 `quality_standards`，把"判据"绑定到具体行。
- 缺陷 `inspection_id` → 引用 `quality_inspections`，缺陷必来源某次检验。
- 措施 `defect_id` → 引用 `quality_defects`，措施必针对某条缺陷。

## 源码剖析

重点看**检验判定逻辑**——这是整个模块的核心算法。`server/standalone.ts:11615-11628`：

```ts
let total_items = items?.length || 0;
let passed_items = 0;
let failed_items = 0;
items?.forEach((item: any) => {
  if (item.result === 'passed') passed_items++;
  if (item.result === 'failed') failed_items++;
});

let result = 'pending';
if (total_items > 0) {
  if (failed_items === 0) result = 'passed';
  else if (passed_items === 0) result = 'failed';
  else result = 'partial';
}
```

判定规则：全部合格 → `passed`；全部不合格 → `failed`；有合格有不合格 → `partial`；无明细 → `pending`。前端把 `result` 渲染成"合格/不合格/部分合格/待检"（见 `InspectionsPage.tsx:22`）。

再看**纠正措施闭环**。后端 `server/standalone.ts:11723-11729`：

```ts
router.put('/quality-corrective-actions/:id', (req, res) => {
  const { status, effect_evaluation } = req.body;
  const updates: any = { status, effect_evaluation };
  if (status === 'completed') updates.completed_date = new Date().toISOString();
  db.update('quality_corrective_actions', req.params.id, updates);
  res.json({ success: true });
});
```

状态推进 `pending → in_progress → completed`，只有置为 `completed` 才写入 `completed_date` 并附带效果评估 `effect_evaluation`，这就是"验证后才能关闭"的代码保证。前端 `CorrectiveActionsPage.tsx:78-88` 的"完成"按钮即触发此 PUT。

缺陷侧的闭环导航更直观：`DefectsPage.tsx:176` 每行有"措施"按钮，跳转到 `/quality/corrective-actions?defect_id=xxx`，把缺陷 ID 带进表单，自动关联。

## 原理剖析

为什么质量要设计成"标准→检验→缺陷→措施"四段而非一张大表？本质是**关注点分离 + 可追溯**：

- 标准独立存，是因为同一标准可能被多张检验单引用，改标准不应改写历史检验。
- 检验与缺陷分离，是因为一次检验可能产生 0 到 N 条缺陷，两者是 1:N。
- 缺陷与措施分离，是因为一条缺陷可能要开多条措施（纠正+预防），且措施有独立的状态机和截止日。

数据一致性依赖外键字段，但本系统未用数据库外键约束，而是靠前端选择框 + 后端字段存储。所以"孤儿数据"（如已删检验的缺陷）在代码层不强制拦截——这是轻量实现的取舍。

🚀 联动点：`quality_inspections.type` 支持 `incoming / in_process / final`，对应来料检、过程检、成品检，可把 `reference_code` 填成采购单号或生产单号，从而与采购、生产模块在业务上打通。

## 管理者视角

作为管理者，你不必会写代码，但要盯住三条价值线：

- **质量追溯降低客诉**：每一笔客诉都能回溯到具体检验单、缺陷单、责任人。看板上的"严重缺陷数""待处理缺陷数"（`DefectsPage.tsx:84-85`）是每日必看指标。
- **CAPA 闭环防再发**：纠正措施不止"修这次"，更要"防下次"。关注 `effect_evaluation`（效果评估）是否填写，未填即关闭等于没闭环。
- **与采购/生产联动**：来料检（incoming）不达标，应触发供应商质量反馈；过程检（in_process）频发缺陷，应回流到工艺/BOM 优化。质量数据是企业持续改进的输入，不是质检员的孤岛。

## 注意事项

⚠️ **标准要先建**：检验明细里的 `standard_id` 引用标准表。虽然后端不强制校验标准存在，但先建标准再检，才能让"判据"可复用、可审计。

🔑 **缺陷要分级**：`defect_level` 分 `minor / major / critical`（轻微/重大/严重）。严重缺陷必须升级处理，别和轻微缺陷混在同一条待办里淹没。

✅ **纠正措施需验证关闭**：状态到 `completed` 才写 `completed_date`，且应附 `effect_evaluation`。只在脑子里"觉得改好了"就点完成，等于埋下再发隐患。

⚠️ 路由提醒：本模块走专用 `/api/quality-*` 端点，不走通用 `catch-all /api/:table`，直接对 `quality_*` 表发通用 CRUD 会 403。

## 小结与练习

本文讲了质量管理四子页：标准定尺度、检验按标准出结论、缺陷登记不合格、措施闭环防再发。源码关键位置：`standalone.ts:11549-11729`（四表 CRUD）、`InspectionsPage.tsx`（检验 UI）、`CorrectiveActionsPage.tsx`（措施闭环）。

练习：
1. 新建一条 `product` 类标准，再开一张 `incoming` 检验单引用它，录 3 个检验项（2 合格 1 不合格），确认结论为 `partial`。
2. 基于该检验单登记一条 `critical` 缺陷，并从缺陷页"措施"按钮开一条 `preventive` 措施。
3. 把措施推进到 `completed`，观察 `completed_date` 与 `effect_evaluation` 是否落库。
