## 核心概念

人事管理是 Sowork AI 企业智能ERP系统 中「以人为核心」的模块，它不另立一套 HR 子系统，而是复用底座的通用数据层。先记住三个关键点：

- **没有 `/api/hr` 域前缀**。所有人事数据都走 catch-all 路由 `/api/:table`，由服务端 `ALLOWED` 白名单决定哪些表可被读写（`server/standalone.ts:4070` 直接 `if (!ALLOWED.includes(table))` 返回 `{ error: 'Invalid table' }`）。
- **表即接口**。你在前端看到的一个子页面，背后就是一张（或几张）已注册的数据表；前端只管渲染，后端只管 CRUD。
- **配置即功能**。字段、子集、模板、提醒都是「数据」，不是硬编码，因此同一套代码能适配不同行业。

人事模块的 9 个入口，在 `client/src/components/Layout.tsx:56-65` 这样声明：

```
/personnel
 ├─ 人事首页      /personnel          → employees（+ contracts / employee_changes 聚合）
 ├─ 字段自定义    /personnel/field     → field_definitions
 ├─ 劳动合同      /personnel/contract  → contracts（contract_templates 已注册但前端未用）
 ├─ 员工变动      /personnel/change    → employee_changes（另误调 /api/reminder_rules）
 ├─ 提醒管理      /personnel/reminder  → reminders
 ├─ 人员子集      /personnel/subset    → employee_subsets（subset_records 已注册但前端未用）
 ├─ 打印模板      /personnel/print     → print_templates
 ├─ 考核评估      /personnel/assessment→ assessment_tools（questions/results 已注册但前端未用）
 └─ 人才报表      /personnel/talent    → talent_reports（+ performance_cycles）
```

这些表全部登记在 `server/standalone.ts` 的 `ALLOWED` 数组中：合同类在 `:1075`、考核类在 `:1094`、人才类在 `:1096`、人事变动/字段/子集在 `:1102`、打印/提醒在 `:1103`。

> ⚠️ **注意事项（模块级）**：`contract_templates`、`subset_records`、`assessment_questions`、`assessment_results`、`talent_profiles`、`reminder_logs` 这 6 张表虽然都进了 `ALLOWED` 白名单，但在 v1.2.0 的前端人事页面里**没有任何一处读写它们**。它们属于「预留但未接线」的表，二次开发时可以直接复用，但别误以为当前界面已在用。

## 人事首页（员工主档聚合）

人事首页对应 `client/src/pages/PersonnelPage/PersonnelPage.tsx`，路由 `/personnel`（`app.tsx:203`），是员工主档的总入口。

### 界面布局与操作流程

顶部 5 张统计卡：员工总数、在职人数、待入职、已离职、本月入职（`PersonnelPage.tsx:165-171`）。其下是筛选区（姓名/工号/手机号、部门、状态）与一张员工表（工号、姓名、部门、职位、职级、入职日期、手机号、状态、操作）。点姓名或「查看」弹出员工详情抽屉，内有 5 个 Tab：基本信息 / 工作信息 / 合同记录 / 薪资信息 / 异动记录；点「新增员工 / 编辑」弹出 `EmployeeForm`；点「删除」二次确认后 `DELETE /api/employees/:id`。

操作流程：进 `/personnel` → 用筛选定位员工 → 点姓名看合同/异动 → 如需建档点「新增员工」填姓名、工号、部门、职位、职级、入职日期等必填项后保存。

### 底层逻辑与数据模型

核心表 `employees`（注册于 `ALLOWED`）。首页的 `viewEmployee` 会并行拉取 `GET /api/contracts?employeeId=xxx` 与 `GET /api/employee_changes?employeeId=xxx`（`PersonnelPage.tsx:106-107`），把合同与异动明细挂到当前员工上。员工状态机为 `active / pending / inactive / terminated`（见 `statusMap` `:50-55`：`pending`=待入职、`inactive`=离职、`terminated`=已辞退）。

### 源码剖析

保存员工的请求体直接透传给 catch-all（`PersonnelPage.tsx:115-132`）：

```ts
const method = editingEmployee ? 'PUT' : 'POST';
const url = editingEmployee ? `/api/employees/${editingEmployee.id}` : '/api/employees';
fetch(url, { method, headers: {'Content-Type':'application/json'}, body: JSON.stringify(emp) });
```

`EmployeeForm` 默认 `status:'pending'`（`:511`），即新增员工默认是「待入职」，需 HR 在「工作信息」Tab 手动改为 `active` 才计入在职；否则「在职人数」统计不会包含他。

### 原理剖析

首页是「只读聚合 + 主档维护」的混合体：它既承担员工 CRUD（增删改），又通过`viewEmployee`把分散在 `contracts`、`employee_changes` 的数据按员工维度聚合展示。把高频的人事动作前置到首页，一线 HR 无需跳转多个子页即可完成日常建档。

### 管理者视角

- 员工主档是所有人事数据的锚点：`contracts` 靠 `employeeId` 关联、`employee_changes` 靠 `employeeId` 关联、考勤/薪资都从 `employees` 取部门与状态。主档错了，下游全盘错。
- 🚀 建议把「本月入职」「已离职」两张卡当成月度人力进出看板，配合「待入职」可提前安排入职手续。

> ⚠️ **注意事项**：员工详情里的「薪资信息」Tab 当前是占位（`PersonnelPage.tsx:371-378` 显示「当前薪资信息暂未开放，请联系HR部门查询」），薪资数据需到「薪酬管理」模块查看，这里点不开。

字段自定义对应 `client/src/pages/PersonnelPage/FieldConfigPage.tsx`，路由 `/personnel/field`（`app.tsx:204`），底层主表 `field_definitions`。

### 界面布局与操作流程

顶部 4 张统计卡：总字段数、可见字段、必填字段、系统字段。下方 antd 表格列：ID、字段名称、字段标识（fieldKey）、字段类型、分组、显示顺序、可见、必填、系统字段、操作（查看/编辑/删除）。「新增字段」弹窗含：字段名称、字段标识、字段类型、所属分组、显示顺序、是否可见、是否必填、选项值、默认值、校验规则。

操作流程：点「新增字段」→ 填字段名、字段标识（如 `bloodType`，新增后不可改）、类型（8 种之一）、分组（8 组之一）→ 保存即生效。系统字段（`isSystem=1`）不可删、标识不可改。

### 底层逻辑与数据模型

落库表 `field_definitions`（`ALLOWED` `:1102`）。关键字段：`name`、`fieldKey`、`type`、`groupName`、`visibility`、`required`、`isSystem`、`options`、`defaultValue`、`validation`、`displayOrder`（`FieldConfigPage.tsx:38-52`）。前端把类型抽成常量：

```ts
// FieldConfigPage.tsx:14,16-36
const TABLE = "field_definitions";
const FIELD_TYPES = [
  {value:"text",label:"文本"}, {value:"number",label:"数字"}, {value:"date",label:"日期"},
  {value:"select",label:"下拉选择"}, {value:"textarea",label:"多行文本"},
  {value:"checkbox",label:"复选框"}, {value:"radio",label:"单选框"}, {value:"file",label:"文件上传"},
];
const GROUP_OPTIONS = [ /* basic/work/contact/education/family/bank/contract/custom 共 8 组 */ ];
```

### 源码剖析

保存走 catch-all：`POST /api/field_definitions` 新建、`PUT /api/field_definitions/:id` 编辑（`FieldConfigPage.tsx:130-144`）。后端写入时，`POST /:table` 会用 `PRAGMA table_info` 校验列，自动补 `createdAt/updatedAt`（`standalone.ts:4106-4113`）：

```ts
const tableInfo = db.query(`PRAGMA table_info("${table}")`);
const cols = tableInfo.map((c:any)=>c.name);
data = Object.fromEntries(Object.entries(data).filter(([k])=>cols.includes(k))); // 只留表中真实存在的列
```

### 原理剖析

「元数据驱动」是 ERP 灵活适配行业的根基：`field_definitions` 本身就是一行数据，渲染时由 `config?.fields.filter(f=>f.visible)` 动态拼出表单。新增一个「血型」字段，无需改任何代码，员工主档与人员子集即可自动带上它。⚠️ 注意：本页只「定义」字段，真实取值仍落在 `employees` 的扩展列或各子集的 `data` JSON 里——定义与取值是两层，迁移时要一并处理。

### 管理者视角

- 🚀 不同行业要的字段不同：制造业要「工号/产线」、贸易公司要「护照号/签证」。用字段自定义即可零代码适配，避免为每个客户改表结构。
- 🔑 用 `groupName` 把字段归到「基本信息/合同信息/银行」等分组，能让员工详情页井井有条；用 `validation`（如 `email,phone,idcard`）在前端做格式校验。

## 劳动合同（签订 / 续签 / 到期 / 生日）

劳动合同对应 `client/src/pages/PersonnelPage/ContractManagementPage.tsx`，路由 `/personnel/contract`（`app.tsx:205`），底层主表 `contracts`。

### 界面布局与操作流程

页面用 5 个 Tab 组织（`:586-603`）：合同列表、合同签订、批量导入、到期提醒、生日管理。列表 Tab 含状态筛选（签订中/生效中/已到期/已终止）与「待续签合同 N 份」计数，以及黄色「⚠️ 合同到期预警」区块。签订 Tab 选员工（仅在职）、合同类型、起止日期、期限（月）、提前提醒天数后提交。

操作流程：先到「合同签订」选员工、选类型（固定期限/无固定期限/以完成一定任务）、填起止日期与提醒天数 → 提交生成一条 `contracts` 记录；列表里可对「生效中」合同做续签（以原 endDate 为新起点、signCount+1）或终止。

### 底层逻辑与数据模型

合同状态机定义在 `ContractManagementPage.tsx:16-32`：

```ts
interface Contract {
  contractType: 'fixed_term' | 'no_term' | 'task_based';
  status: 'signing' | 'active' | 'expired' | 'terminated';
  remindDays: number;
}
```

理论上的流转是 `signing → active → expired/terminated`。列表 Tab 的「合同到期预警」只统计 `status === 'active'` 且 `endDate` 在 30 天内的合同（`:196-199`）。

### 源码剖析

提交与续签都把状态写死为 `signing`：

```ts
// ContractManagementPage.tsx:112（handleSubmit）
status: 'signing',
// ContractManagementPage.tsx:155（handleRenew）
status: 'signing',
```

全文件**没有任何一处把状态置为 `active`**（只有 `handleTerminate` 置 `terminated`，`:171`）。也就是说，一份合同从签订到续签，状态永远停在 `signing`。

> ⚠️ **源码事实 / 疑似 bug（重要）**：因为状态永远到不了 `active`，「合同列表」里的「生效中」筛选、`getExpiringContracts` 的到期预警（要求 `status==='active'`）**永远不会命中**。你在前端签的每一份合同都只会显示在「签订中」，且黄色到期预警区块恒为空。要修，应在 `handleSubmit`/`handleRenew` 后（或由一个后端定时任务）把 `status` 置为 `active`，并在 `endDate` 过期后置为 `expired`。当前版本若想看到到期预警，只能手动把库里 `contracts.status` 改成 `active` 再等到期——这显然不是预期行为。

另外，`contract_templates` 虽在 `ALLOWED`（`:1075`），但本页从不读写它；「批量导入」也仅是前端占位（选文件 + 展示 CSV 格式要求，`:401-429`），没有真正解析上传文件调用接口。

### 原理剖析

合同是法律风险的闸口：状态机本应让「签订中 → 生效 → 到期/终止」自动流转，HR 据此判断「谁的合同快到期要续签」。本页把状态停在 `signing` 是状态机「只进不退」的典型实现缺陷——留痕有了，但状态推进断了。

### 管理者视角

- 🔑 **合同到期提醒**直接关系法律风险：本应靠 `active + endDate≤30天` 提前预警，让 HR 提前启动续签，杜绝「合同裸奔」。但受上述 bug 影响，当前前端的预警不工作，**不要依赖界面黄色预警**，应改用下一节「提醒管理」的 `reminders` 规则，或定期查 `contracts` 原始表。
- ✅ 生日管理 Tab 是纯前端计算（从 `employees.birthday` 取当天/当月生日，`:203-222`），可用来做员工关怀，与合同无关。

## 员工变动（入转调离记录）

员工变动对应 `client/src/pages/PersonnelPage/EmployeeChangePage.tsx`，路由 `/personnel/change`（`app.tsx:206`），底层主表 `employee_changes`。

### 界面布局与操作流程

3 个 Tab（`:534-554`）：异动申请、异动记录、异动提醒。申请 Tab 列表只显示 `pending` 的待审批单，行内可「批准/拒绝/撤销」；记录 Tab 按状态+类型筛选展示全部历史；提醒 Tab 列出「转正/退休/工龄」待提醒员工。申请弹窗选员工、异动类型（转正/调岗/借调/离职/退休/复职），调岗/借调时还可填目标部门、目标岗位、目标职级、生效日期、原因。

操作流程：新建变动单 → 选类型、填前后部门/岗位/职级/薪资 → 提交进入「待审批」（`status:'pending'`）→ HR 在列表批准或拒绝。

### 底层逻辑与数据模型

落库表 `employee_changes`（`ALLOWED` `:1102`）。变动类型映射 `changeTypeMap`（`:49-56`）：`positive` 转正、`transfer` 调岗、`secondment` 借调、`resign` 离职、`retire` 退休、`reinstate` 复职。记录带审批状态 `pending/approved/rejected`，审批后留痕 `approverName / approveTime`（`handleApprove` `:150-161`）。

### 源码剖析

提交时状态写 `pending`（`EmployeeChangePage.tsx:123`），审批接口把状态改为 `approved` 或 `rejected` 并写审批人：

```ts
// EmployeeChangePage.tsx:152-156
await fetch(`/api/employee_changes/${change.id}`, {
  method:'PUT',
  body: JSON.stringify({ status:'approved', approverName:'Admin', approveTime: new Date().toISOString() }),
});
```

注意 `approverName` 被**硬编码为 `'Admin'`**，没有取当前登录用户——多 HR 协作时无法区分是谁批的。

该页 `loadData` 还会 `fetch('/api/reminder_rules')`（`:92`），但 `reminder_rules` **不在 `ALLOWED` 白名单**（`server/standalone.ts` 里只有 `reminders` 与 `reminder_logs`）。该请求会被 catch-all 拒绝返回 `{error:'Invalid table'}`，`Array.isArray` 为 false → `setReminderRules([])`，其返回值在「异动提醒」Tab 里**根本没被使用**（提醒列表由 `getReminderList` 直接从 `employees` 计算，`:193-238`）。

> ⚠️ **源码事实 / 疑似 bug**：`EmployeeChangePage` 对 `/api/reminder_rules` 的调用是**死代码**——表未注册，结果被忽略，提醒 Tab 实际用的是前端按 `hireDate/birthday` 算出的转正/退休/工龄列表，与任何提醒规则配置无关。若你期望「异动提醒」受「提醒管理」里的规则驱动，当前版本做不到。

### 原理剖析

「变动即留痕」是合规审计的核心：无论批准拒绝都写入审批人和时间，满足劳动法对员工入转调离的举证责任。类型用枚举+中文映射表渲染，新增类型只需在 `changeTypeMap` 加一项。

### 管理者视角

- ✅ **变动记录要留痕**：员工入转调离必须进 `employee_changes` 并走审批，否则离职结算、工龄认定、三期女工保护会无据可查。
- 🔑 审批人硬编码 `Admin` 是实施隐患：上线前应改为读取当前登录用户 `localStorage`/上下文，否则审计链「谁批的」失真。

## 提醒管理（到期 / 生日 / 转正等规则）

提醒管理对应 `client/src/pages/PersonnelPage/ReminderManagementPage.tsx`，路由 `/personnel/reminder`（`app.tsx:207`），底层主表 `reminders`。

### 界面布局与操作流程

顶部 4 张统计卡：提醒总数、启用中、合同提醒、培训提醒。下方 antd 表格列：ID、提醒名称、提醒类型、所属模块、提前天数、状态、创建时间、操作（查看/编辑/启停/删除）。「新增提醒」弹窗含：提醒名称、提醒类型（8 种）、所属模块（6 个）、目标模块、提前天数、目标角色、是否启用、目标用户ID、提醒模板（支持 `${employeeName}` 等变量）。

### 底层逻辑与数据模型

关键字段：`name`、`type`、`module`、`targetModule`、`advanceDays`、`isActive`、`targetRoles`、`targetUsers`、`template`（`:36-48`）。本页 CRUD 全部走 catch-all（`POST/PUT/DELETE /api/reminders`，`:142-147`、`:113-125`）。

### 源码剖析

启停提醒只改 `isActive` 位（`:113-125`）：

```ts
await fetch(`/api/${TABLE}/${record.id}`, {
  method:'PUT', body: JSON.stringify({ isActive: record.isActive ? 0 : 1 }),
});
```

⚠️ 注意：与第 14 篇「员工变动」页误调的 `reminder_rules` 不同，本页才是对的真表 `reminders`。但 `reminder_logs`（同样在 `ALLOWED` `:1103`）在本页**没有任何写入**——也就是说「提醒是否真的触达过」缺乏日志留痕，无法事后核对「设了但没响」。

### 原理剖析

提醒是「可配置 + 可追踪」的开关：规则存在 `reminders`，触发由后台定时任务（或 AI 预警中心）扫描后推送。模板用 `${变量}` 占位，便于一处定义、批量套用到不同员工。

### 管理者视角

- 🔑 上线前务必确认提醒真的在跑：因 `reminder_logs` 无写入，无法在界面验证「已触达」。建议自建一张触达日志或在定时任务里补写 `reminder_logs`，否则「设了但不响」会无据可查。
- ✅ 把合同到期、试用期转正、证书到期三类设为启用，可覆盖大部分人事时效性风险。

## 人员子集（学历 / 经历 / 家庭 / 证书等分组信息）

人员子集对应 `client/src/pages/PersonnelPage/EmployeeSubsetPage.tsx`，路由 `/personnel/subset`（`app.tsx:208`，且包了 ErrorBoundary），底层主表 `employee_subsets`。

### 界面布局与操作流程

3 个 Tab（`:472-483`）：子集管理、修改申请、字段配置。子集管理 Tab 顶部 6 个子集类型按钮（学历信息/工作经历/家庭成员/证书资质/培训记录/奖惩记录），下方按当前类型的可见字段渲染表格，行内可删除；「+ 添加」弹窗选员工后，按该类型的字段动态出表单项。修改申请 Tab 列出 `pending` 的待审批子集记录，可「通过/拒绝」。

### 底层逻辑与数据模型

落库表 `employee_subsets`（`ALLOWED` `:1102`）。子集类型与字段是**前端硬编码**的 `subsetTypeMap`（`:32-101`），例如 `education` 含 school/degree/major/graduationYear/educationType。每条记录结构：

```ts
// EmployeeSubsetPage.tsx:12-21,144-151
interface SubsetRecord { id; employeeId; employeeName?; subsetType:string; data:Record<string,any>; status:'pending'|'approved'|'rejected'; }
// 提交时
{ employeeId, employeeName, subsetType, data: formData, status:'pending' }
```

即所有子集值都塞进一个 `data` JSON 字段，而非展开成列。⚠️ 与旧版教程描述不同：v1.2.0 前端**只写 `employee_subsets`**，`subset_records` 表虽在 `ALLOWED`（`:1102`）并有建表语句，但本页从不读写它——子集明细直接以 JSON 存在 `employee_subsets.data` 里。

### 源码剖析

提交逻辑（`:140-166`）：

```ts
await fetch('/api/employee_subsets', { method:'POST',
  body: JSON.stringify({ employeeId: formEmployeeId, employeeName, subsetType, data: formData, status:'pending' }) });
```

审批只把 `status` 改成 `approved`/`rejected`（`:178-202`）。**「字段配置」Tab 是空壳**：它的 `fieldConfigs` 是组件内 `useState`（`:347`），「保存配置」按钮**没有 `onClick`**（`:383-387`），且 `subsetTypeMap` 是写死的常量——也就是说你在这里勾选/取消字段可见性，刷新后全部还原，配置不会落库。

> ⚠️ **源码事实 / 缺陷**：「字段配置」Tab 当前是纯前端演示，改动不持久；子集字段的可见性由 `subsetTypeMap` 写死决定（`renderSubsetTab` 用 `config?.fields.filter(f=>f.visible)`，`:222`），与任何数据库配置无关。若你要让子集字段可后台配置，需要新建一张字段定义表并改写本页。

### 原理剖析

子集把「员工主档装不下、又按类别成组」的信息（学历、家庭、证书…）抽出来，用 `subsetType + data(JSON)` 的宽表模式承载。好处是一张表能装任意多类子集，坏处是 JSON 不可直接做 SQL 统计——适合「录入/展示」，不适合「按字段聚合分析」。

### 管理者视角

- 🚀 用子集固化「高潜」「核心岗」「临退休」「持证人员」等群体：例如把「证书资质」子集里持某证的人圈出来，做资质合规盘点。
- 🔑 子集记录有审批流（`pending→approved`），适合让员工自助填报、HR 审核，避免主档被随意改。

const vars = { '${employeeName}': employee.name, '${department}': employee.department, /* ... */ '${companyName}': '（内置占位公司名）', '${printDate}': new Date().toISOString().slice(0,10) };

for (const [key,value] of Object.entries(vars)) result = result.replace(new RegExp(key.replace('$','\\$'),'g'), value);

> ⚠️ **源码事实 / 疑似 bug（重要，且触及红线说明）**：源码在 `PrintTemplatePage.tsx:152` 与 `:181` 两处，把公司名**硬编码为一个内置占位字符串**（并非从系统「公司信息」配置读取）。这意味着打印出来的工卡/证明会显示该占位名，而非贵司真实名称；且该占位串与产品名「Sowork AI 企业智能ERP系统」无关，属历史遗留样例值。正确做法应改为从系统配置读取 `companyName`，否则每家客户部署都要改源码。🚀 二次开发切入点：把 `vars['${companyName}']` 改成读取 `/api/system-config` 之类接口即可全局部署生效。
>
> 另：「导出PDF」按钮是空壳（`PrintTemplatePage.tsx:647-649` 仅 `message.info('导出为PDF功能开发中')`），当前只能走浏览器 `window.print()` 另存为 PDF。

### 管理者视角

- 🚀 批量工卡：选 N 名员工 → 预览 → 一次打印，是入职批量发卡的利器。
- 🔑 把常用模板（在职证明、劳动合同）设为默认，HR 日常出件不用每次重选。

## 与其他模块的衔接

人事不是孤岛，它向上游与下游都连着其它模块：

- **组织架构成员主档（13 篇）**：`employees` 的 `department/position/rank` 来自组织机构模块；人事建员工时填的部门/职位要与组织树一致，否则考勤、薪资、子集都会按错维度聚合。
- **考勤模块（15 篇）**：考勤一切计算靠 `employeeId` 关联 `employees`；员工状态 `active/inactive` 决定能否打卡、是否计入出勤。
- **薪酬模块（16 篇）**：薪资以 `employees` 为锚，合同里的薪资字段、变动里的 `toSalary` 是调薪依据（注意本页变动记录的 `fromSalary/toSalary` 仅留痕，不直接改薪资表）。
- **绩效 / 招聘模块**：`performance_cycles` 被人才报表复用；`assessment_tools` 本应被绩效考核引用；招聘的 `candidates/talent_pools` 与人才库呼应（同列 `ALLOWED` `:1077`）。
- **系统/合规模块**：`audit_logs`、`login_logs` 记录操作与登录，配合 `employee_changes`、`contracts` 构成「谁、何时、改了什么人事数据」的完整可追溯链。

🔑 记住：人事模块对外只通过 `employees / contracts / employee_changes / employee_subsets / reminders / print_templates / assessment_tools / talent_reports` 这几张表「输出结论」，其余模块来读即可，耦合被牢牢控制在表边界内。

## 管理者视角（总览）

从管理角度看，人事模块解决三件事：

- **合规用人降低风险**：合同状态机 + 到期提醒规避「合同裸奔」；员工变动全留痕满足劳动法举证。⚠️ 但如前所述，当前前端合同状态永远停在 `signing`、且 `reminder_logs` 无写入，这两处会让「自动预警」失真——上线前必须修或改用定期查表。
- **人才盘点支撑决策**：人员子集把高潜/核心/临退休固化成可圈选群体；人才报表九宫格 + 继任计划让梯队建设可视。
- **灵活适配不同行业**：字段自定义 + 打印模板，让「加字段、出工卡、开证明」零代码完成。

⚠️ 给实施顾问的全局提醒：antd 风格的人事子页（字段、提醒、打印、人才报表）在读取列表时取 `result.data`，而 catch-all 的 `GET /:table` 直接 `res.json(data)` 返回**裸数组**（`standalone.ts:4087`）。若后端未对列表响应做统一的 `{data}` 包装，这些页面 `result.data` 会是 `undefined`、列表渲染为空——`PrintTemplatePage` 的员工加载已兼容两种格式（`Array.isArray(result)?result:(result.data||[])`，`:123`），其余四个 antd 页面未兼容。排障时先开 Network 看 `/api/xxx` 返回的是数组还是 `{data:[]}`。

## 注意事项（汇总）

- ⚠️ **合同状态机断链**：`ContractManagementPage` 签订/续签都写 `status:'signing'`，无代码置 `active`，导致「生效中」筛选与到期预警永不命中（详见「劳动合同 源码剖析」）。需后端/前端补状态推进。
- ⚠️ **到期提醒查错表**：`/ai/alert-records`（`standalone.ts:3312-3313`）扫的是 `emp_contracts` 表，而前端写的是 `contracts` 表；`emp_contracts` 在当前建表脚本里并不存在（只有 `contracts` 有 `CREATE TABLE`，见 `standalone.js:24860`）。即 AI 预警中心的合同到期扫描与界面用的是两张不同的、且其中一张可能为空/不存在的表——两套实现彼此脱节。
- ⚠️ **`reminder_rules` 调用落空**：`EmployeeChangePage` 调 `/api/reminder_rules`，该表不在 `ALLOWED`，且结果未被使用，属死代码（详见「员工变动 源码剖析」）。
- ⚠️ **预留表未接线**：`contract_templates`、`subset_records`、`assessment_questions`、`assessment_results`、`talent_profiles`、`reminder_logs` 已注册但前端未读写，别误以为界面在用。
- ✅ **字段存 JSON/扩展列**：`field_definitions` 是元数据，真实值落在员工主档扩展列或 `employee_subsets.data` JSON 里，迁移要一并处理。
- ⚠️ **变动审批人硬编码 `Admin`**：`EmployeeChangePage` 审批写死 `approverName:'Admin'`，应改读当前登录用户。
- ✅ **子集审批流可用**：子集记录 `pending→approved` 留痕，适合员工自助填报 + HR 审核。
- ⚠️ **打印模板公司名硬编码**：`PrintTemplatePage` 把公司名写死为内置占位串（非系统配置），工卡/证明会显示该占位名；应改为读系统配置。
- ⚠️ **antd 页面 `result.data` 取值风险**：见上方「管理者视角（总览）」说明。

5. **提醒管理**：进 `/personnel/reminder`，新增一条「合同到期」提醒（提前 30 天、目标角色 HR、启用），写 `reminders`。
6. **人员子集**：进 `/personnel/subset`，选「学历信息」给一名员工添加记录（状态 `pending`），到「修改申请」通过；再打开「字段配置」勾选字段后刷新，验证配置不持久（对应缺陷）。
7. **打印模板**：进 `/personnel/print`，新建一份「在职证明」模板（变量 `${employeeName}/${department}`），用「员工工卡打印」预览一名员工。
8. **考核评估**：进 `/personnel/assessment`，新建一个「技能测评」工具（题目数 20、时长 30、启用），验证只落 `assessment_tools`。
9. **人才报表**：进 `/personnel/talent`，选一个考核周期，新建报告并粘贴九宫格 JSON `{"高潜高绩效":["张三"]}`，看「完整度」进度条变化。
