# 43. 综合事务：通知公告 / 文档 / 问卷 + 办公

本篇带你深入 **Sowork AI 企业智能ERP系统**（v1.2.0）的「综合事务」模块。你会登录 `admin/admin123`，端口 `3000`，后端以 `npm install --legacy-peer-deps` 装好依赖后启动。综合事务是企业内部协同的总入口：通知公告、文档管理、问卷调查，以及会议室预订、办公用品领用等办公事务，全部在这里落地。

## 学习目标

学完你能做到：

- 发布一条带「已读回执」的通知公告，并查看全员阅读率。
- 在文件夹中上传文档，按用户/角色/部门配置访问权限。
- 创建问卷或投票，收集回复并查看柱状图、饼图统计。
- 理解 `announcements` / `documents` / `surveys` 等表与通用 CRUD 的映射关系。
- 从管理者视角评估这套机制对信息触达与提效的价值。

## 核心概念

综合事务在代码里叫「办公」模块，前端路由定义在 `client/src/components/Layout.tsx:116-120`：

```text
/office
 ├── /office/announcement   通知公告   (AnnouncementPage)
 ├── /office/document       文档管理   (DocumentPage)
 └── /office/survey         问卷调查   (SurveyPage)
```

三类业务各有专属页面，但它们的数据全部跑在统一的「表驱动」引擎上。后端把表名注册进 `ALLOWED` 白名单（`server/standalone.ts:1091-1092` 综合事务模块、`1105` 办公管理模块）。多数请求最终落到 catch-all 路由 `/api/:table`（`server/standalone.ts:4065` 起）。

| 业务 | 主表 | 配套表 |
|------|------|--------|
| 通知公告 | `announcements` | `announcement_reads` |
| 文档管理 | `documents` | `document_folders`、`document_permissions`、`file_storage` |
| 问卷调查 | `surveys` | `survey_questions`、`survey_options`、`survey_responses` |
| 会议室/用品 | `meeting_rooms`、`meetings`、`office_supplies`、`supply_requests` | — |

🔑 记住一个规律：凡在白名单里的表，都能用 `GET /api/:table`、`POST /api/:table`、`PUT /api/:table/:id`、`DELETE /api/:table/:id` 直接增删改查。

## 界面布局与操作流程

**通知公告（AnnouncementPage）**

左侧是统计卡片（总数/已发布/草稿/总阅读），右侧是表格。`新增公告` 弹窗里填标题、类型（通知/新闻/人事任免/制度/活动/紧急）、优先级、正文，可勾选「置顶」，状态默认 `draft`。保存后点「发布」即置为 `published`。

点「查看」打开抽屉：正文下方有阅读进度条（已读人数 / 全员人数），并列出已读员工。你一打开查看，前端就自动调 `POST /api/announcements/:id/read` 标记自己已读。

**文档管理（DocumentPage）**

页面分左右：左边是文件夹树（可新建/改名/删除），右边是文档列表。点「上传文件」弹窗，支持拖拽批量上传，并能选目标文件夹与访问权限。列表里每行可预览、下载、配权限、删除。

**问卷调查（SurveyPage）**

顶部统计卡（总数/进行中/总回复/已结束）。`新建问卷/投票` 选类型 `survey`（问卷）或 `vote`（投票评选），可设匿名、截止日期。建好后点「管理题目」加题目（单选/多选/填空），投票型再加候选项。「立即投票」让员工填写，「查看统计」出图表。

**会议室与办公用品**

`meeting_rooms`、`meetings`、`office_supplies`、`supply_requests` 属于办公管理模块（`server/standalone.ts:1105`），同样走通用 `/api/:table` 接口。你可以用统一的表视图进行会议室建档、会议预订、用品库存与领用申请的管理。

## 底层逻辑与数据模型

公告模型字段（`server/standalone.ts:189` 附近可对照 `surveys` 写法）：

```text
announcements: id, title, content, category, priority,
               status(draft/published/archived), isPinned,
               readCount, authorName, publishAt, createdAt
announcement_reads: id, announcementId, userId, employeeId,
                    employeeName, readAt
```

已读回执的核心：每个员工查看公告时写一条 `announcement_reads` 记录，`announcements.readCount` 自增。阅读率 = `readCount / 全员人数`。

文档模型：

```text
documents: id, name, folderId, type, size, mimeType,
           url, accessLevel(all/custom/...), uploaderId, downloads
document_permissions: id, documentId, targetType(user/role/department/all),
                      targetId, permission(read), grantedBy
file_storage: id, documentId, originalName, filePath, size ...
```

问卷模型（`surveys` 定义在 `server/standalone.ts:189-194`）：

```text
surveys: id, title, type(survey/vote), status(draft/active/closed),
         isAnonymous, deadline, responseCount
survey_options: id, surveyId, optionText, optionOrder
survey_responses: id, surveyId, userId, answers(JSON), employeeName
```

## 源码剖析

**公告已读回执逻辑**（`server/standalone.ts:2184-2196`）：

```ts
router.post('/announcements/:id/read', (req, res) => {
  const { employeeId, employeeName } = req.body;
  const existing = (db as any).db.prepare(
    'SELECT * FROM announcement_reads WHERE announcementId = ? AND employeeId = ?'
  ).get(id, employeeId);
  if (!existing) {
    db.prepare('INSERT INTO announcement_reads ...').run(...);
    db.prepare('UPDATE announcements SET readCount = readCount + 1 WHERE id = ?').run(id);
  }
  res.json({ success: true });
});
```

🚀 注意 `if (!existing)`：同一人对同一公告只计一次已读，重复查看不会刷高阅读量。前端 `AnnouncementPage.tsx:109-120` 在打开抽屉时即调用此接口并刷新列表。

**问卷统计**（`server/standalone.ts:2199-2208`）：

```ts
router.get('/surveys/:id/stats', (req, res) => {
  const survey = db.prepare('SELECT * FROM surveys WHERE id = ?').get(id);
  const options = db.prepare('SELECT * FROM survey_options WHERE surveyId = ? ORDER BY optionOrder').all(id);
  const responses = db.prepare('SELECT * FROM survey_responses WHERE surveyId = ?').all(id);
  res.json({ survey, options, responseCount: responses.length, responses });
});
```

前端 `SurveyPage.tsx:289-360` 拿到 `responses` 后，遍历每个选项，用 `JSON.parse(r.answers).flat().includes(optionText)` 统计票数，再用 ECharts 画柱状图与饼图。投票提交走 `POST /surveys/:id/vote`（`server/standalone.ts:2211-2227`），已投过的用户会改 `answers` 而非新增行。

**文档上传**（`server/standalone.ts:2353-2398`）用 `multer` 接收 `multipart/form-data`，先写 `documents`，再写 `file_storage` 保存真实路径，返回 `/api/documents/download/:id` 作为访问 URL。权限写入在 `/documents/:id/permissions`（`server/standalone.ts:2660-2676`），先 `DELETE` 旧权限再批量 `INSERT`，支持 user/role/department/all 四种 `targetType`。

## 原理剖析

综合事务的「万能」来自两件事：

1. **白名单 + catch-all 路由**。所有业务表注册进 `ALLOWED`（`server/standalone.ts:1091-1105`），`/api/:table` 路由（`4065-4115`）用 `db.findAll / findWhere / insert / update / delete` 通用方法完成 CRUD，并自动补全 `createdAt/updatedAt`。这让会议室、用品这些没有独立页面的业务，也能即建即用。

2. **专用路由优先于 catch-all**。公告的已读、问卷的投票与统计、文档的上传与权限，都是「先注册专用路由、最后才落到 `/:table`」的典型模式（专用路由写在 `2176-2709`，catch-all 在 `4065` 之后）。这样既享受通用 CRUD，又能为关键动作写定制逻辑。

✅ 这种分层让新业务只需建表 + 进白名单，几乎零后端代码即可上线。

## 管理者视角

- **信息触达全员**：公告 + 已读回执，让你一眼看出哪些人还没看重要制度，阅读率低于 50% 时页面还会提示「建议提醒未读员工」。
- **文档集中管理降丢失**：文件夹 + 权限 + `file_storage` 物理隔离，告别散落微信/邮件的版本混乱。
- **问卷收集反馈**：投票评选、满意度调研一键发起，实时柱状/饼图让结果立等可取。
- **会议室/用品提效**：预订与领用走同一套表引擎，数据可追溯、可统计，减少线下扯皮。

## 注意事项

⚠️ **文档权限要配**：上传时默认不一定是「全员可见」，敏感文件（合同、薪资）务必在「权限」里指定用户/角色/部门，否则可能越权可见。

🔑 **问卷匿名性**：`isAnonymous=1` 时统计页把回复者显示为「匿名用户」；若需实名追溯，请保持匿名关闭。投票型务必确认匿名设置符合公司合规要求。

✅ **公告可已读回执**：发布重要通知后，务必通过「查看 → 阅读情况」跟进未读名单，必要时结合企业微信/邮件二次提醒。

⚠️ **会议室/用品走通用接口**：它们没有独立前端页面，需通过表视图或后续扩展接入；直接操作 `meetings`/`supply_requests` 时注意字段与状态约定，避免脏数据。
