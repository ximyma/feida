## 核心概念

商城前台是一套独立的「站外电商门户」，顾客无需登录 ERP 后台即可浏览和下单。它和 ERP 内部模块（销售、库存、售后）共享同一套后端 API，但页面栈完全独立。

| 概念 | 说明 | 对应代码 |
| --- | --- | --- |
| 前台路由 | 12 个 `/shop` 开头的页面，裸奔在 `AuthGuard` 之外 | `client/src/app.tsx:183-196` |
| 购物车 | 以 `localStorage` 的 `shop_cart` 为唯一真相，未登录也能用 | `CartPage.tsx:27` |
| 全局态 | `ShopContext` 持有用户、购物车角标、收藏、分类品牌 | `ShopContext.tsx:12-25` |
| 订单 | 下单即写 `shop-orders`，支付回写 `pay_status` | `CheckoutPage.tsx:172` |
| 用户中心 | 登录态聚合订单/余额/收藏/分销/营销 | `ShopUserCenter.tsx:33` |

🔑 关键一句话：**前台不强制登录，但「下单/支付/查订单」会把顾客当作 `user_id: 'guest'` 落到订单表，登录后才与正式会员打通。**

## 界面布局与操作流程

买家视角的完整链路如下（ASCII 树）：

```
商城首页 /shop
 ├─ 搜索 /shop/search?keyword=...
 ├─ 分类 /shop/category/:id
 ├─ 品牌 /shop/brand/:id
 └─ 商品详情 /shop/goods/:id
      ├─ 加购 → 购物车 /shop/cart (localStorage)
      └─ 立即购买 → 结算 /shop/checkout
                       └─ 提交 → 订单表 → 支付 /shop/pay/:id
                                            └─ 支付成功 → 订单列表 /shop/orders
                                                                 └─ 用户中心 /shop/user
```

流程三步走：

1. **逛**：首页 `ShopHomePageV2` 拉 Banner、热销、新品、推荐、DIY 装修区块；分类/品牌/搜索都复用同一页（见 `app.tsx:187-189` 指向同一个组件）。
2. **买**：详情页选规格 SKU、定数量，点「加入购物车」或「立即购买」。
3. **结**：购物车 → 结算（选地址、看运费）→ 支付 → 用户中心查看订单与售后。

各页面入口在顶部导航用 `<Link to="/shop/...">` 串联，无需任何权限判断——这正是 `AuthGuard` 被刻意排除在 `/shop/*` 之外的原因（`app.tsx:183` 注释「商城前台（无需登录）增强版」）。

## 底层逻辑与数据模型

前台不是孤立的，它和后端商城表直接对话。核心数据流：

```
页面组件 ──fetch──> /api/shop-goods ──> 商品表(含 SKU/库存/促销)
            ──fetch──> /api/shop-orders ──> 订单表 + 订单项
            ──fetch──> /api/shop-addresses ──> 收货地址表
            ──fetch──> /api/shop-balance ──> 会员余额/积分
```

购物车模型最简单：`localStorage['shop_cart']` 是一个 `{ [goodsId]: quantity }` 的字典。加购时直接 `cart[key] = (cart[key]||0) + quantity`（`ProductDetailPageV2.tsx:152-155`）。结算页再拿这些 id 批量 `fetch('/api/shop-goods/:id')` 补全价格、库存，得到真正的下单明细。

⚠️ 注意：购物车是**前端本地态**，只有进入结算页那一刻才向后端核对库存与现价。这带来一个经典问题——库存实时性（见注意事项）。

订单数据结构（节选自 `ShopOrdersPage.tsx:6-25`）：

| 字段 | 含义 |
| --- | --- |
| `order_no` | 订单号（业务主键展示用） |
| `pay_amount` / `total_amount` | 实付 / 商品总金额 |
| `pay_status` | `unpaid` → `paid` 由支付回写 |
| `order_status` | `pending/paid/shipped/completed/cancelled` |
| `items[]` | 订单项：商品名、数量、单价、图片 |

## 源码剖析

### 1. 前台路由为何不受保护

`app.tsx:174-196` 的一段值得逐行看。`AuthGuard` 包裹的是 `<Layout />` 后台（`app.tsx:198`），而 `/shop/*` 全部写在它**之前**，直接挂在 `<Routes>` 上。所以游客访问 `/shop` 不会被重定向到 `/login`。

```tsx
// client/src/app.tsx:183-196
{/* 商城前台（无需登录）增强版 */}
<Route path="/shop" element={<ShopHomePageV2 />} />
<Route path="/shop/goods/:id" element={<ProductDetailPageV2 />} />
<Route path="/shop/cart" element={<CartPage />} />
<Route path="/shop/checkout" element={<CheckoutPage />} />
<Route path="/shop/pay/:id" element={<PayPage />} />
<Route path="/shop/user" element={<ShopUserCenter />} />
<Route path="/shop/orders" element={<ShopOrdersPage />} />
// ...其余 /shop/favorites、/shop/order/:id 等同理
```

🚀 实践提示：这条设计让你能直接把 `/shop` 当成对外门店链接发给客户，而 ERP 后台仍被 `AuthGuard` 牢牢守住。

### 2. 购物车状态：localStorage 作主，Context 作角标

`CartPage.tsx:26-47` 是购物车加载的核心。它读本地 `shop_cart`，再用 `Object.keys` 取出商品 id，并发 `fetch('/api/shop-goods/:id')` 拿最新商品信息，最后 `map` 成带数量的 `CartItem`。

```tsx
// client/src/pages/ShopHomePage/CartPage.tsx:26-41
const loadCart = async () => {
  const savedCart = localStorage.getItem('shop_cart');
  if (savedCart) {
    const cart = JSON.parse(savedCart);
    const productIds = Object.keys(cart);
    if (productIds.length > 0) {
      const products = await Promise.all(
        productIds.map(id => fetch(`/api/shop-goods/${id}`).then(r => r.json()))
      );
      const items = products.map((p) => ({
        ...p, quantity: cart[p.id] || 1
      })).filter(item => item.id);
      setCartItems(items);
    }
  }
};
```

而购物车角标（导航上的小红点数字）由 `ShopContext.tsx` 统一维护：`cartCount` 在初始化时就把 `shop_cart` 各值求和（`ShopContext.tsx:34-41`），任何页面加购后通过 `window.dispatchEvent(new Event('cartUpdated'))` 广播，角标组件监听刷新。这是「本地态 + 全局广播」的轻量状态管理范式，避免了引入 Redux。

### 3. 下单调用：一条 POST 写订单

结算页 `CheckoutPage.tsx:158-231` 的 `handleSubmit` 是整条链路最关键的写操作。它把收货地址拼成字符串、遍历购物车明细，一次性 `POST /api/shop-orders`。

```tsx
// client/src/pages/ShopHomePage/CheckoutPage.tsx:172-191
const res = await fetch('/api/shop-orders', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    user_id: 'guest',
    user_name: selectedAddress.contact_name,
    user_phone: selectedAddress.contact_phone,
    shipping_address: addressStr,
    remark,
    payment_method: paymentMethod,
    items: cartItems.map(item => ({
      goods_id: item.id,
      goods_name: item.name,
      quantity: item.quantity,
      price: item.price,
      image_url: item.main_image
    }))
  })
});
```

下单成功后：普通购物车流程会 `localStorage.removeItem('shop_cart')` 清空本地车并广播 `cartUpdated`（`:200-201`）；若选「在线支付」则弹模拟支付框，调 `POST /api/shop-orders/:id/pay` 标记已付（`:210-217`）。

### 4. 支付回写：PUT 改状态

`PayPage.tsx:56-79` 的 `handlePay` 在模拟支付完成后，用 `PUT /api/shop-orders/:id` 把 `pay_status` 和 `order_status` 都改成 `paid`，并写入支付方式与支付时间。注意它先 `setTimeout` 模拟 1.5 秒支付网关延迟，再发请求——真实环境这里应替换为支付宝/微信的异步通知（回调验签，见注意事项）。

### 5. 用户中心：聚合视图

`ShopUserCenter.tsx:58-64` 在挂载时并发加载地址、订单、余额、营销、分销五类数据。其中 `loadBalance`（`:104-111`）调 `/api/shop-balance?user_id=` 拉余额与流水，与第 38 篇的「余额积分」模块同源。订单网格直接复用 `/api/shop-orders`，与第 36 篇后台「销售订单」、第 37 篇「售后退款」共享一张订单表——这就是前台与后台的数据一致性根基。

## 原理剖析

为什么购物车放 `localStorage` 而不是后端？因为**前台允许未登录下单**。如果购物车存在后端，必须先有用户会话。用本地字典，顾客从加购到结算全程零门槛，仅在最后一刻（结算页）才真正落库为 `guest` 订单。代价是库存只在结算时校验一次——这也是电商经典的「下单锁库存」问题在这一简化实现里的体现。

支付为何拆成「下单写订单」+「支付回写状态」两步？这是支付领域标准模式：**订单创建与支付成功是两个事件**。订单先以 `unpaid` 存在，支付网关（真实环境是异步回调）再把它翻成 `paid`。这样即使支付中途掉线，订单不会丢，顾客回到 `/shop/orders` 还能继续付。

用户中心的「聚合」本质是**读聚合**：它不新建数据，只是把分散在订单、余额、收藏、分销等多个 API 的结果拼到一个 Tab 视图里。这种设计让顾客在一个页面看到全局，而不必理解 ERP 后台的模块划分。

## 管理者视角

作为管理员，你要从「转化」和「复购」两个维度看待前台：

- **前台体验决定转化**：首页加载慢、分类不清晰、结算步骤繁琐，都会直接丢掉订单。注意 `ShopHomePageV2` 首页一次性并发 5+ 个 `/api/shop-goods` 请求（热销/新品/推荐/装修区块），商品量大时要在后端加缓存。
- **用户中心提升复购**：收藏、余额、积分、分销团队都在这里触达顾客。第 38 篇的余额积分玩法，只有顾客能进 `/shop/user` 才真正产生粘性。
- **前后台数据一致是底线**：顾客在前台看到的订单状态，必须和第 36/37 篇后台「订单/售后」完全一致。任何后台改单、发货、退款操作，都要通过同一张 `shop-orders` 表回流到前台。建议定期核对前台订单数与后台销售订单数。

## 注意事项

⚠️ **库存实时性**：购物车用本地态，结算时才核对库存（`CartPage` 没有实时锁库存）。若你部署了 CDN/页面缓存，顾客可能看到已售罄商品。生产环境应在 `addToCart` 和 `handleSubmit` 处加后端库存校验与乐观锁。

🔑 **支付回调必须验签**：本文 `PayPage` 是模拟支付（点击即成功）。真实接入支付宝/微信时，`/api/shop-orders/:id/pay` 这类「标记已付」的接口绝不能由前端直接信任——必须依赖支付网关的**异步回调 + 签名校验**来翻状态，否则任何人都能伪造支付成功。

✅ **用户中心聚合订单与售后**：`ShopOrdersPage` 已内置退款/退货弹窗（`:50-58` 的 `refundModalOpen`、`returnShipOpen`），它调用售后 API，与第 37 篇后台售后流程呼应。确保前台退款入口与后台审核状态同步，避免顾客提交退款后前台仍显示「已完成」。

🚀 补充：分类/品牌/搜索三个路由（`app.tsx:187-189`）都指向 `ShopHomePageV2`，靠 `useSearchParams` 区分（`ShopHomePageV2.tsx:44-48`）。调试时直接改 URL 的 `?category=` `?brand=` `?keyword=` 即可，不必逐个页面点。
