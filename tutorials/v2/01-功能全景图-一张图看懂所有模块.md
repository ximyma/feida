# 01 · 功能全景图：一张图看懂所有模块

本文是「Sowork AI 企业智能ERP系统」（下文简称 Sowork AI）技术教程系列的第 1 篇。它写给两类人：想搞清楚「这系统能管我哪些事」的**企业管理者**，以及想读懂「菜单与权限怎么落地」的**技术开发**。读完后，你能对着一张能力全景图，说出每一个模块解决了什么业务痛点。

> 环境基线：版本 **v1.2.0**；默认账号 `admin/admin123`、`hr_admin/hr123456`；服务端口 **3000**（可用 `SERVER_PORT` 覆盖）；依赖安装一律 `npm install --legacy-peer-deps`，**严禁**裸 `npm install`。技术栈：前端 React 19 + React Router 7 + Ant Design 5，后端 Node 22 + Express 4 + better-sqlite3/PostgreSQL，自带一套 Odoo 风格 ORM。

## 学习目标

- 你能在 5 分钟内，对照全景图说清 Sowork AI 覆盖哪些业务域。
- 你理解左侧导航的「模块 → 子页」是怎么从代码里长出来的。
- 管理者能据此判断：哪些模块直接对应自家痛点，上线后能省掉哪些手工活。
- 开发者能定位菜单与角色权限的真实代码位置，不再凭猜测改需求。

## 核心概念

Sowork AI 不是「一个 HR 系统」或「一个商城」，而是把企业里 7 大业务域塞进同一个登录态、同一套权限、同一套数据底座。

| 概念 | 一句话解释 | 对应代码 |
| --- | --- | --- |
| 模块（Module） | 左侧导航一级菜单，如「人事管理」 | `subMenuItems` 的 key（`Layout.tsx:55`） |
| 子页（Subpage） | 模块展开后的二级菜单，如「工资表」 | `subMenuItems[key]` 数组项（`Layout.tsx:55-256`） |
| 权限点（Perm） | 控制某模块是否对某人可见 | `MODULE_PERM`（`Layout.tsx:323-349`） |
| 角色（Role） | 一组权限点的集合 | `USER_ROLE_PERMS`（`Layout.tsx:351-358`） |

🔑 关键认知：模块清单是**白名单式硬编码**在 `Layout.tsx` 里的，不是数据库动态拉取。改菜单 = 改源码，不是改配置。

## 界面布局与操作流程

登录后，左侧是导航栏，右侧是工作区。顶部一级菜单（仪表盘、组织、产品档案等）平铺，下方是分组模块，点击展开看到子页。

```
┌──────────────┬───────────────────────────────────────┐
│  Sowork AI   │   首页仪表盘 / 组织管理 / 产品档案 ...    │
│  ─────────── │   ───────────────────────────────────  │
│ ▸ 人事管理    │                                        │
│ ▸ 考勤管理    │         （点开模块 → 点开子页 → 干活）   │
│ ▸ 薪酬管理    │                                        │
│   ...        │                                        │
└──────────────┴───────────────────────────────────────┘
   左侧导航                  右侧工作区
```

常规操作流：登录 → 在左侧找到业务域 → 展开模块 → 进入子页 → 列表/表单 CRUD。所有业务数据走后端通用接口，下面「原理剖析」会讲清。

## 底层逻辑与数据模型

Sowork AI 的全部业务表，都由自研 Odoo 风格 ORM 管理。前端不写死每张表的字段，而是按 `table` 名向通用接口拿元数据、自动渲染列表与表单。

模块与表的对应关系举例：

| 模块 | 典型表（table） | 说明 |
| --- | --- | --- |
| 人事管理 | `employee`、`contract` | 员工主档与合同 |
| 薪酬管理 | `salary_table`、`salary_formula` | 工资表与公式 |
| 仓储管理 | `warehouse`、`stock_in`、`stock_out` | 仓、入、出 |
| 销售管理 | `customer`、`sales_order` | 客户与订单 |

⚠️ 注意路由事实（开发必读）：**后端没有 `/api/hr`、`/api/erp` 这类按域前缀的路由组！** 业务数据统一走通用 CRUD catch-all：

```
GET    /api/:table      # 列表/查询
POST   /api/:table      # 新增
PUT    /api/:table/:id  # 修改
DELETE /api/:table/:id  # 删除
```

只有少量命名路由组：`attendance/salary/schedule/leave/training` 归属 HR 域，`ai/lowcode/rbac/i18n/mrp/odoo` 归属平台域。其余业务模块都是这张「通用表接口 + 白名单 `ALLOWED`」在支撑。

## 源码剖析

菜单数据集中定义在 `client/src/components/Layout.tsx`：

- 顶部平铺菜单 `menuItems`：`Layout.tsx:46-53`（首页仪表盘、组织管理、产品档案、数据统计、低代码平台、应用管理）。
- 分组模块与子页 `subMenuItems`：`Layout.tsx:55-256`，是一个 `Record<路径, 子页数组>`。例如 `/personnel` 下有 9 个子页（`Layout.tsx:56-66`），`/plm` 下有 10 个（`Layout.tsx:133-144`）。
- 模块中文名映射 `newModuleLabels`：`Layout.tsx:284-308`，如 `/salary → 薪酬管理`。
- 模块图标 `newModuleIcons`：`Layout.tsx:258-282`。

**菜单如何按角色过滤（核心机制）：**

1. `MODULE_PERM`（`Layout.tsx:323-349`）把每个模块路径映射到一个权限点，如 `/salary → 'p_salary'`。
2. `USER_ROLE_PERMS`（`Layout.tsx:351-358`）预置 6 种角色的权限点集合，例如 `hr_admin` 只有 HR 相关权限，`employee` 仅 `p_personnel/p_attendance/p_approval/p_apps`。
3. `userPerms`（`Layout.tsx:360-391`）从 `sessionStorage.__current_user` 解析当前用户的 `roleIds`，聚合成权限点数组。
4. `hasModule(path)`（`Layout.tsx:393-397`）判断：有 perm 则看用户是否持有，无 perm 则默认可见。
5. 渲染侧（`Layout.tsx:556-577`）对每个模块写 `{hasModule('/xxx') && renderNewModuleNav('/xxx')}`，无权限直接不渲染；`/system` 仅 `isAdmin` 可见（`Layout.tsx:577`）。
6. 单模块的展开/子页由 `renderNewModuleNav`（`Layout.tsx:471-516`）负责，遍历 `subMenuItems[path]` 输出 `NavLink`。

✅ 一句话：菜单是「权限点白名单 ∩ 当前用户权限」的结果，代码全在 `Layout.tsx`，不依赖后端接口。

## 原理剖析

为什么 Sowork AI 用一个通用 `/api/:table` 就能撑起 20+ 模块？答案是「元数据驱动」：

```
表名 table ──▶ ORM 读取字段定义 ──▶ 前端按字段类型自动渲染
                                     列表(列) / 表单(输入控件)
```

开发者新增一个业务对象，往往只需在 ORM 注册一张表 + 在 `subMenuItems` 加一项，无需为每个模块写一套独立前后端。代价是权限与菜单仍是前端硬编码（见上节），这是当前版本的「诚实边界」。

路由表（前端）集中定义在 `client/src/app.tsx:173-390`，每个子页对应一个 `<Route>`。注意：即便模块没在侧边栏渲染（如 `/product` 仅出现在 `menuItems` 平铺项，`Layout.tsx:49`，其 `subMenuItems['/product']` 虽定义但未参与侧边栏展开渲染），对应路由依然存在，URL 可直接访问。

## 管理者视角

下面按 7 大域，回答「解决什么痛点 / 上系统前怎么凑合 / 用了省什么」。

| 业务域 | 解决的痛点 | 上系统前怎么凑合 | 用了 Sowork AI 省了什么 |
| --- | --- | --- | --- |
| **HR 域**（人事/考勤/薪酬/绩效/招聘/后勤/培训/审批） | 员工信息散落 Excel，考勤靠打卡机导出再人工算薪 | 微信发通知、纸质请假条、Excel 汇总工资 | ✅ 入转调离合同一体，排班/薪资公式自动算，审批线上留痕 |
| **ERP 域**（产品/PLM/仓储/销售/采购/生产/财务/质量） | 进销存与财务脱节，BOM 靠人肉维护 | 多个 Excel 对账，月底手工做凭证 | ✅ 从款号到 BOM 到工单全链路，库存实时，凭证自动 |
| **CMS 域** | 官网内容更新要找外包 | FTP 传静态页、改代码 | ✅ 栏目/文章/Banner 后台自助发布 |
| **商城域** | 想做自营电商但缺系统 | 挂靠第三方平台交佣金 | ✅ 商品/订单/优惠券/分销自有闭环 |
| **AI 域** | 数据不会看、重复问答多 | 人工跑报表、群里反复问 | 🚀 AI 助手+BI 分析+智能预警 |
| **低代码域** | 零散需求总要排期开发 | 等 IT 排期、或买 SaaS | 🚀 拖拽搭应用，模型集成中心接 ORM |
| **系统管控域** | 账号乱、权限不清、无审计 | 共用账号、无日志 | ✅ 角色权限/审计日志/企微钉钉集成 |

🚀 总结：对中小制造企业，HR + ERP 两块就能替换掉至少 3 套零散工具（考勤机系统、Excel 薪资、进销存软件），这是上线最直观的 ROI。

## 注意事项

- ⚠️ 安装必须 `npm install --legacy-peer-deps`，裸 `npm install` 在 React 19 + Ant Design 5 组合下会依赖冲突失败。
- ⚠️ 默认账号 `admin/admin123` 权限极高（含 `/system`），演示环境务必改密，生产禁止公网暴露 3000 端口。
- 🔑 菜单是前端硬编码，改模块可见性要改 `Layout.tsx` 的 `MODULE_PERM` / `USER_ROLE_PERMS`，不是改数据库。
- 🔑 业务数据走通用 `/api/:table`，但受 `ALLOWED` 白名单约束，未注册的表名直接访问会被拒。
- ✅ 角色权限判断在 `Layout.tsx:393-397`，调试「看不到某模块」先从这里排查 `userPerms`。
