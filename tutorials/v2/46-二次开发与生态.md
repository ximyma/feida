本篇属于《Sowork AI 企业智能ERP系统》v1.2.0 代码级双读者教程。你将学会在系统中亲手扩展业务——既能用代码写完整模块，也能用向导零代码搭插件，并了解如何把既有 Odoo 资产迁移进来、如何参与开源社区。环境默认账号 `admin/admin123`，服务端口 `3000`，依赖安装请使用 `npm install --legacy-peer-deps`。

## 学习目标

完成本篇后，你能：

- 区分 `modules/`（手写业务模块）与 `addons/`（低代码/插件）两条扩展路径。
- 照着约定手写一个业务模块，并理解它如何自动变成可操作的页面。
- 跑通 Odoo → Sowork AI 企业智能ERP系统 的模型迁移流水线。
- 从管理者视角判断"该自研还是该用社区插件"。
- 以 Apache-2.0 合规方式向社区提交你的模块。

## 核心概念

系统把"业务"拆成两层扩展面，二者共用同一套 ORM 注册机制：

| 维度 | modules/（手写模块） | addons/（低代码插件） |
| --- | --- | --- |
| 定位 | 完整控制的业务模块 | 向导生成的轻量插件 |
| 作者 | 平台/核心团队 | 业务用户、实施方 |
| 约定 | `exports.models` 数组或 `export default` | 每个模型一个 `exports.model` 对象 |
| 配套 | manifest + 数据 + hooks | manifest + `app.json` 菜单 |
| 前端 | 几乎零（模型即页面） | 完全零（向导生成） |
| 生效 | 需重启/热重载 | 扫描即加载 |

🔑 一句话记住：**手写模块给你全部权力，低代码插件给你全部速度。**

## 界面布局与操作流程

两条路径在磁盘上清晰分开：

```
feida/
├── modules/            ← 手写业务模块
│   ├── hr_core/        manifest.json + models/models.js
│   ├── blog/           manifest.json + models/ + data/
│   └── odoo_base/      由 Odoo 转换而来的 40 个模型
└── addons/             ← 低代码/插件（呼应第 11/12 篇）
    ├── app_mru06doi/   manifest.json + app.json + models/*.js
    ├── demo_erp/
    └── analytic_odoo/
```

你在浏览器 `http://localhost:3000` 登录后，插件模块会依据 `app.json` 的 `menu` 自动出现在左侧应用菜单；手写模块则依据 `manifest.json` 的 `application` 与 `sequence` 出现。你不需要写一行前端路由——这是后面"原理剖析"要讲清的关键。

操作流程（新增一个业务模块）：

1. 🚀 简单需求：进入低代码向导，填字段 → 系统落到 `addons/<app>/models/*.js` + `app.json`。
2. 复杂需求：在 `modules/` 下新建目录，写 `manifest.json` 与 `models/models.js`，重启服务。
3. 迁移需求：用 Odoo 转换工具把 `.py` 模型批量落进 `modules/odoo_base/`。

## 底层逻辑与数据模型

无论哪种路径，一个"业务模块"本质就是一组 **ORM 模型定义**。以 `modules/hr_core/models/models.js` 为例，它导出 `exports.models` 数组：

```js
exports.models = [
  {
    _name: 'employees',          // 表名，也是页面标识
    _description: '员工档案',
    _rec_name: 'realName',
    _fields: {
      realName: { type: 'char', label: '姓名', required: true, index: true },
      baseSalary: { type: 'float', label: '基本工资',
                    groups: ['super_admin','hr_admin','hr_staff'] },
      status: { type: 'selection', label: '状态', default: 'active',
                selection: [{label:'在职',value:'active'},{label:'离职',value:'inactive'}] },
    },
  },
  // departments / positions / ranks ...
];
```

字段类型（`char/text/integer/float/date/selection/many2one/...`）直接决定页面渲染的控件。注意 `groups` 字段——它让"基本工资""身份证号"只对特定角色可见，权限模型与数据模型同源。

低代码插件走的是单文件约定。看 `addons/app_mru06doi/models/opportunity.js`：

```js
exports.model = {
  _name: "opportunity",
  _description: "商机信息",
  _fields: {
    customer_id: { type: "many2one", label: "所属客户", relation: "customer" },
    amount:      { type: "char", label: "金额" },
  }
};
```

差别在于：手写模块用 `exports.models`（复数、数组），插件用 `exports.model`（单数、对象）。这正是加载器要识别的两种约定。

## 源码剖析

核心加载器在 `server/modules/orm/model-registry.ts`。它从模块目录扫描 `models/`，逐个 `require` 并识别导出形态——这是"同一套机制承载两条路径"的真相：

```ts
// server/modules/orm/model-registry.ts:51
loadFromModule(moduleDir: string): BaseModel[] {
  const modelsDir = path.join(moduleDir, 'models');
  if (!fs.existsSync(modelsDir)) return [];
  const files = fs.readdirSync(modelsDir).filter(f => f.endsWith('.js') || f.endsWith('.ts'));
  for (const file of files) {
    const fp = path.join(modelsDir, file);
    try {
      delete require.cache[require.resolve(fp)];   // 清缓存，保证热重载拿到最新
      const mod = require(fp);
      const modelDefs: ModelOptions[] = [];
      if (mod.models && Array.isArray(mod.models)) {        // 手写模块
        modelDefs.push(...mod.models);
      } else if (mod.model && mod.model._name) {           // 低代码插件
        modelDefs.push(mod.model as ModelOptions);
      } else if (mod.default && mod.default._name) {        // export default
        modelDefs.push(mod.default as ModelOptions);
      } else if (Array.isArray(mod)) {                      // 裸数组兜底
        modelDefs.push(...(mod as ModelOptions[]));
      }
      for (const modelDef of modelDefs) {
        if (modelDef._name) {
          this.register(modelDef);                          // 建表/注册
          loaded.push(this.models.get(modelDef._inherit || modelDef._name)!);
        }
      }
    } catch (e) { console.error(`[ModelRegistry] 加载失败: ${fp}`, (e as Error).message); }
  }
  return loaded;
}
```

两个细节值得你记住：`model-registry.ts:62` 删除 `require.cache` 是为了**热重载**时能重新加载改动后的模型文件；`model-registry.ts:65-73` 依次兼容 `models` / `model` / `default` / 裸数组四种导出，所以手写模块和插件才能共用一个加载函数。

启动时 `server/standalone.ts:101-135` 分别扫描 `modules/` 与 `addons/`，对后者还会打上 `def._addon = true` 标记（`standalone.ts:130`），让前端区分"核心模块"与"插件"。

## 原理剖析

**为什么加一个业务模块几乎零前端代码？** 答案是"模型即页面"：

- 你在 `models.js` 里声明一个带 `_fields` 的模型，注册器就为它 `ensureTable()` 自动建表（`model-registry.ts:29`）。
- 前端通用渲染器读取字段元数据（`type/label/selection/groups`），动态生成列表、表单、搜索框——没有为某个模型写专属页面。
- `app.json` 的 `menu` 只是指定"哪个模型挂哪个菜单、用什么图标"，剩下的列表/详情/增删改由通用引擎统一提供。

于是从"定义字段"到"可用页面"之间，你只差一次服务加载。手写模块多了一点自由（可以写 `hooks/post_init.js` 灌种子数据、写 `data/*.sql`），插件则把这一切收敛进向导。

**Odoo 资产怎么迁移进来？** 流水线在 `tools/`：

```
odoo-parser.py  解析 .py 模型 → 中间 JSON
        │
        ▼
odoo2feida.js   JSON → modules/<name>/manifest.json + models/*.js
```

`odoo-parser.py:8-14` 把 Odoo 的 `Char/Integer/Many2one` 等类型映射成 Sowork AI 企业智能ERP系统 的 `char/integer/many2one`。`odoo2feida.js:52-65` 再把模型分成"独立模型"与"继承模型"分别落盘，并在 `odoo2feida.js:83` 标出"缺少 relation  many2one"这类需人工核对的点——呼应第 12 篇的 Odoo 集成。你看 `modules/odoo_base/` 正是这条流水线产出的 40 个 `res.*` / `ir.*` 模型。

> ⚠️ 注：`odoo2feida.js` 为仓库真实脚本（更名前命名，非产品名）；产品对外名称统一为 **Sowork AI 企业智能ERP系统**。本地目录树中的 `feida/` 即你的项目根目录（机器路径），对外不称产品名。详见大纲"历史内部命名说明"。

## 管理者视角

从 TCO（总拥有成本）看两条路径：

- 🚀 **生态扩展寿命**：手写模块进 `modules/`，随版本演进可获得完整 API 支持；低代码插件进 `addons/`，适合随业务快速试错、用完即弃。
- ✅ **非核心需求用低代码自搭**：CRM、进销存之外的长尾流程，交给业务部门用向导生成，IT 不必卷入。
- **社区插件降低 TCO**：官方以 Apache-2.0 开源，任何人可贡献、复用现成插件，避免重复造轮子。

⚠️ 但别把"能低代码"误解成"不用治理"：插件一旦承载核心数据，就要纳入备份、权限与版本规划，否则会成为影子资产。

## 注意事项

- ⚠️ 手写模块通常需**重启服务**（或依赖热重载）才能生效；改完 `models.js` 后确认控制台出现 `[ORM] <模块>: +N 模型`。
- 🔑 **别混约定**：`modules/` 用 `exports.models`（数组），`addons/` 用 `exports.model`（单对象）。混写会导致加载器识别不到模型。
- ✅ **优先低代码，复杂再手写**：80% 的需求向导即可满足；只有当需要 `hooks`、`data` 种子、自定义逻辑或深度继承（`_inherit`）时，才落地到 `modules/`。
- ⚠️ 从 Odoo 迁移的模型，迁移后请人工核对 `many2one.relation` 与 `selection` 选项，工具只保证"结构到位"，业务语义仍需你确认。
- 🔑 向社区贡献前，确认你的模块 `license` 为 `Apache-2.0` 且不含硬编码的敏感配置，再通过 PR 提交。

## 小结与练习

你已看清 Sowork AI 企业智能ERP系统 的双轨扩展：手写模块（`modules/`，`exports.models`，完整控制）与低代码插件（`addons/`，`exports.model`，向导生成），二者由同一 `ModelRegistry.loadFromModule` 统一加载；"模型即页面"让你加业务几乎零前端；`odoo-parser.py` + `odoo2feida.js` 把 Odoo 资产平滑迁入门；Apache-2.0 让社区插件持续降低总拥有成本。
