# 10. 知识库、BI分析与智能预警

本篇带你掌握 **Sowork AI 企业智能ERP系统**（版本 v1.2.0）三大智能化能力：知识库、BI 分析、智能预警。你将看到它们如何复用第 09 篇讲的 SSE 流式与 RAG 引擎，又被封装成业务功能。

环境约定：账号 `admin/admin123`，服务端口 `3000`，安装依赖请用 `npm install --legacy-peer-deps`。本系统没有 `/api/hr` 这类域前缀路由，业务数据统一走 catch-all 路由 `/api/:table`（见 `standalone.ts:4065`），而 AI 能力集中在 `/api/ai/*`。

## 学习目标

读完本篇你能做到：

- 用代码级视角理解「上传文档 → 切分 → 混合检索 → RAG 问答」全链路。
- 看懂 BI 分析的数据源、报表定义与图表三者关系，并知道如何接第 42 篇仪表盘。
- 理解智能预警的规则引擎，以及它如何直接查业务库生成告警记录。
- 明白本篇是「业务封装」，第 09 篇是底层的「引擎」。

## 核心概念

三大能力各解决一类问题：

| 能力 | 解决什么问题 | 核心机制 | 关键路由 |
|------|--------------|----------|----------|
| 知识库 | 重复问答、制度查找 | 文档切分 + 混合检索 + RAG | `/api/ai/kb`、`/api/ai/rag-chat` |
| BI 分析 | 经营不可视、决策靠拍 | 数据源 + 报表定义 + ECharts | `/api/report_definitions`、`/api/data_sources` |
| 智能预警 | 风险事后才发现 | 规则引擎 + 阈值触发 | `/api/ai/alert-rules`、`/api/ai/alert-records` |

它们共享第 09 篇的底座：`ai-service.js` 里的 `chatCompletion`（`ai-service.js:111`）与混合检索 `hybridSearch`（`ai-service.js:294`）。本篇只是把这些原子能力编排成看得懂、用得上的功能。

## 界面布局与操作流程

知识库页面 `AIKnowledgePage.tsx` 是左右两栏布局：

```
┌──────────────┬─────────────────────────────────┐
│ 知识库列表    │ 知识条目 | 文档管理 | 命中测试   │
│ (新建/编辑)   │  (搜索)   (拖拽上传)  (检索验证) │
└──────────────┴─────────────────────────────────┘
```

BI 页面 `AIBIAnalyticsPage.tsx` 自上而下是：指标卡（员工总数/出勤率/平均薪资/绩效达标率）→ ECharts 图表（考勤趋势、部门结构、人员流动）→「AI 智能洞察」按钮 → 智能预警提示区。

预警页面 `AIAlertsPage.tsx` 用 Tab 切换「预警记录」（待处理/已处理/已忽略）与「预警规则」（阈值、启用开关）。

## 底层逻辑与数据模型

知识库核心三张表：`ai_knowledge_bases`（库）、`ai_knowledge_documents`（原始文档）、`ai_knowledge`（切分后的知识条目）。

BI 的数据模型在 `database.service.ts:1480`：

```sql
report_definitions( id, name, category, tableName,
  chartType, xField, yFields, filters, fields, config )
data_sources( id, name, type, tableName, filePath,
  apiUrl, apiHeaders, fields, status )
```

一句口诀：**data_sources 决定「数据从哪来」，report_definitions 决定「画成什么图」**。这正好衔接第 42 篇仪表盘。

预警的 `alert_rules` 在代码中是静态规则数组（`standalone.ts:3295`），而 `alert-records` 会直接联表查出真实业务异常（如合同 30 天内到期）。

## 源码剖析

**① 知识库：上传即切分。** `POST /api/ai/kb/:kbId/upload`（`standalone.ts:2825`）按扩展名选解析器（PDF/DOCX/XLSX/文本），读出的文本调用 `recursiveSplitText` 切成 800 字、重叠 150 字的片段，再逐条写入 `ai_knowledge`：

```ts
// standalone.ts:2912
const chunks = aiService.recursiveSplitText(content, 800, 150);
for (let i = 0; i < Math.min(chunks.length, 50); i++) {
  db.insert('ai_knowledge', { id: itemId, kb_id: kbId, ... });
}
```

切分本身是递归按 `。！？；` 等分隔符下钻（`ai-service.js:228`），保证语义不被粗暴切断。

**② RAG 问答。** `POST /api/ai/rag-chat`（`standalone.ts:3263`）先用 `buildRagContext`（`ai-service.js:322`）做混合检索拼出上下文，再塞进 system 提示词调用大模型：

```ts
// standalone.ts:3275
const ragContext = aiService.buildRagContext(query, items);
const systemMsg = `${aiService.getRuntimeConfig().systemPrompt}\n\n...${ragContext}`;
const result = await aiService.chatCompletion(allMessages);
```

混合检索 `hybridSearch`（`ai-service.js:294`）把「语义余弦相似度」与「关键词命中」按权重融合，这正是第 09 篇 RAG 引擎被业务封装的位置。

**③ BI 报表。** `GET /report_definitions`（`standalone.ts:2053`）转调 `getReportDefinitions`（`database.service.ts:4402`）读表；前端 `AIBIAnalyticsPage.tsx` 用 `ReactECharts` 渲染 `option`。注意：页面里指标卡调用真实 catch-all 路由（`/api/employees`、`/api/salaries`），部分示例图表序列为演示数据，真正可扩展的入口是 `data_sources` / `report_definitions` 两张表。

**④ 智能预警。** `GET /api/ai/alert-records`（`standalone.ts:3308`）直接 JOIN 合同表与员工表，扫出 30 天内到期合同并生成记录：

```ts
// standalone.ts:3312
const contracts = db.query(
  "SELECT e.realName, e.deptName, c.endDate FROM emp_contracts c \
   JOIN employees e ON c.empId = e.id \
   WHERE c.status='active' AND c.endDate <= date('now','+30 days')");
```

## 原理剖析

把链路画清楚：

```
文档 ──上传──> 解析 ──切分──> ai_knowledge
                                  │
查询 ──混合检索(hybridSearch)──> 拼上下文(buildRagContext)
                                  │
                              chatCompletion ──> RAG 回答
```

预警则是「规则 → 查业务库 → 比对阈值 → 生成记录」的单向管线。规则在代码里定义（`standalone.ts:3295`），引擎按 `threshold` 判断越界，越界即落一条 `alert-records`。

关键在于：知识库和预警都**不重新发明 AI**，而是站在第 09 篇的 `chatCompletion` 与 `hybridSearch` 上做编排——这就是「引擎」与「业务封装」的分层。

## 管理者视角

- **知识库**：把员工手册、薪酬制度、考勤办法一次性入库，新人提问由 AI 直接答，减少 HR 重复答疑。
- **BI 分析**：出勤率、薪资分布、人员流动一眼看清，经营从「凭感觉」变「看数据」。
- **智能预警**：库存告急、合同到期、连续迟到、加班超时，系统在异常发生前就提醒你，从「救火」变「防火」。

## 注意事项

⚠️ 知识库回答质量直接取决于你上传的文档质量——垃圾进，垃圾出。

🔑 预警规则必须结合真实业务阈值（如 `threshold: 36` 表示月加班 36 小时），拍脑袋设值会误报或漏报。

✅ BI 的数据源 `data_sources` 默认 `type='internal'`，可直接指向现有表，无需另建数仓即可出图。

🚀 改完规则或上传新文档后，记得在前端点「刷新」，预警与检索结果会实时反映最新数据。

## 小结与练习

本篇你看到：知识库用「切分 + 混合检索 + RAG」把文档变成可问答资产；BI 用「数据源 + 报表定义 + ECharts」让经营可视；预警用「规则 + 阈值」主动防风险。三者都站在第 09 篇的引擎之上。

练习：

1. 新建一个知识库并上传一份 PDF，观察 `ai_knowledge` 表新增了多少条目（用命中测试验证检索效果）。
2. 在 `data_sources` 插入一条指向 `employees` 的记录，再写一条 `report_definitions`，尝试让 BI 画出部门人数饼图。
3. 把「合同到期提醒」阈值从 30 天改成 60 天，重启服务后看 `alert-records` 返回记录是否变多。
