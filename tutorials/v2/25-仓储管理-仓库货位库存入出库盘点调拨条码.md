## 核心概念

仓储管理要解决一个根本问题：**账上的库存数字，必须与实际货架上的实物一致**。系统用一张 `inventory` 表记录「真相」，所有业务动作都通过单据去修改它，而非直接手改。这正是 ERP 区别于「用 Excel 记账」的地方——每一笔变动都有来源、有痕迹。

业务动作与库存变化的关系：

| 动作 | 入口页面 | 对库存的影响 |
|------|----------|-------------|
| 采购/生产/退货入库 | 入库管理 | `+` 增加 |
| 销售/领料/退货出库 | 出库管理 | `-` 减少 |
| 盘点 | 库存盘点 | 校正差异（可正可负） |
| 调拨 | 库存调拨 | 仓库间移动（一减一加） |
| 扫码 | 条码管理 | 仅识别，不直接改数 |

🔑 一句话：**单据是过程，库存是结果。** 你在「库存查询」里看到的数字，是无数张单据累加出来的快照。

关于接口路由，有一点必须先讲清：Sowork AI 的仓储数据**大部分走 catch-all 通用接口 `/api/:table`**（例如仓库、货位、库存、条码的增删改查），但**入出库、盘点、调拨这类带事务语义的动作有专用接口**（`POST /api/inventory/transaction`、盘点 `complete`、调拨 `complete` 等）。前者负责「档案与记录的维护」，后者负责「库存数量的原子变动」。你在前端 Network 面板里会同时看到两类请求，不必困惑。

另外，`inventory` 里用 `sku_id` 与 `material_id` 二选一来表示「被库存的对象」：成品通常用 SKU，原材料/半成品用物料。同一行只能指向其一，系统靠 `sku_id || material_id` 这个兜底逻辑统一取用。

### 仓储数据模型总览表

下面是所有相关表及其关键字段。字段名与后端 `db` 操作、前端 `interface` 完全一致，可对照源码使用。

| 表名 | 作用 | 关键字段 |
|------|------|----------|
| `warehouses` | 仓库档案 | `id, code, name, type(raw/wip/finished/tool), location, area, capacity, manager_name, status(active/inactive), remark` |
| `locations` | 货位 | `id, warehouse_id, code, row_no, shelf_no, level_no, area, capacity, status(available/occupied/reserved/deleted), remark` |
| `inventory` | 库存唯一真相 | `id, warehouse_id, location_id, sku_id, material_id, qty, locked_qty, unit, batch_no, expiry_date, cost_price` |
| `inventory_transactions` | 出入库流水 | `id, warehouse_id, location_id, sku_id, material_id, type(in/out), qty, unit, cost_price, batch_no, source_doc_id, source_doc_type, operator_id, operator_name, created_at` |
| `stock_checks` | 盘点单头 | `id, code, warehouse_id, location_id, status(draft/in_progress/completed), operator_id, operator_name, created_at, completed_at` |
| `stock_check_items` | 盘点明细 | `id, check_id, inventory_id, sku_id/material_id, system_qty, actual_qty, diff_qty` |
| `transfer_orders` | 调拨单头 | `id, code, from_warehouse_id, to_warehouse_id, status(pending/approved/completed), operator_name, created_at, approved_at` |
| `transfer_order_items` | 调拨明细 | `id, transfer_id, sku_id, material_id, qty, unit` |
| `barcodes` | 条码 | `id, code, type(sku/material/location), target_id, warehouse_id, location_id, batch_no, expiry_date, qty` |

⚠️ 注意 `inventory` 表的 `location_id`、`locked_qty`、`expiry_date` 三个字段容易被忽略：货位可空（不指定货位的库存也算合法），`locked_qty` 是锁定预留量（当前页面只读展示），`expiry_date` 支持批次到期管理。

## 仓储首页/概览

### ① 界面布局

左侧导航「仓储管理」下含 9 个页面（见 `client/src/components/Layout.tsx:145-155`），其中 `/warehouse` 是首页概览（`client/src/pages/WarehousePage/WarehousePage.tsx`）。首页把 8 个功能按「仓库管理 / 库存管理 / 仓储作业」三组卡片展示，并带一个「库存统计概览」四宫格（仓库数量、库存总量、SKU 种类、本月出入库）。顶部还有一句副标题「仓库、库存、条码一体化管理」，点明模块定位。

### ② 操作流程

首页是导航中枢，不做数据写入。你点击任意卡片即可进入对应子页。四宫格的统计数字目前是**前端静态占位值 0**（`WarehousePage.tsx:79-94`），真实的实时统计要看「库存查询」页面或后端 `/api/inventory/statistics` 接口。换句话说，首页当前更像「功能地图」，而非「数据仪表盘」。

### ③ 底层逻辑与数据模型

首页本身不落库，它只是路由入口与分组展示。四宫格若要做成真实统计，应调用 `GET /api/inventory/statistics`（`server/standalone.ts:7038-7050`），该接口返回 `total_qty / total_value / sku_count / warehouse_count`。当前前端未接线，所以恒为 0。

### ④ 源码剖析

四宫格数据来源是写死的 `0`：

```tsx
// client/src/pages/WarehousePage/WarehousePage.tsx:79-94
<div className="text-2xl font-bold text-blue-600">0</div>
<div className="text-sm text-gray-600 mt-1">仓库数量</div>
```

而后端其实已提供统计能力：

```ts
// server/standalone.ts:7043-7048
const stats = {
  total_qty: inventory.reduce((sum, i) => sum + (i.qty || 0), 0),
  total_value: inventory.reduce((sum, i) => sum + (i.qty * i.cost_price), 0),
  sku_count: new Set(inventory.map(i => i.sku_id).filter(Boolean)).size,
  warehouse_count: new Set(inventory.map(i => i.warehouse_id).filter(Boolean)).size
};
```

### ⑤ 原理剖析

首页的「卡片 + 四宫格」是一种**信息架构**：把高频操作前置、把汇总指标显眼化。当前四宫格是占位实现，但这恰恰说明统计口径已经由后端准备好——你只需把前端那个 `0` 换成接口返回值，概览就能实时反映全局库存健康度。🚀 对实施顾问来说，这是一个低风险、高感知的二开切入点：客户一登录就能看到「我现在有多少货、值多少钱」。

## 仓库档案

### ① 界面布局

`client/src/pages/WarehousePage/WarehousesPage.tsx`。顶部是搜索框 + 「新增仓库」按钮；下方表格列：仓库编码、名称、类型、位置、面积/容量、负责人、状态、操作。状态用绿/灰徽章区分「启用/停用」，类型用中文映射（原材料仓/半成品仓/成品仓/工具仓）。

### ② 操作流程

1. 点「新增仓库」，填编码（如 `WH001`）、名称、类型、位置、面积、容量、负责人。
2. 列表里可编辑、可删除（删除是弹窗二次确认）。
3. 仓库是后续所有货位、库存的归属根节点，建议先建好仓库再建货位。
4. 搜索框按编码或名称实时过滤，方便仓库多了以后快速定位。

### ③ 底层逻辑与数据模型

落库表 `warehouses`，关键字段见总览表。后端提供完整 CRUD：`GET /api/warehouses`（支持 `type`、`status` 过滤）、`GET /:id`、`POST`、`PUT /:id`、`DELETE /:id`。注意 `type` 字段影响业务语义——成品仓放成品、原材料仓放物料，虽然不是系统强制约束，但利于统计与权限规划。

### ④ 源码剖析

新建仓库时，后端用时间戳生成 `id`（`wh_` 前缀）并允许前端用 `code` 覆盖默认编码：

```ts
// server/standalone.ts:6978-6983
router.post('/warehouses', (req, res) => {
  const id = `wh_${Date.now()}`;
  const code = req.body.code || `WH${String(Date.now()).slice(-6)}`;
  db.insert('warehouses', { id, code, ...req.body, created_at: new Date().toISOString() });
  res.json({ success: true, id });
});
```

⚠️ **删除是软删除**：后端并不物理删除行，而是把 `status` 改为 `inactive`：

```ts
// server/standalone.ts:6988-6991
router.delete('/warehouses/:id', (req, res) => {
  db.update('warehouses', req.params.id, { status: 'inactive' });
  res.json({ success: true });
});
```

前端在拉取仓库列表时，货位/库存等页面会用 `w.status === 'active'` 过滤，停用的仓库不会出现在下拉里。

### ⑤ 原理剖析

「软删除 + 状态过滤」是 ERP 系统的常见设计：仓库可能被引用在大量历史单据中，物理删会破坏外键与审计链。把状态置为 `inactive` 既从业务视图隐藏，又保留了数据完整性。所以你「删掉」的仓库，在数据库里依然存在——这也是为什么「删除」后库存查询的仓库下拉里它消失了，但底层 `inventory` 行仍可能引用它。✅ 若需彻底清理，应走数据治理流程而非界面删除。

## 货位管理

### ① 界面布局

`client/src/pages/WarehousePage/LocationsPage.tsx`。结构与仓库档案类似，多了「所属仓库」下拉筛选。表格列：所属仓库、货位编码、货位位置（排/架/层）、面积/容量、状态、操作。新增弹窗强制要求先选仓库再编码。

### ② 操作流程

1. 新增货位须先选「所属仓库」，再填编码（如 `A01-01-01`）、排/架/层、面积、容量。
2. 货位状态有四种：`available` 可用、`occupied` 占用、`reserved` 预留、`deleted` 已删除。
3. 货位把「仓库」这个粗粒度进一步细化到「货架上的某一格」，便于上架、拣货定位。
4. 列表支持先按仓库、再按关键词（编码或仓库名）两级过滤。

### ③ 底层逻辑与数据模型

落库表 `locations`，通过 `warehouse_id` 归属仓库，关键字段见总览表。后端 CRUD 在 `server/standalone.ts:6993-7017`，与仓库同构：`GET /api/locations`（支持 `warehouse_id`、`status` 过滤）、`GET /:id`、`POST`、`PUT /:id`、`DELETE /:id`。

### ④ 源码剖析

货位新增同样用时间戳 `id`（`loc_` 前缀），并把整段 `req.body` 透传：

```ts
// server/standalone.ts:7005-7009
router.post('/locations', (req, res) => {
  const id = `loc_${Date.now()}`;
  db.insert('locations', { id, ...req.body, created_at: new Date().toISOString() });
  res.json({ success: true, id });
});
```

删除同样是软删除，状态改为 `deleted`：

```ts
// server/standalone.ts:7014-7017
router.delete('/locations/:id', (req, res) => {
  db.update('locations', req.params.id, { status: 'deleted' });
  res.json({ success: true });
});
```

前端「库存查询」拉取货位时用了 `l.status !== 'deleted'` 过滤（`InventoryPage.tsx:48`），所以已删除货位不会进入库存定位的可选项。

### ⑤ 原理剖析

货位编码建议采用「排-架-层」规则（如 `A01-01-01`），这样 PDA 扫码或人工找货时，字符串本身就携带空间信息。货位状态 `occupied/reserved` 当前由用户手动维护——系统并不会在出入库时自动翻转它，因此它更多是一个**提示标志**，真正的可用量以 `inventory.qty` 为准。🔑 一句话：货位是「空间坐标」，库存是「数量真相」，两者通过 `inventory.location_id` 关联，但彼此独立更新。

## 库存查询

### ① 界面布局

`client/src/pages/WarehousePage/InventoryPage.tsx`。顶部四张统计卡：库存种类、库存总量、库存金额、仓库数量（这里是**真实计算**值，区别于首页占位）。下方支持按关键词、仓库、货位三级筛选，表格列：仓库、货位、SKU/物料、批次号、数量、锁定、成本价、到期日。数量为负时显示红色，为正显示绿色。

### ② 操作流程

1. 用搜索框按「批次号 / SKU 编码」模糊搜索。
2. 先用仓库下拉过滤，选中仓库后会出现货位二级下拉，可继续下钻。
3. 数量是实时从 `inventory` 读出的，任何入出库、盘点、调拨后刷新即可见变化。
4. 库存金额 = 数量 × 成本价，是按当前快照估算的在库价值。

### ③ 底层逻辑与数据模型

数据来自 `GET /api/inventory`（`server/standalone.ts:7020-7035`），支持 `warehouse_id / location_id / sku_id / material_id / keyword` 过滤。前端统计：

```ts
// client/src/pages/WarehousePage/InventoryPage.tsx:65-66
const totalQty = filtered.reduce((sum, i) => sum + (i.qty || 0), 0);
const totalValue = filtered.reduce((sum, i) => sum + ((i.qty || 0) * (i.cost_price || 0)), 0);
```

### ④ 源码剖析

后端库存查询就是一次全表扫描 + 多维过滤：

```ts
// server/standalone.ts:7020-7035
router.get('/inventory', (req, res) => {
  const { warehouse_id, location_id, sku_id, material_id, keyword } = req.query;
  let list = db.findAll('inventory');
  if (warehouse_id) list = list.filter(l => l.warehouse_id === warehouse_id);
  if (location_id) list = list.filter(l => l.location_id === location_id);
  if (sku_id) list = list.filter(l => l.sku_id === sku_id);
  if (material_id) list = list.filter(l => l.material_id === material_id);
  if (keyword) { /* 按 batch_no / sku_id 模糊匹配 */ }
  res.json(list);
});
```

注意这里读取的 `inventory` 行带有原概览未强调的 `locked_qty`（锁定）与 `expiry_date`（到期日）字段——它们是库存查询页真实展示的列。

### ⑤ 原理剖析

库存查询是「只读快照」。它不修改任何数据，只是把 `inventory` 表的当前状态投影成人类可读的表格。✅ 养成习惯：每次做完入/出库或盘点，都回到这里验证一遍数字是否如预期——它是你检验所有单据副作用的「真相仪表盘」。🚀 更进一步：把 `keyword` 过滤配合批次号，你就能做「某批货现在在哪、还有多少、何时到期」的追溯，这对食品、鞋服等有时效的行业尤为重要。

## 入库管理

### ① 界面布局

`client/src/pages/WarehousePage/StockInPage.tsx`。列表展示历史入库流水（仓库、SKU/物料、数量带绿色 `+`、批次、来源单据、操作人、时间）。「新增入库」弹窗支持：入库仓库、来源类型（采购/生产/退货/其他）、来源单号、操作人，以及可多行的入库明细（SKU、物料、数量、批次、成本、备注）。

### ② 操作流程

1. 选入库仓库、来源类型（如采购入库选 `purchase`）、填来源单号（可关联采购单号）。
2. 在明细里逐行加 SKU/物料、数量、批次、成本。可「添加一行」批量入多种货。
3. 确认后系统对每行调用一次事务接口，列表即时刷新。
4. 列表的「来源单据」列可回链到采购/生产系统，形成跨模块追溯。

### ③ 底层逻辑与数据模型

入库走**专用事务接口** `POST /api/inventory/transaction`，而非普通表 CRUD。每行明细生成一条 `inventory_transactions`（`type='in'`），并联动 `inventory`。前端提交体见 `StockInPage.tsx:65-83`。`source_doc_type` 仅作分类标签，不参与库存计算，但影响后续统计与审计。

### ④ 源码剖析

后端核心逻辑：先写流水，再按「仓库 + SKU/物料 + 货位」定位库存行，有则累加、无则新建：

```ts
// server/standalone.ts:7078-7091
if (inventory.length > 0) {
  const inv = inventory[0];
  const newQty = type === 'in' ? (inv.qty || 0) + qty : (inv.qty || 0) - qty;
  db.update('inventory', inv.id, { qty: newQty, updated_at: new Date().toISOString() });
} else {
  const invId = `inv_${Date.now()}`;
  db.insert('inventory', {
    id: invId, warehouse_id, location_id, sku_id, material_id,
    qty: type === 'in' ? qty : -qty,
    unit: unit || '双', batch_no, cost_price: cost_price || 0,
    created_at: new Date().toISOString(), updated_at: new Date().toISOString()
  });
}
```

注意 `type='in'` 加、`type='out'` 减。入库时前端**传了 `location_id`**（`StockInPage.tsx:71`），因此入库库存可以精确到货位。

### ⑤ 原理剖析

把「一笔入库单」拆成「多条事务」，是 ERP 的关键抽象：一张单对应 N 行商品，每行独立写流水、独立改库存。这样即使同一单据里混了多个 SKU，事后也能逐行追溯「谁、在什么仓库、何时、入了多少、来自哪张单」。🔑 同时，库存行是「按维度唯一」的：同一个 `仓库+货位+物料` 组合只有一行，多次入库只是不断累加 `qty`，而不是新增多行——这正是 `findWhere` 先查后插的意义。

## 出库管理

### ① 界面布局

`client/src/pages/WarehousePage/StockOutPage.tsx`。与入库页面几乎镜像：列表数量带红色 `-`，来源类型切换为销售/领料/退货/其他，明细行少了「批次、成本」两列（出库通常按现有库存扣减，不在此重填成本）。

### ② 操作流程

1. 选出库仓库、来源类型（如销售出库选 `sales`）、填销售单号。
2. 明细行填 SKU/物料、数量、备注，确认后逐行调用事务接口。
3. 列表刷新，数量显示为 `-N`。
4. 与入库一样，来源单号可关联销售/生产模块，形成闭环。

### ③ 底层逻辑与数据模型

同样走 `POST /api/inventory/transaction`，但 `type='out'`。⚠️ 关键差异：**出库表单不含货位字段**（`StockOutPage.tsx:27` 的 `formData` 只有 `warehouse_id`，提交体 `StockOutPage.tsx:66-78` 也未传 `location_id`）。这意味着出库只扣减「仓库级」库存行，不会精确扣到某个货位。

### ④ 源码剖析

出库复用与入库同一段增减逻辑，只是 `type` 为 `out`：

```ts
// server/standalone.ts:7080
const newQty = type === 'in' ? (inv.qty || 0) + qty : (inv.qty || 0) - qty;
```

若定位到的库存行不存在，出库会直接插入一条**负数**库存（`qty: -qty`，`standalone.ts:7086`）。这正是「先入库、后出库」顺序重要的原因。

### ⑤ 原理剖析

出库产生的负库存是「允许但需警惕」的状态：它意味着系统在没货的情况下记录了发货，会掩盖缺货风险。✅ 实务中你应在出库前先到「库存查询」确认 `qty ≥ 出库量`。当前版本未做「库存不足拦截」，把校验责任交给了操作员——这是一个值得注意的设计取舍。⚠️ 另外，由于出库不传 `location_id`，若同一物料在同一仓库既有货位库存、又有无货位库存，系统只按 `warehouse_id + 物料` 定位，可能扣到不唯一的那一行——多货位场景要特别当心。

## 库存盘点

### ① 界面布局

`client/src/pages/WarehousePage/StockCheckPage.tsx`。列表展示盘点单（单号、仓库、货位、状态、操作人、创建时间、操作）。「新建盘点」弹窗只含：仓库、货位、操作人三项。「完成盘点」按钮在状态非 `completed` 时显示。状态三态：草稿 `draft`、盘点中 `in_progress`、已完成 `completed`。

### ② 操作流程

1. 选仓库、货位、填操作人，创建盘点单（状态 `draft`，编号形如 `SCxxxxxx`）。
2. 点「完成盘点」，确认后状态变 `completed`。

### ③ 底层逻辑与数据模型

盘点单头落 `stock_checks`，明细落 `stock_check_items`。后端 `POST /api/stock-checks`（`standalone.ts:7120-7130`）建头；`POST /api/stock-checks/:id/complete`（`standalone.ts:7148-7169`）校正库存。理论上，盘点前应暂停该仓库相关出入库，让「系统数」与「实物数」在同一时刻对齐，差异才有意义。

### ④ 源码剖析

完成盘点时，遍历明细，把 `diff_qty` 并入对应 `inventory` 行：

```ts
// server/standalone.ts:7155-7165
const items = db.findWhere('stock_check_items', { check_id: req.params.id });
items.forEach((item) => {
  if (item.diff_qty !== 0 && item.inventory_id) {
    const inv = db.findById('inventory', item.inventory_id);
    if (inv) {
      db.update('inventory', item.inventory_id, {
        qty: (inv.qty || 0) + (item.diff_qty || 0),
        updated_at: new Date().toISOString()
      });
    }
  }
});
```

### ⑤ 原理剖析

🔑 **重要事实（与原概览描述不一致）**：当前前端 `StockCheckPage` **只创建盘点单头，没有提供录入 `stock_check_items` 明细的界面**（弹窗仅有仓库/货位/操作人，`StockCheckPage.tsx:159-170`）。而后端 `complete` 必须依赖 `stock_check_items` 里带 `inventory_id` 和 `diff_qty` 的行才会动库存。因此，仅用界面创建的盘点单，点「完成盘点」后 `inventory` 数字**不会变化**——因为它没有可被校正的明细。

如果你要在 v1.2.0 真正跑通「盘点校正」，需要通过 API 直接写入 `stock_check_items`（例如用 `PUT /api/stock-checks/:id` 携带 `items` 数组，`standalone.ts:7131-7146`），再调用 `complete`。这是本模块一个需要留意的实现缺口，建议后续版本补上明细录入 UI，否则盘点功能在纯界面操作下形同虚设。

## 库存调拨

### ① 界面布局

`client/src/pages/WarehousePage/TransferPage.tsx`。列表：调拨单号、调出仓库、调入仓库、状态、操作人、创建时间、操作。状态机三态：待审批 `pending`、已审批 `approved`、已完成 `completed`。「新建调拨」弹窗选调出/调入仓库、操作人、可多行调拨明细。

### ② 操作流程

1. 选调出、调入仓库（两者不能相同，前端会拦截，`TransferPage.tsx:75`），填明细。
2. 创建后状态为 `pending`，点「审批」变 `approved`。
3. 再点「完成」，库存才真正在两端移动，状态变 `completed`。

### ③ 底层逻辑与数据模型

调拨单头 `transfer_orders`、明细 `transfer_order_items`。后端：`POST /api/transfer-orders`（`standalone.ts:7209-7229`）、`POST /:id/approve`（`:7249-7255`）、`POST /:id/complete`（`:7256-7308`）。调拨单编码形如 `TFxxxxxx`，与盘点单 `SC`、条码 `BAR` 同样用时间戳后缀保证唯一。

### ④ 源码剖析

`complete` 先校验必须是 `approved`，否则拒绝：

```ts
// server/standalone.ts:7257-7261
const order = db.findById('transfer_orders', req.params.id);
if (!order || order.status !== 'approved') {
  res.status(400).json({ success: false, message: '只能完成已审批的调拨单' });
  return;
}
```

随后对每条明细：源仓库 `-qty`，目标仓库 `+qty`（目标无行则插入）：

```ts
// server/standalone.ts:7266-7303 （节选）
// 扣减源仓库
db.update('inventory', inv.id, { qty: (inv.qty || 0) - (item.qty || 0), ... });
// 增加目标仓库
db.update('inventory', inv.id, { qty: (inv.qty || 0) + (item.qty || 0), ... });
// 目标无行则 insert 新建
```

### ⑤ 原理剖析

调拨的本质是**库存所有权的空间迁移**：全局总量守恒，只是 `warehouse_id` 从 A 变到 B。它必须「先审批后执行」，是因为移动一旦发生就代表实物已在途，审批是一道责任闸口。🔑 注意一个细节：源仓库扣减时，系统按 `from_warehouse_id + 物料` 定位（不校验货位），目标仓库按 `to_warehouse_id + 物料` 定位或新建——所以调拨也是仓库级搬运，不精确到货位。⚠️ 若源仓库该物料库存不足，扣减后会出现负库存，后端并不拦截，需要你在审批前人工核对。

## 条码管理

### ① 界面布局

`client/src/pages/WarehousePage/BarcodesPage.tsx`。列表：条码编号、类型、关联对象、仓库、批次、数量、到期日、操作。类型有三种：产品条码 `sku`、物料条码 `material`、货位条码 `location`。支持按类型筛选与增删改。新增弹窗含：编号、类型、关联对象 ID、所属仓库、批次、到期日、数量。

### ② 操作流程

1. 「新增条码」填编号（留空则后端自动生成 `BAR` 前缀）、选类型、填关联对象 ID（`target_id`，即 SKU/物料/货位的主键）、所属仓库、批次、到期日、数量。
2. 条码本身不触碰 `inventory`，它只是一把「扫描即可定位对象」的钥匙。
3. 货位条码可用于「扫货位找库存」，产品/物料条码可用于「扫商品办收发货」。

### ③ 底层逻辑与数据模型

落库表 `barcodes`，关键字段：`code, type, target_id, warehouse_id, location_id, batch_no, expiry_date, qty`。后端 CRUD：`GET/POST/PUT/DELETE /api/barcodes`（`standalone.ts:7172-7193`）。`target_id` 的含义随 `type` 变化：类型 `sku` 时指向 SKU 表主键，`material` 指向物料，`location` 指向货位。

### ④ 源码剖析

新建条码时，`code` 可由前端指定，否则后端兜底：

```ts
// server/standalone.ts:7180-7185
router.post('/barcodes', (req, res) => {
  const id = `bar_${Date.now()}`;
  const code = req.body.code || `BAR${String(Date.now()).slice(-8)}`;
  db.insert('barcodes', { id, code, ...req.body, created_at: new Date().toISOString() });
  res.json({ success: true, id, code });
});
```

查询支持按 `type / target_id / warehouse_id` 过滤（`standalone.ts:7172-7179`），方便 PDA 扫码后反查某条码对应哪个对象、在哪个仓库。

### ⑤ 原理剖析

🔑 条码是「识别层」而非「数量层」：`barcodes` 表与 `inventory` 表解耦，一张条码通过 `target_id` 指向某个 SKU/物料/货位，但扫码动作本身不会改库存。真正改库存的，仍是前面讲过的入出库/盘点/调拨单据。条码的价值在于**把人工输入 SKU 编码，替换成一次扫描**，减少错录、提速收发货。🚀 在鞋服行业，常给每款每色每码生成独立条码，配合 PDA 实现「扫一件、记一件」的精准拣货，这正是 Sowork AI 条码模块的设计意图。

## 管理者视角

从管理角度看，仓储模块解决三件事：

- **实时库存降低缺货与积压**：「库存查询」与 `GET /api/inventory/statistics`（`server/standalone.ts:7038-7050`）可随时给出总量、总值、SKU 数、覆盖仓库数，帮助采购与销售提前决策。🚀 建议把统计接口接入首页四宫格，让管理者一登录就看到全局健康度。
- **盘点保障账实相符**：定期盘点并 `complete`，能把损耗、错发、漏记的差异一次性校正，避免「账上有、架上无」。⚠️ 但如前所述，当前 UI 无法录入盘点明细，需通过 API 补明细才能真校正——这是上线前必须解决的点。
- **条码提效防错**：为物料/成品生成条码后，PDA 扫码即可识别，减少人工录入错误，提升收发货速度。

⚠️ 管理者要盯住两点：一是出入库必须走单据，禁止任何人直接改 `inventory` 的 `qty`；二是盘点/调拨执行期间应明确责任人与时间窗，避免并发流水冲掉差异。

## 注意事项

- ⚠️ **出入库必须走单据**，绝不直接手改 `inventory.qty`。直接改库会破坏流水追溯，且无法审计。
- 🔑 **出库不精确货位**：当前出库接口不接收 `location_id`（`StockOutPage.tsx:66-78`），只扣仓库级库存。若你依赖货位级精度，需自行扩展或确保入库时也按仓库聚合看待。
- ⚠️ **盘点 UI 缺口**：仅用界面建的盘点单无明细，`complete` 不动库存；需经 API 写入 `stock_check_items` 才能校正（见上文「库存盘点 ⑤」）。
- ✅ **条码打印便于 PDA 扫码**：批量生成条码并贴标，是仓储移动作业（收货、上架、拣货）提效的前提。
- ⚠️ **调拨单需先 `approve` 再 `complete`**：未审批执行 `complete` 会被后端拒绝（`server/standalone.ts:7257-7261`）。
- 🔑 **删除都是软删除**：仓库置 `inactive`、货位置 `deleted`，数据仍保留，仅从业务视图隐藏（`standalone.ts:6988-6991`、`7014-7017`）。
- ⚠️ **负库存允许但需警惕**：未入库就出库会产生负数库存（`standalone.ts:7086`），虽系统不拦截，但会掩盖缺货风险，应尽早补正。
- 🔑 **库存行按维度唯一**：同一 `仓库+货位+物料` 只有一行，多次出入库累加而非新增，定位失败会误建/误插行，操作时务必保证维度一致。

## 小结与练习

本篇你学到：Sowork AI 企业智能ERP系统 的仓储管理由「首页概览 + 8 个子页」组成；`inventory` 表按 `仓库 + 货位 + 物料/SKU` 维度记录数量，是唯一真相；所有库存变动都通过单据 + 流水（`inventory_transactions` 及各单据明细）完成，保证可追溯；条码只做识别、不改数量；专用事务接口与 catch-all 通用接口各司其职。

练习（逐个子页，可操作）：

1. **仓库档案**：新建一个仓库「成品仓」（`type=finished`），在其下建一个货位「A-01」。
2. **货位管理**：为该货位填排/架/层（如 排A/架01/层01），理解编码规则。
3. **库存查询**：先空跑一次，记住当前总量，作为后续核验基线。
4. **入库管理**：对「A-01」执行一笔入库（某 SKU，数量 100，批次 B1），到库存查询核验为 100。
5. **出库管理**：执行一笔出库（数量 30），核验变为 70，并到流水查看 `type=out` 记录。
6. **库存盘点**：新建盘点单指向该仓库，并通过 API 写入一条 `stock_check_items`（含 `inventory_id` 与 `diff_qty=-5`），再 `complete`，验证 `inventory` 被校正为 65。
7. **库存调拨**：新建调拨单，把 20 件从「成品仓」调到另一仓库，依次审批、完成，核验两仓一减一加、全局总量不变。
8. **条码管理**：为该 SKU 生成一张产品条码，思考如何贴标用于 PDA 收货扫码。
9. **管理者视角**：调用 `GET /api/inventory/statistics`，对照「库存查询」的四宫格真实值，理解统计口径。
