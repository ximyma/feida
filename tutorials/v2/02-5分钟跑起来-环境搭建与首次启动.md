## 核心概念

先建立四个最小概念，后面不再展开。

- **单端口托管**：生产模式下，前端 SPA 与后端 API 共用同一个端口（默认 3000），没有跨域烦恼。
- **catch-all 业务路由**：后端**不**按 `/api/hr`、`/api/erp` 这类域前缀分流；所有业务数据走统一的通配路由 `/api/:table`。前端按表名拼 URL。
- **零运维数据库**：首次启动自动建库 + 播种（seed），库文件落在 `data/ehr.db`（SQLite）。
- **构建即拷贝**：后端用 `tsc` 编译，但两个手写 `.js` 文件必须手动 `cp` 进 `dist`，否则运行期 `require` 失败。

```
项目根目录（启动前后）
├── package.json          # 脚本与依赖的"唯一真相源"
├── 启动系统.bat          # Windows 一键启动（Python 生成，纯 ASCII）
├── gen_launch_bat.py     # 生成 启动系统.bat 的脚本
├── gen_start_bat.py      # 生成简化版 start.bat 的脚本
├── server/
│   ├── standalone.ts     # 主入口：路由 + 静态托管 + 自动 seed
│   ├── ai-service.js     # 手写 JS，需 cp 进 dist
│   └── code-agent-tools.js
├── dist/                 # 构建产物（npm run build 后才有）
└── data/ehr.db           # 首次启动自动生成
```

## 底层逻辑与数据模型

你不需要打开数据库就能用，但了解两点有助于排错：

- **自动 seed**：`standalone.ts` 在监听端口前，会检查 `data/ehr.db` 是否存在。不存在则建表并写入默认账号（`admin`、`hr_admin` 等）、RBAC 权限点、CMS/商城示例数据。这意味着**第一次启动会慢几秒**，属正常。
- **数据即表**：因为业务走 `/api/:table`，你前端看到的"人事列表""商品列表"，本质都是对某个数据库表的 CRUD。表结构由 ORM 自动建立，无需你手写 SQL 建表。

## 源码剖析

### `package.json` 的 scripts 全解

`package.json` 中 `version` 为 `1.2.0`，关键脚本如下：

| 脚本 | 实际命令 | 作用 |
|------|---------|------|
| `dev` | `concurrently` 并行跑 server + client | 开发模式，前端热更新 |
| `build` | `build:server && build:client` | 全量构建 |
| `build:server` | `tsc -p tsconfig.server.json` + 两次 `cpSync` | 编译后端并拷贝手写 JS |
| `build:client` | `vite build` | 打包前端 SPA |
| `start` | `node dist/server/standalone.js` | 生产启动（**本文终点**） |

**为什么 `build:server` 要额外 `cp` 两个文件？** 看这条脚本：

```json
"build:server": "tsc -p tsconfig.server.json && node -e \"require('fs').cpSync('server/ai-service.js','dist/server/ai-service.js'); require('fs').cpSync('server/code-agent-tools.js','dist/server/code-agent-tools.js')\""
```

`tsc` 只编译 `.ts` 文件。`ai-service.js` 与 `code-agent-tools.js` 是**手写的纯 JavaScript**，不在 TS 编译范围内，因此不会被产出到 `dist/`。但 `standalone.js`（编译后）运行时会 `require` 这两个文件。如果只跑 `tsc` 不拷贝，启动即报 `Cannot find module`。所以构建脚本用一行 `node -e` 把这两个文件同步复制进 `dist/server/`，保证运行期可加载。

### `npm start` 实际做什么

`npm start` 等价于 `node dist/server/standalone.js`。它依次做三件事：

1. 连接/创建 `data/ehr.db`，执行建表与 seed；
2. 注册 API 路由（含 catch-all 的 `/api/:table`）与静态资源托管（把 `dist/client` 当作根路径返回）；
3. 监听 `SERVER_PORT`（默认 3000），打印访问地址。

### `启动系统.bat` 是什么、怎么来的

`启动系统.bat` 是 Windows 用户的一键启动器，逻辑是：

1. 若 `node_modules` 不存在 → 自动 `npm install --legacy-peer-deps`；
2. 若 `dist/server/standalone.js` 不存在 → 自动 `npm run build`；
3. 用 `netstat` + `taskkill` 清理 3000 端口上的旧进程（防端口占用）；
4. 最终 `node dist\server\standalone.js` 启动。

它**不是手写**的，而是由 `gen_launch_bat.py` 生成的。为什么用 Python 生成？因为项目约定 `.bat` 文件必须是**纯 ASCII 编码**（避免中文在非 UTF-8 代码页下乱码导致启动失败）。`gen_start_bat.py` 则生成更简化的 `start.bat`（仅设置 `SERVER_PORT` 默认 3000 再 `node dist/server/standalone.js`）。两个生成器都强制 `encoding="ascii"`，确保产物可被任何 Windows 环境直接双击运行。

## 原理剖析

### 依赖 prune 陷阱：为什么裸 `npm install` 会白屏

这是本文最重要的技术点。Sowork AI 的前端重度依赖 `antd`（已在 `dependencies` 中声明 `^5.29.3`）、`@ant-design/icons`、`echarts` 等。但项目同时使用了 React 19 与一整套 Radix UI 组件，并混入了一些历史遗留的非常规依赖条目，导致 npm 的 **peer 依赖解析** 存在冲突。

当你运行**裸** `npm install` 时，npm 会：

1. 做严格的 peer 依赖校验；
2. 按 `package.json` 为"唯一真相源"重新计算依赖树；
3. 把计算后"多余/冲突"的包**剪除（prune）**掉。

结果就是：本该被前端 import 的 `antd` 等核心包被判定为"多余"而删除 → 前端构建产物里失去组件库 → 浏览器打开后**整页白屏**，控制台报 `Module not found: antd`。

✅ 正确做法是一律加 `--legacy-peer-deps`，跳过严格的 peer 校验，保留完整依赖树：

```bash
npm install --legacy-peer-deps
```

🔑 如果已经手滑用了裸 `npm install` 并白屏了，恢复命令就是重装一次正确的：

```bash
rm -rf node_modules package-lock.json
npm install --legacy-peer-deps
npm run build
```

## 注意事项

⚠️ **必须用 `--legacy-peer-deps`**：裸 `npm install` 会 prune 掉 `antd` 等核心包，导致前端白屏。任何环境下安装依赖都不要省略该参数。

⚠️ **默认密码要改**：`admin/admin123`、`hr_admin/hr123456` 是种子账号，上线前务必在「账户设置」中修改，否则等于把后台大门敞开。

🔑 **端口可用 `SERVER_PORT` 覆盖**：3000 被占用时，不要去改代码，用环境变量即可：

```bash
SERVER_PORT=3400 npm start
```

✅ **SQLite 零运维**：无需安装 MySQL/PostgreSQL，库文件 `data/ehr.db` 首次启动自动生成，备份只需复制该文件。
