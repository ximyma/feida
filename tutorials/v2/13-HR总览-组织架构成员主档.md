# 13 · HR 总览——组织架构成员主档

## 学习目标

你将在本篇掌握四件事。

- 理解 HR 模块在 **Sowork AI 企业智能ERP系统** 中的整体位置：组织 → 员工主档 → 各 HR 子域。
- 看懂 `departments`／`positions`／`ranks`／`employees` 四张核心表的字段与关联关系。
- 学会通过 catch-all 路由 `/api/employees` 对员工主档做增删改查（CRUD）。
- 读懂组织树如何用递归函数渲染，并能照葫芦画瓢改出自己的树。

环境准备：账号 `admin/admin123`，服务端口 `3000`，依赖安装使用 `npm install --legacy-peer-deps`，本篇对应版本 `v1.2.0`。

## 核心概念

HR 模块在系统里呈"金字塔"结构：先搭组织骨架，再填员工血肉，最后长出各类业务子树。

```
Sowork AI 企业智能ERP系统
└─ HR 总览
   ├─ 组织管理  (/organization)   部门 / 岗位 / 职级 / 编制
   ├─ 人事管理  (/personnel)      员工主档 / 合同 / 变动 / 子集
   └─ HR 子域
      ├─ 考勤 (/attendance)   班次 / 排班 / 规则 / 假期
      ├─ 薪酬 (/salary)       工资表 / 配置 / 公式 / 企业缴纳
      ├─ 绩效 (/performance)  KPI / 周期 / 记录 / 等级
      └─ 招聘 / 培训 / 审批 …
```

🔑 一条红线你务必记牢：**系统没有 `/api/hr` 这种按域前缀的路由**。HR 数据全部走 catch-all 路由 `/api/:table`，表名必须在服务端 `ALLOWED` 白名单中（例如 `employees`、`departments`、`positions`、`ranks`）。只有少量命名组（如 `attendance`、`salary`、`schedule`、`leave`、`training`）走独立接口。换句话说，"表名即资源路径"。

## 界面布局与操作流程

登录后，左侧菜单顶层第 2 项是 **组织管理**（`/organization`）；其下方还有 **人事管理** 模块，展开后是子菜单：人事首页、字段自定义、劳动合同、员工变动、提醒管理、人员子集、打印模板、考核评估、人才报表。

组织管理页有 5 个 Tab：

| Tab | 作用 |
| --- | --- |
| 组织架构图 | 左侧部门树 + 右侧部门/岗位/员工穿透详情 |
| 部门管理 | 部门表格，支持划转/合并/调序/删除 |
| 岗位管理 | 岗位表格，挂靠部门 |
| 职级体系 | 职级与薪资区间 |
| 编制管理 | 部门/岗位计划编制 vs 实际人数对比 |

员工主档则在 **人事管理 → 人事首页**（即 `PersonnelPage`）以列表呈现，点姓名弹出详情，含基本、工作、合同、薪资、异动五个子页签。

🚀 操作流程建议：先到组织管理建好部门与岗位，再到人事管理新增员工并选部门/岗位——顺序不能反。

## 底层逻辑与数据模型

四张表的字段如下（关键列）。注意 `employees` 用 `realName` 作为记录名（`_rec_name`），且部分敏感字段带了 `groups` 权限组。

**employees（员工档案）**

| 字段 | 类型 | 说明 | 可见权限组 |
| --- | --- | --- | --- |
| realName | char | 姓名（必填、索引） | 全部 |
| phone | char | 手机号 | 全部 |
| email | char | 邮箱 | 全部 |
| gender | selection | 男/女 | 全部 |
| birthDate / joinDate | date | 出生/入职日期 | 全部 |
| deptId / deptName | char | 所属部门（索引） | 全部 |
| positionId / positionName | char | 岗位 | 全部 |
| rankId | char | 职级 | 全部 |
| status | selection | 在职/离职/试用（默认在职） | 全部 |
| baseSalary | float | 基本工资 | super_admin/hr_admin/hr_staff |
| idCard | char | 身份证号 | super_admin/hr_admin |
| bankAccount | char | 银行账号 | super_admin/hr_admin/finance |

**departments（组织架构）**：`name`（必填）、`parentId`（上级部门，索引）、`sortOrder`（排序）、`managerId`（负责人）。

**positions（岗位管理）**：`name`（必填）、`deptId`（所属部门）、`sortOrder`。

**ranks（职级体系）**：`name`（必填）、`level`（级次）、`minSalary`/`maxSalary`（薪资区间）。

员工主档是其它一切 HR 子表的"外键源头"：考勤按 `employeeId` 记、工资按 `employeeId` 发、合同/异动按 `employeeId` 关联。理解了这一点，你就理解了整个 HR 数据流。

组织树的递归渲染在 `OrganizationPage.tsx:129-150` 构建：

```ts
// OrganizationPage.tsx:129-150
const treeData = useMemo(() => {
  const buildTree = (parentId: string | null, level: number): any[] => {
    return departments
      .filter(d => d.parentId === parentId)
      .sort((a, b) => (a.sortOrder || 0) - (b.sortOrder || 0))
      .map(d => ({ ...d, level, children: buildTree(d.id, level + 1) }));
  };
  return [{ id: 'company', name: companyName, level: 0, parentId: null,
            children: buildTree(null, 1) }];
}, [departments, employees, companyName]);
```

## 原理剖析

为什么是"catch-all + 白名单"这种设计？因为它用最少的代码覆盖了几十张表。新增一张 HR 表，只需把表名加进 `ALLOWED`，前端即可直接 `fetch('/api/新表')`，无需为每个域写一套路由。白名单 `ALLOWED` 是安全闸门：不在名单里的表名一律拒绝，避免任意表被越权读写。

字段上的 `groups` 是行内字段级权限。例如 `idCard` 只有 `super_admin`/`hr_admin` 可见，普通 `hr_staff` 在接口返回里会被自动剥离。这解释了为什么同一条员工记录，不同角色看到的字段不一样。

组织树用"父 id 自引用"建模（`parentId` 指向同表 `id`），配合递归 `buildTree` 拼出多层级树。这种结构支持无限层级，且划转/合并只需改 `parentId`，整棵子树自动跟随。

## 管理者视角

🚀 组织架构成员主档的最大价值，是成为全公司人资数据的**单一数据源**。过去 hr 用 Excel 维护花名册，考勤用另一份、工资又一份，三处对不齐是常态。现在：部门、岗位、员工一处录入，考勤、薪酬、绩效全部引用同一份主档，再也不会"人名对不上工号"。

入转调离的数据连贯性因此成立：员工入职即在主档建卡 → 考勤自动认领 `employeeId` → 发薪按同一 `employeeId` 出工资 → 离职只改 `status='inactive'`，历史合同与异动记录全部保留可追溯。管理者看到的不再是割裂的表格，而是一条完整的人事生命周期。

## 注意事项

⚠️ **员工主档是外键源头**：删除员工会波及考勤、工资、合同等所有关联表，删除前务必确认无在途业务。

🔑 **先建部门/职位，再建员工**：`employees` 的 `deptId`/`positionId` 指向已存在的组织记录，组织没搭好就录员工，会出现"无部门人员"。

✅ **树形组织用递归渲染**：`buildTree` 与 `renderTreeNode` 互为递归，改展示逻辑时务必保持父子调用对称，否则会出现层级错乱或无限循环。

🚀 **敏感字段受 groups 限制**：`baseSalary`/`idCard`/`bankAccount` 对低权限角色不可见，调试接口时"看不到字段"通常是权限所致，而非数据缺失。

## 小结与练习

本篇你学到了：HR 模块以"组织 → 员工主档 → 子域"为骨架；数据全部走 `/api/:table` 且表名在 `ALLOWED` 白名单；`employees` 等四张表字段与权限组；组织树靠递归构建与渲染。

1. 用 `curl http://localhost:3000/api/departments` 验证白名单放行，再试一个不在 `ALLOWED` 的表名看是否被拒。

3. 给 `employees` 模型新增一个 `nickName` 字段（带 `label`），重启后在人事首页新增员工表单中应能出现该字段——验证"模型即 UI"的联动。
4. 以 `hr_staff` 角色登录，确认员工详情里 `idCard` 不展示，理解 `groups` 字段级权限。
