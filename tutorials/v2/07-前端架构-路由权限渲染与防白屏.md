# 07 · 前端架构：路由、权限渲染与防白屏

本文是 **Sowork AI 企业智能ERP系统**（以下简称「本系统」）v1.2.0 技术教程的第七篇。你将看到前端的「路由表—权限点—菜单过滤—按钮控制」是如何一条链路串起来的，以及为什么我们不惜用双层 `ErrorBoundary` 也要把「白屏」这个顽疾摁死。

运行环境提示：服务端默认端口 `3000`，默认账号 `admin / admin123`；前端依赖请使用 `npm install --legacy-peer-deps` 安装（antd v5 与部分依赖存在 peer 冲突，必须带该参数）。

## 学习目标

- 读懂 `index.tsx → app.tsx → Layout.tsx` 的路由与 Providers 装配链。
- 掌握「前端权限」的双层模型：本地即时计算 + 后端 `/api/rbac/resolve` 权威合并。
- 理解 `usePermission` 的 `can()` / `hasPermission()` 如何控制菜单与按钮显隐。
- 搞清「防白屏」的来龙去脉，以及 `treeData` 与 `<TreeNode>` 的取舍。
- 从管理者视角，确认「前端隐藏 ≠ 安全」这一生死线。

## 核心概念

你只要记住三个词：**路由表、权限点、错误边界**。

- **路由表**：`react-router-dom` 的 `<Routes>`，按路径把 URL 映射到页面组件。
- **权限点（permission point）**：形如 `p_personnel`、`cms:article:delete` 的字符串。模块级用 `p_*`；细粒度用 `模块:资源:动作`。
- **错误边界（ErrorBoundary）**：React 的"兜底组件"，兜住子树的渲染异常，避免整页白屏。

整个前端权限流是一条单向管道：

```
登录 → localStorage 写入 user(含 roleIds)
   ↓
usePermission 本地 computePermissions(roleIds)   ← 秒出，防闪烁
   ↓
POST /api/rbac/resolve 后端权威合并              ← 兜底权威源
   ↓
Layout 用 hasModule() 过滤侧边菜单
   ↓
业务页面用 can('cms:article:delete') 控制按钮显隐
```

🔑 关键认知：**前端这套渲染只是"体验层"，真正的拦截永远在后端 `resolve`**。前端隐藏按钮，只是让没有权限的人"看不见"，而不是"做不到"。

## 界面布局与操作流程

启动后，浏览器打开 `http://localhost:3000`，你会进入带左侧菜单的「壳」布局。菜单是否出现，取决于当前账号的角色。

典型菜单树（节选）如下：

```
首页仪表盘           (始终可见)
├─ 组织管理          (p_org)
├─ 人事管理          (p_personnel)
│   ├─ 人事首页
│   ├─ 劳动合同
│   └─ 考核评估
├─ 薪酬管理          (p_salary)
├─ 系统管理          (仅 super_admin / tech_admin 可见)
│   ├─ 用户管理
│   └─ 角色权限
└─ 更多应用          (始终可见)
```

用 `admin/admin123` 登录是超级管理员，能看到全部模块；换成 `hr_staff` 账号，则只能看到人事、考勤等少数模块——这就是 `Layout` 在渲染时按权限过滤的结果。

点进任意页面，新增/删除/发布等按钮的有无，则由该页面内部的 `can(...)` 决定。你「看不到」并非「不能调接口」，这是后面要反复强调的双层模型。

## 底层逻辑与数据模型

权限的"真相源"分两极：

| 层级 | 来源 | 作用 | 可靠性 |
|------|------|------|--------|
| 本地计算 | `computePermissions(roleIds)` | 首屏秒出，防按钮闪烁 | 可被伪造，仅体验 |
| 后端权威 | `POST /api/rbac/resolve` | 合并细粒度权限点 | 唯一可信拦截 |

角色到权限的映射在前后端各存一份、保持一致：

```ts
// usePermission.tsx:57-69  ROLE_PERMISSIONS
super_admin:  [...MODULE_KEYS, ...ALL_PERMISSION_POINTS],
hr_admin:     [p_personnel, p_attendance, ... 'cms:article:edit', ...],
hr_staff:     ['p_personnel','p_attendance','p_recruitment','cms:article:view', ...],
employee:     []
```

注意路由事实：**本系统没有按域前缀的 `/api/hr`**。所有业务表走 catch-all 的 `/api/:table`（如 `/api/employees`、`/api/contracts`），权限校验统一落在后端中间件对这张表的读写拦截上。这也是为什么前端只负责"显隐"，后端才负责"放行/拒绝"。

## 源码剖析

**1) 入口装配 `index.tsx`**

`BrowserRouter` 包在最外层，再叠 `AppConfigProvider`、`ShopProvider`，最后才是 `App`：

```tsx
// index.tsx:3,11-17
import { BrowserRouter } from 'react-router-dom';
<BrowserRouter>
  <AppConfigProvider>
    <ShopProvider>
      <App />
    </ShopProvider>
  </AppConfigProvider>
</BrowserRouter>
```

**2) 路由表与顶层错误边界 `app.tsx`**

整个 `<Routes>` 被一个顶层 `ErrorBoundary` 包裹（行 172 开、行 391 闭）。未登录访问受保护页，由 `AuthGuard` 踢回登录页：

```tsx
// app.tsx:163-167  登录守卫
function AuthGuard({ children }) {
  const currentUser = sessionStorage.getItem('__current_user');
  if (!currentUser) return <Navigate to="/login" replace />;
  return <>{children}</>;
}
```

受保护路由套一层 `AuthGuard<Layout/>`，内部才是业务页（行 198-387）。注意大量子路由又各自包了 `ErrorBoundary`，这就是「双层边界」：

```tsx
// app.tsx:208  单个页面级边界
<Route path="/personnel/subset" element={
  <ErrorBoundary fallbackName="员工子集"><EmployeeSubsetPage /></ErrorBoundary>
} />
```

**3) 菜单按角色过滤 `Layout.tsx`**

模块路径 → 权限码映射（行 323-349），以及角色 → 权限集合（行 351-358）。运行时把当前用户的 `roleIds` 折算成权限数组 `userPerms`（行 360-391），再用 `hasModule()` 决定菜单是否渲染：

```tsx
// Layout.tsx:393-397
const hasModule = (path: string): boolean => {
  if (path === '/apps') return true;            // 更多应用始终可见
  const perm = MODULE_PERM[path];
  return perm ? userPerms.includes(perm) : true;
};
```

`/system` 模块只对管理员开放，用的是独立判断而非权限码：

```tsx
// Layout.tsx:399,577
const isAdmin = currentUser.userType === 'super_admin' || currentUser.userType === 'tech_admin';
...
{isAdmin && renderNewModuleNav('/system')}
```

**4) `usePermission` 的本地 + 后端合并**

`useEffect` 里先本地 `computePermissions` 同步算出首批权限（防闪烁），再异步请求后端权威结果并"并集合并"：

```tsx
// usePermission.tsx:157-166
fetch('/api/rbac/resolve', { method:'POST',
  headers:{'Content-Type':'application/json'},
  body: JSON.stringify({ roleIds })
}).then(r => r.ok ? r.json() : null).then(d => {
  if (d && Array.isArray(d.permissions)) {
    setPermissions(prev => Array.from(new Set([...prev, ...d.permissions])));
    if (Array.isArray(d.siteScope) && d.siteScope.length) setSiteScope(d.siteScope);
  }
}).catch(() => { /* 离线回退本地 */ });
```

对外暴露的判定函数（行 175-204）：

```tsx
// usePermission.tsx:175-184,197,200-204
const hasPermission = (permission) => {
  if (!user) return false;
  if (user.roleIds?.includes('role_super_admin')) return true;  // 超管直通
  return permissions.includes(permission);
};
const can = hasPermission;                       // 按钮显隐语义别名
const canAccessSite = (siteCode) =>             // 站点范围判定，'*' 全站
  siteScope.includes('*') || siteScope.includes(siteCode);
```

**5) 原生 fetch 封装 `useAPI.ts`**

本系统**不依赖 axios**，用原生 `fetch` 包了一层 `apiRequest`：

```ts
// useAPI.ts:3,6-13
const API_BASE = '/api';
async function apiRequest<T>(endpoint, options?) {
  const response = await fetch(`${API_BASE}${endpoint}`, {
    headers: { 'Content-Type': 'application/json', ...options?.headers }, ...options,
  });
  if (!response.ok) throw new Error(`API Error: ${response.status}`);
  return response.json();
}
// useAPI.ts:165-170  业务常用四个动词
export const api = {
  get:    (e)       => apiRequest(e),
  post:   (e, body) => apiRequest(e, { method:'POST',   body: JSON.stringify(body) }),
  put:    (e, body) => apiRequest(e, { method:'PUT',    body: JSON.stringify(body) }),
  delete: (e)       => apiRequest(e, { method:'DELETE' }),
};
```

**6) 双层错误边界 `ErrorBoundary.tsx`**

类组件，捕获子树异常并降级为 antd `Result`：

```tsx
// ErrorBoundary.tsx:15-22
static getDerivedStateFromError(error) { return { hasError: true, error }; }
componentDidCatch(error, errorInfo) {
  console.error('[ErrorBoundary]', this.props.fallbackName || 'page', error, errorInfo);
}
// ErrorBoundary.tsx:24-55  render() 出错时返回 Result 而非崩溃
```

## 原理剖析

**为什么是双层边界？** 顶层边界（app.tsx:172）兜住"路由级"崩溃；页面级边界（各子路由）兜住"单页"崩溃。后者让「员工子集页炸了」只显示该页的报错卡片，而左侧菜单、其他功能照常可用——而不是整个后台白屏。

**为什么要防白屏？** 在 antd v5 下，`Tree`/`Tabs` 等组件若误用子节点写法（如 `<TreeNode>` 直接嵌套、`TabPane` 不当组合），渲染期抛错会沿 React 组件树向上冒泡。没有边界时，异常一路炸到根，整页白屏、控制台一堆红、用户只能强刷。我们用 `treeData` 属性式写法替代 `<TreeNode>` 子节点，再辅以 `ErrorBoundary`，把"冒泡爆炸"变成"局部降级"。

🚀 一句话：**treeData 是写法层面的规避，ErrorBoundary 是兜底层面的保险**，两者叠加才稳妥。

**本地 + 后端为什么并集？** 本地计算保证「首屏不闪烁」（否则按钮先全显再消失，体验很差）；后端 `resolve` 保证「细粒度权限点（如 `cms:article:delete`）的权威来源」。两者并集，既快又准。

## 管理者视角

作为管理者，你最该盯住一句话：**权限在前后端必须同源**。本系统中，前端 `ROLE_PERMISSIONS`（usePermission.tsx:57）与后端 RBAC 角色表是同一套语义、同一份目录（`PERMISSION_CATALOG` 与后端 `RBAC_CATALOG` 对应）。

这带来什么好处？当某员工被误操作赋予了 `hr_admin`，他「看到」的按钮会变多，但**他真正能写库的动作，仍由后端 `resolve` 与中间件拦截决定**。也就是说，即使有人用 DevTools 把隐藏按钮强行显示出来、伪造请求，后端也会因缺少对应权限点而拒绝——这就把"越权操作"事故锁死在服务器端。

反之，如果你们自行二开时把拦截逻辑只写在前端（比如仅靠 `can()` 隐藏按钮、后端不校验），那就等于把公司数据的大门钥匙挂在门把手上。✅ 永远记住：前端是"体验开关"，后端才是"保险柜"。

## 注意事项

⚠️ **前端隐藏按钮只是体验，真正拦截靠后端 `resolve`**。任何涉及资金、人事、薪酬的写操作，二开时务必确认后端有对应校验，不要只在前端 `can()` 处做文章。

🔑 **CMS / 栏目页必须用 `treeData` 属性，不要用 `<TreeNode>` 子节点写法**。这是 antd v5 下引发整页白屏的主要元凶之一，配合 `ErrorBoundary` 才能彻底规避。

✅ **双层 `ErrorBoundary`（顶层 + 页面级）是防白屏的硬保障**，新增页面时请沿用现有写法，给每个易崩页面套上 `<ErrorBoundary fallbackName="...">`。

⚠️ 路由没有 `/api/hr` 这类域前缀；业务统一走 catch-all `/api/:table`，权限校验应挂在表级中间件，而不是按 URL 前缀分流。

🚀 安装依赖务必 `npm install --legacy-peer-deps`，否则 antd v5 的 peer 冲突会导致构建失败。

## 小结与练习

你已掌握：路由表由 `app.tsx` 集中声明并套 `AuthGuard`；`Layout` 用 `MODULE_PERM` + 角色映射过滤菜单；`usePermission` 通过「本地计算 + `/api/rbac/resolve` 合并」产出权限集合；`can()` 控制按钮显隐；`ErrorBoundary` 双层包裹防止白屏。

动手练习：

1. 用 `hr_staff` 账号登录，观察左侧菜单与 `admin` 的差异，对应到 `Layout.tsx` 的 `USER_ROLE_PERMS`（行 351-358）。
2. 在某页面用 `can('cms:article:delete')` 包裹一个按钮，登录不同角色查看显隐变化。
3. 故意在 `CMSAdminPage` 用 `<TreeNode>` 写法，看是否触发白屏；改为 `treeData` 后复测。
4. 打开浏览器 DevTools，把被 `can()` 隐藏的按钮 DOM 强制显示并点击，确认后端仍拒绝——验证"双层模型"。
