# 21 · 流程审批——请假/加班/离职审批流引擎

本篇带你从使用者视角和代码视角，彻底搞懂 **Sowork AI 企业智能ERP系统**（v1.2.0）里的流程审批模块：请假、加班、离职三类申请如何提交、如何流转、如何归档。你不仅能照着界面操作，还能读懂底层那套「模板 → 实例 → 节点 → 审批人」的 workflow 引擎。

## 学习目标

学完你能做到：

- 用 `admin/admin123` 在 `http://localhost:3000` 提交一份请假/加班/离职申请。
- 在「审批记录」里看清一份申请经过哪些节点、谁批的、谁卡的。
- 在源码里定位 `workflow-engine.ts` 的 `startWorkflow`、`processNodeAction`，讲清状态机如何推进。
- 理解为什么「审批流模板要先配」「节点审批人缺失会卡单」。

## 核心概念

审批的本质是一台状态机：一份申请从 `running` 出发，沿节点逐个 `pending → approved`，走到 `end` 变 `completed`；任意节点 `reject` 则整体 `rejected`。

| 概念 | 含义 | 对应表 |
| --- | --- | --- |
| 流程模板 | 画好的审批路线图 | `workflow_definitions` |
| 流程实例 | 某次具体申请 | `workflow_instances` |
| 节点实例 | 路线图上的某一站 | `workflow_instance_nodes` |
| 审批人 | 该站由谁签 | `workflow_node_assignees` |
| 审批意见 | 每站的留言 | `workflow_comments` |
| 老三样业务 | 请假/加班/离职/记录 | `leave_records` 等 + `approval_*` |

🔑 关键认知：业务表（`leave_records`、`overtime_records`、`resignation_records`）存「申请内容」，而 `workflow_*` 系列表存「审批过程」。两者通过 `businessId` 关联。

## 界面布局与操作流程

左侧菜单「流程审批」(`/approval`) 下挂四个子页（`client/src/components/Layout.tsx:105-111`）：

```
流程审批 (/approval)
├── 审批首页      /approval
├── 请假申请      /approval/leave
├── 加班申请      /approval/overtime
├── 离职申请      /approval/resignation
└── 审批记录      /approval/record
```

以请假为例，你在 `LeavePage` 点击「新建」，填写类型/起止日/天数/事由，提交后状态进入 `pending`（`LeavePage.tsx:28` 的 `STATUS_MAP`）。

🚀 流程：填单 → 系统按模板 `wf_leave` 自动生成实例 → 直属主管 `pending` → 主管同意 → HR `pending` → HR 同意 → `completed`（请假落考勤）。

审批人侧：在「审批记录」或待办里看到自己的任务，点「同意/拒绝/委托」。拒绝即整体结束；委托把该节点转给别人继续 `pending`。

## 底层逻辑与数据模型

引擎启动时预置了三个默认模板（`server/workflow-engine.ts:103` `seedDefaultWorkflows`）：

```
wf_leave      请假：start → 直属主管 → HR → end
wf_overtime   加班：start → 直属主管 → end
wf_resignation 离职：start → 主管 → HR → 部门负责人 → end
```

数据落库靠 catch-all 路由。注意 **路由事实**：系统没有 `/api/hr` 这种按域前缀的接口，所有表数据走 `/api/:table`。所以请假页直接 `fetch('/api/leave_records?...')`（`LeavePage.tsx:90`）。

但这些表名受保护：`RESERVED_ROUTES` 含 `'workflows','approval','workflow'`（`server/standalone.ts:4067`），catch-all 会让位给专用路由（`/workflows` 在 `standalone.ts:4405` 起）。

节点流转的核心表关系：

```
workflow_definitions (nodes/edges JSON)
        │ 实例化
        ▼
workflow_instances (status: running/completed/rejected/cancelled)
        │ 每站一张
        ▼
workflow_instance_nodes (status: pending/approved/rejected)
        │ 一人或多审批人
        ▼
workflow_node_assignees (谁该签)
```

## 源码剖析

引擎在 `server/workflow-engine.ts`，不混在 `standalone.ts` 里。提交申请时调用 `startWorkflow`（`workflow-engine.ts:296`）：它查模板、解析 `nodes/edges`、找到 `start` 的下一站，建实例并把 `currentNodeId` 指向第一业务节点，再 `activateNode` 激活。

```ts
// workflow-engine.ts:330 起，创建实例
const instance = {
  id: instanceId, definitionId: params.definitionId,
  businessId: params.businessId, businessType: params.businessType,
  title: params.title, applicantId: params.applicantId,
  status: 'running', currentNodeId: firstNodeId,   // ← 关键：当前停在第一站
  formData: JSON.stringify(params.formData || {}),
  startedAt: now, completedAt: null,
};
```

审批人点击「同意」时进入 `processNodeAction`（`workflow-engine.ts:454`）。拒绝直接结束：

```ts
// workflow-engine.ts:485
if (action === 'reject') {
  db.query(`UPDATE workflow_instances SET status='rejected', completedAt=? WHERE id=?`,
    [now, instanceId]);
  return { success: true, status: 'rejected', message: '审批已拒绝，流程结束' };
}
```

同意则顺着 `edges` 找下一站：`activateNode` 把目标节点置 `pending`（`workflow-engine.ts:366`；普通顺序节点走 `workflow-engine.ts:551` 的第一条边），网关分支走 `evaluateConditions`（`workflow-engine.ts:558` 用 `new Function` 安全求值 `formData.amount < 5000`）。

审批人如何解析？`resolveAssignees`（`workflow-engine.ts:413`）：`role` 类型按 `dept_manager→role_dept_manager` 映射查 `users` 表；查不到则回落到 `系统管理员`。这正是「卡单」的伏笔。

## 原理剖析

把一份申请想成一台有限状态机：

```
[提交] → running
  └─ node.pending ──approve──▶ next node.pending
                     └─reject──▶ rejected（终态）
  └─ 走到 end / 无后续边 ──▶ completed（终态）
  └─ 申请人撤回 ──▶ cancelled（终态）
```

引擎靠三张表驱动：`instances.status` 是全局态，`instance_nodes.status` 是各站态，`edges` 是转移条件。它支持三种高级能力：

- 排他网关：金额 `<5000` 走小额审批，`>=5000` 加财务并行（`workflow-engine.ts:522`）。
- 并行网关：多节点同时 `pending`，全部 `approved` 才汇聚（`workflow-engine.ts:530`）。
- 委托/加签：节点 `delegate` 后重置为 `pending` 换人（`workflow-engine.ts:491`）。

✅ 全程每次节点变更都写 `workflow_instance_nodes` 和 `workflow_comments`，天然留痕，可审计。

## 管理者视角

为什么要把纸质签字搬上线？

- **权责清晰**：每一步谁批、何时批、批语是什么，落库可查，扯皮时有据。
- **可审计**：审批记录全留痕，HR 与合规随时回溯。
- **数据联动**：请假通过 → 写入考勤假期额度；离职通过 → 触发人事档案状态变更、停权。审批不再是孤岛，而是业务闭环的开关。

对管理者而言，预置的三套模板开箱即用；若公司层级不同，可在「审批流设计器」(`WorkflowDesigner.tsx`) 拖拽节点改路线，改完即生效、版本自增（`standalone.ts:4137` `newVersion = existing.version + 1`）。

## 注意事项

⚠️ **审批流模板要先配**：没有对应 `workflow_definitions`，`startWorkflow` 直接抛「工作流定义不存在」（`workflow-engine.ts:308`）。上线前先确认模板已 `published`。

🔑 **节点审批人缺失会卡单**：`resolveAssignees` 查不到角色对应用户时回落到「系统管理员」，若管理员也不处理，该节点永远 `pending`，整条流停在中途。配置模板时务必确认 `dept_manager`/`hr_admin` 等角色有人。

✅ **审批记录全留痕**：节点与意见表只增不删（`DELETE` 实例受「运行中不可删」保护，`standalone.ts:4512`）。不要试图手动清库来掩盖审批，留痕是合规资产。

其他：删除模板前需确认无 `running` 实例；并行网关要等齐全部分支才前进，别漏配某条边的审批人。

## 小结与练习

你已理解：审批 = 业务表（申请内容）+ `workflow_*`（过程）+ 状态机（流转）。引擎入口是 `startWorkflow`/`processNodeAction`/`advanceToNode`，路由靠 `/api/:table` catch-all 加 `/workflows` 专用接口。

练习：

1. 用 `admin/admin123` 提交一份病假申请，跟踪「审批记录」里 `running → completed` 的节点变化。
2. 在 `workflow-engine.ts` 给 `wf_overtime` 加一个 HR 节点，重启服务（依赖安装用 `npm install --legacy-peer-deps`）观察新模板生效。
3. 在 `resolveAssignees` 故意留空某角色，提交申请，观察是否回落到「系统管理员」并定位卡单根因。
