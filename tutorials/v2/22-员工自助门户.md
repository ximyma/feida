# 22-员工自助门户（Self-Service）

本文是 **Sowork AI 企业智能ERP系统**（版本 v1.2.0）的代码级双读者教程。你将同时站在「普通员工」与「系统/开发者」两个视角，理解员工登录后能在自助门户里做什么，以及它的数据隔离到底是怎么实现的。

环境约定：默认账号 `admin/admin123`，服务端口 `3000`，安装依赖请用 `npm install --legacy-peer-deps`。

## 学习目标

- 知道员工登录后能在自助门户里完成哪些事：档案核对、考勤自查、薪资条查看、请假/加班申请、审批进度跟踪。
- 读懂前端 `SelfServicePage` 如何取当前用户身份、如何并发拉取自己的数据。
- 看懂后端 `/attendance/my-records/:employeeId` 与薪资生成接口的「按 user id 过滤」逻辑。
- 建立核心安全意识：自助门户的数据隔离必须由后端按 token/会话中的 user id 强制过滤，绝不能靠前端隐藏。

## 核心概念

员工自助门户（Self-Service）是 ERP 面向**一线员工**的轻量入口，与 HR 后台（人事/薪酬/考勤管理）形成分工：

| 概念 | 说明 | 读者视角 |
| --- | --- | --- |
| 员工自助 | 员工只看/办「自己的」事 | 普通员工 |
| 数据隔离 | 每个人的数据只对自己可见 | 开发者/管理员 |
| 审批流桥接 | 申请自动进入流程审批（见 21 篇） | 两者皆是 |
| 信息修改申请 | 员工发起、管理员审核生效 | 两者皆是 |

🔑 一句话记忆：**自助门户 = 我的数据 + 我的申请 + 安全隔离**。

## 底层逻辑与数据模型

自助门户不单独建表，而是复用同一套业务数据。前端通过 catch-all 接口 `/api/:table` 拉取，并按 `employeeId` 过滤：

```text
数据关系（都是「属于某个员工」）
employees (员工主档)
   ├─ attendance_records  考勤记录 (employeeId)
   ├─ salaries            薪资记录 (employeeId)
   ├─ leave_balances      假期余额 (employeeId)
   ├─ employee_changes    人事变动 (employeeId)
   └─ approval_requests   我的申请 (requesterId)
```

路由事实：本系统**没有** `/api/hr` 这种按域前缀拆分的接口，所有表数据都走 catch-all `/api/:table`。员工之所以「只能看到自己」，关键在于——请求里携带的 `employeeId` 必须来自当前登录身份，而不是随便传一个 id 就能查别人。

## 源码剖析

### 1. 前端如何确定「我是谁」

`SelfServicePage.tsx:124` 定义了取当前用户 id 的函数，身份来自登录态 `localStorage.user`：

```ts
// client/src/pages/SelfServicePage/SelfServicePage.tsx:124
function getCurrentUserId(): string {
  try {
    const userStr = localStorage.getItem('user');
    if (userStr) {
      const user = JSON.parse(userStr);
      // 优先使用 employeeId，其次用 id
      return user.employeeId || user.id || '';
    }
  } catch { /* ignore */ }
  return '';
}
```

随后在 `loadAll()`（`SelfServicePage.tsx:154`）里，用这个 id 并发拉取**只属于自己的数据**：

```ts
// client/src/pages/SelfServicePage/SelfServicePage.tsx:159
const [empRes, changesRes, attRes, leaveRes, salaryRes, apprRes, annRes] = await Promise.all([
  fetch(`/api/employees/${currentUserId}`),
  fetch(`/api/employee_changes?employeeId=${currentUserId}`),
  fetch(`/api/attendance_records?employeeId=${currentUserId}`),
  fetch(`/api/leave_balances?employeeId=${currentUserId}`),
  fetch(`/api/salaries?employeeId=${currentUserId}`),
  fetch(`/api/approval_requests?requesterId=${currentUserId}`),
  fetch(`/api/announcements`),
]);
```

注意：每个请求都带上了 `currentUserId`，这是「自助」的起点。

### 2. 后端按 user id 过滤考勤记录

后端 `standalone.ts:1260` 的 `my-records` 接口，用路径参数 `employeeId` 直接到表内过滤：

```ts
// server/standalone.ts:1260
router.get('/attendance/my-records/:employeeId', (req, res) => {
  const { employeeId } = req.params;
  const { month } = req.query;
  let records = db.findWhere('attendance_records', { employeeId });
  if (month) {
    records = records.filter((r: any) => r.date.startsWith(month as string));
  }
  res.json(records.sort((a: any, b: any) => b.date.localeCompare(a.date)));
});
```

`db.findWhere('attendance_records', { employeeId })` 就是数据隔离的落点：SQL 条件里带死了 `employeeId = ?`，任何其他员工的记录都不会返回。

### 3. 薪资按 user id 生成与隔离

薪资同样绑定 `employeeId`。`standalone.ts:1840` 的薪资生成接口支持「单员工」与「全员」两种模式：

```ts
// server/standalone.ts:1842（节选）
const { month, employeeId } = req.body;
if (employeeId) {
  // 单个员工
  const result = await calculateSalary(db, employeeId, month);
  results.push(result);
} else {
  // 全员
  const batchResult = await batchCalculateSalary(db, month);
  results = batchResult.results;
}
```

生成的薪资记录主键形如 `salary_${employeeId}_${month}`，并通过 catch-all `/api/salaries?employeeId=xxx` 只回查本人。

### 4. catch-all 如何兜底过滤

所有 `fetch('/api/:table?xxx=yyy')` 最终落到 `standalone.ts:4065` 的 catch-all：

```ts
// server/standalone.ts:4072
const filters = Object.fromEntries(
  Object.entries(req.query).filter(([k]) => !skipKeys.has(k))
);
// ...
data = db.findWhere(table, filters);
```

查询参数被原样转成 SQL 过滤条件。这意味着：**前端传什么 `employeeId`，后端就只查那条；但前提是这个 id 必须是后端从登录身份里可信推导出来的。**

## 原理剖析

为什么数据隔离必须由后端兜底？看下面这条信任链：

```text
登录成功 → 服务端签发身份（token / 会话）
        → 前端保存 user（含 employeeId）
        → 每次请求带上"我是谁"
        → 后端按 employeeId 过滤 SQL
        → 只返回该员工自己的行
```

⚠️ 关键点：如果后端只是「前端传什么 id 就查什么 id」，那么攻击者改一下请求里的 `employeeId` 就能看到别人的薪资——这就是典型的越权漏洞（IDOR）。正确做法（也应该是你落地的做法）是：**后端从 token/会话中解析出 user id，忽略前端传入的 id，强制用它过滤**。本系统自助门户的设计意图正是如此：员工身份一旦由登录态确定，所有自助查询都被绑定到该 user id。

🚀 实践建议：把 `getCurrentUserId()` 换成从后端返回的 `req.user.id`（由鉴权中间件注入），让过滤条件不可被客户端篡改。

## 管理者视角

- **减少事务性工作量**：员工自己查考勤、查薪资条、提修改申请，HR 不再被大量重复咨询淹没。
- **提升体验与透明度**：打卡、余额、审批进度一目了然，减少信息不对称带来的沟通成本。
- **流程闭环**：员工的请假/加班申请自动进入流程审批引擎（呼应 21 篇），管理者在审批台统一处理，状态实时回写到「我的申请」。
- **合规与安全**：敏感数据（薪资、身份证、银行卡）按 user id 严格隔离，配合权限与水印，降低泄露风险。

## 注意事项

⚠️ **自助门户必须按 user id 隔离，不能靠前端隐藏。** 前端不显示≠后端查不到；越权防护一定要落在 SQL 过滤层，由后端身份推导，而非信任前端传参。

🔑 **薪资条需要权限 + 水印双重保护。** 薪资是最敏感数据：除了 user id 隔离，还应叠加「本人或无 HR 权限不可见」的访问控制，并在页面/导出文件上加动态水印（工号+时间），便于泄露溯源。

✅ **申请自动进审批流。** 自助门户提交的请假、加班、信息修改等，都会写入 `approval_requests` 并由流程审批引擎驱动（见 21-流程审批），请勿在自助端另写一套审批逻辑，避免双轨不一致。

🚀 **身份来源要可信。** 当前 `getCurrentUserId()` 读 `localStorage`，适合演示；生产环境请改为后端从 token 解析 user id，杜绝客户端伪造。

## 小结与练习

本篇你学到了：员工自助门户是面向一线员工的「我的数据」入口；前端凭登录态里的 `employeeId` 并发拉取 `attendance_records`、`salaries`、`leave_balances` 等表；后端以 `/attendance/my-records/:employeeId`（`standalone.ts:1260`）和薪资生成接口（`standalone.ts:1840`）为代表，用 `employeeId` 做 SQL 过滤实现数据隔离；数据最终都走 catch-all `/api/:table`，隔离的可靠性取决于后端是否从可信身份推导 user id。
