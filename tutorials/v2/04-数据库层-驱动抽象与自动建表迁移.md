# 04｜数据库层：驱动抽象与自动建表迁移

本篇带你深入 **Sowork AI 企业智能ERP系统**（产品首现全称，下文简称 Sowork AI）的数据库层。你将看到 v1.2.0 版本如何用一层统一的驱动接口屏蔽 SQLite 与 PostgreSQL（PG）的差异，以及「模型即表」是如何做到注册即建表、加字段即自动迁移的。

## 学习目标

读完本篇，你应该能够：

- 说清 `IDatabaseDriver` 接口为什么存在，它屏蔽了什么。
- 看懂 `db-factory.ts` 如何依据环境变量在 SQLite / PG 之间切换。
- 解释「模型注册即建表、新字段自动 ALTER」的底层机制。
- 掌握表字段的存储约定（`id TEXT PRIMARY KEY`、JSON 存 TEXT、时间戳 TEXT）。
- 作为管理者，能为「零运维 SQLite」与「上云 PG」做出选型判断。

## 核心概念

数据库层由两个支柱构成：

1. **驱动抽象**：用 `IDatabaseDriver` 接口统一两种数据库的能力，业务代码只依赖接口，不关心底层是 SQLite 还是 PG。
2. **自动建表/迁移**：用 ORM 模型的 `_fields` 描述结构，启动或注册时执行 `CREATE TABLE IF NOT EXISTS` 与 `ALTER TABLE ADD COLUMN`。

二者关系可用一棵树表示：

```
业务/ORM
   │
   ▼
IDatabaseDriver  (统一接口)
   ├── SQLiteDriver  (better-sqlite3, 同步)   ← 默认
   └── PgDriver      (pg Pool, 异步)          ← 上云
```

🔑 关键认知：业务层永远只调用 `db.query / insert / update / delete`，至于底下是文件还是服务端，它不关心。

## 界面布局与操作流程

数据库层没有独立 UI，但它贯穿你每次操作。其「界面」就是启动日志与数据目录：

```
启动 Sowork AI 后控制台输出
─────────────────────────────
[DB] SQLite: data/ehr.db          ← 默认驱动
[Database] SQLite at .../data/ehr.db
```

当你想切换到 PG，只需设置环境变量并重启（端口仍是 3000，默认账号 `admin/admin123`）：

```bash
# 方式一：连接串
DATABASE_URL=postgres://user:pass@localhost:5432/sowork npm run start

# 方式二：分拆变量
DB_DRIVER=postgres DB_HOST=localhost DB_PORT=5432 DB_NAME=sowork DB_USER=sowork DB_PASS=xxx npm run start
```

安装依赖时请使用：

```bash
npm install --legacy-peer-deps
```

启动后，所有以 `/api/:table` 为前缀的 catch-all 请求（注意：系统中**没有** `/api/hr` 这类按域前缀的路由）都会经由这层驱动落到对应表上。

## 底层逻辑与数据模型

驱动抽象的本质是「面向接口编程」。接口定义在 `database-driver.ts`：

```ts
export interface IDatabaseDriver {
  query(sql: string, params?: any[]): any[] | Promise<any[]>;
  get(...): any | null | Promise<any | null>;
  run(...): { changes: number; ... } | Promise<...>;
  exec(sql: string): void | Promise<void>;
  prepare(sql: string): IPreparedStatement;
  getTableInfo(table: string): Array<any> | Promise<Array<any>>;
  listTables(): string[] | Promise<string[]>;
  insert / update / delete / backup / close;
}
export type DriverType = 'sqlite' | 'postgres';   // ✅ 只有这两个
```

注意返回类型里大量出现 `any[] | Promise<any[]>`。这是刻意设计：SQLite 驱动（`DatabaseService`）是**同步**的，PG 驱动（`PgDriver`）是**异步**的。接口用联合类型兼容二者，上层用 `await` 即可通吃。

为什么**没有 MySQL 驱动**？这是源码事实，不是省略：

- `DriverType` 的联合类型只声明 `'sqlite' | 'postgres'`；
- `db-factory.ts` 的判断分支只有 `postgres` 与「默认 SQLite」两路，没有任何 `mysql` / `mysql2` 分支；
- 全仓库数据库模块内 `grep mysql` 零命中。

设计意图写在文件头注释里：「为 SQLite → PostgreSQL 迁移做准备」——产品定位是从单机文件库平滑上云到 PG，没有引入第三种数据库的动因。

自动建表则来自 `BaseModel.ensureTable()`。它读 `_fields`，把每个字段转成列定义，拼出 `CREATE TABLE` 后执行。表字段有三个强约定：

| 约定 | 写法 | 说明 |
|------|------|------|
| 主键 | `id TEXT PRIMARY KEY` | 业务生成 `nanoid`，不依赖自增整数 |
| JSON | `TEXT DEFAULT '[]'` | 数组/对象序列化进 TEXT，非空默认空数组 |
| 时间戳 | `created_at TEXT DEFAULT CURRENT_TIMESTAMP` | 时间戳按字符串存，跨库一致 |

`employees` 表的真实建表片段（节选自 `database.service.ts:23`）印证了这点：`tags TEXT DEFAULT '[]'`，`createdAt TEXT DEFAULT CURRENT_TIMESTAMP`。

## 源码剖析

### 1. 连接初始化与 WAL 模式

SQLite 的初始化在 `database.service.ts:7`：

```ts
onModuleInit() {
  const dbDir = path.join(process.cwd(), 'data');
  const dbPath = path.join(dbDir, 'ehr.db');
  // ... 建目录
  this.db = new Database(dbPath);
  this.db.pragma('journal_mode = WAL');   // ← 13 行：开启 WAL
  this.createTables();
  this.migrateColumns();
  // ...
}
```

🔑 `journal_mode = WAL` 把「写前日志」模式打开，读写可并发，显著提升多用户同时操作的吞吐量。这是 SQLite 单文件库能撑住中小团队并发的关键开关。

### 2. 驱动工厂的环境切换

`db-factory.ts:18` 是切换中枢：

```ts
export function createDatabase() {
  const driver = process.env.DB_DRIVER || 'sqlite';
  const url = process.env.DATABASE_URL || '';
  if (driver === 'postgres' || url.startsWith('postgres')) {
    // ... 构造 PgConfig
    return { db: new PgDriver(config), type: 'postgres' };
  }
  // 默认 SQLite
  const svc = new DatabaseService();
  svc.onModuleInit();
  return { db: svc as any, type: 'sqlite' };
}
```

默认走 SQLite；一旦检测到 `DB_DRIVER=postgres` 或 `DATABASE_URL` 以 `postgres` 开头，就换成 `PgDriver`。业务代码拿到的都是 `IDatabaseDriver`，后续零改动。

### 3. PG 驱动的语法适配

PG 与 SQLite 语法不同，`PgDriver` 用一个 `pgSql()` 做自动转换（`pg-driver.ts:167`）：

```ts
// INSERT OR REPLACE → ON CONFLICT (id) DO UPDATE
// datetime('now')   → NOW()
```

同时 `types.setTypeParser` 把 PG 的数值类型 `int8 / numeric / float8 / int4` 强制解析为 JS `number`，抹平「PG 默认返回字符串」的差异。

### 4. 注册即建表

`model-registry.ts:24` 的 `register()` 在加载模型时直接触发：

```ts
register(options) {
  const model = new BaseModel(this.db, options);
  if (!options._inherit) {
    model.ensureTable();        // ← 非继承模型，立即建表
  }
  ...
}
```

### 5. 自动迁移：幂等 ADD COLUMN

`orm-model.ts:75` 的 `migrateColumns()` 是「加字段不丢数据」的核心：

```ts
migrateColumns(): void {
  if (!this._inherit) return;
  const existing = this.getTableColumns(this._inherit);
  for (const [name, field] of Object.entries(this._fields)) {
    if (!existing.includes(name)) {                 // 只补不存在的列
      try {
        this.db.exec(`ALTER TABLE ${this._inherit} ADD COLUMN ${fieldToColumnDDL(name, field)}`);
      } catch (e) { /* 列已存在，忽略 */ }
    }
  }
}
```

列定义由 `fields.ts:55` 的 `fieldToColumnDDL()` 生成。注意 `try/catch` 吞掉异常——即便并发重复执行，重复 `ADD COLUMN` 也会因「列已存在」被忽略，迁移天然**幂等**。

## 原理剖析

把两条主线连起来看：

- **抽象层**回答「写一次代码，跑两种库」。代价是接口方法必须取两者能力交集（所以没有数据库专属语法泄漏到业务层），收益是切换数据库只是改环境变量。
- **迁移层**回答「改模型不写 SQL」。模型 `_fields` 是单一事实来源（Single Source of Truth）：建表 DDL、字段校验、默认值、类型转换（`coerceField`）全从它派生。新字段只需在 `_fields` 加一项，重启即 `ALTER` 补齐，老数据原样保留。

为什么迁移只做 `ADD COLUMN`？因为删列/改列会破坏已有数据，而 ERP 数据不可逆。设计选择「只增不删」，把破坏性操作留给人工审慎处理——这是有意为之的保守策略。

## 管理者视角

作为管理者，你不必读懂每一行代码，但要理解选型含义：

✅ **SQLite（默认，零运维）**：单个文件 `data/ehr.db`，无需安装数据库服务、无需 DBA。适合初创团队、内网单机、演示验证。配合 WAL，十几人并发录入毫无压力。

🚀 **PostgreSQL（上云，可扩展）**：当你需要多实例部署、高并发、专业备份与容灾时，切到 PG。改两行环境变量即可，业务代码不动。呼应本系列第 45 篇——**数据备份是 ERP 的生命线**：SQLite 直接复制 `ehr.db`（及 `-wal`/`-shm`）即可冷备；PG 则走 `pg_dump`（驱动中 `backup()` 已封装）。

一句话：先用 SQLite 跑起来，业务增长后再平滑上云 PG，迁移成本极低。

## 注意事项

⚠️ **生产环境建议用 PostgreSQL。** SQLite 单文件虽方便，但多进程高并发写入有天花板，且文件损坏即全量损失。面向正式生产、多人高频使用的场景，请切换到 PG 并配置定期 `pg_dump`。

⚠️ **迁移是 ADD COLUMN 幂等，不会删列。** 你在 `_fields` 里删除一个字段，数据库列不会自动消失（避免误删数据）。废弃字段会长期留存在表中，需要清理时请人工 `ALTER TABLE DROP COLUMN` 并评估影响。同理，字段类型变更也不会回滚旧数据。

🔑 **WAL 模式提升并发。** 默认已开启，请勿在配置中关闭 `journal_mode`，否则写操作会阻塞读，多人同时操作会卡顿。

⚠️ **切换驱动前请先备份。** 从 SQLite 迁到 PG 是「换库不换表结构」，但数据本身需要你用导出/导入工具搬运，驱动不会自动把 SQLite 数据灌进 PG。

## 小结与练习

本篇你掌握了：驱动抽象用 `IDatabaseDriver` 屏蔽 SQLite/PG 差异（仅此两种，无 MySQL）；`db-factory` 按环境变量切换；`ensureTable`/`migrateColumns` 实现注册即建表、新字段幂等 `ADD COLUMN`；字段三约定为 `id TEXT PRIMARY KEY`、JSON→TEXT、时间戳 TEXT。
