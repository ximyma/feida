# 37. 售后：退货退款状态机

> 环境约定：产品名 **Sowork AI 企业智能ERP系统**（本文统称「系统」），版本 **v1.2.0**，后台账号 **admin/admin123**，服务端口 **3000**，安装依赖用 `npm install --legacy-peer-deps`。

## 学习目标

读完本篇，你将能够：

- 说清售后单的两种类型（仅退款 `refund`、退货退款 `return`）和各自的状态流。
- 在后台「售后管理」页完成审核、标记收货、确认退款、拒绝等操作。
- 理解 `return_received` 状态如何触发**自动回库**，并与第 36 篇「订单与支付」的库存扣减形成闭环。
- 看懂服务端防重复提交、防重复回库这两道安全闸。

## 核心概念

售后是下单—支付之后的逆向流程。系统用一张表 `shop_order_aftersale` 承载所有售后单，用字段 `status` 表达状态机当前节点，用字段 `type` 区分业务类型。

两种类型映射到两条不同的状态链：

```
仅退款 refund:
  pending ──approved──▶ refunded
                  └──▶ rejected

退货退款 return（多两段物流/收货）:
  pending ──approved──▶ return_shipped ──▶ return_received ──▶ refunded
                  └──────────────────────────────────────▶ rejected
```

关键差异：退货退款多了 `return_shipped`（用户寄回）和 `return_received`（商家收货）两个必须环节，而**自动回库只在退货退款、且商家确认收货时发生**。

状态含义速查表：

| 状态 | 中文 | 含义 | 谁触发 |
|------|------|------|--------|
| pending | 待审核 | 用户刚提交申请 | 用户 |
| approved | 已同意 | 商家通过申请 | 后台 |
| return_shipped | 已寄回 | 用户填写退货物流单号 | 用户 |
| return_received | 已收货 | 商家确认收到退货 | 后台 |
| refunded | 退款完成 | 财务/后台确认退款 | 后台 |
| rejected | 已拒绝 | 商家驳回申请 | 后台 |

🔑 记住：`rejected` 是任何非终态节点的合法出口，但 `refunded` 之后不再流转。

## 界面布局与操作流程

**后台侧**（`client/src/pages/Admin/ShopAdminPage.tsx`）：

入口在顶部 Tab「售后管理」（key=`aftersale`）。列表展示售后单号、订单号、类型、退款金额、状态标签。点击任意一行弹出「售后处理」Modal（`ShopAdminPage.tsx:1262`），根据当前状态动态渲染按钮：

```tsx
{aftersaleCur.status==='pending' && <Button onClick={()=>doAftersale('approved')}>同意</Button>}
{aftersaleCur.status==='approved' && type==='return' && <Button onClick={()=>doAftersale('return_received')}>标记收货</Button>}
{['approved','return_shipped','return_received'].includes(status) && <Button onClick={()=>doAftersale('refunded',{refund_method})}>确认退款</Button>}
{['pending','approved'].includes(status) && <Button onClick={...}>拒绝</Button>}
```

`doAftersale`（`ShopAdminPage.tsx:78`）本质就是向 `/api/shop-order-aftersale/:id` 发 `PUT`，把目标 `status` 和 `reviewer:'admin'` 传给服务端，由服务端推进状态机。

**用户侧**（`client/src/pages/ShopHomePage/ShopOrdersPage.tsx`）：

在「我的订单」详情里，系统用 Antd `Steps` 组件画出进度时间线（`aftersaleSteps`，`ShopOrdersPage.tsx:106`）。退货类型显示 5 步（申请→审核→寄回→收货→退款），仅退款显示 3 步。当状态为 `approved` 且是退货类型时，页面出现「填写退货物流」按钮（`ShopOrdersPage.tsx:386`），用户提交物流公司+单号，触发 `status:'return_shipped'` 的 `PUT`（`submitReturnShip`，`ShopOrdersPage.tsx:121`）。

🚀 一句话：用户负责 `pending→return_shipped`，后台负责 `approved→return_received→refunded` 与 `rejected`。

## 底层逻辑与数据模型

售后单落表 `shop_order_aftersale`，核心字段：

| 字段 | 作用 |
|------|------|
| id | 单号，形如 `as_1700000000000` |
| order_id | 关联订单 |
| type | `refund` / `return` |
| status | 状态机当前节点 |
| refund_amount | 退款金额 |
| return_tracking_no / return_tracking_company | 退货物流单号与承运商（用户填） |
| received_at | 收货时间戳，**同时是回库去重哨兵** |
| refund_method | `original` 原路退回 / `balance` 退到余额 |
| reject_reason / admin_remark | 拒绝理由与处理备注 |

每次状态推进都会通过 `logOrder` 往 `shop_order_logs` 写一条可追溯记录（标签 `aftersale_approved`/`aftersale_received`/`aftersale_refunded` 等，`ShopAdminPage.tsx:59`）。这使得「谁、何时、改成了什么」全程留痕。

回库动作由 `restoreAftersaleStock(orderId)` 统一完成：它加载该订单所有 `shop_order_items`，逐条调用 `applyStockChange(warehouseId, product_id, sku, quantity, 'return', remark, 'system')`。而 `applyStockChange`（`standalone.ts:8872`）用**正数** `num` 把库存加回去，并向 `shop_stock_logs` 写入一条 `type='return'` 的流水——这正是第 36 篇所说的「库存账实联动」的逆向一笔。

## 源码剖析

服务端路由集中在 `server/standalone.ts:10959` 之后的「售后管理」段落。

**状态机定义与回库函数**（`standalone.ts:10960-10971`）：

```ts
// 售后状态机：
//   仅退款(refund): pending → approved → refunded | rejected
//   退货退款(return): pending → approved → return_shipped → return_received → refunded | rejected
const restoreAftersaleStock = (orderId: string) => {
  const wh = db.findOne('shop_warehouse', { is_default: 1 }) || db.findAll('shop_warehouse')[0];
  const items = db.findWhere('shop_order_items', { order_id: orderId }) as any[];
  items.forEach((it) => {
    applyStockChange(wh ? wh.id : null, it.product_id, it.sku || '',
      it.quantity || 0, 'return', '售后退货回滚 #' + orderId, 'system');
  });
};
```

**防重复提交**（`standalone.ts:10984-10989`）：提交申请时，只要该订单存在「未拒绝」的售后单就返回 400。

```ts
const existing = (db.findWhere('shop_order_aftersale', { order_id }) as any[])
  .filter((r) => r.status !== 'rejected');
if (existing.length) return res.status(400).json({ success: false, message: '该订单已有售后申请，无法重复提交' });
```

**状态迁移 + 自动回库**（`standalone.ts:11034-11044`）是本文重点：

```ts
} else if (st === 'return_received') {
  // 商家收货确认 → 商品回库（仅退货类型，且此前未回滚过）
  if (as.type === 'return' && !as.received_at) restoreAftersaleStock(as.order_id);
  db.update('shop_orders', as.order_id, { order_status: 'refunding' });
  logOrder(as.order_id, 'aftersale_received', '商家已收货，待退款', operator);
} else if (st === 'refunded') {
  // 退货类型若此前未走收货回滚，则在此补回滚，避免丢库存
  if (as.type === 'return' && !as.received_at) restoreAftersaleStock(as.order_id);
  db.update('shop_order_aftersale', req.params.id, { refunded_at: now });
  db.update('shop_orders', as.order_id, { order_status: 'refunded' });
  logOrder(as.order_id, 'aftersale_refunded', '退款完成', operator);
}
```

两个分支都带 `!as.received_at` 守卫，保证退货库存**最多回滚一次**；`refunded` 分支还充当「兜底」——万一没走过 `return_received`，退款时仍会把库存补回，避免丢库存。此外系统保留了旧别名路由 `/shop-refunds`（GET/POST/PUT，`standalone.ts:11059` 起），字段已对齐，避免老调用直接 500。

## 原理剖析

为什么回库时机选在 `return_received` 而非 `refunded`？逻辑很清晰：钱可以晚退，货必须先收。用户在 `approved` 后寄回，后台在确认真实收到实物时才改 `return_received` 并回库——此时账面库存立刻恢复，可再次售卖；而退款 `refunded` 往往要等财务确认，二者解耦，互不阻塞。

为什么需要 `!as.received_at` 去重？因为「标记收货」和「确认退款」是两个独立按钮，理论上都可能触发回库；用 `received_at` 作为哨兵，回库过一次该字段被置位，后续分支的 `!as.received_at` 即不成立，杜绝重复加库存导致账实不符。

为什么申请层就防重？`rejected` 的单子视为「已了结」，允许用户重新发起；而 `pending/approved/...` 的单子若已存在，再次提交就是重复工单，服务端用 400 直接挡下，把判断成本留在后端而非依赖前端禁用按钮。

## 管理者视角

✅ 售后体验直接决定复购率：状态标签（待审核/已同意/已寄回/已收货/退款完成/已拒绝）让客户全程可见进度，减少「我的退货到哪了」的催问工单。

✅ 自动回库保障了账实相符：退货一旦收货，库存立即回流并可再售，不会因人工漏记而产生「幽灵缺货」或超卖。

✅ 状态透明减少扯皮：每一步都写 `shop_order_logs`，谁在何时改了什么一查便知，纠纷时可追溯。

🔑 从经营角度，建议把「确认退款」权限收口给财务或特定角色，因为 `refunded` 同时改写订单终态 `refunded` 并释放库存占用，是资金与库存的双重出口。

## 注意事项

⚠️ **回库去重要靠 `received_at`**：`return_received` 与 `refunded` 两处回库都受 `!as.received_at` 守卫保护。运营上务必先「标记收货」再「确认退款」，让去重哨兵正确生效，避免任何重复加库存的可能。

🔑 **退款需财务确认**：`refunded` 是资金出口，应在后台管控流程中由财务或授权人员点击「确认退款」，不可与「标记收货」混为一步自动完成。

✅ **物流单号可追踪**：用户侧填写的 `return_tracking_no` / `return_tracking_company` 会随售后单透出到后台处理弹窗，退货纠纷时可用它向承运商核验，务必提醒用户如实填写。

⚠️ **拒绝要留理由**：`rejected` 必须带 `reject_reason`，否则后续客户追问无据。系统也要求后台拒绝弹窗填写理由（`ShopAdminPage.tsx:1291`）。

## 小结与练习

本篇你掌握了：售后单两张状态链（`refund` 三步、`return` 五步）、后台 Modal 按钮如何驱动 `PUT /api/shop-order-aftersale/:id`、以及 `return_received` 如何通过 `restoreAftersaleStock`→`applyStockChange('return')` 自动回库并与第 36 篇库存扣减形成闭环。两道安全闸（申请层 400 防重、回库层 `received_at` 去重）是这套状态机的稳定基石。

练习：
1. 用 admin 登录，开一个退货退款单，走到 `return_received`，去数据库查 `shop_stock_logs` 是否多出 `type='return'` 记录。
2. 故意对同一个订单重复提交售后申请，确认前端收到 400「无法重复提交」。
3. 在 `return_received` 之前直接点「确认退款」，验证 `refunded` 分支的兜底回库是否把库存补回。
