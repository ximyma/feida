## 核心概念："配置即数据库"

传统开发加一个业务模块要：设计表 → 写迁移 → 写后端 CRUD → 写前端页面 → 联调。低代码平台把这一切压缩成**填一个表单**：

```
业务人员在向导里填：
  模块名「客户关系管理」
  模型「客户」+ 字段（客户名 char、电话 char、所属行业 selection…）
        │
        ▼  POST /api/lowcode/create-module
  ① 生成 addons/客户关系管理/models/customer.js   （模型定义文件）
  ② reg.register(modelDef)                        （动态注册到 ORM → 自动建表）
  ③ 生成 manifest.json                            （模块元数据）
        │
        ▼  POST /api/lowcode/save-app-config
  ④ 生成 app.json                                 （前端菜单/渲染配置）
        │
        ▼  前端读 app.json
  ⑤ 自动渲染出「客户」列表页 + 增删改查表单        （零前端代码）
```

🔑 关键洞察：**低代码平台复用的正是第 04/06 章那套自研 ORM**。因为 ORM 是"模型即表结构、启动即建表"，所以"运行时新增模型"天然可行——这是自研 ORM 相比 Prisma（需 codegen）最大的红利。

## 源码剖析一：`create-module` 生成模型并热注册

核心接口（`server/standalone.ts:708-758`）。逐段读这段"运行时造模块"的代码：

```ts
// server/standalone.ts:708-758
router.post('/lowcode/create-module', (req, res) => {
  const { moduleName, models } = req.body;
  if (!moduleName || !Array.isArray(models)) {
    res.status(400).json({ error: '缺少 moduleName 或 models' }); return;
  }
  try {
    const fs = require('fs');
    const path = require('path');
    const modelsDir = path.join(process.cwd(), 'addons', moduleName, 'models');
    fs.mkdirSync(modelsDir, { recursive: true });

    for (const modelDef of models) {
      if (!modelDef._name) continue;
      const filePath = path.join(modelsDir, modelDef._name + '.js');

      // 如果已有模型文件，合并字段（编辑模式保留全部字段）
      let finalDef = modelDef;
      if (fs.existsSync(filePath)) {
        try {
          delete require.cache[require.resolve(filePath)];
          const existing = require(filePath);
          const existingDef = existing.model || existing;
          if (existingDef._fields) {
            finalDef = { ...existingDef, _fields: { ...existingDef._fields, ...modelDef._fields } };
          }
        } catch (_) { /* 读取失败则覆盖 */ }
      }

      const content = '// 低代码生成: ' + finalDef._name + '\n' +
        'exports.model = ' + JSON.stringify(finalDef, null, 2) + ';\n';
      fs.writeFileSync(filePath, content, 'utf-8');
      // 动态注册到 ORM
      const reg = (global as any).__feida_models;
      if (reg) {
        reg.register(modelDef);
        const def = reg.getDef(modelDef._name);
        if (def) { def._addon = true; def._addonName = moduleName; }
      }
    }
    // 生成 manifest
    const manifest = {
      name: moduleName, name_en: moduleName, version: '1.0.0', author: '低代码',
      category: 'LowCode', depends: [], summary: '低代码生成的模块',
      installable: true, auto_install: true, application: true, sequence: 700,
    };
    fs.writeFileSync(path.join(process.cwd(), 'addons', moduleName, 'manifest.json'), JSON.stringify(manifest, null, 2));
    res.json({ success: true, module: moduleName, models: models.length });
  } catch (e: any) { res.status(500).json({ error: e.message }); }
});
```

拆解四个关键动作：

1. **写模型文件**：把前端传来的 `modelDef`（含 `_name` / `_fields`）序列化成 `exports.model = {...}` 写到 `addons/<模块>/models/<模型>.js`。**这是持久化**——重启后模块仍在。
2. **编辑模式合并字段**：如果文件已存在，先 `delete require.cache` 清缓存再 `require` 出旧定义，用 `{...旧字段, ...新字段}` 合并——**保证编辑时不丢已有字段**。
3. **`reg.register(modelDef)` 热注册**：把新模型立刻注册进全局 ORM 注册表 `global.__feida_models`。注册的副作用就是 `ensureTable()` **自动建表**（见第 04、06 章）——所以**无需重启就有了新表**。
4. **打标 `_addon`**：标记这是 addon 模块的模型，便于后续区分内置模型与低代码模型。

⚠️ 注意一个细节：**写文件用的是 `finalDef`（合并后），热注册用的是 `modelDef`（原始传入）**。持久化保留全字段，运行时注册以本次提交为准——理解这个区别能避免"改了字段但内存没更新"的困惑。

## 源码剖析二：`add-field` 给现有表加字段

单独加一个字段不需要重建模块，走 `add-field`（`server/standalone.ts:760-778`）：

```ts
// server/standalone.ts:760-778
router.post('/lowcode/add-field', (req, res) => {
  const { table, fieldName, fieldDef } = req.body;
  if (!table || !fieldName || !fieldDef) {
    res.status(400).json({ error: '缺少 table/fieldName/fieldDef' }); return;
  }
  try {
    const rawDb = (global as any).__feida_raw_db;
    const typeMap: Record<string, string> = { char:'TEXT', text:'TEXT', integer:'INTEGER', float:'REAL', boolean:'INTEGER DEFAULT 0', date:'TEXT', datetime:'TEXT', selection:'TEXT', many2one:'TEXT DEFAULT \'\'' };
    const sqlType = typeMap[fieldDef.type] || 'TEXT';
    rawDb.exec('ALTER TABLE "' + table + '" ADD COLUMN "' + fieldName + '" ' + sqlType);
    // 更新内存中的模型定义
    const reg = (global as any).__feida_models;
    if (reg) {
      const def = reg.getDef(table);
      if (def) def._fields[fieldName] = fieldDef;
    }
    res.json({ success: true, table, field: fieldName });
  } catch (e: any) { res.status(500).json({ error: e.message }); }
});
```

逐段读：

1. **`typeMap` 类型映射**：把低代码的字段类型（`char`/`integer`/`selection`/`many2one`…）映射成 SQLite 列类型。这是"业务语言 → 数据库语言"的翻译层。注意 `boolean` → `INTEGER DEFAULT 0`（SQLite 无布尔），`many2one` → `TEXT DEFAULT ''`（存关联记录的 id 字符串）。
2. **`rawDb.exec('ALTER TABLE ...')`**：直接对物理表加列，**立即生效、不丢数据**（复用第 04 章的增量迁移思路）。
3. **同步内存定义 `def._fields[fieldName] = fieldDef`**：物理表加了列，内存里的模型定义也要同步，否则 ORM 不认识这个新字段。**物理 + 逻辑双写**是这里的关键。

## 源码剖析三：`app.json` 驱动前端菜单

后端生成了模型和表，前端怎么知道要渲染哪些菜单？靠 `app.json`。看一个真实的低代码生成产物（`addons/app_mru06doi/app.json`）：

```json
// addons/app_mru06doi/app.json
{
  "name": "客户关系管理",
  "description": "客户关系管理",
  "icon": "👥",
  "color": "#1677ff",
  "menu": [
    { "label": "客户信息", "table": "customer_information", "icon": "📋", "desc": "管理客户信息" },
    { "label": "联系人信息", "table": "contactinfo", "icon": "📋", "desc": "管理联系人信息" },
    { "label": "商机信息", "table": "opportunity", "icon": "📋", "desc": "管理商机信息" },
    { "label": "跟进记录", "table": "followinfo", "icon": "📋", "desc": "管理跟进记录" }
  ]
}
```

`menu` 数组的每一项就是一个菜单：`label`（显示名）+ `table`（对应的模型表名）+ `icon`。前端读到这个配置，就为每个 `table` 渲染出**标准的列表页 + 增删改查表单**——数据读写走的正是第 02 章的 catch-all `/:table` 通用 CRUD。

对应的模型文件长这样（`addons/app_mru06doi/models/contactinfo.js`）：

```js
// addons/app_mru06doi/models/contactinfo.js
// 低代码生成: contactinfo
exports.model = {
  "_name": "contactinfo",
  "_description": "联系人信息",
  "_fields": {
    "contact_name": { "type": "char", "label": "联系人姓名", "required": false },
    "customer_id": { "type": "many2one", "label": "所属客户", "required": false, "relation": "customer" },
    ...
  }
};
```

🔑 **一条闭环**：`app.json`(菜单) + `models/*.js`(表结构) + catch-all `/:table`(通用 CRUD) = 一个完整的业务模块，**零手写代码**。

## 源码剖析四：模块加载的入口

低代码模块和内置模块共用同一套加载机制（`server/modules/orm/model-registry.ts:50-86`）：

```ts
// server/modules/orm/model-registry.ts:50-86
loadFromModule(moduleDir: string): BaseModel[] {
  const modelsDir = path.join(moduleDir, 'models');
  if (!fs.existsSync(modelsDir)) return [];
  const loaded: BaseModel[] = [];
  const files = fs.readdirSync(modelsDir).filter(f => f.endsWith('.js') || f.endsWith('.ts'));
  for (const file of files) {
    const fp = path.join(modelsDir, file);
    try {
      delete require.cache[require.resolve(fp)];   // 清缓存，加载最新文件
      const mod = require(fp);
      const modelDefs: ModelOptions[] = [];
      if (mod.models && Array.isArray(mod.models)) {
        modelDefs.push(...mod.models);             // exports.models = [...]（内置模块写法）
      } else if (mod.model && mod.model._name) {
        modelDefs.push(mod.model as ModelOptions); // exports.model = {...}（低代码写法）
      } else if (mod.default && mod.default._name) {
        modelDefs.push(mod.default as ModelOptions);
      } else if (Array.isArray(mod)) {
        modelDefs.push(...(mod as ModelOptions[]));
      }
      for (const modelDef of modelDefs) {
        if (modelDef._name) {
          this.register(modelDef);                 // 注册 → 自动建表
          loaded.push(this.models.get(modelDef._inherit || modelDef._name)!);
        }
      }
    } catch (e) {
      console.error(`[ModelRegistry] 加载失败: ${fp}`, (e as Error).message);
    }
  }
  return loaded;
}
```

注意它**兼容多种导出形式**：`exports.models = [...]`（内置模块，数组）、`exports.model = {...}`（低代码，单个）、`export default`、直接数组。这让手写模块和低代码生成模块都能被统一加载。启动时（第 01 章 `standalone.ts:97-113`）遍历 `modules/` 和 `addons/` 目录调用它，实现"放进目录即生效"。

## 实战代码：用 API 手动造一个模块

不用界面，直接调 API 造一个"设备台账"模块：

```bash
# 1. 创建模块 + 模型（自动建表）
curl -X POST http://localhost:3000/api/lowcode/create-module \
  -H "Content-Type: application/json" \
  -d '{
    "moduleName": "asset_mgmt",
    "models": [{
      "_name": "asset_ledger",
      "_description": "设备台账",
      "_fields": {
        "asset_name": { "type": "char", "label": "设备名称", "required": true },
        "buy_date": { "type": "date", "label": "购入日期" },
        "price": { "type": "float", "label": "购入价格" }
      }
    }]
  }'
# → {"success":true,"module":"asset_mgmt","models":1}

# 2. 立刻就能用通用 CRUD 写数据（无需重启）
curl -X POST http://localhost:3000/api/asset_ledger \
  -H "Content-Type: application/json" \
  -d '{"asset_name":"激光切割机","price":128000}'

# 3. 后加一个字段
curl -X POST http://localhost:3000/api/lowcode/add-field \
  -H "Content-Type: application/json" \
  -d '{"table":"asset_ledger","fieldName":"location","fieldDef":{"type":"char","label":"存放位置"}}'
```

> ⚠️ 第 2 步能成功的前提是 `asset_ledger` 在通用 CRUD 白名单 `ALLOWED` 内，或被 addon 加载逻辑放行——低代码模块通常通过 addon 注册路径生效。

## 管理者视角

| 关注点 | 低代码平台给出的答案 |
|--------|-------------------|
| **业务响应速度** | 业务部门自己就能建"设备台账""客户跟进"这类模块，不用排期等开发。需求当天上线。 |
| **IT 成本** | 大量长尾的、简单的 CRUD 类应用不再消耗宝贵的开发人力，开发团队专注核心系统。 |
| **数据统一** | 低代码建的模块和核心系统共用一个数据库、一套权限、一套 AI 知识库，不产生数据孤岛。 |
| **边界与风险** | 低代码适合"标准表单 + 列表"类应用；复杂业务逻辑（如多级审批联动、复杂计算）仍需开发介入。生成的模型文件应纳入版本管理，避免误删。 |

给决策者一句话：**低代码平台把"业务部门的长尾需求"从 IT 排期里解放出来**，是提升组织数字化速度的杠杆——但要划清它与核心开发的边界。

## 注意事项

- ⚠️ **模型文件是真实产物，要纳入备份**：`addons/<模块>/models/*.js` 和 `app.json` 是磁盘文件，误删会导致模块丢失。生产环境应把 `addons/` 一起备份。
- ⚠️ **编辑模块用合并语义**：`create-module` 对已存在文件是"合并字段"，不会删旧字段。要删字段需手动处理（SQLite 删列较麻烦，一般保留为废弃字段）。
- ⚠️ **`many2one` 存的是关联 id 字符串**：类型映射为 `TEXT DEFAULT ''`，前端需自行渲染成下拉选择。
- 🔑 低代码之所以能"运行时建表"，根基是第 06 章的自研 ORM（模型即表结构 + 启动即建表）。理解 ORM 才能理解低代码。
