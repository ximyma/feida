## 学习目标

- 读懂 Sowork AI 如何用 `better-sqlite3` 同步打开数据库并完成"建表 + 迁移 + 种子"一条龙
- 理解 SQLite ↔ PostgreSQL 双驱动抽象接口 `IDatabaseDriver` 的设计
- 精读 `migrateColumns()` 用 `ALTER TABLE ADD COLUMN` 实现的"零停机增量迁移"
- 掌握本项目的表结构约定（`id TEXT PRIMARY KEY`、JSON 存 TEXT、时间戳约定）
- （管理者）理解"自动迁移"如何降低版本升级的运维风险

## 核心概念：数据库层要解决三个问题

企业系统的数据库层不只是"连上库"，它要回答三个问题：

```
① 连接与初始化   → 用哪个驱动？库文件放哪？启动时做什么？
② 表结构管理     → 谁负责建表？升级加字段怎么办（不能丢数据）？
③ 驱动可移植     → 小企业用 SQLite，大企业能不能平滑换 PostgreSQL？
```

Sowork AI 对这三个问题的回答，全在 `server/modules/database/` 这个目录里。这一章我们逐个源码验证。

## 源码剖析一：连接与初始化一条龙

入口是 `DatabaseService.onModuleInit()`（`server/modules/database/database.service.ts:1-19`）：

```ts
// server/modules/database/database.service.ts:1-19
import Database from 'better-sqlite3';
import * as path from 'path';

export class DatabaseService {
  public db!: Database.Database;

  onModuleInit() {
    const dbDir = path.join(process.cwd(), 'data');
    const dbPath = path.join(dbDir, 'ehr.db');
    const fs = require('fs');
    if (!fs.existsSync(dbDir)) fs.mkdirSync(dbDir, { recursive: true });
    this.db = new Database(dbPath);
    this.db.pragma('journal_mode = WAL');
    this.createTables();
    this.migrateColumns();
    this.migrateShopProductFks();
    this.seedDefaultData();
    console.log(`[Database] SQLite at ${dbPath}`);
  }
```

逐行读这段"点火代码"：

- **`new Database(dbPath)`**：用 `better-sqlite3` **同步**打开 `data/ehr.db`。同步是关键——它让后续建表不需要 `await`，也让 SQL 查询能写成 `const rows = db.query(sql)` 这种直白形式（无回调、无 Promise 地狱）。
- **`pragma('journal_mode = WAL')`**：开启 WAL（Write-Ahead Logging）日志模式。它让"读"和"写"可以并发（读不阻塞写），显著提升多用户场景下的吞吐——这也是你在项目里看到 `data/ehr.db-wal`、`data/ehr.db-shm` 两个伴生文件的原因。
- **四步初始化**：`createTables()` 建表 → `migrateColumns()` 增量迁移 → `migrateShopProductFks()` 修外键 → `seedDefaultData()` 播种默认数据。**每次启动都跑一遍**，但都是幂等的（`IF NOT EXISTS`、`if 列不存在才加`）。

🔑 "每次启动都建表迁移"听起来很吓人，其实是这套系统**免运维升级**的关键：用户拿到新版本代码，直接启动，缺的表自动补、缺的列自动加，不需要手动跑迁移脚本。

## 源码剖析二：双驱动抽象 `IDatabaseDriver`

小企业用 SQLite 够了，但当企业长大、并发上来，SQLite 的单写锁会成为瓶颈。Sowork AI 为此预留了 **PostgreSQL 切换能力**，靠的是一个统一接口 `IDatabaseDriver`（`server/modules/database/database-driver.ts:10-31`）：

```ts
// server/modules/database/database-driver.ts:10-31
export interface IDatabaseDriver {
  query(sql: string, params?: any[]): any[] | Promise<any[]>;
  get(sql: string, params?: any[]): any | null | Promise<any | null>;
  run(sql: string, params?: any[]): { changes: number; lastInsertRowid?: number | bigint } | Promise<{ changes: number }>;
  exec(sql: string): void | Promise<void>;
  prepare(sql: string): IPreparedStatement;
  getTableInfo(table: string): Array<any> | Promise<Array<any>>;
  listTables(): string[] | Promise<string[]>;
  insert(table: string, data: Record<string, any>): { id: string } | Promise<{ id: string }>;
  update(table: string, id: string, data: Record<string, any>): { changes: number } | Promise<{ changes: number }>;
  delete(table: string, id: string): { changes: number } | Promise<{ changes: number }>;
  backup(targetPath: string): Promise<void>;
  close(): void;
}

export type DriverType = 'sqlite' | 'postgres';
```

设计上的两个巧思：

1. **返回类型是"同步 | Promise"联合类型**（`any[] | Promise<any[]>`）：这是为了兼容 SQLite 的同步 API 和 PostgreSQL 的异步 API。同一个接口，两种驱动各自实现。
2. **`DriverType = 'sqlite' | 'postgres'`**：⚠️ 只有这两种。项目里虽然装了 `mysql2` 包，但数据库驱动层**并不支持 MySQL**——`mysql2` 是别处（如数据源连接器）用的。教程里不要误写"支持 MySQL"。

切换逻辑在工厂函数里（`server/modules/database/db-factory.ts:18-45`）：

```ts
// server/modules/database/db-factory.ts:18-45
export function createDatabase(): { db: IDatabaseDriver; type: 'sqlite' | 'postgres' } {
  const driver = process.env.DB_DRIVER || 'sqlite';
  const url = process.env.DATABASE_URL || '';

  if (driver === 'postgres' || url.startsWith('postgres')) {
    let config: PgConfig;
    if (url.startsWith('postgres')) {
      config = url as any;
    } else {
      config = {
        host: process.env.DB_HOST || 'localhost',
        port: parseInt(process.env.DB_PORT || '5432'),
        database: process.env.DB_NAME || 'feida',
        user: process.env.DB_USER || 'feida',
        password: process.env.DB_PASS || '',
      };
    }
    console.log(`[DB] PostgreSQL: ${config.host}:${config.port}/${config.database}`);
    return { db: new PgDriver(config), type: 'postgres' };
  }

  // 默认 SQLite
  const dbPath = process.env.DB_PATH || url.replace('sqlite://', '') || 'data/ehr.db';
  console.log(`[DB] SQLite: ${dbPath}`);
  const svc = new DatabaseService();
  svc.onModuleInit();
  return { db: svc as any, type: 'sqlite' };
}
```

🔑 切换只靠**环境变量**：设 `DB_DRIVER=postgres` 或给一个 `postgres://...` 的 `DATABASE_URL`，就走 `PgDriver`；否则默认 SQLite。**业务代码一行都不用改**——这就是驱动抽象的价值。

## 源码剖析三：集中式建表 `createTables()`

所有核心表在一个方法里用 `this.db.exec()` 一次性建好（`server/modules/database/database.service.ts:21-92`，节选 `employees` 与 `users`）：

```ts
// server/modules/database/database.service.ts:21-58
private createTables() {
  this.db.exec(`
    CREATE TABLE IF NOT EXISTS employees (
      id TEXT PRIMARY KEY, name TEXT NOT NULL, employeeId TEXT UNIQUE NOT NULL,
      department TEXT, position TEXT, rank TEXT, status TEXT DEFAULT 'active',
      hireDate TEXT, phone TEXT, email TEXT, salaryLocation TEXT DEFAULT 'shenzhen',
      ...
      tags TEXT DEFAULT '[]', remark TEXT,
      deptId TEXT, positionId TEXT,
      createdAt TEXT DEFAULT CURRENT_TIMESTAMP, updatedAt TEXT DEFAULT CURRENT_TIMESTAMP
    );
    ...
    CREATE TABLE IF NOT EXISTS users (
      id TEXT PRIMARY KEY, username TEXT UNIQUE NOT NULL, realName TEXT NOT NULL,
      phone TEXT, email TEXT, password TEXT, userType TEXT DEFAULT 'employee',
      roleIds TEXT DEFAULT '[]', status TEXT DEFAULT 'active', employeeId TEXT,
      lastLoginAt TEXT, lastLoginIp TEXT, createdAt TEXT DEFAULT CURRENT_TIMESTAMP
    );
```

从这段建表 SQL 能提炼出**三条贯穿全项目的表结构约定**：

| 约定 | 写法 | 为什么 |
|------|------|--------|
| **主键统一 `id TEXT PRIMARY KEY`** | `id TEXT PRIMARY KEY` | 用字符串主键（如 `emp_1721600000`、nanoid），跨表迁移/合并不会主键冲突，也便于前端生成临时 id。 |
| **数组/对象存 `TEXT DEFAULT '[]'` 或 `'{}'`** | `roleIds TEXT DEFAULT '[]'`、`tags TEXT DEFAULT '[]'` | SQLite 无原生 JSON 列，用 TEXT 存 JSON 字符串，读出后 `JSON.parse`。简单可移植。 |
| **时间戳 `TEXT DEFAULT CURRENT_TIMESTAMP`** | `createdAt TEXT DEFAULT CURRENT_TIMESTAMP` | 存 ISO 字符串，前端直接显示，无需时区转换的坑。 |

⚠️ 注意 `roleIds TEXT DEFAULT '[]'`——用户的角色是以 **JSON 数组字符串**存的（如 `'["role_hr_admin"]'`）。第 12 章讲 RBAC resolve 时，你会看到后端把它 `JSON.parse` 回数组再解析权限。这条约定是理解权限系统的前提。

## 源码剖析四：零停机增量迁移 `migrateColumns()`

这是本章最重要的一段。企业系统最怕"升级丢数据"。Sowork AI 的迁移策略是**只加列、从不删列、从不重建表**（`server/modules/database/database.service.ts:3276-3302`）：

```ts
// server/modules/database/database.service.ts:3276-3302
private migrateColumns() {
  const addCol = (table: string, col: string, def: string) => {
    try {
      const info = this.db.prepare(`PRAGMA table_info(${table})`).all() as any[];
      if (!info.find((c) => c.name === col)) {
        this.db.exec(`ALTER TABLE ${table} ADD COLUMN ${col} ${def}`);
      }
    } catch (e) { /* ignore */ }
  };
  addCol('cms_articles', 'is_bold', 'INTEGER DEFAULT 0');
  addCol('shop_goods', 'video_url', 'TEXT DEFAULT \'\'');
  addCol('cms_articles', 'custom_fields', 'TEXT DEFAULT \'{}\'');
  addCol('cms_articles', 'sensitive_hits', 'TEXT DEFAULT \'[]\'');
  // 敏感词表（建表期之后才新增的表，也在此补建）
  this.db.exec(`CREATE TABLE IF NOT EXISTS cms_sensitive_words (
    id TEXT PRIMARY KEY, word TEXT UNIQUE, level INTEGER DEFAULT 1, created_at TEXT
  )`);
  ...
  const swCount = (this.db.prepare('SELECT COUNT(*) AS c FROM cms_sensitive_words').get() as any).c;
  if (swCount === 0) {
    const demo = ['暴力', '色情', '赌博', '政治敏感词', '诈骗'];
    const ins = this.db.prepare('INSERT OR IGNORE INTO cms_sensitive_words (id, word, level, created_at) VALUES (?, ?, 1, ?)');
    demo.forEach((w, i) => ins.run('sw_' + (i + 1), w, new Date().toISOString()));
  }
```

逐段拆解这套迁移机制：

1. **`addCol` 辅助函数**：先 `PRAGMA table_info(表)` 查出当前所有列，**只有当目标列不存在时**才 `ALTER TABLE ... ADD COLUMN`。这保证了幂等——启动 100 次也只加一次。
2. **`try/catch` 吞异常**：即使某个 `ALTER` 失败（比如列因并发已存在），也不会中断整个启动流程。
3. **迁移里还能补建新表**：像 `cms_sensitive_words` 是后期才加的功能表，放在 `migrateColumns` 里 `CREATE TABLE IF NOT EXISTS` 补建，老库升级即自动拥有。
4. **补建后顺手播种**：新表建好后检查 `COUNT(*)`，为空则写入演示数据。

ORM 层也有一套**对应的迁移机制**，用于模块化模型（`server/modules/orm/orm-model.ts:53-87`）：

```ts
// server/modules/orm/orm-model.ts:53-87
ensureTable(): void {
  const hasInherit = this._inherit;
  const cols: string[] = [];
  if (!hasInherit) {
    // 新表: 完整创建
    cols.push('id TEXT PRIMARY KEY');
    for (const [name, field] of Object.entries(this._fields)) {
      cols.push(fieldToColumnDDL(name, field));
    }
    cols.push('created_at TEXT DEFAULT CURRENT_TIMESTAMP');
    cols.push('updated_at TEXT DEFAULT CURRENT_TIMESTAMP');
    try {
      (this.db as any).db.exec(`CREATE TABLE IF NOT EXISTS ${this._name} (${cols.join(', ')})`);
    } catch (e: any) { /* race */ }
  } else {
    this.migrateColumns();   // 继承: 只加增量列
  }
}

migrateColumns(): void {
  if (!this._inherit) return;
  const existing = this.getTableColumns(this._inherit);
  for (const [name, field] of Object.entries(this._fields)) {
    if (!existing.includes(name)) {
      try {
        (this.db as any).db.exec(`ALTER TABLE ${this._inherit} ADD COLUMN ${fieldToColumnDDL(name, field)}`);
      } catch (e: any) { /* 列已存在, 忽略 */ }
    }
  }
}
```

🔑 这里体现了 **Odoo 式"模型即表结构"**：模型的 `_fields` 声明经 `fieldToColumnDDL()` 转成 DDL 自动建表；用 `_inherit` 继承已有表时，只 `ALTER TABLE ADD COLUMN` 增量补字段——和 `database.service.ts` 的 `migrateColumns()` 思路完全一致。第 06、08 章会深入这套 ORM。

## 实战代码：给一张表安全地加字段

假设你要给 `employees` 加一个"微信号"字段。**不要**手改建表语句（老库不会重建）。正确做法是在 `migrateColumns()` 里加一行：

```ts
// 在 server/modules/database/database.service.ts 的 migrateColumns() 中追加
addCol('employees', 'wechat', "TEXT DEFAULT ''");
```

重启服务，老库自动多出 `wechat` 列，**已有员工数据一条不丢**。这就是增量迁移的威力。

## 管理者视角

| 关注点 | 数据库层给出的答案 |
|--------|-------------------|
| **升级会不会丢数据** | 不会。迁移策略是"只加列、从不删列/重建表"，历史数据永远保留。升级 = 换代码重启，无需 DBA 手动跑脚本。 |
| **数据存在哪、怎么备份** | 默认全在 `data/ehr.db` 单文件（含 `-wal`/`-shm` 伴生文件）。冷备份 = 停服拷贝三个文件；热备份见接口 `backup()`。 |
| **企业长大了扛不住怎么办** | 改一个环境变量 `DB_DRIVER=postgres` 即可切到 PostgreSQL，业务代码零改动。平滑度极高。 |
| **风险** | SQLite 是单写锁，高并发写入（如秒杀）会排队。若并发写密集，建议直接上 PostgreSQL。 |

给决策者一句话：**"自动增量迁移 + 单文件数据 + 一键换库"三件套，把数据库运维成本压到了最低**——中小企业无需专职 DBA。

## 注意事项

- ⚠️ **加字段只能走 `migrateColumns()`**，不能改 `createTables()` 里的建表语句——因为 `CREATE TABLE IF NOT EXISTS` 对已存在的老库不生效，新字段加不上。
- ⚠️ **JSON 字段读写要 `JSON.parse` / `JSON.stringify`**：`roleIds`、`tags`、`custom_fields` 存的是字符串，直接当数组用会出错。
- ⚠️ **WAL 模式下有三个文件**：`ehr.db`、`ehr.db-wal`、`ehr.db-shm`。备份要**三个一起拷**，或先执行一次 checkpoint 合并。
- ⚠️ **驱动层只支持 SQLite / PostgreSQL**，`DriverType = 'sqlite' | 'postgres'`。别被 `package.json` 里的 `mysql2` 误导。
- 🔑 迁移函数**每次启动都跑**且幂等，这是"免运维升级"的设计核心，不是 bug。
