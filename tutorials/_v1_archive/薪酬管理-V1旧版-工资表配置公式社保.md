# 16-薪酬管理：工资表 / 配置 / 公式 / 社保

本篇带你以代码级视角吃透 **Sowork AI 企业智能ERP系统**（以下简称 Sowork AI）v1.2.0 的薪酬模块。你用 `admin/admin123` 登录，访问 `http://localhost:3000`，从薪酬首页一路走到企业缴纳（社保公积金）。

## 核心概念

薪酬模块有 4 个核心实体，记住它们你就握住全局：

| 概念 | 承载表 | 作用 |
|------|--------|------|
| 薪资项 | `salary_items` | 工资条里的每一行（基本工资/加班费/迟到扣款…） |
| 薪资公式 | `salary_items.formula` | 决定该行金额如何算出来 |
| 工资表 | `salaries` | 每月算薪结果留档，一人一月一条 |
| 社保 | `insurance_*` / `company_contributions` | 个人与公司的五险一金 |

薪资项分五类（前端枚举）：`earnings`（应发）、`allowance`（补贴）、`deductions`（扣款）、`insurance`（社保）、`tax`（税）。计算时按类别累加，决定"税前/扣减/实发"。

### 薪酬数据模型总览表

| 表名 | 关键字段 | 写入方 |
|------|----------|--------|
| `salary_items` | id, name, code, type, formula, defaultValue, isTaxable, sortOrder, isActive, category | 薪资配置/公式页 |
| `salaries` | id=`salary_{员工}_{月份}`, employeeId, month, baseSalary, positionSalary, performance, overtime, mealAllowance, transportAllowance, otherAllowance, socialInsurance, medicalInsurance, housingFund, otherDeduction, tax, grossSalary, netSalary, company*(6项), companyTotal, status | `/salary/generate` |
| `insurance_schemes` | city, isActive, 个人/公司费率方案 | 引擎 `calculateInsurance` 读取 |
| `insured_employees` | 谁参保、用哪套方案、基数 | 注册于 ALLOWED（UI 未直接维护） |
| `insurance_ledger` | 社保缴纳台账：个人/公司金额留痕 | 注册于 ALLOWED（UI 未直接维护） |
| `company_contributions` | id, employeeId, month, pension, medical, unemployment, injury, maternity, housingFund, enterpriseAnnuity, total | 企业缴纳页手动维护 |
| `payroll_records` | employeeId, period, basePay, overtime, bonus, deduction, socialSecurity, housingFund, tax, netPay, status | 模型扩展（发放记录） |

员工表通过 `_inherit` 被薪资模块扩展（`modules/hr_payroll/models/models.js:4`），新增 `baseSalary`、`socialSecurity`、`housingFund`、`bankAccount` 等字段，其中敏感字段带 `groups` 权限限定——这是薪资保密的权限基础。

## 薪酬首页

### ① 界面布局

首页组件是 `client/src/pages/SalaryPage/SalaryPage.tsx`。顶部是 6 张统计卡：本月人数、应发总额、社保公积金、代扣个税、实发总额，以及"草稿/确认/已发"计数。下方"⚡ 快捷操作"4 张卡片，分别跳转到工资表、公式引擎、薪资配置、社保公积金。再下方是一个大卡片，内含 4 个标签页：`📊 薪资总览`、`📝 薪资记录`、`⚙️ 薪资项目`、`📈 薪资分析`。

界面顶部还有一个月份选择器（默认当前月），全局过滤数据。

### ② 操作流程

1. 选择统计月份，看 6 张卡的总盘。
2. 在"薪资记录"页签里，对 `draft` 状态的记录做**确认 / 编辑 / 删除 / 发放**。
3. 在"薪资项目"页签里，按 5 类（应发/补贴/扣款/社保/税）查看并新增/编辑薪资项。
4. 在"薪资分析"页签里看薪资档位分布（5k 以下到 20k 以上）和收入/扣款构成。

薪资记录的确认走 `confirmSalary`（`SalaryPage.tsx:108`），本质是 `PUT /api/salaries/{id}` 把 `status` 改为 `confirmed`；发放 `paySalary`（`:113`）改 `status=paid` 并记录 `paidAt`；批量确认 `batchConfirm`（`:148`）对当月所有草稿统一确认。

### ③ 底层逻辑与数据模型

首页只读两张表：`salaries`（通过 `/api/salaries` 拉取，`:82`）和 `salary_items`（`:83`）。统计指标在 `currentMonthRecords` 上做 `reduce` 聚合（`SalaryPage.tsx:91-106`）。注意首页的 `ISalaryRecord` 接口（`:15-36`）字段与 `salaries` 表一一对应。

### ④ 源码剖析

首页的编辑保存 `handleSaveRecord`（`:124-146`）是**前端自己重算**的：

```ts
const grossSalary = baseSalary + positionSalary + performance + overtime
  + mealAllowance + transportAllowance + otherAllowance;
const netSalary = grossSalary - socialInsurance - medicalInsurance
  - housingFund - otherDeduction - tax;
```

它把所有项简单加总，再扣掉三险一金与个税。这与后端 `calculateSalary` 的累加逻辑不同（见"薪资公式"节），属于首页手工录改场景的简化口径。

### ⑤ 原理剖析

首页本质是**看板 + 手工台账层**：它既展示 `generate` 生成的算薪结果，又允许 HR 对个别记录手动调整。批量"确认/发放"只是推进 `status` 状态机（draft→confirmed→paid），并不重新算薪——真正的自动算薪在"薪资公式"页触发。

## 工资表

### ① 界面布局

`client/src/pages/SalaryPage/SalaryTablePage.tsx` 以 4 个标签页组织：`📋 个人月工资`、`🏢 部门汇总`、`📊 工资报表`、`📝 薪资调整`。

"个人月工资"左侧有月份、部门、状态三个筛选器，加"重置/导出 CSV"按钮；上方一行汇总卡（人数/应发总额/扣除总额/实发总额/平均工资）；主体是宽表格，列含基本工资、岗位工资、绩效、加班、补贴、应发、社保公积金、个税、实发、状态、操作。"部门汇总"按部门聚合人数与工资。"工资报表"是 6 个可点报告卡（个人明细、部门汇总、年度汇总、发放记录、社保台账、调薪记录）。"薪资调整"目前是占位引导页。

### ② 操作流程

1. 选月份 + 部门/状态筛选，看个人工资明细。
2. 点"详情"右侧抽屉看完整工资条；点"发放"标记 `paid`；点删除从 `salaries` 移除。
3. 点"📥 导出CSV"，浏览器下载 `工资表_{月份}.csv`（含 BOM 头，Excel 不乱码）。
4. 切到"部门汇总"看各部门平均实发；切"工资报表"查看各类报表卡。

### ③ 底层逻辑与数据模型

数据全部来自 `salaries` 表（`SalaryTablePage.tsx:73` 用 `GET /api/salaries?month=...`）。`Salary` 接口（`:10-38`）除基础项外还有 `companyPension…companyHousingFund`、`companyTotal` 等公司侧字段——它们由 `/salary/generate` 写入（见后文），但本页个人表格未展示，仅保存在库里。部门维度靠前端把 `employeeId` join 到 `employees` 的 `department` 字段得出（`getEmpDept`，`:90`）。

### ④ 源码剖析

导出 CSV 的 `handleExport`（`:135-149`）把过滤后的记录拼成表头 + 行，用 `Blob` 下载：

```ts
const csv = ['\ufeff'+headers.join(','), ...rows.map(r => r.join(','))].join('\n');
const blob = new Blob([csv], { type: 'text/csv;charset=utf-8' });
```

部门汇总 `getDepartmentSummary`（`:100-115`）按部门分组累加 `netSalary/grossSalary/base`，并算平均实发。注意它用 `getEmpDept(s.employeeId)` 而非 `employeeName` 反查部门，因此员工表存在是汇总正确的前提。

### ⑤ 原理剖析

工资表是 `salaries` 的**只读报表层**。它不重新算薪，只对已生成记录做筛选、导出、状态推进。状态机仍是 draft→confirmed→paid，与首页共用同一套字段。

## 薪资配置

### ① 界面布局

`client/src/pages/SalaryPage/SalaryConfigPage.tsx` 有 4 个标签页：`工资项管理`、`职级工资`、`工资方案`、`薪资分析`。

"工资项管理"是主表，列出每个薪资项的名称、编码、类型、数据类型、小数位、计税、状态，并提供新增/编辑/删除。"职级工资"是一个"职级 × 应发项"的矩阵输入框（用于定薪参考）。"工资方案"展示默认/高管/销售三套静态方案卡 + 公式示例。"薪资分析"可按部门 + 月份调用 `/api/salary_analysis` 做统计（人数/平均/总额/最高/最低）。

### ② 操作流程

1. 在"工资项管理"点"+ 新增工资项"，填名称、编码、类型、默认值、是否计税、是否启用，可选填公式。
2. 保存即 `POST /api/salary_items`；编辑即 `PUT /api/salary_items/{id}`（`handleItemSubmit`，`:77-104`）。
3. 在"职级工资"里按职级填各应发项标准金额（注意：保存仅 `alert` 提示，未真正落库，属 UI 占位）。
4. 在"薪资分析"选部门 + 月份，点"开始分析"看分布。

### ③ 底层逻辑与数据模型

核心仍是 `salary_items` 表。`SalaryItem` 接口（`SalaryConfigPage.tsx:3-16`）字段：`id, name, code, type, dataType, decimalPlaces, formula, defaultValue, isTaxable, sortOrder, isActive, category`。

⚠️ **类型枚举不一致**：本页的 `type` 联合类型是 `earnings | deduction | insurance | tax`（`:7`），**缺少 `allowance` 且用单数 `deduction`**；而"薪资公式"页与引擎默认项用的是 `deductions`（复数）、并含 `allowance`（`salary-formula-engine.ts:855-861`）。这意味着在本页新建的"补贴项"无法选择，且若选了 `deduction` 单数，与引擎约定 `deductions` 对不上。这是前端与引擎之间一个真实的命名口径差异，配置时应优先在"薪资公式"页建项。

### ④ 源码剖析

保存逻辑 `handleItemSubmit`（`:77`）把表单拼成 `{...formData, isActive, isTaxable}` 后写库。注意它**没有调用 `validate-formula`**——公式语法校验只在"薪资公式"页保存时做（见下节）。因此在本页填错公式不会即时报错，要到"公式"页保存或实际计算时才暴露。

```ts
const url = editingItem ? `/api/salary_items/${editingItem.id}` : '/api/salary_items';
const method = editingItem ? 'PUT' : 'POST';
```

### ⑤ 原理剖析

薪资配置是 `salary_items` 的**元数据管理面**。它定义"工资条有哪些行、每行是什么类型、默认值多少"，但不触发计算。真正的公式求值与算薪在"薪资公式"页。两页都读写同一张 `salary_items`，但语义侧重不同。

## 薪资公式

### ① 界面布局

`client/src/pages/SalaryPage/SalaryFormulaPage.tsx` 是算薪主引擎页。右上角有月份选择器 + `计算薪资` + `生成薪资记录` 两个主按钮；下方 3 个标签页：`📋 薪资项配置`、`📝 公式模板`、`🔍 计算预览`。

"薪资项配置"列出所有薪资项，含公式、计税、状态，可编辑/预览/删除，并有"公式说明"提示（支持变量与 IF/ROUND/MAX/MIN/ABS）。"公式模板"给出 10 个常用模板（如 `加班小时数 * 50`、`IF(迟到次数 > 3, 200, 0)`）。"计算预览"可输入公式即时测试，并显示全员汇总。

### ② 操作流程

1. 在"薪资项配置"点"新增薪资项"，填公式后，前端会先调 `validate-formula` 校验（`:124`）。
2. 在"公式模板"一键套用模板到公式框并预览。
3. 点右上"计算薪资"（`batch-calculate`），对全体在职员工算一遍；点"生成薪资记录"（`generate`）把结果落库到 `salaries`。
4. 在"计算预览"输入任意公式 + 测试数据，看 `preview-formula` 返回结果。

### ③ 底层逻辑与数据模型

公式引擎在 `server/salary-formula-engine.ts`。`SalaryContext`（`salary-formula-engine.ts:26-37`）是公式能引用的全部变量：`baseSalary`、`positionSalary`、`performance`、`overtimeHours`、`lateCount`、`absentCount`、`leaveDays`，以及 `employee` 对象。保存项时这些中文变量名会被 `replaceVariables`（`:106`）替换成真实数值。

### ④ 源码剖析

保存薪资项时，前端 `handleSaveItem`（`:118-155`）先请求 `POST /api/salary/validate-formula`：

```ts
const validRes = await fetch('/api/salary/validate-formula', { method:'POST',
  body: JSON.stringify({ formula: values.formula }) });
if (!validData.valid) { messageApi.error(`公式语法错误: ${validData.error}`); return; }
```

后端 `validateFormula`（`salary-formula-engine.ts:813`）做两件事：检查括号匹配（`:820-826`），再用测试上下文试执行（`:841`）。保存项接口 `/salary/items`（`standalone.ts:1777`）也会强制校验（`:1783-1789`），公式错误直接 400 拒绝——这是防止"算错工资"的关键防线。

"计算薪资"按钮走 `handleCalculate`（`:184`）→ `POST /api/salary/batch-calculate`（`:698`）→ `batchCalculateSalary`（`salary-formula-engine.ts:764`），遍历在职员工逐个 `calculateSalary`。"生成薪资记录"走 `handleGenerateRecords`（`:221`）→ `POST /api/salary/generate`（`:1840`），把结果写库，主键 `salary_{员工}_{月份}`，重复生成自动覆盖（`:1878-1883`）。

核心是 `calculateSalary`（`salary-formula-engine.ts:468`）：取员工 → 取启用的 `salary_items` → `buildSalaryContext` 汇总考勤 → 逐项 `evaluateFormula` 求值 → `calculateEmployeeInsurance` 算社保 → `calculateTax` 算个税 → 汇总。逐项累加逻辑：

```ts
for (const item of activeItems) {
  if (item.formula) {
    const result = evaluateFormula(item.formula, context);
    value = result.success ? result.value : (item.defaultValue || 0);
  } else value = item.defaultValue || 0;
  if (item.type === 'earnings' || item.type === 'allowance') totalEarnings += value;
  else if (item.type === 'deductions' || item.type === 'insurance' || item.type === 'tax') totalDeductions += value;
}
```

`evaluateFormula`（`:79`）先 `replaceVariables` 把"基本工资"等中文变量替换为数字，再 `preprocessExpression`（`:153`）把中文括号 `（）`、运算符 `＋－×÷` 转为半角，并处理 IF/ROUND/MAX/MIN/ABS，最后 `evaluateExpression`（`:250`）用 `sanitizeExpression`（`:291`）只允许数字与运算符后安全求值。

🚀 默认开箱即用 9 个薪资项由 `getDefaultSalaryItems`（`salary-formula-engine.ts:851`）提供，如 `加班费 = 加班小时数 * 50`（`:856`）、`迟到扣款 = 迟到次数 * 50`（`:860`）。

### ⑤ 原理剖析

一次算薪是：`取员工 → 取启用薪资项 → buildSalaryContext 汇总当月考勤/加班/请假拼出变量 → 逐项 evaluateFormula 求值并累加应发/扣减 → calculateEmployeeInsurance 算五险一金 → calculateTax 算个税 → 汇总成 salaries 一条 draft 记录`。

⚠️ **两个需诚实提示的源码事实**：
- `calculateSalary` 中 `taxableIncome = totalEarnings`（`:537`），社保抵扣取 `socialSecurity = insurance.personal.total`，但 `housingFund = items['housing_fund']`（`:539`）。由于住房公积金属 `insurance` 计算结果、并非一个 `salary_items` 的 code，除非你额外建一个 code 为 `housing_fund` 的项，否则个税计算里公积金抵扣常为 0。
- `netSalary = grossSalary - totalDeductions - tax`（`:544`）。`totalDeductions` 只累加 `salary_items` 里 type 为 deductions/insurance/tax 的项；而 `calculateEmployeeInsurance` 的结果**并未作为薪资项计入 items**，所以个人五险一金在后端算薪口径里**没有被从实发中扣除**。这与首页手工编辑的扣减逻辑不一致。生产使用前建议 HR 在首页/工资表对社保做人工核对，或补充一条 insurance 类薪资项把个人社保纳入扣减。

## 企业缴纳（社保）

### ① 界面布局

`client/src/pages/SalaryPage/CompanyContributionPage.tsx` 顶部是 5 张统计卡：缴费记录数、企业缴费合计、养老保险合计、公积金合计、个人部分合计。下方一张表，列含养老、医疗、失业、工伤、生育、住房公积金、企业年金、合计，以及查看/编辑/删除操作。右上角支持按姓名搜索、按月份筛选、刷新、新增缴费。

### ② 操作流程

1. 点"新增缴费"，选员工 + 缴费月份，逐项填企业承担的养老/医疗/失业/工伤/生育/公积金/年金。
2. 保存时前端自动求和 `total = 养老+医疗+失业+工伤+生育+公积金+年金`（`handleSubmit`，`:148-169`）。
3. 点"查看"右侧抽屉看完整缴费明细；点编辑/删除维护记录。
4. 用姓名搜索或月份筛选快速定位。

### ③ 底层逻辑与数据模型

本页操作的是 `company_contributions` 表（常量 `TABLE = "company_contributions"`，`:12`）。`IRecord` 接口（`:14-30`）字段：`id, employeeId, month, pension, medical, unemployment, injury, maternity, housingFund, enterpriseAnnuity, total, createdAt`，外加前端 join 的 `employeeName/department`。

🔑 **两套社保机制并存**：Sowork AI 里社保其实有两套实现。一套是引擎侧的 `calculateInsurance`（`salary-formula-engine.ts:409`），按 `insurance_schemes` 城市费率算个人+公司两部分，被 `calculateSalary` 调用；另一套就是本页的 `company_contributions`，由 HR **手动逐人逐月**维护企业承担部分。旧版概览提到的 `insured_employees` / `insurance_ledger` 仅在 `ALLOWED`（`:1073`）注册、并被 `calculateInsurance` 通过 `insurance_schemes` 间接使用，但我在 5 个子页组件里**没有找到直接维护它们的 UI**——企业缴纳页实际落地在 `company_contributions`。这是与旧概览描述不一致的地方，请以实际 UI 为准。

### ④ 源码剖析

保存 `handleSubmit`（`:148`）把各险种 `Number()` 化后求和，再 `POST/PUT /api/company_contributions`：

```ts
const total = pension + medical + unemployment + injury + maternity + housingFund + enterpriseAnnuity;
const payload = { employeeId, month, pension, medical, unemployment, injury, maternity, housingFund, enterpriseAnnuity, total };
```

加载时 `fetchData`（`:65`）用 `GET /api/company_contributions`，并在前端把 `employeeId` join 成姓名/部门（`:76-80`）。统计卡 `stats`（`:196-202`）对列表做 reduce 聚合。

### ⑤ 原理剖析

企业缴纳页是**公司侧社保公积金的手工台账**。它与算薪链路（`calculateSalary`）解耦：算薪时公司承担部分（`companyContributions`）虽被算出来写进 `salaries.companyTotal`，但本页并不自动同步，需要 HR 另行录入或核对。费率合规的责任落在 `insurance_schemes`（引擎读取）与人工录入的准确性上。

## 管理者视角

- **算薪自动化**：过去 HR 逐人 Excel 加减，Sowork AI 一次"批量计算"（`/salary/batch-calculate`）即对全体在职员工出表，把"算薪日"从数天压到几分钟，再"生成"落库成 `salaries` 月度档案。
- **社保合规**：城市费率集中在 `insurance_schemes`，政策变动只改方案、不改代码；企业侧另用 `company_contributions` 留痕，审计可追溯。注意别长期依赖深圳默认值。
- **薪资保密**：模型字段 `groups` 把基本工资、卡号限定给 HR/财务，普通员工只能在自助端看到自己的工资条（呼应 RBAC 权限体系）。
- **薪资项可配置**：业务变化（新增补贴、调整扣款规则）由 HR 在"薪资公式"页改公式即可，无需研发介入；公式模板与预览降低门槛。

---

## 注意事项

⚠️ **公式语法错误会算错工资**。保存前务必先用"薪资公式"页的校验（即 `validate-formula`），括号不匹配或变量名写错都会被拦下；上线前可用 `preview-formula` 拿测试数据试算。

🔑 **社保基数按政策**。个人/公司比例务必以当地最新政策维护进 `insurance_schemes`；别长期依赖深圳默认值，异地员工会算错。企业缴纳页的 `company_contributions` 是人工录入，需与引擎结果对账。

✅ **工资表可按月留档**。`/salary/generate` 按 `salary_{员工}_{月份}` 写入 `salaries`，重复生成自动覆盖、不重复建条；确认无误后把 `status` 从 draft 改为 confirmed/paid，即形成不可随意篡改的月度档案。

⚠️ **权限别开太大**。基本工资、银行卡号属敏感字段，只授权给 `hr_admin / finance`，避免薪酬泄露。

⚠️ **类型口径要统一**。在"薪资配置"页新建薪资项时，类型枚举缺少 `allowance` 且用 `deduction`（单数），与引擎/公式页的 `deductions` 不一致。建议薪资项统一在"薪资公式"页维护。

## 小结与练习

你已看清：薪资项（`salary_items`）是工资条骨架，公式（`.formula`）是血肉，引擎把它们与考勤数据、`calculateInsurance`、`calculateTax` 串成一条月薪；工资表是企业侧社保的手工台账。两页共用 `salaries` 状态机。下面给每个子页一个可操作练习：

1. **薪酬首页**：用 admin 登录，选一个历史月份，在"薪资记录"页对一条 draft 点"确认"，再到数据库确认 `salaries` 该条 `status` 变为 `confirmed`。
2. **工资表**：在"个人月工资"选本月，点"导出CSV"，打开文件确认表头含"实发工资"且人名/部门正确。
3. **薪资配置**：在"工资项管理"新增一个 `earnings` 类项"交通补贴 2"，默认值 300，保存后到 `salary_items` 表确认落库。
4. **薪资公式**：写一条 `绩效工资 = 基本工资 * 0.2`，先点预览看结果，再保存（会触发 `validate-formula`）；然后对全员"计算薪资"+"生成薪资记录"，确认 `salaries` 新增 draft 记录。
5. **企业缴纳**：新增一条某员工的 `company_contributions`，填入养老/医疗/公积金等，保存后看顶部"企业缴费合计"是否随之和变化。
