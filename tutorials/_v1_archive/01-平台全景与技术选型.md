## 核心概念：一套代码，五大业务域

Sowork AI 企业智能ERP系统是一套**开箱即用的企业数字化全栈平台**。它不是"HR 系统"这么简单，而是以 HR 为核心，横向长出了五个业务域：

```
                 Sowork AI 单体全栈平台（一个 Node 进程）
   ┌──────────┬──────────┬──────────┬──────────┬──────────┐
   │   HR     │   ERP    │   CMS    │  商城     │   AI     │
   │ 人事/考勤 │ 制造/进销存│ 内容门户  │ ShopXO超集│ 对话/RAG │
   │ 薪酬/绩效 │ 采购/财务 │ 富文本/SEO│ 订单/售后 │ 工具循环 │
   └──────────┴──────────┴──────────┴──────────┴──────────┘
        └──────── 自研 ORM + RBAC + i18n + 低代码 ────────┘
                          SQLite / PostgreSQL
```

🔑 **关键判断：为什么是单体，不是微服务？**

微服务解决的是"超大团队 + 超高并发 + 独立伸缩"的问题。而 Sowork AI 的目标用户是**中小企业私有化部署**——他们要的是"一条命令跑起来、一个文件夹能备份、一台 4 核机器扛得住"。单体全栈在这个场景下有三个压倒性优势：

1. **零分布式复杂度**：没有服务发现、没有分布式事务、没有跨服务链路追踪。数据库就是一个 `data/ehr.db` 文件。
2. **部署即拷贝**：`dist/` + `data/` 打包走人，换台机器解压即用。
3. **一路读到底**：出了问题，从 HTTP 入口到 SQL 执行全在一个代码库里，`grep` 一下就能定位。

代价是"无法按模块独立扩容"——但对目标用户来说，这个代价根本不会触发。

## 源码剖析一：从 `package.json` 读技术栈

技术选型不看文档、看 `package.json` 最诚实。核心依赖（真实版本）：

```json
// package.json · dependencies（节选真实版本）
"react": "^19",              // 前端框架（React 19）
"react-router-dom": "^7",    // 前端路由
"antd": "^5",                // Ant Design 5 组件库
"@tiptap/core": "3.22.4",    // 富文本编辑器（钉死 3.22.4，见下方注意事项）
"express": "^4",             // 后端 HTTP 框架
"better-sqlite3": "^11",     // 同步 SQLite 驱动（默认数据库）
"pg": "^8",                  // PostgreSQL 驱动（可选切换）
"multer": "...",             // 文件上传
"nanoid": "...",             // 主键生成
"sharp": "...",              // 图片处理（缩略图/水印）
```

构建与启动脚本（`package.json` 第 7~16 行，**逐字**）：

```json
"scripts": {
  "dev": "concurrently -n server,client -c blue,green \"npm run dev:server\" \"npm run dev:client\"",
  "dev:server": "tsx watch server/standalone.ts",
  "dev:client": "vite --config vite.config.ts",
  "build": "npm run build:server && npm run build:client",
  "build:server": "tsc -p tsconfig.server.json && node -e \"require('fs').cpSync('server/ai-service.js','dist/server/ai-service.js'); require('fs').cpSync('server/code-agent-tools.js','dist/server/code-agent-tools.js')\"",
  "build:client": "vite build --config vite.config.ts",
  "start": "node dist/server/standalone.js",
  "type:check": "tsc --noEmit"
}
```

逐段拆解你需要理解的三个关键点：

- **`dev:server` 用 `tsx watch`**：开发期直接跑 TypeScript 源码（无需先编译），改文件即热重启。
- **`build:server` 后面那段 `node -e ...cpSync(...)`**：这是个**重要伏笔**。`server/ai-service.js` 和 `server/code-agent-tools.js` 是**手写的 `.js` 文件**（不是 TS 编译产物），`tsc` 不会处理它们，所以构建时必须手动 `cpSync` 拷贝到 `dist/`。忘了这一步，AI 功能在生产环境就会"文件找不到"。
- **`start` 只启一个进程**：`node dist/server/standalone.js`——前端已被 Vite 打包成静态文件，由这个 Node 进程一并托管。生产环境**单端口、单进程**。

## 源码剖析二：一个进程如何同时跑起五大业务域

答案在 `server/standalone.ts` 顶部的启动编排（第 61~155 行）。这段代码是整个系统的"点火顺序"，逐段读：

```ts
// server/standalone.ts:61-67
const app = express();
const db = new DatabaseService();
db.onModuleInit();                          // ① 打开 SQLite + 建表 + 迁移 + 种子

// 全局暴露数据库引用(供API使用)
(global as any).__feida_db = db;
(global as any).__feida_raw_db = (db as any).db;
```

第一步就 `new DatabaseService()` 并同步 `onModuleInit()`——因为用的是 `better-sqlite3`（同步驱动），建表可以同步完成，不需要 `await`。然后把 db 挂到 `global`，让后面几千行的路由闭包都能直接拿到。

```ts
// server/standalone.ts:86-95（模块系统，参照 Odoo ir.module.module）
const { ModuleRegistry } = require('./modules/module-registry');
const moduleRegistry = new ModuleRegistry(db, path.join(__dirname, '..', '..', 'modules'));
moduleRegistry.discoverAll();
// 自动安装依赖已满足的模块
for (const m of moduleRegistry.list()) {
  if (m.auto_install && moduleRegistry.checkDependencies(m)) {
    try { moduleRegistry.install(m.name); } catch (e: any) { console.error('[Module] install failed:', m.name, e.message); }
  }
}
```

第二步扫描 `modules/` 目录，把每个业务模块（HR、blog、forum……）注册进来并按依赖自动安装——这是 **Odoo 式模块化**的核心。

```ts
// server/standalone.ts:97-113（ORM 模型注册）
const { ModelRegistry } = require('./modules/orm/model-class');
const modelRegistry = new ModelRegistry(db);
const modulesDir = path.join(__dirname, '..', '..', 'modules');
if (require('fs').existsSync(modulesDir)) {
  for (const entry of require('fs').readdirSync(modulesDir, { withFileTypes: true })) {
    if (!entry.isDirectory()) continue;
    const dir = path.join(modulesDir, entry.name);
    try {
      const loaded = modelRegistry.loadFromModule(dir);
      if (loaded.length > 0) console.log(`[ORM] ${entry.name}: +${loaded.length} 模型`);
    } catch (e: any) { console.error('[ORM] 模型加载失败:', entry.name, e.message); }
  }
}
registerCoreModels(modelRegistry);   // 手动注册未模块化的核心表
```

第三步把每个模块 `models/` 目录下的模型定义加载进 ORM 注册表（第 06 章会逐行读这套 ORM）。

```ts
// server/standalone.ts:142-155
const { Environment } = require('./modules/orm/environment');
const env = Environment.createProxy(new Environment((db as any).db, modelRegistry, { uid: 1 }));
(global as any).__feida_env = env;
...
// CMS & Shop 种子数据
const seedCmsShop = require('./seed-cms-shop');
seedCmsShop.default(db);
```

第四步创建 Odoo 风格的 `Environment`（记录集引擎入口），最后播种 CMS/商城的初始数据。

✅ 读完这 90 行，你就掌握了整个系统的点火链：
**建库建表 → 发现模块 → 注册模型 → 建 ORM 环境 → 播种数据 → 启动 HTTP**。五大业务域"同时跑起来"，本质就是它们的表、模型、路由都在这一个进程里被依次装配。

## 实战代码：用一条 SQL 验证"五域同库"

跑起来后（第 03 章讲搭建），打开数据库看看五大业务域的表是不是真在同一个库里：

```bash
# 用 sqlite3 直接看表清单（Windows 可用 DB Browser for SQLite）
sqlite3 data/ehr.db ".tables" | tr ' ' '\n' | grep -E "^(employees|shop_goods|cms_articles|ai_knowledge|production_plans)$"
```

预期输出（五域各命中一张代表表）：

```
employees          # HR
shop_goods         # 商城
cms_articles       # CMS
ai_knowledge       # AI 知识库
production_plans   # ERP 制造
```

这五张表同在 `data/ehr.db` 一个文件里——这就是"单体全栈"最直观的证据。

## 管理者视角

| 维度 | 单体全栈对企业意味着什么 |
|------|------------------------|
| **上手成本** | 不需要 K8s、不需要中间件集群。一台普通服务器（4 核 8G）即可承载中小企业全员使用。 |
| **数据资产** | 全部数据在一个 `data/ehr.db`（或一个 PostgreSQL 库）。备份 = 拷一个文件；迁移 = 换台机器解压。数据主权 100% 在企业自己手里。 |
| **功能覆盖** | 一套系统替代了"HR SaaS + 电商 SaaS + 官网 CMS + BI 工具"的组合，省下多份订阅费与多套账号体系的对接成本。 |
| **风险** | 单点：进程挂了全站不可用。缓解方式见第 14 章（进程守护 + 定时备份 + 可选双机）。 |
| **选型建议** | 若企业规模 < 500 人、并发 < 200，单体全栈是性价比最高的选择；超过此规模再考虑第 04 章的 PostgreSQL 切换与横向拆分。 |

一句话给决策者：**它把"买五套系统 + 做集成"压缩成了"部署一套系统"**，这是它最大的商业价值。

## 注意事项

- ⚠️ **Tiptap 版本必须钉死 `3.22.4`**：新增任何 `@tiptap/extension-*` 扩展都要用 `3.22.4`，写 `^3.22.4` 会解析到 3.27.x 触发 peer 冲突（第 10 章详述）。
- ⚠️ **`package.json` 历史上不完整**：`antd`、`sharp`、`pg` 等曾被源码 import 但未声明。**永远用 `npm install --legacy-peer-deps`，且不要裸跑 `npm install X` 去"修复"**——会把未声明包当垃圾清掉，导致前端编译报 `failed to resolve import "antd"`（第 03 章有完整源码级解释）。
- ⚠️ **`ai-service.js` / `code-agent-tools.js` 是手写 JS**：不经 `tsc` 编译，靠 `build:server` 里的 `cpSync` 拷贝。改了它们记得重新 `build`。
- 🔑 生产环境是**单端口单进程**：前端静态资源由 Node 进程托管，不需要单独的 Nginx（可选加，见第 14 章）。
