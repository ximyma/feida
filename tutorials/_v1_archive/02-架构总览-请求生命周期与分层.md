## 核心概念：一次请求的完整旅程

先建立全局地图。当浏览器发出 `GET /api/employees` 时，请求在服务端经过这样一条链路：

```
浏览器
  │  GET /api/employees
  ▼
express()  ── app.use(express.json())            ① 解析 JSON body
  │        ── app.use(loginSecurityMiddleware)    ② 登录安全（限速/锁定）
  ▼
app.use('/api', apiAuthMiddleware, apiRouter())   ③ API 鉴权 + 业务路由
  │
  ├─ 具名路由匹配？ /rbac/* /ai/* /shop-* /cms-* ...  ── 命中则处理
  │
  └─ 落到 catch-all /:table  ── ALLOWED 白名单校验 ── db.findAll('employees')
  ▼
res.json(data)  ── 返回
```

而当浏览器请求 `GET /admin/cms`（前端页面路由）时，走的是**另一条链路**：

```
app.use(express.static(clientDir))   ── 静态文件没命中
  ▼
app.get('*', ...)  ── SPA 回退，返回 client/index.html
  ▼
浏览器接管，由 React Router 渲染 /admin/cms 页面
```

🔑 记住这个分叉：**`/api/*` 走后端 API，其它一切走 SPA 回退**。这是单体全栈"前后端同端口"的核心机制。

## 源码剖析一：全局中间件链只有两个

很多人以为企业系统的中间件会堆一大摞。Sowork AI 恰恰相反——**全局只挂两个**（`server/standalone.ts:159-161`）：

```ts
// server/standalone.ts:159-161
app.use(express.json());
app.use(loginSecurityMiddleware);
const upload = multer({ dest: uploadDir });
```

逐行读：

- **`express.json()`**：把请求体解析成 `req.body`。所有 POST/PUT 的 JSON 都靠它。
- **`loginSecurityMiddleware`**：登录安全中间件（第 05 章逐行精读），负责 IP 限速与账号锁定。它挂在**全局最前面**，意味着**每个请求**都先过一遍安全检查。
- **`multer`**：只是创建了上传实例，并没有 `app.use`——它按需挂在具体的文件上传路由上，不污染全局。

设计哲学：**全局中间件越少，请求链路越可预测，排障越快**。鉴权不放全局，而是放在 `/api` 这一层（见下文），因为前台 CMS/商城的很多接口是允许匿名访问的。

## 源码剖析二：单一 `apiRouter()` 集中式路由

这是 Sowork AI 一个容易被误解的设计。它**没有**把路由按模块拆成几十个子 `Router` 文件，而是把**全部业务路由集中定义在一个 `apiRouter()` 函数内**：

```ts
// server/standalone.ts:208-209
function apiRouter() {
  const router = express.Router();
  // ... 此后数千行，cms-* / shop-* / rbac/* / ai/* / lowcode/* 全部在这里定义
```

然后统一挂载（`server/standalone.ts:11856-11860`）：

```ts
// server/standalone.ts:11856-11860
app.use('/uploads', express.static(uploadDir));

app.use('/api', apiAuthMiddleware, apiRouter());

app.use('/addons', express.static(path.join(__dirname, '..', '..', 'addons'), { maxAge: 0, etag: false }));
```

注意 `app.use('/api', apiAuthMiddleware, apiRouter())` 这一行的**三段结构**：

1. `'/api'`：所有业务接口的统一前缀。
2. `apiAuthMiddleware`：**API 层鉴权中间件**——只对 `/api/*` 生效，前台页面不受影响。
3. `apiRouter()`：返回那个装满路由的 `express.Router` 实例。

🔑 **为什么集中式而非拆分？** 这是刻意的权衡：

- **优点**：路由注册顺序完全可控（这对 catch-all 路由至关重要，见下文"铁律"）；跨模块共享闭包变量（如 `db`、`applyStockChange`）无需层层传参；`grep 'router.post'` 一次看全所有接口。
- **代价**：`standalone.ts` 单文件近 12000 行。但配合编辑器的符号跳转与 `grep`，维护成本可控；且避免了微拆分带来的"文件迷宫"。

## 源码剖析三：catch-all 通用 CRUD 与白名单闸口

Sowork AI 有 200+ 张表，如果每张表都手写增删改查会爆炸。它的解法是**一套 catch-all 路由服务所有"标准表"**（`server/standalone.ts:4065-4088`）：

```ts
// server/standalone.ts:4065-4088 — GET /:table
router.get('/:table', (req, res, next) => {
  const { table } = req.params;
  const RESERVED_ROUTES = new Set(['workflows', 'approval', 'workflow', 'ai', 'mrp', 'sitemap.xml']);
  // 带连字符的路径交给专用路由处理
  if (table.includes('-') || RESERVED_ROUTES.has(table)) { return next(); }
  if (!ALLOWED.includes(table)) { res.json({ error: 'Invalid table' }); return; }
  const skipKeys = new Set(['_', 'limit', 'offset', 'page', 'pageSize', 'sort', 'order', 'fields']);
  const filters = Object.fromEntries(Object.entries(req.query).filter(([k]) => !skipKeys.has(k)));
  const limitVal = req.query.limit ? parseInt(req.query.limit as string) : undefined;
  const offsetVal = req.query.offset ? parseInt(req.query.offset as string) : undefined;
  let data;
  try {
    if (Object.keys(filters).length > 0) {
      data = limitVal ? db.findWhere(table, filters, limitVal) : db.findWhere(table, filters);
    } else {
      data = db.findAll(table);
      if (limitVal) data = data.slice(offsetVal || 0, (offsetVal || 0) + limitVal);
    }
  } catch(e) {
    data = db.findAll(table);
    if (limitVal) data = data.slice(offsetVal || 0, (offsetVal || 0) + limitVal);
  }
  res.json(data);
});
```

逐段拆解三个精妙之处：

1. **`if (table.includes('-') || RESERVED_ROUTES.has(table)) return next()`**：带连字符的路径（如 `cms-articles`、`shop-goods`）和保留字（`ai`、`workflow`）**不由 catch-all 处理**，而是 `next()` 放行给后面更具体的路由。这就是"通用兜底 + 专用覆盖"的分层。
2. **`if (!ALLOWED.includes(table)) return { error: 'Invalid table' }`**：**安全闸口**。只有在白名单 `ALLOWED` 里的表名才允许被通用读写，防止攻击者用 `/api/sqlite_master` 之类探测系统表。
3. **`skipKeys` 过滤**：把 `limit/offset/page/sort` 这些分页排序参数从"查询条件"里剔除，剩下的才当作字段过滤条件传给 `db.findWhere`。

写入侧（`server/standalone.ts:4097-4115`）更能体现"防御式设计"：

```ts
// server/standalone.ts:4097-4115 — POST /:table
router.post('/:table', (req, res, next) => {
  const { table } = req.params;
  const RESERVED_ROUTES = new Set(['workflows', 'approval', 'workflow']);
  if (table.includes('-') || RESERVED_ROUTES.has(table)) { return next(); }
  if (!ALLOWED.includes(table)) { res.json({ error: 'Invalid table' }); return; }
  const rawBody = req.body;
  if (!rawBody.id) rawBody.id = `${table.slice(0,3)}_${Date.now()}`;
  // 只保留目标表真实存在的字段
  let data: Record<string, any> = { ...rawBody };
  try {
    const tableInfo = db.query(`PRAGMA table_info("${table.replace(/"/g, '""')}")`);
    const cols = tableInfo.map((c: any) => c.name);
    data = Object.fromEntries(Object.entries(data).filter(([k]) => cols.includes(k)));
    if (cols.includes('createdAt') && !data.createdAt) data.createdAt = new Date().toISOString();
    if (cols.includes('updatedAt')) data.updatedAt = new Date().toISOString();
  } catch {}
  res.json(db.insert(table, data));
});
```

🔑 注意 `PRAGMA table_info` 那段：写入前先查表的真实列，**只保留数据库里存在的字段**再插入。这让前端可以"多传字段无所谓"，后端自动过滤，避免 `no such column` 报错——这是"约定优于配置"的典型体现。

白名单本身很长（`server/standalone.ts:1060` 起），覆盖基础、考勤、薪资、招聘、绩效、后勤、进销存、CMS、电商等模块表：

```ts
// server/standalone.ts:1060 起（节选头部）
const ALLOWED = [
  'employees','departments','positions','ranks',
  'users','roles','permissions',
  'shift_types','schedules','attendance_records','leave_records','leave_balances',
  // ... 一直到 blog_posts / blog_categories / blog_comments
];
```

## 源码剖析四：静态资源与 SPA 回退

请求旅程的另一条分支——非 `/api` 请求怎么处理（`server/standalone.ts:11862-11883`）：

```ts
// server/standalone.ts:11862-11883
const clientDir = path.join(__dirname, '../client');
app.use(express.static(clientDir, {
  setHeaders: (res, filePath) => {
    if (filePath.endsWith('.js') || filePath.endsWith('.mjs')) {
      res.setHeader('Content-Type', 'application/javascript; charset=utf-8');
    } else if (filePath.endsWith('.css')) {
      res.setHeader('Content-Type', 'text/css; charset=utf-8');
    } else if (filePath.endsWith('.svg')) {
      res.setHeader('Content-Type', 'image/svg+xml');
    }
  },
}));

app.get('*', (_req, res) => {
  const indexPath = path.join(__dirname, '../client/index.html');
  if (require('fs').existsSync(indexPath)) {
    res.sendFile(indexPath);
  } else {
    res.status(200).send('<h1>...</h1><p>请先运行 npm run build 构建前端</p>');
  }
});
```

- **`express.static` 的 `setHeaders`**：强制修正 MIME 类型。注释里写得很直白——是为了修复浏览器报的 `Expected JavaScript module but got octet-stream`（某些环境下 `.js` 被当成八位字节流，ESM 模块加载失败）。
- **`app.get('*')`**：所有没被前面路由命中的 GET 请求，一律返回 `index.html`。这样 React Router 的 `/organization`、`/admin/cms` 等前端路由刷新页面也不会 404。

监听收尾（`server/standalone.ts:11885-11892`）：

```ts
// server/standalone.ts:11885-11892
const PORT = Number(process.env.SERVER_PORT) || 3000;
const HOST = process.env.SERVER_HOST || '0.0.0.0';
const server = http.createServer(app);
server.listen(PORT, HOST, () => {
  console.log(`[Bootstrap] Server running on http://${HOST}:${PORT}`);
  console.log(`[Bootstrap] API ready at http://${HOST}:${PORT}/api`);
});
```

端口来自 `SERVER_PORT`（默认 3000），绑定 `0.0.0.0` 意味着局域网内其它机器也能访问。

对照上面两条链路图，你能清楚看到 `/api/employees`（catch-all）和 `/api/cms-articles`（专用）分别走了哪条路。

## 管理者视角

| 关注点 | 架构给出的答案 |
|--------|---------------|
| **系统好不好维护** | "约定优于配置"——新增一张标准表只要加进 `ALLOWED` 白名单就自动有了增删改查，不用写代码。功能迭代快、人力成本低。 |
| **安全边界在哪** | 三道闸口：全局登录安全中间件、`/api` 层鉴权、catch-all 白名单。攻击者无法通过通用接口触达系统表。 |
| **性能瓶颈** | 单进程内存路由匹配，无网络跳转，延迟极低。瓶颈通常在数据库（SQLite 写锁），见第 04 章的 PG 切换方案。 |
| **风险提示** | 集中式路由文件大，新人上手需要一点时间；建议配合本教程的"源码锚点行号"快速定位。 |

给决策者一句话：**这套架构用"少量约定 + 一个白名单"换来了极低的功能扩展成本**，非常契合需要快速响应业务变化的中小企业。

## 注意事项

- ⚠️ **catch-all 路由的"铁律"**：任何具名路由（`/rbac/*`、`/i18n/*`、`/cms-*`、`/shop-*`）都**必须注册在 `/:table`（约 4065 行）之前**。否则请求会被 catch-all 抢先匹配，报 `Invalid table`。第 05、12 章会反复强调这条。
- ⚠️ **新增标准表别忘了加白名单**：建了表但没加进 `ALLOWED`，通用接口会返回 `Invalid table`。
- ⚠️ **前端路由不要以 `/api` 开头**：否则会被后端 API 层拦截，无法回退到 SPA。
- 🔑 带连字符的接口路径（如 `cms-articles`）是"专用路由"的标志——它们绕过 catch-all，有自己的业务逻辑。
