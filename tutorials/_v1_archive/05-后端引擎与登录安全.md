## 核心概念：单入口 + 中间件流水线

Sowork AI 后端只有一个入口 `server/standalone.ts`。所有请求按注册顺序流过一条"中间件流水线"：

```
HTTP 请求
  │
  ├─ loginSecurityMiddleware   # 仅拦 /api/login：IP 限速 + 账户锁定
  ├─ /api/* → 鉴权 → apiRouter  # 业务路由（含 rbac/i18n/cms/shop…）
  ├─ /uploads/*  → 静态文件
  ├─ /addons/*   → 插件配置文件
  └─ /*          → 托管 dist/client + SPA fallback（刷新子路由不 404）
```

> 🔑 **顺序即安全边界**：先拦登录爆破，再拦 API 越权，最后才托管页面。把守卫放在路由之前，等价于"全局门禁"。

## 源码剖析一：登录安全中间件

登录接口是最容易被爆破的入口。`login-security.ts` 做了两层防护，先看常量：

```ts
// server/modules/login-security.ts:11-14
const MAX_FAILURES = 5;         // 连续失败 5 次锁定
const LOCK_MINUTES = 30;        // 锁定 30 分钟
const RATE_LIMIT_PER_IP = 10;   // 单 IP 每分钟最多 10 次尝试
const RATE_WINDOW_SEC = 60;     // 速率窗口 60 秒
```

### 第一层：IP 速率限制（内存滑动窗口）

```ts
// server/modules/login-security.ts:77-92
const ipRateMap = new Map<string, { count: number; resetAt: number }>();
export function checkIpRateLimit(ip: string) {
  const now = Date.now();
  const entry = ipRateMap.get(ip);
  if (!entry || now > entry.resetAt) {                       // 窗口过期→重置
    ipRateMap.set(ip, { count: 1, resetAt: now + RATE_WINDOW_SEC * 1000 });
    return { allowed: true };
  }
  entry.count++;
  if (entry.count > RATE_LIMIT_PER_IP) {                     // 超限→拒绝并给等待秒数
    return { allowed: false, waitSeconds: Math.ceil((entry.resetAt - now) / 1000) };
  }
  return { allowed: true };
}
```

**逐段理解**
- 用一个 **内存 Map** 记录每个 IP 的 `{已尝试次数, 窗口重置时间}`。
- 窗口过期（`now > resetAt`）就重新计数；否则累加，超过 10 次就拒绝。
- ⚠️ **边界**：这是**单进程内存**方案。若你用 PM2 cluster / 多副本部署，各进程的 `ipRateMap` 互相独立，限速会被放大 N 倍。多副本场景需换成 Redis 计数（这是二次开发常见改造点）。

### 第二层：账户失败锁定（落库持久）

失败计数存数据库，重启不丢：

```ts
// server/modules/login-security.ts:46-68
export function recordLoginFailure(username, ip, reason) {
  db.insert('login_logs', { id:`log_${Date.now()}`, username, ip, success:0, fail_reason: reason });
  const existing = db.query('SELECT id, fail_count FROM login_lockouts WHERE username = ?', [username])[0];
  const newCount = (existing?.fail_count || 0) + 1;
  if (existing) {
    if (newCount >= MAX_FAILURES) {                          // 达阈值→写 locked_until
      const lockUntil = new Date(Date.now() + LOCK_MINUTES*60000).toISOString();
      db.update('login_lockouts', existing.id, { fail_count:newCount, locked_until:lockUntil });
    } else db.update('login_lockouts', existing.id, { fail_count:newCount });
  } else { /* 首次失败，插入一行 */ }
}
```

锁定检查会**顺带做过期清理**（自愈，无需定时任务）：

```ts
// server/modules/login-security.ts:32-43
export function isAccountLocked(username: string) {
  const row = db.query('SELECT locked_until FROM login_lockouts WHERE username = ?', [username])[0];
  if (!row?.locked_until) return { locked: false };
  const until = new Date(row.locked_until);
  if (until < new Date()) { db.delete('login_lockouts', username); return { locked: false }; }  // 过期即清
  return { locked: true, remainingMinutes: Math.ceil((until.getTime()-Date.now())/60000) };
}
```

### 把两层拼进中间件

```ts
// server/modules/login-security.ts:95-116
export function loginSecurityMiddleware(req, res, next) {
  if (req.path !== '/api/login' && req.path !== '/api/auth/login') return next();  // 只管登录
  const ip = req.ip || req.socket?.remoteAddress || 'unknown';
  const rate = checkIpRateLimit(ip);
  if (!rate.allowed) return res.status(429).json({ success:false, error:`请求过于频繁，请${rate.waitSeconds}秒后重试` });
  const username = req.body?.username || req.body?.account;
  if (username) {
    const lock = isAccountLocked(username);
    if (lock.locked) return res.status(423).json({ success:false, error:`账户已锁定，请${lock.remainingMinutes}分钟后重试` });
  }
  next();
}
```

> 🔑 两个状态码值得记：**429**（Too Many Requests，IP 限速）与 **423**（Locked，账户锁定）。前端可据此给不同提示。

## 源码剖析二：catch-all 路由铁律

Sowork AI 有一条**通用 CRUD 路由** `/:table`，让白名单内的表不用手写接口就能增删改查。但它有个致命顺序问题——如果注册太早，会把 `/rbac/resolve`、`/i18n/messages` 这类具名路由**全部吞掉**（`:table` 会匹配成 `table='rbac'`）。

所以源码里到处是这样的注释警告：

```ts
// server/standalone.ts:3772
// ===== 多语言 i18n 骨架（#107）=====（注册于 catch-all 之前，避免被 /:table 拦截）
// server/standalone.ts:3814
// ===== RBAC 细化权限骨架（#108）=====（注册于 catch-all 之前，避免被 /:table 拦截）
```

> ⚠️ **铁律**：所有具名业务路由（`/rbac/*`、`/i18n/*`、`/lowcode/*` 等）**必须**注册在通用 `/:table` catch-all **之前**。这是二次开发新增接口时"接口莫名 404"的头号原因——你加的路由被 catch-all 抢先匹配了。

## 源码剖析三：RBAC 权限点目录与种子

权限不是"角色=一堆页面"，而是**权限点**（`域:资源:动作`）。目录直接硬编码在源码里，清晰可审计：

```ts
// server/standalone.ts:3815-3838（节选）
const RBAC_CATALOG = [
  { moduleKey: 'cms', moduleName: '内容管理', points: [
    { key: 'cms:article:delete', label: '删除文章' },
    { key: 'cms:channel:manage', label: '栏目管理' },
    { key: 'cms:comment:moderate', label: '评论审核' }, /* … */
  ]},
  { moduleKey: 'shop', moduleName: '商城', points: [
    { key: 'shop:goods:manage', label: '商品管理' },
    { key: 'shop:order:refund', label: '退款处理' }, /* … */
  ]},
  /* hr / sys … */
];
const RBAC_ALL = RBAC_CATALOG.flatMap(g => g.points.map(p => p.key));
```

角色种子里有一段**极其关键的注释**，解释了一个真实踩过的坑：

```ts
// server/standalone.ts:3869-3895（节选）
// roles 表迁移：补充 siteScope 列（站点级权限，#130）
if (!cols.includes('siteScope')) {
  db.prepare("ALTER TABLE roles ADD COLUMN siteScope TEXT DEFAULT '[\"*\"]'").run();
}
// …
// siteScope：RBAC_ROLES 均为内置系统角色，其站点范围始终以代码定义为准
// （ALTER ADD COLUMN DEFAULT 曾把旧行统一填成 ["*"]，故此处强制回填修复）。
db.prepare('UPDATE roles SET permissionIds = ?, siteScope = ? WHERE id = ?')
  .run(JSON.stringify(merged), JSON.stringify(r.sites), ex.id);
```

**为什么内置角色要"强制回填"而不是"尊重已有值"？**
因为 `ALTER TABLE ADD COLUMN ... DEFAULT '["*"]'` 会把**所有旧行**的 `siteScope` 填成"全部站点"。此后无法区分"这是管理员真的改成了全站"还是"这是迁移填的默认值"。为避免把某个本应受限的内置角色错误放开成全站，代码选择**对内置角色始终以 `RBAC_ROLES` 代码定义覆盖**；自定义角色则不受影响，由管理员经 API 维护。

> 🔑 这段是"代码级教程"的典型价值：光看功能你只知道"有站点权限"，读源码才知道**为什么内置角色的站点范围不能改数据库、只能改代码**。

resolve 端点把多个角色的权限点求并集返回（前端 `usePermission` 就调它，见第 7 章）：

```ts
// server/standalone.ts:3957-3973
router.post('/rbac/resolve', (req, res) => {
  const roleIds = req.body?.roleIds || [];
  const points = new Set(); const sites = new Set(); let allSites = false;
  for (const rid of roleIds) {
    const code = String(rid).replace(/^role_/, '');
    const r = db.prepare('SELECT permissionIds, siteScope FROM roles WHERE code = ?').get(code);
    if (r) {
      JSON.parse(r.permissionIds).forEach(p => points.add(p));       // 权限点并集
      const ss = normSites(r.siteScope);
      if (ss.includes('*')) allSites = true; else ss.forEach(s => sites.add(s));
    }
  }
  res.json({ permissions:[...points], siteScope: allSites?['*']:[...sites], sites: RBAC_SITES });
});
```

> 多角色时权限点取**并集**、站点范围里只要有一个 `*` 就是全站——符合"权限叠加"的直觉。

## 管理者视角

- **合规价值**：登录失败锁定 + 全量 `login_logs` 是等保/审计的硬性要求，Sowork AI 开箱即备，省去二次开发。
- **权限颗粒度**：权限点做到"删除文章/退款/发货"级别，能满足"财务只能看订单不能退款"这类真实分权诉求，降低内部越权风险。
- **风险提示**：默认 IP 限速是单进程内存实现，若做高可用多副本部署，需告知技术团队改为集中式计数，否则限速形同虚设。

## 注意事项

- ⚠️ **catch-all 铁律**：新接口务必注册在 `/:table` 之前，否则 404。
- ⚠️ **IP 限速是内存态**：多副本部署需换 Redis。
- ⚠️ **内置角色 siteScope 改数据库无效**：以 `RBAC_ROLES` 代码为准，要改改代码。
- 🔑 **状态码约定**：429=IP 限速，423=账户锁定，401=未授权，403=无权限。
- 🔑 锁定过期**自愈清理**，无需定时任务。
