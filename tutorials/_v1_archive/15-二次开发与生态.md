## 核心概念：两套并行的扩展机制

Sowork AI 提供两条二次开发路径，面向不同人群：

```
modules/   Odoo 风格手写模块   → 面向开发者，功能完整、可含 SQL 种子数据
addons/    低代码向导生成       → 面向业务人员，第 08 章的 create-module 产物
```

两者殊途同归——都被同一套自研 ORM（第 06 章）加载、都会自动建表、都通过 `app.json` 驱动前端菜单。区别只在"谁来写、写多少手工代码"。

目录现状：

```
modules/   blog, elearning, forum, hr_core, hr_payroll,
           l10n_cn, odoo_base, odoo_res_partner, shoe_garment
addons/    analytic_odoo, app_mru06doi(低代码CRM), demo_erp
```

`modules/` 里能看到明显的 Odoo 血统：`odoo_base`、`odoo_res_partner`、`l10n_cn`（中国本地化）——这些是参照 Odoo 对应模块移植来的。

## 源码剖析一：模块清单 manifest.json

每个模块的入口是 `manifest.json`，它对标 Odoo 的 `__manifest__.py`。以 `modules/blog/manifest.json` 为例：

```json
// modules/blog/manifest.json
{
  "name": "博客",
  "name_en": "Blog",
  "version": "1.0.0",
  "author": "Sowork AI",
  "license": "Apache-2.0",
  "category": "Website/Blog",
  "depends": [],
  "summary": "企业博客系统: 分类、标签、评论、RSS",
  "description": "支持多栏目博客、文章标签、评论审核、RSS订阅、SEO优化。参照 Odoo website_blog。",
  "installable": true,
  "application": true,
  "sequence": 200,
  "data": [ "data/blog.sql" ],
  "auto_install": true
}
```

关键字段（Odoo 开发者会觉得眼熟）：

1. **`depends`**：依赖的其他模块列表。空数组表示无依赖。加载器会按依赖顺序初始化（依赖先行）。
2. **`data`**：初始化数据文件（这里是 `data/blog.sql`）——模块安装时执行的种子 SQL，比如内置分类、示例文章。
3. **`installable` / `application`**：是否可安装、是否作为独立应用出现在菜单。
4. **`sequence`**：加载/显示顺序，数字越小越靠前。
5. **`auto_install`**：为 true 时随系统自动安装，无需手动启用。
6. **`description` 里明确写"参照 Odoo website_blog"**——这是本项目的诚实做法：移植自哪个 Odoo 模块，清清楚楚。

🔑 `manifest.json` 是模块的"身份证 + 装配说明书"。加载器读它决定：装不装、依赖谁、先后顺序、要不要跑种子数据。理解它就理解了模块的生命周期。

## 源码剖析二：模型定义 DSL（models.js）

模块的核心是数据模型。`modules/blog/models/models.js` 用一套声明式 DSL 定义模型，**模型即表结构**（第 06 章 ORM 会据此自动建表）：

```js
// modules/blog/models/models.js
// 博客模块 — ORM模型
exports.models = [
  {
    _name: 'blog_categories',
    _description: '博客分类',
    _rec_name: 'name',
    _fields: {
      name:        { type: 'char', label: '名称', required: true },
      slug:        { type: 'char', label: 'URL别名' },
      description: { type: 'text', label: '描述' },
      sortOrder:   { type: 'integer', label: '排序' },
    },
  },
  {
    _name: 'blog_posts',
    _description: '博客文章',
    _rec_name: 'title',
    _fields: {
      title:       { type: 'char', label: '标题', required: true },
      summary:     { type: 'text', label: '摘要' },
      content:     { type: 'text', label: '内容' },
      category_id: { type: 'char', label: '分类' },
      author_name: { type: 'char', label: '作者' },
      status:      { type: 'selection', label: '状态', default: 'draft',
                     selection: [{label:'草稿',value:'draft'},{label:'已发布',value:'published'}] },
      published_at:{ type: 'datetime', label: '发布时间' },
    },
  },
  {
    _name: 'blog_comments',
    _description: '博客评论',
    _fields: {
      post_id:     { type: 'char', label: '文章ID', index: true },
      author_name: { type: 'char', label: '评论者' },
      content:     { type: 'text', label: '内容' },
      status:      { type: 'selection', label: '状态', default: 'pending',
                     selection: [{label:'待审',value:'pending'},{label:'通过',value:'approved'}] },
    },
  },
];
```

DSL 要素逐个说清（对照第 06 章 ORM）：

1. **`exports.models`（数组）**：一个文件可定义多个模型。加载器（`model-registry.ts` 的 `loadFromModule`，第 08 章讲过）支持 `exports.models`/`exports.model`/`export default` 三种导出形式。
2. **`_name`**：模型名 = 数据库表名（`blog_categories` → 表 `blog_categories`）。
3. **`_description` / `_rec_name`**：人类可读描述 / 记录的"显示字段"（用于下拉展示等）。
4. **`_fields`**：字段定义。每个字段有 `type`、`label`、可选 `required`/`default`/`index`。
5. **字段类型**：`char`（短文本）、`text`（长文本）、`integer`、`datetime`、`selection`（枚举，带 `selection` 选项数组）。这些类型由 ORM 映射到 SQLite 列类型。
6. **`index: true`**：为该字段建索引（如 `blog_comments.post_id` 常用于按文章查评论）。

低代码生成的 addon 用的是同一套 DSL，只是导出 `exports.model`（单个）。看 `addons/app_mru06doi/models/contactinfo.js`：

```js
// addons/app_mru06doi/models/contactinfo.js
// 低代码生成: contactinfo
exports.model = {
  "_name": "contactinfo",
  "_description": "联系人信息",
  "_fields": {
    "contact_name": { "type": "char", "label": "联系人姓名", "required": false },
    "customer_id":  { "type": "many2one", "label": "所属客户", "relation": "customer" },
    "gender":       { "type": "char", "label": "性别" },
    "department":   { "type": "char", "label": "部门" },
    "position":     { "type": "char", "label": "职务" }
    /* … */
  }
};
```

注意 `many2one` 类型带 `relation: 'customer'`——这是**关系字段**，声明"联系人属于某客户"，对标 Odoo 的多对一关系。低代码向导（第 08 章）生成的就是这种 JSON 结构的 DSL，和手写模块**完全同构**。

🔑 **手写模块和低代码 addon 用同一套模型 DSL**，这是整个扩展生态的统一基石：无论你是资深开发者手写 `models.js`，还是业务人员点向导生成 addon，产物都是 ORM 能识别的同一种模型定义。一套 DSL，两种生产方式。

## 源码剖析三：前端菜单驱动（app.json）

模块定义了数据和模型，还需要在前端有入口。这靠 addon 里的 `app.json`（第 08 章讲过），以 `addons/app_mru06doi/app.json` 为例：

```json
// addons/app_mru06doi/app.json
{
  "name": "客户关系管理",
  "description": "客户关系管理",
  "icon": "👥",
  "color": "#1677ff",
  "menu": [
    { "label": "客户信息",   "table": "customer_information", "icon": "📋", "desc": "管理客户信息" },
    { "label": "联系人信息", "table": "contactinfo",         "icon": "📋", "desc": "管理联系人信息" },
    { "label": "商机信息",   "table": "opportunity",         "icon": "📋", "desc": "管理商机信息" },
    { "label": "跟进记录",   "table": "followinfo",          "icon": "📋", "desc": "管理跟进记录" }
  ]
}
```

`menu` 数组的每一项把一个"菜单标签"绑定到一张"表"。前端读 `app.json`，为每个菜单项渲染一个通用的增删改查页面——数据走第 02 章的 catch-all `/:table` CRUD。**所以低代码模块无需写任何前端代码，就有完整的列表/表单界面**。

## 源码剖析四：从 Odoo 移植的路径

`modules/` 里的 `odoo_base`、`odoo_res_partner`、`l10n_cn` 揭示了本项目的一条重要生态路径：**参照 Odoo 源码移植功能模块**。移植不是照搬代码（Odoo 是 Python），而是：

```
① 读 Odoo 对应模块的模型定义（Python __manifest__.py + models/*.py）
② 把 Odoo 的字段、关系翻译成本项目的 models.js DSL
③ 补一份 manifest.json（depends/data/sequence）
④ 需要种子数据就写 data/*.sql
⑤ 放进 modules/，ORM 自动建表，即可用
```

`res_partner`（Odoo 的合作伙伴/联系人核心模型）被移植成 `odoo_res_partner`，`l10n_cn`（Odoo 中国会计本地化）被移植成同名模块——**功能借鉴 Odoo 成熟设计，实现落到本项目轻量技术栈上**。这让 Sowork AI 能快速吸收 Odoo 生态里被验证过的业务模型，而不必从零设计。

🔑 这条路径的价值：**Odoo 有上千个成熟业务模块（会计、库存、制造、CRM…），它们的数据模型是数十年业务沉淀的结晶**。通过"读 Odoo 模型 → 翻译成 DSL → 落地为本项目模块"，Sowork AI 站在了巨人的肩膀上，同时保持了自己单体轻量、易私有化的优势。

## 实战代码：手写一个新模块

假设要加一个"公告"模块：

```bash
# 1. 建目录
mkdir -p modules/notice/models

# 2. 写模型 modules/notice/models/models.js
cat > modules/notice/models/models.js <<'EOF'
exports.models = [{
  _name: 'notice_items', _description: '公告', _rec_name: 'title',
  _fields: {
    title:      { type: 'char', label: '标题', required: true },
    content:    { type: 'text', label: '内容' },
    level:      { type: 'selection', label: '级别', default: 'normal',
                  selection: [{label:'普通',value:'normal'},{label:'重要',value:'urgent'}] },
    publish_at: { type: 'datetime', label: '发布时间' },
  },
}];
EOF

# 3. 写清单 modules/notice/manifest.json（name/version/depends:[]/auto_install:true）
# 4. 重启服务 → ORM 自动建表 notice_items
# 5. 通用 CRUD 立即可用：
curl -X POST http://localhost:3000/api/notice_items \
  -H "Content-Type: application/json" \
  -d '{"title":"系统升级通知","level":"urgent"}'
```

> 注：本例用 `cat` 只为教程演示。实际二次开发建议用编辑器写文件（尤其 Windows 下避免编码问题）。

无需写任何路由、无需改数据库脚本——定义模型 → 自动建表 → catch-all CRUD 接管。这就是第 06 章 ORM + 第 02 章 catch-all 路由联手带来的扩展红利。

---

## 运行演示：新模块自动可用

按上面步骤加完 `notice` 模块并重启，你会观察到：

1. 启动日志里 ORM 报告注册了新模型 `notice_items` 并 `CREATE TABLE`。
2. 直接 `POST /api/notice_items` 就能写数据，`GET /api/notice_items` 能读——**catch-all 路由自动接管，一行 CRUD 代码没写**。
3. 若再补一个 `app.json` 声明菜单，前端侧边栏立刻出现"公告"入口和现成的增删改查界面。

从"定义模型"到"前后端全通"，中间没有写任何胶水代码——这就是这套架构对二次开发者最大的善意。

## 注意事项

- ⚠️ **手写模块和低代码 addon 用同一套模型 DSL**，别把两套机制搞混——`modules/` 手写、`addons/` 低代码生成，但底层模型格式一致。
- ⚠️ **`manifest.json` 的 `depends` 要写全**：依赖别的模型却不声明依赖，加载顺序可能出错导致建表/外键异常。
- ⚠️ **模型改字段用 ORM 的 ALTER 迁移**（第 06 章），别直接改已有字段类型——SQLite 对 ALTER 支持有限。
- ⚠️ **Windows 下写模块文件注意编码**：用编辑器保存 UTF-8，别用会引入 BOM 或乱码的方式。
- 🔑 移植 Odoo 模块时，重点是"翻译数据模型"而非搬代码——把 Odoo 的字段/关系映射成本项目的 `char/text/selection/many2one` DSL。
