## 核心概念：单体应用的部署优势

前面章节反复强调 Sowork AI 是**单体全栈应用**——一个 Node 进程同时提供 API、静态前端、AI 能力。这个架构选择在部署阶段兑现了红利：

```
微服务部署：N 个服务 + 服务发现 + 网关 + 配置中心 + 消息队列……
Sowork AI：   1 个 node 进程 + 1 个 SQLite 文件
```

部署简单到什么程度？**一个 Docker 容器、一个数据卷，就是全部**。对中小企业私有化部署，这意味着一台普通服务器就能跑起完整的 HR+ERP+CMS+商城+AI 系统，运维几乎为零。

## 源码剖析一：构建脚本链

部署前要先构建。`package.json:7-16` 的 scripts 定义了整条构建链：

```json
// package.json:7-16
"scripts": {
  "dev": "concurrently -n server,client -c blue,green \"npm run dev:server\" \"npm run dev:client\"",
  "dev:server": "tsx watch server/standalone.ts",
  "dev:client": "vite --config vite.config.ts",
  "build": "npm run build:server && npm run build:client",
  "build:server": "tsc -p tsconfig.server.json && node -e \"require('fs').cpSync('server/ai-service.js','dist/server/ai-service.js'); require('fs').cpSync('server/code-agent-tools.js','dist/server/code-agent-tools.js')\"",
  "build:client": "vite build --config vite.config.ts",
  "start": "node dist/server/standalone.js",
  "type:check": "tsc --noEmit"
}
```

逐条拆解关键脚本：

1. **`dev`**：用 `concurrently` 同时起后端（`tsx watch`，改代码自动重启）和前端（`vite` 热更新）。开发用。
2. **`build:server`** ——这里有个**必须理解的细节**：先 `tsc` 把 TypeScript 编译到 `dist/`，**然后用 `node -e` 执行 `cpSync` 把两个手写 JS 拷进去**。为什么？因为 `server/ai-service.js` 和 `server/code-agent-tools.js` 是**手写的 `.js`（不是 `.ts`）**，tsc 不会处理它们（第 09 章讲过），必须手动拷贝，否则 `dist` 里缺这两个文件、AI 功能全废。
3. **`build:client`**：vite 把 React 前端打包成静态资源到 `dist/client/`。
4. **`start`**：直接跑编译产物 `dist/server/standalone.js`，它会同时服务 API 和 `dist/client` 静态文件（第 02 章的 SPA fallback）。

🔑 **`build:server` 里的 `cpSync` 是最容易被忽略的部署陷阱**：如果有人把它改成纯 `tsc`，或手动清理 dist 后只跑 tsc，AI 服务就会在生产环境报"找不到模块"。改构建脚本前务必记住这两个手写 JS 的存在。

## 源码剖析二：多阶段 Dockerfile

Docker 部署用多阶段构建，兼顾"构建需要的工具"和"运行镜像的精简"。完整 `Dockerfile`：

```dockerfile
# Dockerfile
# ── 构建阶段 ──
FROM node:22-alpine AS builder
RUN apk add --no-cache python3 make g++
WORKDIR /app
COPY package.json ./
RUN npm install --legacy-peer-deps
COPY . .
RUN npm run build:server && npm run build:client
RUN npm prune --omit=dev

# ── 运行阶段 ──
FROM node:22-alpine
RUN apk add --no-cache dumb-init
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./
RUN mkdir -p /app/data
ENV SERVER_PORT=3000
EXPOSE 3000
VOLUME ["/app/data"]
ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "dist/server/standalone.js"]
```

逐段精读：

**构建阶段（builder）**
- `apk add python3 make g++`：**为什么 Node 镜像要装 Python 和编译器？** 因为 `better-sqlite3` 是原生模块（C++），`npm install` 时要用 node-gyp 现场编译，依赖 Python + g++。这是纯 JS 项目常忽略的一环。
- `npm install --legacy-peer-deps`：**这个 flag 不能省**（项目记忆里的最高优先级陷阱）——本项目有 peer 依赖冲突（如 Tiptap 版本），裸 `npm install` 会失败或错误 prune 依赖。
- `COPY package.json ./` 先于 `COPY . .`：这是 Docker 层缓存优化——只要 package.json 没变，`npm install` 层就能复用缓存，不必每次重装。
- `npm prune --omit=dev`：构建完成后删掉 devDependencies（tsx、vite 等），为运行镜像瘦身。

**运行阶段**
- **重新 `FROM node:22-alpine`**：丢弃构建阶段的 python/g++ 等重型工具，运行镜像只留必需品，体积大幅缩小。
- `COPY --from=builder`：只从构建阶段拷 `dist`、`node_modules`（已 prune）、`package.json`——**编译产物进镜像，源码不进**。
- `dumb-init`：作为 PID 1 进程，正确转发信号（SIGTERM）给 node，保证 `docker stop` 时优雅退出、不留僵尸进程。
- `VOLUME ["/app/data"]`：声明数据卷——SQLite 数据库文件（`data/ehr.db`）挂在这里，**容器删了数据还在**。

🔑 多阶段构建的精髓：**构建环境和运行环境分离**。构建要 python/g++/devDeps，运行只要 node + 产物。分离后运行镜像更小、攻击面更少、启动更快。

## 源码剖析三：docker-compose 数据持久化

`docker-compose.yml` 把部署简化成一条命令：

```yaml
# docker-compose.yml
services:
  feida:
    build: .
    container_name: feida-hr
    ports:
      - "3000:3000"
    volumes:
      # 持久化数据库文件
      - ./data:/app/data
    environment:
      - SERVER_PORT=3000
      - TZ=Asia/Shanghai
    restart: unless-stopped
```

要点：

1. **`volumes: ./data:/app/data`**：把宿主机的 `./data` 目录挂载到容器 `/app/data`。**这是数据安全的命门**——SQLite 文件存在宿主机上，容器重建、镜像升级，数据都毫发无损。删了这行，容器一删数据就没了。
2. **`TZ=Asia/Shanghai`**：设时区。否则容器默认 UTC，所有 `new Date().toISOString()`（前面各章大量使用）会差 8 小时。
3. **`restart: unless-stopped`**：进程崩溃或服务器重启后自动拉起，除非手动停。生产环境必备。

部署就三步：

```bash
docker compose up -d --build   # 构建并后台启动
docker compose logs -f         # 看日志
docker compose down            # 停止（数据仍在 ./data）
```

## 源码剖析四：Windows 一键启动脚本（自愈逻辑）

对不用 Docker 的 Windows 用户，`启动系统.bat` 提供一键启动，且带**自愈能力**。注意（项目铁律）：**这个 .bat 全 ASCII、由 Python 脚本生成**（用户规则：禁止中文、禁止 PowerShell 写入、必须 Python 生成）。核心逻辑：

```bat
:: 启动系统.bat（节选）
REM Check if node_modules exists
if not exist "node_modules\" (
    echo [INFO] Installing dependencies...
    call npm install --legacy-peer-deps
    ...
)
REM Check if built
if not exist "dist\server\standalone.js" (
    echo [INFO] Building project...
    call npm run build
    ...
)
REM Kill existing process on port 3000
for /f "tokens=5" %%a in ('netstat -ano ^| findstr ":3000.*LISTENING"') do (
    echo [INFO] Found old process PID=%%a on port 3000, killing...
    taskkill /F /PID %%a >nul 2>nul
    timeout /t 2 /nobreak >nul
)
node dist\server\standalone.js
```

自愈逻辑逐条：

1. **缺依赖自动装**：`if not exist "node_modules\"` → 自动 `npm install --legacy-peer-deps`（又是这个 flag）。
2. **没构建自动建**：`if not exist "dist\server\standalone.js"` → 自动 `npm run build`。
3. **端口被占自动清**：用 `netstat` 找占用 3000 端口的进程，`taskkill /F` 强杀——避免"上次没退干净，这次起不来"。
4. **启动失败自动重试**：首次 `node` 退出非 0，再清一次端口重试一次（`启动系统.bat:47-57`）。

🔑 这个脚本体现了"面向非技术用户的部署"思路：**把所有可能出错的前置条件（没装依赖、没构建、端口占用）都自动处理掉**，用户双击即用。这对开源项目降低使用门槛至关重要。

## 实战代码：三种部署方式对照

```bash
# ① Docker（推荐生产）
docker compose up -d --build
# → http://localhost:3000

# ② 手动（Linux/Mac 生产）
npm install --legacy-peer-deps
npm run build
SERVER_PORT=3000 npm start

# ③ Windows 一键（本地/演示）
双击 启动系统.bat
```

三种方式殊途同归：最终都是 `node dist/server/standalone.js` + 一个 SQLite 文件。

## 运行演示：验证数据持久化

这是最能体现单体 + 数据卷价值的实验：

```bash
docker compose up -d --build
# 创建一篇文章（数据写入 ./data/ehr.db）
curl -X POST http://localhost:3000/api/cms-articles -d '{"title":"持久化测试"}' -H "Content-Type: application/json"
docker compose down            # 停并删除容器
docker compose up -d           # 重新起一个全新容器
curl http://localhost:3000/api/cms-articles   # 文章还在！
```

容器被完全删除重建，但因为数据挂在宿主机 `./data`，文章依然存在。**这就是 `volumes` 那一行的价值**。

## 管理者视角

| 关注点 | 部署方案给出的答案 |
|--------|--------------------|
| **数据主权** | SQLite 文件 + 数据卷，全部数据留在企业自己的服务器/宿主机，不依赖任何云服务。可完全私有化、断网运行。 |
| **运维成本** | 单进程 + 单文件，一台普通服务器即可。无需 DBA、无需运维集群，`docker compose up` 就完事。 |
| **升级安全** | 数据与镜像分离，升级只换镜像，数据卷不动，回滚也简单。 |
| **上手门槛** | Windows 用户双击 `启动系统.bat` 即用，自动装依赖/构建/清端口，非技术人员也能部署演示。 |

给决策者一句话：**"一个容器 + 一个数据文件 + 完全私有化"，让 Sowork AI 的部署运维成本降到中小企业可承受的最低点，同时把数据主权 100% 留在企业手里**。

## 注意事项

- ⚠️ **`npm install` 必须带 `--legacy-peer-deps`**（Dockerfile、bat、手动都是）——这是全项目最高优先级陷阱，裸 install 会破坏依赖树。
- ⚠️ **`build:server` 的 `cpSync` 不能删**：两个手写 JS（ai-service/code-agent-tools）靠它进 dist，删了 AI 功能报错。
- ⚠️ **数据卷 `./data:/app/data` 是数据命门**：删了这行，容器一删数据全丢。
- ⚠️ **Docker 构建需 python3+g++**（编译 better-sqlite3 原生模块），基础镜像要装。
- ⚠️ **.bat 文件必须全 ASCII 且用 Python 生成**（用户规则），别用 PowerShell 或直接写中文，会乱码。
- 🔑 生产环境设 `TZ=Asia/Shanghai` + `restart: unless-stopped`，否则时间差 8 小时、崩溃不自愈。
