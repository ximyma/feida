## 核心概念：为什么用"裸 fetch + 断言"而非 Jest？

Sowork AI 的测试没有用 Jest/Mocha/Vitest 任何框架，而是 25 个独立的 `test-*.js` 脚本，每个都是纯 Node + fetch。这是一个刻意的选择：

```
传统单元测试：mock 数据库、mock 依赖，测函数内部逻辑
Sowork AI 的选择：真启动服务，用 HTTP 打真接口，验证端到端行为
```

**为什么？** 因为这个系统的复杂度集中在"路由 + 数据库 + 业务状态机"的交互上（前几章反复看到：库存联动、售后状态机、RBAC 播种）。这些逻辑用 mock 单元测试很难覆盖真实路径，反而是**黑盒 API 测试**能一把抓住"从请求到落库"的完整行为。而且零框架依赖，符合全系统"够用即最优"的一贯气质。

25 个测试脚本覆盖各模块：

```
test-rbac.js          RBAC 权限（本章精读）
test-aftersale.js     售后状态机（第 11 章逻辑）
test-cms-tags.js      标签云聚合（第 10 章逻辑）
test-smart-tags.js    智能标签提取
test-distribution.js  分销佣金
test-warehouse.js     仓库库存
test-i18n.js          国际化
test-site-scope.js    站点级权限（第 12 章逻辑）
test-perm-points.js   权限点驱动
test-v12-features.js  v1.2 新特性
...（共 25 个）
```

## 源码剖析一：测试脚本的标准骨架

每个测试脚本都遵循同一个极简骨架。以 `test-rbac.js:1-8` 开头为例：

```js
// test-rbac.js:1-8
require('./fetch-polyfill');
// #108 RBAC 细化权限验证
const BASE = process.env.BASE || 'http://localhost:3000';
let pass = 0, fail = 0;
function ok(name, cond, extra) { if (cond) { pass++; console.log(`  ✅ ${name}`); } else { fail++; console.log(`  ❌ ${name}` + (extra ? ` — ${extra}` : '')); } }

async function main() {
  console.log(`#108 RBAC 测试 @ ${BASE}`);
```

四个约定，逐个说清：

1. **`require('./fetch-polyfill')`**：老 Node 版本没有全局 fetch，这个 polyfill 补上（新版 Node 22 自带，但保留兼容）。
2. **`BASE = process.env.BASE || 'http://localhost:3000'`**：测试目标地址可通过环境变量覆盖。默认打本地 3000。运行时 `BASE=http://localhost:3400 node test-rbac.js` 就能测别的端口。
3. **`ok(name, cond, extra)` 断言函数**：条件真则 `pass++` 打 ✅，假则 `fail++` 打 ❌ 并显示额外信息。**这就是整个"测试框架"——五行代码**。
4. **`pass/fail` 计数 + 退出码**：结尾 `process.exit(fail === 0 ? 0 : 1)`，让 CI 能通过退出码判断成败。

结尾模式（`test-rbac.js:66-69`）：

```js
// test-rbac.js:66-69
  console.log(`\n#108 结果: ${pass} 通过, ${fail} 失败`);
  process.exit(fail === 0 ? 0 : 1);
}
main().catch(e => { console.error('测试异常:', e); process.exit(1); });
```

🔑 `main().catch(...)` 兜住所有未捕获异常，保证脚本崩溃也返回非零退出码——CI 不会把"崩溃"误判成"通过"。

## 源码剖析二：一个完整用例的编排（test-rbac.js）

`test-rbac.js` 是学习"如何编排一个端到端用例"的好范本。它按业务流程组织了 8 组断言（`test-rbac.js:10-64`），逐组解读：

**第 1 组 · 权限目录（10-16 行）**
```js
const r1 = await fetch(`${BASE}/api/rbac/permissions`);
const d1 = await r1.json();
ok('permissions 返回 200', r1.status === 200);
ok('分组含 cms 模块', Array.isArray(d1) && d1.some(g => g.moduleKey === 'cms'));
const cms = (Array.isArray(d1) ? d1.find(g => g.moduleKey === 'cms') : null);
ok('cms 含细粒度点 cms:article:delete', !!cms && cms.points.some(p => p.key === 'cms:article:delete'));
```
先验证"权限目录返回正确且含预期权限点"——这是第 12 章 `RBAC_CATALOG` 播种是否生效的直接检验。

**第 2 组 · 角色与默认权限映射（18-26 行）**
```js
const sa = d2.find(r => r.code === 'super_admin');
ok('super_admin 含 cms:article:delete', !!sa && sa.permissionIds.includes('cms:article:delete'));
const emp = d2.find(r => r.code === 'employee');
ok('employee 无细粒度 cms 权限', !!emp && !emp.permissionIds.some(p => p.startsWith('cms:')));
```
**正反两面都测**：超级管理员**应有**某权限，普通员工**应无**任何 CMS 权限。用 `p.startsWith('cms:')` 前缀匹配——这正呼应第 12 章"三段式命名便于前缀批量判断"的设计。

**第 4-5 组 · 角色 CRUD（33-46 行）**
```js
const r3 = await fetch(`${BASE}/api/rbac/roles`, { method: 'POST', ... body: JSON.stringify({ name: '测试角色', code, ... }) });
ok('创建角色 200', r3.status === 200, 'status=' + r3.status);
const newId = d3.id;
// 更新
const r4 = await fetch(`${BASE}/api/rbac/roles/${newId}`, { method: 'PUT', ... });
ok('更新后含 cms:article:delete', d4b.permissionIds.includes('cms:article:delete'));
```
创建一个**带时间戳的唯一 code**（`'test_role_' + Date.now()`）避免和已有角色冲突，然后测 POST/PUT/GET 全链路。

**第 6 组 · resolve 解析（48-54 行）**
```js
const r5 = await fetch(`${BASE}/api/rbac/resolve`, { method: 'POST', ... body: JSON.stringify({ roleIds: ['role_employee'] }) });
ok('resolve employee 不含 cms:article:delete', ... && !d5.permissions.includes('cms:article:delete'));
```
验证第 12 章 `/rbac/resolve` 的并集逻辑：员工解析出的权限不含删除文章，超管解析出的含。

**第 7-8 组 · 边界与清理（56-64 行）**
```js
const r6 = await fetch(`${BASE}/api/rbac/roles/${sa.id}`, { method: 'DELETE' });
ok('系统角色删除被拒(403)', r6.status === 403, 'status=' + r6.status);
// 清理测试数据
const r7 = await fetch(`${BASE}/api/rbac/roles/${newId}`, { method: 'DELETE' });
ok('删除自定义角色 200', r7.status === 200);
```
**测边界**：系统角色不可删（应 403）；**测清理**：删掉自己创建的测试角色，并验证删除后查询 404。

🔑 这个脚本示范了一个好的端到端用例应有的要素：**正反用例、边界用例、自造唯一测试数据、测完清理、每步带 `extra` 便于定位失败**。

## 源码剖析三：用副作用验证业务逻辑

对于有状态副作用的功能，测试不只看接口返回，还要**查副作用是否正确**。以售后回滚（第 11 章）为例，`test-aftersale.js` 的思路是：

```
1. 记录商品当前库存
2. 走完"申请→审核→收货→退款"售后流程
3. 查 shop_stock_logs，断言恰好新增一条 type='return' 的回库流水
4. 断言商品库存回到售前水平（不多不少）
```

这正是第 11 章"用库存流水做对账"思路的测试化落地。**接口返回 `success:true` 不代表库存真的对——必须查副作用**。同理：

- `test-cms-tags.js` 创建带标签的文章后，查 `/cms-tags` 断言标签计数正确增加，再软删除后断言计数减少。
- `test-warehouse.js` 手动调整库存后，断言 `shop_stock_logs` 有对应流水且 `after_stock` 正确。

🔑 **副作用验证是黑盒测试的精髓**：不信任接口自报的"成功"，而是去查它应该改变的数据是否真的按预期改变了。

## 实战代码：运行测试

前提：先启动服务（`npm start` 或 `启动系统.bat`），测试脚本打的是真接口。

```bash
# 默认打 localhost:3000
node test-rbac.js
#   ✅ permissions 返回 200
#   ✅ 分组含 cms 模块
#   ...
#   #108 结果: 17 通过, 0 失败

# 指定端口
BASE=http://localhost:3400 node test-aftersale.js

# 批量跑全部测试（Bash）
for f in test-*.js; do echo "=== $f ==="; node "$f"; done
```

退出码约定：全绿 → 0，有失败 → 1。CI 里 `node test-rbac.js && node test-aftersale.js && ...` 任一失败即中断。

## 运行演示：一次全量回归的样子

v1.0.0 发布前跑过一次全量回归：22 套件、165+ 断言、0 失败（v1.2 又扩充了 `test-v12-features.js` 等，现共 25 个）。终端输出大致是：

```
=== test-rbac.js ===       #108 结果: 17 通过, 0 失败
=== test-aftersale.js ===  售后 结果: 20 通过, 0 失败
=== test-cms-tags.js ===   标签 结果: 8 通过, 0 失败
...
```

全绿意味着：库存联动、售后状态机、RBAC 播种、标签聚合等核心业务在这次改动后行为未回退。**这就是回归测试给开源项目的底气**。

> ⚠️ 历史坑：`test-api.js` 曾硬编码 `localhost:3000` 且断言无空值保护，服务没起时直接 `exit=1` 误报。已重写为用 `BASE` + 健壮断言。**测试脚本自己也要防御性编程**。

## 管理者视角

| 关注点 | 测试体系给出的答案 |
|--------|--------------------|
| **质量信心** | 25 套自动化回归，每次改动可一键验证核心业务未被破坏，敢于持续迭代。 |
| **开源可信度** | 开源项目最怕"能跑但没人敢改"。完整的测试脚本让贡献者和使用者都有信心。 |

| **业务正确性** | 用副作用验证（查库存流水、查标签计数），确保的是"业务真的对"而非"接口没报错"。 |

给决策者一句话：**这套轻量测试让 Sowork AI 从"一个人的项目"具备了"可被社区安全共建"的质量基础**。

## 注意事项

- ⚠️ **测试前必须先启动服务**——这是黑盒 API 测试，不启动服务全部失败。
- ⚠️ **测试会写真数据库**：`test-rbac.js` 等会创建/删除测试角色。建议对独立测试库运行，或确保脚本测完自清理。
- ⚠️ **用 `BASE` 而非硬编码端口**，并对返回做空值保护（`Array.isArray(d) && ...`），否则服务异常时会误报或崩溃。
- 🔑 有副作用的功能，断言要查副作用（库存流水、标签计数），别只信接口返回的 `success`。
- 🔑 造测试数据用时间戳保唯一（`'test_' + Date.now()`），测完删除，避免污染和重复运行冲突。

## 练习

1. 阅读 `test-rbac.js`，指出它在哪几步"造了测试数据"、哪几步"清理了测试数据"，并解释为什么清理很重要。
2. `ok('employee 无细粒度 cms 权限', !emp.permissionIds.some(p => p.startsWith('cms:')))` 用了前缀匹配。说明这依赖了第 12 章权限点命名的什么规范。
3. **读源码猜作用**：`main().catch(e => { ...; process.exit(1); })`。猜猜如果去掉这个 catch，测试脚本内部抛异常时 CI 会得到什么退出码，为什么危险。
4. 动手：仿照 `test-rbac.js` 的骨架，为第 10 章"标签云"写一个最小测试：创建一篇带标签 `test_tag` 的文章 → 查 `/cms-tags` 断言含该标签 → 软删除该文章 → 再查断言计数减少。
