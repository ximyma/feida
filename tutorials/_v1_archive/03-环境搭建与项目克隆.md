## 核心概念：先看清项目结构

Sowork AI 是**单体全栈**——一个仓库同时装着后端（TypeScript）和前端（React 19）。克隆完成后你会看到：

```
feida/
├── package.json              # 🔑 依赖 + 全部构建/启动脚本
├── .npmrc                    # 国内镜像配置（安装提速）
├── vite.config.ts            # 前端构建配置
├── tsconfig.server.json      # 后端 TS 编译配置
├── server/                   # 后端
│   ├── standalone.ts         # 🚀 唯一入口：路由 + 静态托管 + 自动建库
│   ├── ai-service.js         # AI 服务（LLM/RAG/工具循环）
│   └── modules/orm/          # 自研 ORM 引擎（见第 6 章）
├── client/                   # 前端 React SPA
│   └── src/{app.tsx,pages/,components/,hooks/}
├── data/                     # SQLite 数据库（首启自动生成 ehr.db）
└── dist/                     # 构建产物
```

## 安装步骤

### 1. 安装 Node.js（务必 22.x）

```bash
# 方法一：官网 https://nodejs.org/ 下载 22.x LTS

# 方法二：nvm（推荐，可多版本切换）
nvm install 22
nvm use 22

# 验证
node -v   # ✅ 应输出 v22.x
npm -v    # ✅ 应输出 10.x
```

> ⚠️ **不要用低于 18 的 Node**：项目用了 Node 22 的特性（如 `fs.cpSync`，见下文构建脚本），旧版本会直接报错。

### 2. 安装 Git 并克隆

```bash
git clone <your-repo-url> feida
cd feida
```

## 源码剖析一：package.json 的 scripts 逐条拆解

很多人 clone 完直接 `npm run dev` 却不知道背后发生了什么。我们逐条读：

```jsonc
// package.json → scripts
{
  "dev": "concurrently -n server,client -c blue,green \"npm run dev:server\" \"npm run dev:client\"",
  "dev:server": "tsx watch server/standalone.ts",
  "dev:client": "vite --config vite.config.ts",
  "build": "npm run build:server && npm run build:client",
  "build:server": "tsc -p tsconfig.server.json && node -e \"require('fs').cpSync('server/ai-service.js','dist/server/ai-service.js'); require('fs').cpSync('server/code-agent-tools.js','dist/server/code-agent-tools.js')\"",
  "build:client": "vite build --config vite.config.ts",
  "start": "node dist/server/standalone.js"
}
```

**逐条讲解**
- `dev`：用 `concurrently` **同时**拉起前后端两个进程，`-n server,client` 是给日志加名字前缀，`-c blue,green` 给它们不同颜色，方便你在一个终端里区分谁的输出。
- `dev:server`：`tsx watch` 直接运行 `.ts` 源码并监听改动热重载——**开发期后端不需要先编译**。
- `dev:client`：Vite 启动前端 dev server，提供 HMR（改 `.tsx` 秒级刷新）。
- `build:server`：先 `tsc` 按 `tsconfig.server.json` 把后端编译到 `dist/`；🔑 **注意后半段**——`ai-service.js` 和 `code-agent-tools.js` 是**手写 JS（不是 TS）**，`tsc` 不会处理它们，所以用 `fs.cpSync` 手动拷进 `dist/server/`。这就是为什么 `build:server` 依赖 Node 22（`cpSync` 是较新 API）。
- `start`：生产模式只跑一个编译后的入口 `dist/server/standalone.js`，它自己既托管前端静态资源又提供 API（单端口）。

> 🔑 **一句话记住**：开发用 `tsx`（免编译、热更新），生产用 `tsc` 编译后 `node` 跑。前端 dev 用 Vite server，生产用 Vite build 出静态文件由后端托管。

## 源码剖析二：为什么必须 `--legacy-peer-deps`（依赖 prune 陷阱）

这是 Sowork AI 最著名的坑，也是**新手第一次构建必翻车**的地方。先给结论：

```bash
npm install --legacy-peer-deps   # ✅ 唯一正确
npm install                       # ❌ 会导致前端无法编译
```

**从源码层面解释根因**：`package.json` 的 `dependencies` 历史上**不完整**——大量被源码 `import` 的包（如 `antd`、`axios`、`sharp`、`es-toolkit`、`pg`、`mysql2` 等）在早期是"手动装进 node_modules"的，并未全部写进 `dependencies`。

当你跑**裸** `npm install` 时，npm 会以 `package.json` 为"唯一真相"做 reconcile：凡是 `node_modules` 里有、但 `package.json` 没声明的包，都会被判定为 **extraneous（多余）而 prune 掉**。于是 `antd` 被删，`build:client` 立刻报：

```
✗ [vite]: Rollup failed to resolve import "antd"
```

`--legacy-peer-deps` 会放宽 peer 依赖校验、**不去激进 prune 这些"未声明但存在"的包**，从而保住 antd 等核心依赖。

> ⚠️ **两条铁律（血泪教训）**
> 1. 本项目**永远**用 `npm install --legacy-peer-deps`，**绝不**跑裸 `npm install`。
> 2. 万一已经被 prune，用下面这条**针对性**恢复，**不要**整体重装：
>    ```bash
>    npm install antd@^5 @ant-design/icons@^5 es-toolkit@^1 sharp pg mysql2 --legacy-peer-deps
>    ```

### `.npmrc`：为什么安装还这么快

```ini
# .npmrc
registry=https://registry.npmmirror.com/          # npm 包走国内镜像
sharp_binary_host=https://npmmirror.com/mirrors/sharp/     # sharp 二进制走镜像
sqlite3_binary_host_mirror=https://npmmirror.com/mirrors/sqlite3/
```

`sharp`、`better-sqlite3` 这类**带原生二进制**的包，默认会去 GitHub 下预编译产物，国内经常卡住。`.npmrc` 把它们都指向 npmmirror 镜像——这就是为什么本项目在国内也能顺利装上原生依赖。

## 运行演示

### 开发模式（前后端热更新）

```bash
npm install --legacy-peer-deps
npm run dev
# server 日志（蓝色）：tsx watch → [Bootstrap] API ready at http://localhost:3000/api
# client 日志（绿色）：VITE ready → http://localhost:8080
```

访问 `http://localhost:8080`，前端通过 Vite 代理把 `/api` 打到 `:3000`。

### 生产模式（单端口）

```bash
npm run build            # tsc 编译后端 + vite 打包前端 → dist/
npm start                # 单进程托管 SPA + API
# 访问 http://localhost:3000
SERVER_PORT=3400 npm start   # 自定义端口
```

### 首次登录

首启时 `standalone.ts` 会自动建库 + 播种数据并创建默认账号：

| 用户名 | 密码 | 角色 |
|--------|------|------|
| `admin` | `admin123` | 超级管理员 |
| `hr_admin` | `hr123456` | HR 管理员 |

> ⚠️ 上线前**必须**改掉默认密码（见第 12 章权限与第 14 章安全清单）。

## 管理者视角

- **部署门槛**：Sowork AI 开发用 SQLite，**零外部数据库依赖**——意味着一台普通办公电脑/内网服务器就能起一套完整系统，不需要先采购数据库。POC/试点成本极低。
- **上线风险**：依赖 prune 陷阱是"实施踩坑高发点"，让实施同事记住那条恢复命令，能省掉一次典型的"上线前夜环境事故"。
- **国产化友好**：`.npmrc` 全量走国内镜像，内网/离线环境也能可控地把依赖缓存下来。

---

## 注意事项

- ⚠️ **只用 `npm install --legacy-peer-deps`**，裸 `npm install` 会 prune 掉 antd 导致前端编译失败。
- ⚠️ **被 prune 后针对性补装，别整体重装**（恢复命令见上）。
- ⚠️ **Node 必须 ≥ 18、推荐 22**：`build:server` 用了 `fs.cpSync`。
- 🔑 开发模式前端在 **8080**、后端在 **3000**；生产模式统一 **3000**。
- 🔑 数据库首启自动创建，**无需手动初始化**。

## 练习

1. 打开 `package.json`，用自己的话解释 `build:server` 后半段为什么要 `cpSync` 两个 `.js` 文件（提示：它们不是 TS）。
2. 故意跑一次裸 `npm install`，观察 `npm run build:client` 是否报 `resolve import "antd"`，再用恢复命令修复并截图。
3. 阅读 `.npmrc`，列出其中 3 个"原生二进制包镜像"配置项，并说明各自服务于哪个 npm 包。
