## 核心概念：前端要解决的三层问题

企业后台前端的复杂度不在"画页面"，而在这三层：

```
① 装配层   → React 入口如何组织：路由、全局状态、错误边界
② 权限层   → 谁能看到哪个菜单、哪个按钮？前后端如何同源？
③ 健壮层   → 某个子组件渲染崩了，怎么不让整站白屏？
```

Sowork AI 的前端（`client/src/`，共 184 个页面文件 + 194 个组件文件）对这三层都有明确的源码级答案。

## 源码剖析一：入口装配（多层 Provider + 全局边界）

入口文件 `client/src/index.tsx`（1~19 行）用 `BrowserRouter` 和两层业务 Context 包裹整个 App：

```tsx
// client/src/index.tsx:1-19
import React from 'react';
import ReactDOM from 'react-dom/client';
import { BrowserRouter } from 'react-router-dom';
import App from './app';
import { AppConfigProvider } from './contexts/AppConfigContext';
import { ShopProvider } from './pages/ShopHomePage/ShopContext';
import './index.css';

ReactDOM.createRoot(document.getElementById('root')!).render(
  <React.StrictMode>
    <BrowserRouter>
      <AppConfigProvider>
        <ShopProvider>
          <App />
        </ShopProvider>
      </AppConfigProvider>
    </BrowserRouter>
  </React.StrictMode>
);
```

装配顺序有讲究，从外到内：`StrictMode`（开发期严格检查）→ `BrowserRouter`（路由上下文）→ `AppConfigProvider`（系统配置：主题、站点信息）→ `ShopProvider`（商城全局态：购物车）→ `App`。

`App` 内部再套两层"守护"（`client/src/app.tsx:163-178`）：

```tsx
// client/src/app.tsx:163-178
function AuthGuard({ children }: { children: React.ReactNode }) {
  const currentUser = sessionStorage.getItem('__current_user');
  if (!currentUser) return <Navigate to="/login" replace />;
  return <>{children}</>;
}

export default function App() {
  return (
    <I18nProvider>
    <ErrorBoundary fallbackName="应用页面">
    <Routes>
      <Route path="/login" element={<LoginPage />} />

      {/* 网站前台（无需登录） */}
      <Route path="/site" element={<SiteHomePage />} />
      <Route path="/site/articles" element={<ErrorBoundary fallbackName="网站栏目-文章列表"><ArticleListPageV2 /></ErrorBoundary>} />
```

三个要点：

1. **`I18nProvider` 在最外层**：多语言上下文包裹一切，任何页面都能 `t('删除')` 翻译（第 12 章讲 i18n 后端）。
2. **顶层 `ErrorBoundary fallbackName="应用页面"`**：整棵路由树的**最后一道防线**，任何未被子边界捕获的渲染错误都在这里兜住，不至于白屏。
3. **`AuthGuard` 用 `sessionStorage`**：未登录跳 `/login`。注意前台 `/site/*` 路由**不套 AuthGuard**——CMS 门户/商城对匿名访客开放。

登录后的后台路由统一挂在 `AuthGuard → Layout` 下，且每个高风险页面再单独包一层带 `fallbackName` 的 ErrorBoundary（`client/src/app.tsx:198-211, 339-340`）：

```tsx
// client/src/app.tsx:198-340（节选）
<Route element={<AuthGuard><Layout /></AuthGuard>}>
  <Route index element={<DashboardPage />} />
  <Route path="/organization" element={<OrganizationPage />} />
  <Route path="/personnel" element={<PersonnelPage />} />
  ...
  {/* 网站与商城管理后台 */}
  <Route path="/admin/shop" element={<ErrorBoundary fallbackName="商城后台"><ShopAdminPage /></ErrorBoundary>} />
  <Route path="/admin/cms" element={<ErrorBoundary fallbackName="CMS后台-栏目管理"><CMSAdminPage /></ErrorBoundary>} />
</Route>
```

🔑 **双层错误边界策略**：顶层兜底 + 每个复杂页面独立边界。好处是——CMS 后台崩了只显示"CMS 后台出错"，其它菜单照常能用，而不是整站白屏。这是从多次"栏目页白屏"事故中沉淀出的架构决策。

## 源码剖析二：`usePermission` —— 前后端同源的按钮级权限

这是本章最硬核的一段。前端权限有个经典矛盾：**要快（不能等接口才渲染，否则闪烁）**，又**要准（不能只信本地，本地可被篡改）**。Sowork AI 的解法是"本地即时算 + 后端权威合并"（`client/src/hooks/usePermission.tsx:143-172`）：

```tsx
// client/src/hooks/usePermission.tsx:143-172
useEffect(() => {
  const userStr = localStorage.getItem('user');
  if (userStr) {
    try {
      const userData = JSON.parse(userStr);
      setUser(userData);
      let roleIds = userData.roleIds;
      if (typeof roleIds === 'string') { try { roleIds = JSON.parse(roleIds); } catch { roleIds = []; } }
      if (!Array.isArray(roleIds)) roleIds = [];
      // 1) 本地即时计算（避免闪烁）
      setPermissions(computePermissions(roleIds));
      setSiteScope(computeSiteScope(roleIds));
      // 2) 后端权威 resolve（失败则沿用本地）
      fetch('/api/rbac/resolve', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ roleIds })
      }).then(r => r.ok ? r.json() : null).then(d => {
        if (d && Array.isArray(d.permissions)) {
          setPermissions(prev => Array.from(new Set([...prev, ...d.permissions])));
          if (Array.isArray(d.siteScope) && d.siteScope.length) setSiteScope(d.siteScope);
        }
      }).catch(() => { /* 离线回退本地 */ });
    } catch (e) {
      console.error('解析用户信息失败:', e);
    }
  }
  setLoading(false);
}, []);
```

这段代码的精妙在于**两阶段**：

1. **第一阶段（同步）**：从 `localStorage` 读用户角色，本地 `computePermissions(roleIds)` 立刻算出一个权限集并渲染——UI 不闪烁，用户秒见菜单。
2. **第二阶段（异步）**：`POST /api/rbac/resolve` 拿后端**权威**权限集，用 `new Set([...prev, ...d.permissions])` **合并**进来。后端才是真相来源（第 12 章讲 resolve 实现）。
3. **离线降级**：`.catch(() => {})`——如果后端不可达，就沿用本地计算结果，不至于全站无权限。

对外暴露的判定函数（`client/src/hooks/usePermission.tsx:175-204`）：

```tsx
// client/src/hooks/usePermission.tsx:175-204
const hasPermission = useCallback((permission: string): boolean => {
  if (!user) return false;
  // 超级管理员拥有所有权限
  if (user.roleIds?.includes('role_super_admin') || user.roleIds?.includes('role_sys_admin')) {
    return true;
  }
  return permissions.includes(permission);
}, [user, permissions]);

// can(point) —— hasPermission 的语义化别名
const can = hasPermission;

// 站点范围判定：'*' 表示全部站点
const canAccessSite = useCallback((siteCode: string): boolean => {
  if (!user) return false;
  if (user.roleIds?.includes('role_super_admin') || user.roleIds?.includes('role_sys_admin')) return true;
  return siteScope.includes('*') || siteScope.includes(siteCode);
}, [user, siteScope]);
```

- **`can` 就是 `hasPermission` 的别名**——语义化，`can('cms:article:delete')` 读起来像自然语言。
- **超管短路**：`role_super_admin` / `role_sys_admin` 直接返回 `true`，不查权限集。
- **`canAccessSite`**：站点级作用域判定，`'*'` 通配全部站点（第 12 章详解站点 scope）。

在页面里的真实用法（`client/src/pages/Admin/CMSAdminPage.tsx:547-554`）：

```tsx
// client/src/pages/Admin/CMSAdminPage.tsx:547-554
{can('cms:article:edit') && <Button size="small" type="link" onClick={() => handleEditArticle(record)}>{t('编辑')}</Button>}
{can('cms:article:create') && <Button size="small" type="link" onClick={() => handleCopyArticle(record)}>{t('复制')}</Button>}
{can('cms:article:delete') && (
  <Popconfirm title="确认删除？" onConfirm={() => handleDeleteArticle(record.id)}>
    <Button size="small" type="link" danger>{t('删除')}</Button>
  </Popconfirm>
)}
```

🔑 **权限点 + 短路渲染 + 翻译**三合一：`can('权限点') && <Button>{t('文案')}</Button>`。没权限的按钮**根本不渲染**，而不是禁用——更干净。

> ⚠️ 源码现状提醒：`CMSAdminPage.tsx` 目前把 `can` 临时写死为 `() => true`（第 49~51 行 `const can = (point: string) => true`），让所有按钮先可见，靠**后端 API 鉴权兜底**。这是"前端显隐是体验优化、后端校验才是安全底线"的务实体现——教程要如实说明这一点。稳定后可恢复真实 `can()` 控制。

## 源码剖析三：为什么是 fetch，不是 axios

很多人默认企业项目用 axios。Sowork AI 的前端 API 客户端是**原生 `fetch` 的极简封装**（`client/src/hooks/useAPI.ts:1-13`）：

```ts
// client/src/hooks/useAPI.ts:1-13
import { useState, useCallback, useEffect } from 'react';

const API_BASE = '/api';

async function apiRequest<T>(endpoint: string, options?: RequestInit): Promise<T> {
  const response = await fetch(`${API_BASE}${endpoint}`, {
    headers: { 'Content-Type': 'application/json', ...options?.headers },
    ...options,
  });
  if (!response.ok) throw new Error(`API Error: ${response.status}`);
  return response.json();
}
```

以及便捷方法（`client/src/hooks/useAPI.ts:165-170`）：

```ts
// client/src/hooks/useAPI.ts:165-170
export const api = {
  get: <T>(endpoint: string) => apiRequest<T>(endpoint),
  post: <T>(endpoint: string, body: any) => apiRequest<T>(endpoint, { method: 'POST', body: JSON.stringify(body) }),
  put: <T>(endpoint: string, body: any) => apiRequest<T>(endpoint, { method: 'PUT', body: JSON.stringify(body) }),
  delete: <T>(endpoint: string) => apiRequest<T>(endpoint, { method: 'DELETE' }),
};
```

为什么不用 axios？

- **`API_BASE = '/api'` 是相对路径**：前后端同端口（第 02 章），不需要跨域配置，也不需要 axios 的 baseURL/拦截器那套。
- **零额外依赖**：`fetch` 是浏览器原生的，少一个包就少一份体积和维护面。
- **够用**：这套系统的接口都是简单 JSON，不需要请求取消、上传进度等 axios 高级特性。

> ⚠️ 项目里 `client/src/api/index.ts` 是一段脚手架残留（引用 `axiosForBackend`，实际未被使用），唯一真正 `import axios` 的只有 `CanteenPage.tsx` 单个文件。理解这点能避免你误以为"全站用 axios"。

`useAPI<T>(table)` 在此之上还提供了一个 CRUD Hook 工厂（`fetchAll/create/update/remove` 等），页面直接 `const { data, create } = useAPI('employees')` 即可。

## 源码剖析四：ErrorBoundary 防白屏

React 有个特性：**任何子组件渲染抛错，整棵树会被卸载**——表现就是"整页白屏且无提示"。ErrorBoundary 是唯一的解药（`client/src/components/ErrorBoundary.tsx:9-56`）：

```tsx
// client/src/components/ErrorBoundary.tsx:9-56
export default class ErrorBoundary extends React.Component<Props, State> {
  constructor(props: Props) {
    super(props);
    this.state = { hasError: false, error: null, errorInfo: '' };
  }

  static getDerivedStateFromError(error: Error): Partial<State> {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error, errorInfo: React.ErrorInfo) {
    console.error('[ErrorBoundary]', this.props.fallbackName || 'page', error, errorInfo);
    this.setState({ errorInfo: errorInfo.componentStack || '' });
  }

  render() {
    if (this.state.hasError) {
      return (
        <Result
          status="error"
          title={`页面渲染出错${this.props.fallbackName ? ' - ' + this.props.fallbackName : ''}`}
          subTitle="请刷新页面重试；如果问题持续，请联系管理员"
          extra={<Button type="primary" onClick={() => window.location.reload()}>刷新页面</Button>}
        />
        // ... 还会展示 error.name / message / componentStack 便于排障
      );
    }
    return this.props.children;
  }
}
```

工作原理：

- **`getDerivedStateFromError`**：捕获到错误时进入 `hasError` 态。
- **`componentDidCatch`**：打印带 `fallbackName` 的日志（所以你能在控制台看到"是哪个页面崩了"），并保存组件调用栈。
- **`render`**：出错时用 antd `<Result status="error">` 展示友好错误页 + "刷新页面"按钮 + 错误堆栈（开发排障用），而不是白屏。

## 实战代码：给你的新页面加防白屏

自己写新页面时，套用同样的模式：

```tsx
// 在 app.tsx 的路由里

  path="/my-feature"
  element={
    <ErrorBoundary fallbackName="我的新功能">
      <MyFeaturePage />
    </ErrorBoundary>

function MyFeaturePage() {
  const { can } = usePermission();
  return (
    <>
      {can('myfeature:create') && <Button onClick={handleCreate}>{t('新建')}</Button>}
    </>
  );
}
```

## 管理者视角

| 关注点 | 前端架构给出的答案 |
|--------|-------------------|
| **数据可见性合规** | 按钮级权限（`can('权限点')`）+ 站点级作用域（`canAccessSite`）让"谁能看/改哪些数据"精确可控，满足内控与审计要求。 |
| **用户体验稳定性** | 双层 ErrorBoundary 保证单个功能出问题不会拖垮整站，员工日常办公不中断。 |
| **前端性能与体积** | 零 axios 依赖、原生 fetch，包体更小，弱网/内网加载更快。 |
| **风险提示** | 前端权限只是"体验层"，真正的安全靠后端 API 鉴权（第 05/12 章）。切勿把前端隐藏当作安全边界。 |

给决策者一句话：**"按钮级权限 + 防白屏"直接关系到合规性与员工日常可用性**，是企业级前端与玩具项目的分水岭。

## 注意事项

- ⚠️ **前端权限 ≠ 安全**：`can()` 只控制显隐，能被绕过。**后端每个写接口都必须独立鉴权**（这也是 CMSAdminPage 敢把 `can` 临时写死 `()=>true` 的底气——后端兜底）。
- ⚠️ **antd Tree 组件白屏陷阱**：Tree 必须用 `treeData` 数组属性，**禁止**用 `<TreeNode>` 子节点或 `const { TreeNode } = Tree`——在 antd v5 下可能 undefined 导致 "Element type is invalid" 整页白屏。所有树形/栏目页面必须包 ErrorBoundary。
- ⚠️ **新增复杂页面必须包 ErrorBoundary**：尤其是含 Tree、动态表格、富文本的页面。
- 🔑 `usePermission` 是"本地算权 + 后端合并"，后端 `resolve` 是权威。理解这个两阶段才能正确调试权限问题。
